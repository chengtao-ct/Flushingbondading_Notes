在做 IoT 设备时，这个坑几乎 100% 会踩到：网络协议（如 TCP/IP、MQTT）和很多 I2C 传感器（比如常用的温湿度传感器、六轴陀螺仪 MPU6050）输出的数据是 **大端序 (Big-Endian，高位在前)**。 但是，我们的 STM32 和 ESP32 芯片的内存是 **小端序 (Little-Endian，低位在前)**。

**实战场景：** 假设你的单片机通过 I2C 从一个温度传感器读到了两个字节的原始数据，存放在了一个数组里： `uint8_t rx_buf[2] = {0x12, 0x34};` 根据传感器手册，`rx_buf[0]` 是高 8 位 (High Byte)，`rx_buf[1]` 是低 8 位 (Low Byte)。

**你的任务：**

1. **基础拼接：** 请写一行代码，用左移 `<<` 和位或 `|`，把这两个 8 位的碎片，完美地拼成一个 16 位的整型变量 `uint16_t temperature`。（正确结果应该是 `0x1234`）。
    
2. **进阶翻转：** 如果你是通过 DMA 直接把数据干进了一个 16 位的变量里，结果由于端序颠倒，读出来变成了错的 `uint16_t raw_data = 0x3412;`。请写一行宏定义或表达式，**在不借助数组的情况下**，把这个 16 位变量的高 8 位和低 8 位对调，恢复成正确的 `0x1234`。


```c
uint16_t temperature = ((uint16_t)rx_buf[1]<<8)|(rx_buf[2]);
```


**思考**
- 先是提取操作，提取的公式先断头，再&

```c
uint16_t rx_high = (raw_data >>8) & 0xFF;
uint16_t rx_low = (raw_data) & 0xFF;

uint16_t true_data = (rx_low << 8) | rx_high; 
```


```c
// 终极一行流宏定义
#define SWAP_16(x)   (((x) >> 8) | ((x) << 8))

uint16_t true_data = SWAP_16(raw_data);
```
