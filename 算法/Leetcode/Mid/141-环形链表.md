> [!abstract] 题目概述
> **考察方向：** 快慢双指针的终极应用（龟兔赛跑算法）。
>
> **嵌入式场景：** 你手写了一个类似"环形缓冲区（Ring Buffer）"的内存池管理机制，但是在压力测试时系统卡死了。你需要写一个诊断函数，判断当前的链表结构是不是被野指针破坏，形成了一个"死循环（死环）"。
>
> **挑战点：** 空间复杂度必须是 $O(1)$，绝对不能修改原链表。

> [!note] 思考过程
> 判断循环的本质是数值的相同，遍历次数无法确定，提醒我仍然是双指针的解法：思考第一个指针记录的是什么？第二个指针应该是用来遍历的，第二个指针遍历玩有几种结果：空指针，指向自己，指向他人，指向自己可以明确有循环，指向他人，仍然要操作，可是假如在循环里面可以不可能有空指针，所以说现在最大的问题就是处理循环中的非指向自己。
>
> 总结：这个问题有两个要点：第一个如何有循环的话：要找到循环点，如果不是循环的话：要找到NULL，问题是行走次数在链表无法确定，快指针没有退出逻辑，或者换一个思路，假如说有循环的话，要么是他本身，要么永远他被排除在外。

> [!warning] 踩坑记录
> 普通双指针（一动一静）在这道题里彻底失效——环在后面，记录者永远被"排除在外"，遍历者在环里疯狂打转，程序陷入死循环，没有退出逻辑。

## 解法

```c
/**
 * Definition for singly-linked list.
 * struct ListNode {
 * int val;
 * struct ListNode *next;
 * };
 */
#include <stdbool.h>

bool hasCycle(struct ListNode *head) {
    if (head == NULL || head->next == NULL) {
        return false;
    }

    struct ListNode *slow = head;
    struct ListNode *fast = head;

    while (fast != NULL && fast->next != NULL) {
        slow = slow->next;
        fast = fast->next->next;

        if (slow == fast) {
            return true;
        }
    }

    return false;
}
```

## 原理剖析

### 1. 初始思路：定海神针法（Anchor & Scout）

第一反应是：**"第一个指针记录位置（站在原地），第二个指针去遍历寻找。"**

- **如果是直路：** 第二个指针一直走，最终遇到 `NULL`，证明无环。
- **如果是环：** 第二个指针一直走，如果环刚好绕回了第一个指针脚下（指向自己），就证明有环。

### 2. 致命缺陷："永远被排除在外"

推演到了最关键的一步：**"假如说有循环的话，要么是他本身，要么永远他被排除在外。"**

假设链表是这样的形状：像一个**"数字 6"**。`[节点1] -> [节点2] -> [节点3] -> [节点4] -> [节点5] -> (指回节点3)`

- 第一个指针（记录者）站在 `[节点1]`。
- 第二个指针（遍历者）往前跑，跑进了 `3 -> 4 -> 5 -> 3 -> 4...` 的死循环里。
- **灾难发生了：** 环在后面，而你的记录者永远被"排除在外"（留在了节点1）。你的遍历者在环里疯狂打转，它永远遇不到 `NULL`，也永远遇不到记录者。程序就此陷入了**死循环，没有退出逻辑**。

这就是普通双指针解法（一动一静）在这道题里彻底失效的根本原因！

### 3. 破局：从"一动一静"升级为"同向相对运动"

既然"第一个指针站着不动"会导致它可能被永远挡在环外，那该怎么解决？前辈大牛们的思路极其暴力且巧妙：**那就别让第一个指针站着了，让他俩一起跑！只不过一个跑得慢，一个跑得快！**

这就顺理成章地过渡到了**龟兔赛跑（快慢指针）**：

- **破除"被排除在外"的魔咒：** 既然慢指针（龟）也在往前走，那它早晚有一天也会顺着"6"字形的把手，走入那个循环圈里。这时候，龟和兔就**都在环里了**。
- **解决"循环中的相遇"：** 一旦龟和兔都在环里，这就变成了一道小学数学题：在一个环形跑道上，兔子速度是 2，乌龟速度是 1。它们无论距离多远，兔子每跑一轮，就会比乌龟多跑 1 步，两者的距离就会缩短 1。**因此，兔子绝对不会跳过乌龟，必然会有一次刚好踩在乌龟背上（`slow == fast`）！**

### 4. 退出逻辑

引入了快慢指针后，这道题的退出逻辑变得极其干净利落，只有非黑即白的两种：

- **退出逻辑 1（没有环）：** 兔子跑得快，它一直往前冲，只要它或者它下一步踩到了悬崖（`fast == NULL || fast->next == NULL`），游戏直接结束，**返回 false**。
- **退出逻辑 2（有环）：** 兔子和乌龟都掉进了一个没有出口的环形角斗场。它们在里面绕圈，因为存在速度差，兔子必定在某个时刻从背后追上并撞见乌龟（`fast == slow`）。只要撞见，游戏直接结束，**返回 true**。

### 进阶彩蛋：找到循环点

- **LeetCode 141（环形链表 I）：** 面试官只问你"有没有环"。我们只要发现龟兔相遇（`fast == slow`），就可以直接高呼"有环！"，立刻返回 `true`，**不需要知道环的起点在哪**。
- **LeetCode 142（环形链表 II）：** 这是一道难度飙升的 Medium 题。面试官不仅问你有没有环，还会问你："如果有环，**请把环的入口节点（循环点）给我找出来**。"

在 LeetCode 142 中，就要用到非常硬核的数学推导了：当龟兔在环内相遇时，如果我们在链表头部再派出一个指针，和当前的乌龟同时以速度 1 往前走，它们最终相遇的地方，**必然是环的入口！**
