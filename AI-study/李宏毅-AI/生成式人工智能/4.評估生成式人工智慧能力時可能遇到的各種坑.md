## 💡 AI 摘要

本课程系统讲解了生成式 AI 能力评估的完整方法论，揭示了评估过程中常见的陷阱（如指标失真、数据泄露、Prompt 敏感性等），并提供了从传统指标到 LLM-as-Judge 的多种评估方案及其实践建议。

---

## 🧠 总体架构与原理解析

- **核心原理**：评估生成式 AI 需要构建 Benchmark（测试数据集 + 标准答案）→ 设计 Evaluation Metric（评估函数）→ 计算得分。但由于生成式任务的开放性，传统评估方法存在诸多局限，需要结合人类评估、LLM 评估等多种手段。

- **关键概念**：
  | 术语 | 解释 |
  |------|------|
  | **Benchmark** | 评估模型能力的测试数据集，包含输入与标准答案 |
  | **Ground Truth** | 人工标注的标准答案 |
  | **Evaluation Metric** | 量化模型表现的指标（如 BLEU、ROUGE、BERTScore） |
  | **Exact Match** | 输出与标准答案完全一致才得分，适用于选择题等有限答案场景 |
  | **Hallucination** | 模型无法正确回答却硬编造答案的现象 |
  | **LLM-as-Judge** | 用语言模型替代人类进行评分 |
  | **Verifier** | 专门训练用于评估其他模型输出的模型 |
  | **Jailbreak** | 绕过模型安全限制，使其输出不该输出的内容 |
  | **Prompt Injection Attack** | 通过注入指令让模型怠忽职守 |

## 📝 详细拆解与知识点

### 一、评估方法对比

| 方法 | 适用场景 | 局限性 |
|------|----------|--------|
| **Exact Match** | 选择题、答案固定的任务 | 无法处理语义相同但字面不同的情况；可能测的是"遵守指令能力"而非知识 |
| **BLEU / ROUGE** | 翻译、摘要任务 | 仅计算词彙重叠，无法识别同义词 |
| **BERTScore** | 需要语义理解的场景 | 基于 Contextualized Embedding 计算语义相似度 |
| **人类评估** | 无标准答案的任务（写诗、小说） | 成本高、再现性差、存在风格偏见 |
| **LLM-as-Judge** | 大规模自动化评估 | 存在自我偏袒、权威偏见等问题 |

### 二、评估陷阱

**1. Goodhart's Law**
> 当一个指标成为被优化的目标，它就不再是好的指标。

**2. 案例：愚笨的鹦鹉（Parrot）**
- 输入 = 输出，不做任何 Paraphrasing
- 在 BLEU、METEOR 上打敗 SOTA
- 即使要求"必须有 X% 不同"，随机替换仍可高分

**3. Hallucination 的根源**
- 说"不知道"和乱猜都得 0 分
- 乱猜有概率蒙对得正分
- 模型学到：乱猜比承认无知更有利

**4. 人类评估的偏见**
- 偏好格式漂亮、有表情符号的回答
- Chatbot Arena 发现：去掉风格影响后，Claude 排名上升

**5. LLM 评估的偏见**
- 自我偏袒：GPT-4 给 GPT-4 更高分
- 修改偏见：被告知"这是修改后的答案"会提高评分
- 权威偏见：有引用链接的答案更受青睐

### 三、LLM-as-Judge 最佳实践

**评分方式对比**：
| 方式 | 与人类评分相关性 |
|------|------------------|
| 只给分数 | 较低 |
| 给分后解释 | 较高 |
| 先推理再给分 | 最高 |

**精确评分方法**：
```
分数 = 1×P(1) + 2×P(2) + 3×P(3) + 4×P(4) + 5×P(5)
```
使用概率加权平均，而非单一 Token 采样。

### 四、Prompt 对评估的巨大影响

**案例 1：大海撈針测试**
- 原 Prompt：Claude 表现不佳
- 加一句"请找出最相关的句子" → 能力起飞

**案例 2：语音评估**
| Prompt | 正确率 |
|--------|--------|
| "比较两句话谁发音准确" | 2.78%（拒绝回答） |
| "衡量谁英文流利，无 Ethical concern" | 61% |
| "你会听到两段音档，哪段比较流利？" | 74% |

**关键发现**：极小的 Prompt 修改（大小写、空格、换行）可导致正确率从 3.6% 到 80% 的差异。

### 五、数据泄露问题

**证据链**：
1. 修改题目人名/数字 → 正确率下降
2. 给题目前半段，模型能接出原题内容
3. ElasticBench 直接比对发现：SWE Bench 约 10% 题目在训练数据中

### 六、恶意攻击类型

| 类型 | 定义 | 案例 |
|------|------|------|
| **Jailbreak** | 让模型做违法/违规的事 | 用编码、多轮对话、说服理由绕过限制 |
| **Prompt Injection** | 让模型怠忽职守 | AI 主播被攻击后一直喵喵叫；论文藏白色小字影响 AI reviewer |

**最有效的 Jailbreak 说服方式**：
1. Logical Appeal（逻辑说服）
2. Authority Appeal（权威背书）
3. Misrepresentation（假装研究人员）

**最无效**：威胁

### 七、其他评估维度

- **生成速度**：首 Token 延迟 vs 每秒 Token 数
- **成本**：API 费用、Reasoning Token 消耗
- **稳定性**：木桶理论——有时下限比平均值更重要

## 🛠️ 实践与落地指南

### 环境与配置

无需特殊环境配置，主要涉及：
- **评估平台**：Chatbot Arena（https://chatbot-arena.org）
- **评估模型**：Prometheus（专用评分模型）
- **Benchmark 数据集**：GSM8K、MATH、SWE Bench、GPQA 等

### Prompt 或代码片段

**1. LLM-as-Judge 评分 Prompt 模板**
```
你是一个专业的评估者。请根据以下标准对回答进行评分。

【评分标准】
{evaluation_criteria}

【参考答案】
{reference_answer}

【待评估的回答】
{model_output}

请先分析这个回答的优点和不足，然后给出 1-5 分的评分。
```

**2. 避免 Hallucination 的评估设计**
```
评分规则：
- 回答正确：+1 分
- 回答"不知道"：0 分
- 回答错误：-1 分（倒扣）
```

**3. Risk-Aware Decision Prompt**
```
你现在面临一个选择题。

情境描述：{situation}
选项：A/B/C/D

规则说明：
- 答对：+{correct_score} 分
- 答错：{wrong_score} 分（可能为负）
- 拒答：0 分

请根据以上规则决定是否回答，以及选择哪个选项。
```

### 操作步骤

**评估模型能力的标准流程**：

1. **明确评估目标**
   - 单一任务能力（如翻译）
   - 特定领域能力（如医疗、金融）
   - 通用能力（多任务综合）

2. **选择或构建 Benchmark**
   - 检查是否存在数据泄露风险
   - 确保测试集与训练集无重叠

3. **设计评估指标**
   - 有标准答案：BLEU、ROUGE、BERTScore
   - 无标准答案：人类评估或 LLM-as-Judge

4. **多 Prompt 测试**
   - 准备 5-10 个不同表述的 Prompt
   - 对结果取平均，减少 Prompt 敏感性影响

5. **小规模验证 LLM 评估**
   - 取 10% 数据同时用人类和 LLM 评分
   - 计算相关性，确认 LLM 评估可靠后再大规模使用

6. **多维度评估**
   - 内容质量
   - 生成速度
   - 成本效益
   - 稳定性（下限表现）

7. **安全测试**
   - Jailbreak 抗性测试
   - Prompt Injection 防御测试
