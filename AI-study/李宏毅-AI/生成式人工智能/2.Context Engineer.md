# 第一部分：Context Engineer 理论体系

## 一、核心定义：什么是上下文工程？

### 1.1 基本概念

**Context Engineering（上下文工程）** 本质上是 **Prompt Engineering 的进阶版**。

| 维度 | 传统 Prompt Engineering | Context Engineering |
|------|------------------------|---------------------|
| **关注点** | 编写"神奇咒语" | 自动化管理输入信息 |
| **核心方法** | 强调格式和特定指令（如"一步步思考"） | 策略性设计、剪裁和组织输入信息 |
| **应用场景** | 单次问答 | AI Agent 系统性构建 |
| **解决的问题** | 引导单次输出 | 突破 LLM 性能瓶颈 |

### 1.2 核心挑战

随着模型能力增强，传统咒语效果正在减弱，现在需要重点解决：

1. **上下文塞爆（Context Overflow）**
   - Context Window 存在上限
   - 无法一次性输入所有信息

2. **迷失中间（Lost in the Middle）**
   - 模型更容易记住开头和结尾
   - 中间信息容易被忽视

### 1.3 核心原理

LLM 本质上是一个函数：

$$f(\text{Context}) = \text{NextToken}$$

**模型的输出完全取决于输入的 Context（上下文）。**

Context Engineering 就是有策略地设计、剪裁和组织输入给模型的信息，以引导模型产生预期的输出。

## 二、上下文的构成要素

一个完整的上下文（Context）通常由以下七大要素构成：

### 2.1 User Prompt（用户指令）

**说明：** 用户下达的任务指令

**包含内容：**
- 任务说明
- 操作指南
- 前置条件
- 范例（In-context Learning）

### 2.2 System Prompt（系统设定）

**说明：** 开发者的全局配置

**包含内容：**
- 模型身份（如"你是 Claude"）
- 能力限制
- 互动风格
- 禁止事项

### 2.3 对话历史（短期记忆）

**说明：** LLM 的"短期记忆"机制

**作用：**
- 模型根据历史记录进行文字接续
- 保持对话连贯性

### 2.4 长期记忆

**说明：** 通过特定机制调用的过去互动信息

**作用：**
- 使用户感到 AI 具备持续认知
- 跨会话保持记忆

### 2.5 外部信息（RAG）

**说明：** 通过搜索引擎或数据库获取的即时资料

**作用：**
- 弥补模型知识滞后
- 提供最新信息

### 2.6 工具使用结果

**说明：** 模型调用外部函数后返回的观测结果

**典型工具：**
- 计算器
- 日历
- 电脑操控
- API 调用

### 2.7 思考过程

**说明：** 模型在给出答案前的"脑内小剧场"

**包含内容：**
- 推理（Reasoning）
- 规划步骤

## 三、AI Agent 的运行架构

### 3.1 Agent 核心特征

AI Agent 的核心特征是**自主规划与环境互动**。

### 3.2 运行逻辑：持续循环

```
┌─────────────────────────────────────────────────────┐
│                                                     │
│   观察 → 思考 → 行动 → 再观察 → 思考 → 行动 → ...   │
│  Observation  Action  Observation  Action           │
│                                                     │
│                     ↓↓↓                            │
│              直到任务完成                           │
└─────────────────────────────────────────────────────┘
```

### 3.3 详细流程

| 步骤 | 名称 | 说明 |
|------|------|------|
| **1** | **观察（Observation）** | Agent 获取当前环境信息（用户问题、工具返回的错误） |
| **2** | **思考（Reasoning）** | 基于上下文进行推理 |
| **3** | **行动（Action）** | 决定下一步动作（搜索、写代码、操作鼠标） |
| **4** | **反馈与更新** | 行动改变环境，产生新的观察 |
| **5** | **循环** | 将新的观察作为新的 Context 喂给模型 |

### 3.4 Agent 架构组件

```
┌─────────────────────────────────────────────┐
│           用户指令              │
└────────────────────┬────────────────────────┘
                     ↓
         ┌────────────────────┐
         │  Perception（感知）  │
         └──────────┬─────────┘
                    ↓
         ┌────────────────────┐
         │  Brain（LLM 核心）  │  ← Planning（规划）
         │  - 思考             │  ← Decision（决策）
         └──────────┬─────────┘
                    ↓
         ┌────────────────────┐
         │   Action（工具）     │  ← 搜索引擎
         │  - 计算器            │  ← Python解释器
         │  - 文件操作          │  ← API调用
         └──────────┬─────────┘
                    ↓
         ┌────────────────────┐
         │ Observation（反馈） │  ← 工具执行结果
         │  - 搜索结果         │  作为新的 Context
         │  - 计算结果         │  喂回给 LLM
         │  - 错误信息         │
         └──────────┬─────────┘
                    ↓
            回到 Brain，形成循环
```

## 四、上下文工程的三大核心策略

由于模型存在 **Context Window 上限**，且长文本会导致性能下降（如 **Lost in the Middle 现象**），需要采取以下策略：

### 4.1 策略一：选择（Selection）

**核心理念：** 只挑选有用的东西放进 Context，而不是全部塞入。

#### 子策略

| 子策略 | 说明 | 应用场景 |
|--------|------|----------|
| **RAG（检索增强生成）** | 通过搜索引擎只抓取相关的文章片段 | 需要外部知识时 |
| **Reranking** | 利用小模型对搜索结果进行二次筛选，确保最相关的句子排在前面 | 提升检索质量 |
| **记忆挑选** | 根据时间近度（Recency）、重要性（Importance）和相关性（Relevance）挑选长期记忆 | 长期记忆管理 |

#### 记忆挑选三维度

```python
# 伪代码示例
def select_memories(all_memories, query):
    scores = []
    for memory in all_memories:
        recency_score = calculate_recency(memory.timestamp)
        importance_score = memory.importance
        relevance_score = calculate_relevance(memory, query)
        total_score = weighted_sum(recency_score, importance_score, relevance_score)
        scores.append((memory, total_score))

    # 只保留得分最高的 N 条记忆
    return sorted(scores, reverse=True)[:N]
```

---

### 4.2 策略二：压缩（Compression）

**核心理念：** 将琐碎的信息提炼成摘要。

#### 子策略

| 子策略 | 说明 | 应用场景 |
|--------|------|----------|
| **递归压缩** | 每隔一段时间（或上下文接近饱和时），调用 LLM 对之前的对话历史进行摘要提取 | 长对话管理 |
| **结构化管理** | 对于 Computer Use 等包含大量琐碎过程的任务，只保留关键结论 | 工具调用结果 |

#### 递归压缩示例

```
原始对话历史（1000 tokens）：
用户：帮我计算 123 * 456
模型：好的，123 * 456 = 56088
用户：再算一下 789 * 012
模型：789 * 12 = 9468
用户：还有 345 * 678
模型：345 * 678 = 233910
...

压缩后（100 tokens）：
历史摘要：用户进行了多次数学计算，包括 123*456=56088、789*12=9468、345*678=233910 等。
```

---

### 4.3 策略三：多代理协作（Multi-agent）

**核心理念：** 通过任务分拆来管理上下文。

#### 工作机制

```
┌─────────────────────────────────────────────────────────┐
│                    用户复杂任务                           │
└────────────────────┬────────────────────────────────────┘
                     ↓
        ┌────────────────────────┐
        │   CEO 代理（主代理）     │
        │  - 任务规划             │
        │  - 任务分配             │
        └────┬────┬────┬────┬─────┘
             ↓    ↓    ↓    ↓
    ┌────────┐ ┌────────┐ ┌────────┐
    │Programmer│ │Tester │ │Writer │
    │代理      │ │代理    │ │代理    │
    └────────┘ └────────┘ └────────┘

优势：
- 每个子代理只需要关注自己任务相关的上下文
- 有效缩短单个模型的输入长度
- 提高复杂任务的处理能力
```

#### 对比：单 Agent vs 多 Agent

| 维度        | 单 Agent    | 多 Agent            |
| --------- | ---------- | ------------------ |
| **上下文长度** | 需要包含所有任务信息 | 每个 Agent 只需关注自己的部分 |
| **专注度**   | 容易分心       | 每个 Agent 专注于特定领域   |
| **扩展性**   | 难以处理复杂任务   | 可以无限扩展子代理          |
| **复杂度**   | 简单         | 需要设计协调机制           |

## 五、底层原理深度解析

### 5.1 上下文学习（In-Context Learning, ICL）

#### 原理

模型**不需要修改参数**（微调），只需要在上下文中看到"输入-输出"的范例（Demonstrations），就能模仿这种模式。

#### 关键发现

1. **示例的质量 > 数量**
   - 提供几个高质量、多样化的例子
   - 比提供几十个低质量例子更有效

2. **标签的影响**
   - 有时不需要正确的标签，模型也能学会格式
   - 但在逻辑推理任务中，正确逻辑的示例至关重要

#### 示例对比

```
❌ 低质量示例（数量多但质量差）：
输入：翻译成英文
输出：Hello
输入：翻译成英文
输出：World
输入：翻译成英文
输出：Test
...

✅ 高质量示例（数量少但质量高）：
输入：将这句话翻译成美国街头俚语："你好吗？"
输出：What's up, homie?

输入：将这句话翻译成美国街头俚语："这太不可思议了。"
输出：That's lit!
```

---

### 5.2 思维链（Chain of Thought, CoT）

#### 原理

LLM 是"自回归"的（Autoregressive），即生成一个字后再生成下一个字。

**问题：** 对于复杂数学或逻辑问题，如果直接让模型生成答案，它没有足够的"计算步骤"去处理。

**解决：** CoT 强迫模型把中间的推理过程（Intermediate Reasoning Steps）写出来。这些生成的"推理步骤"随后变成了新的 Context，辅助模型生成最终答案。

#### 隐喻

这就像人类心算复杂乘法会出错，但如果有草稿纸写下步骤就能算对。**CoT 就是模型的"草稿纸"。**

#### 示例对比

```
❌ 没有 CoT：
问：小明有 5 个苹果，吃了 2 个，又买了 3 个，现在有多少个？
答：6 个

✅ 使用 CoT：
问：小明有 5 个苹果，吃了 2 个，又买了 3 个，现在有多少个？

请一步步思考：
1. 小明一开始有 5 个苹果
2. 吃了 2 个，剩下：5 - 2 = 3 个
3. 又买了 3 个，现在有：3 + 3 = 6 个

答：6 个
```

### 5.3 从 LLM 到 AI Agent

#### LLM 的局限性

```
LLM 只有"大脑"，没有"手脚"和"眼睛"

❌ 无法联网
❌ 无法操作文件
❌ 记忆有限
❌ 无法与环境交互
```

#### Agent 如何解决

```
┌─────────────────────────────────────────────────────────┐
│                      LLM（大脑）                          │
│                    思考、推理、决策                       │
└────────────────────┬────────────────────────────────────┘
                     ↓
         ┌────────────────────┐
         │     工具层（手脚）   │
         │  - 搜索引擎（眼睛）  │
         │  - 文件操作（手）    │
         │  - 计算器（计算）    │
         │  - API（对外接口）   │
         └────────────────────┘
                     ↓
         ┌────────────────────┐
         │   环境（真实世界）   │
         └────────────────────┘
```

---

### 5.4 角色设定（Persona）的本质

#### 原理

当你说"你是一个资深律师"时，你实际上是在**高维向量空间中，将模型的搜索范围限制在了与"法律"相关的区域**。

这排除了"医生"或"厨师"的词汇概率。

#### 向量空间可视化

```
完整向量空间（所有可能的回答）：
┌─────────────────────────────────────────────────────┐
│  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐  │
│  │ 医生 │  │律师  │  │厨师  │  │程序员│  │ ...  │  │
│  │领域  │  │领域  │  │领域  │  │领域  │  │      │  │
│  └──────┘  └──────┘  └──────┘  └──────┘  └──────┘  │
└─────────────────────────────────────────────────────┘

设定为"律师"后，搜索范围缩小到律师领域：
┌─────────────────────────────────────────────────────┐
│                                                        │
│  ┌────────────────────────────────────────────────┐  │
│  │                  律师领域                       │  │
│  │  法律术语、案例分析、辩护策略、合同审查...       │  │
│  └────────────────────────────────────────────────┘  │
│                                                        │
└─────────────────────────────────────────────────────┘
```

#### 优化建议

| 劣质设定 | 优质设定 |
|----------|----------|
| 你是专家 | 你是拥有20年经验的刑法辩护律师，擅长处理经济纠纷，说话风格严谨且犀利 |
| 你是程序员 | 你是资深后端工程师，精通 Python 和 Go，习惯编写高质量、可维护的代码 |
| 你是作家 | 你是畅销书作家，擅长悬疑小说，文字风格紧凑、节奏感强 |

**目的：** 不仅仅是为了好玩，而是为了**收敛解空间**，让预测的下一个 Token 更精准。

# 第二部分：提示工程实战运用指南

## 策略一：用"少样本（Few-Shot）"替代"纯描述"

### 原理

人类语言描述是模糊的（例如："请写得专业一点"）。但"范例"是具体的。

ICL 证明了模型极擅长模仿 Context 中的模式。

### 对比示例

#### ❌ 劣质 Prompt

```
请帮我把这句话翻译成英文，语气要像那种很地道的美国街头俚语。
```

**问题：**
- "地道"、"街头俚语"是主观描述
- 模型无法精准理解你的期望
- 输出风格不稳定

---

#### ✅ 优化 Prompt（使用 ICL）

```
请参照以下风格将文本翻译成美国街头俚语：

输入：你好吗？
输出：What's up, homie?

输入：这太不可思议了。
输出：That's lit!

输入：我没钱了。
输出：I'm broke AF.

输入：[你的句子]
输出：
```

**优势：**
- 通过具体范例明确风格
- 模型直接模仿模式
- 输出高度稳定

---

### 实战应用场景

| 场景 | 劣质写法 | 优化写法（使用 Few-Shot） |
|------|----------|--------------------------|
| **代码风格** | 写一段 Python 代码 | 提供几个符合规范的代码示例 |
| **邮件语气** | 写一封专业的邮件 | 提供 2-3 封参考邮件 |
| **数据格式** | 按照这种格式输出 | 提供完整的输入-输出对 |
| **总结风格** | 总结这篇文章 | 提供 2 篇不同文章的总结示例 |

### 最佳实践

```markdown
1. 提供 2-5 个高质量示例即可（质量 > 数量）
2. 确保示例的多样性（覆盖不同情况）
3. 输入-输出对要清晰对应
4. 对于复杂任务，示例的标签必须是正确的
```

---

## 策略二：显式强制"思维链（CoT）"

### 原理

对于需要逻辑的任务，直接生成答案是在强行压缩计算量。你需要人为地拉长生成的 Token 长度，给模型"思考的时间"。

### 对比示例

#### ❌ 劣质 Prompt

```
这款手机值得买吗？直接告诉我结论。
```

**问题：**
- 模型没有足够的推理步骤
- 容易产生幻觉或错误判断
- 结论缺乏说服力

---

#### ✅ 优化 Prompt（使用 CoT）

```
请评估这款手机是否值得购买。请按照以下步骤进行思考：

步骤 1：列出该手机的核心参数
- 处理器
- 屏幕
- 电池
- 摄像头
- 价格

步骤 2：分析其优缺点
- 优点：...
- 缺点：...

步骤 3：对比同价位的竞品
- 竞品 A：...
- 竞品 B：...

步骤 4：基于上述分析，给出最终购买建议
- 是否值得购买
- 推荐人群
- 如果不推荐，替代方案
```

**优势：**
- 逻辑错误率显著降低
- 回答更有深度和说服力
- 可以追溯推理过程

---

### 实战应用场景

| 场景 | 适用性 |
|------|--------|
| ✅ 数学计算 | 高（必须使用） |
| ✅ 逻辑推理 | 高（必须使用） |
| ✅ 代码分析 | 高（推荐使用） |
| ✅ 决策建议 | 高（推荐使用） |
| ❌ 简单翻译 | 低（不需要） |
| ❌ 创意写作 | 低（可能限制创意） |

### CoT 模板

```markdown
请一步步思考：

步骤 1：[分析什么]
[思考过程]

步骤 2：[分析什么]
[思考过程]

...

步骤 N：[得出结论]
[最终答案]
```

---

## 策略三：结构化上下文与"去噪"

### 原理

1. **Context Window 是有限的**
2. **模型存在"迷失中间"（Lost in the Middle）现象**
   - 开头和结尾记得清楚
   - 中间容易忘记

Context Engineering 要求我们像整理数据库一样整理 Prompt。

### 核心技巧

#### 技巧 1：使用分隔符

用清晰的视觉分隔符将不同部分隔开：

```markdown
### 指令
[你的任务说明]

---

### 背景资料
[粘贴长文本...]

---

### 范例
输入：...
输出：...

---

### 问题
[你的问题]
```

**推荐分隔符：**
- `###` （标题）
- `---` （水平线）
- `"""` （引用块）

---

#### 技巧 2：关键指令后置

利用模型的 **Recency Bias（近因偏差）**，在 Context 很长时，在最后再次重复核心指令。

```markdown
### 背景资料
[粘贴了一篇 5000 字的文章...]

[... 文章中间部分 ...]

[... 文章结尾部分 ...]

---

### 任务
基于以上文章，回答以下问题：
1. 文章的核心观点是什么？
2. 作者用了哪些论据支持观点？
3. 你同意作者的观点吗？为什么？

### 任务提醒
请仅基于上述文章内容回答，不要添加外部信息。
```

---

#### 技巧 3：信息层次化

```markdown
# 一级标题

## 二级标题

### 三级标题

- 要点 1
- 要点 2
  - 子要点 2.1
  - 子要点 2.2
```

---

### 完整示例


## 策略四：模拟 Agent 思维——拆解任务（Prompt Chaining）

### 原理

Lecture 中提到 Agent 通过多步行动解决问题。如果你不是在写代码构建 Agent，你可以通过拆分 Prompt 来手动模拟这个过程。

**核心思想：** 不要试图用一个超级复杂的 Prompt 解决所有问题。

---

### 拆解方法

#### 单步 Prompt（不推荐）

```markdown
请帮我：
1. 从这篇报告中提取所有数据
2. 分析这些数据的趋势
3. 写一份 2000 字的分析报告
4. 生成 3 个可视化图表
5. 给出 5 个行动建议
```

**问题：**
- 任务过于复杂
- 模型容易分散注意力
- 每个环节质量难以保证

---

#### 拆解为多步 Prompt（推荐）

**步骤 1：提取数据**

```markdown
请从以下报告中提取所有数字数据，并整理成表格：

[报告内容]

格式：
| 指标 | 数值 | 单位 | 时间 |
|------|------|------|------|
```

**→ 输出：数据表格**

---

**步骤 2：分析趋势**

```markdown
基于以下数据表格，分析各项指标的趋势变化：

[步骤1输出的表格]

请重点分析：
1. 哪些指标在上升？
2. 哪些指标在下降？
3. 是否有异常波动？
```

**→ 输出：趋势分析**

---

**步骤 3：生成报告**

```markdown
基于以下趋势分析，写一份 2000 字的分析报告：

[步骤2输出的分析]

报告结构：
1. 摘要（200字）
2. 详细分析（1500字）
3. 结论与展望（300字）
```

**→ 输出：分析报告**

---

**步骤 4：生成图表建议**

```markdown
基于以下分析报告，提出 3 个可视化图表的建议：

[步骤3输出的报告]

每个图表建议包括：
- 图表类型（折线图/柱状图/饼图）
- 数据来源
- 要展示的信息
```

**→ 输出：图表建议**

---

**步骤 5：生成行动建议**

```markdown
基于以上所有分析，给出 5 个具体的行动建议：

[之前的所有输出]

建议要求：
- 具体、可执行
- 有优先级排序
- 说明预期效果
```

**→ 输出：行动建议**

---

### Prompt Chaining 流程图

```
原始任务（复杂）
    ↓
┌─────────────────┐
│ 步骤 1：提取数据 │ → 输出 A
└────────┬────────┘
         ↓
┌─────────────────┐
│ 步骤 2：分析趋势 │ → 输出 B（基于 A）
└────────┬────────┘
         ↓
┌─────────────────┐
│ 步骤 3：生成报告 │ → 输出 C（基于 A+B）
└────────┬────────┘
         ↓
┌─────────────────┐
│ 步骤 4：图表建议 │ → 输出 D（基于 C）
└────────┬────────┘
         ↓
┌─────────────────┐
│ 步骤 5：行动建议 │ → 最终输出（基于 C+D）
└─────────────────┘
```

---

### 优势总结

| 维度 | 单步 Prompt | Prompt Chaining |
|------|-------------|-----------------|
| **注意力** | 分散，容易遗漏 | 每步专注单一目标 |
| **质量** | 各环节质量参差不齐 | 每个步骤质量最优 |
| **可控性** | 难以调试 | 每步可单独优化 |
| **复杂度** | 表面简单，实际困难 | 表面繁琐，实际可控 |

## 策略五：角色设定（Persona）的精准化

### 原理

角色设定（Persona）的本质是"子空间搜索"。

当你说"你是一个资深律师"时，你实际上是在高维向量空间中，将模型的搜索范围限制在了与"法律"相关的区域。这排除了"医生"或"厨师"的词汇概率。

**目的：** 收敛解空间，让预测的下一个 Token 更精准。

---

### 优化对比

#### ❌ 泛化的角色设定

```markdown
你是一个专家，请帮我分析这个问题。
```

**问题：**
- "专家"范围太广
- 模型不知道从哪个领域切入
- 可能输出无关信息

---

#### ✅ 精准的角色设定

```markdown
你是一位拥有 15 年经验的**股权投资律师**，专门处理**初创企业融资**相关法律事务。

你的专业领域包括：
- 公司股权结构设计
- 投资协议起草与审查
- 合规风险评估
- 退出机制规划

你的说话风格：
- 严谨、专业
- 引用法律条文时标注来源
- 对于不确定的事项明确说明

请基于你的专业视角，分析以下投资协议条款：
```

**优势：**
- 明确专业领域（股权投资）
- 明确经验年限（15 年）
- 明确专业方向（初创企业融资）
- 明确工作内容
- 明确说话风格

---

### 角色设定三要素

```markdown
### 1. 明确身份
- 角色：[具体职位/身份]
- 经验：[年限/成就]
- 领域：[专业方向]

### 2. 界定能力边界
- 擅长：[列出 3-5 项核心能力]
- 不涉及：[说明不在你的范围内]

### 3. 定义风格
- 语言风格：[正式/通俗/专业等]
- 表达方式：[详细/简洁/数据驱动等]
- 处理不确定性的方式：[明确说明/推测等]
```

# 第三部分：进阶之路总结

## 从"许愿者"到"架构师"

### 许愿者的思维方式

```
输入 → "帮我做这个" → 等待神奇输出
```

**特点：**
- 期待 AI 自动理解需求
- 不关心内部机制
- 结果不可控
- 无法优化

---

### 架构师的思维方式

```
分析需求 → 设计上下文 → 组织要素 → 优化输出
  ↓            ↓           ↓          ↓
  明确目标      选择策略     构建结构   迭代改进
```

**特点：**
- 理解 AI 工作原理
- 系统性设计输入
- 结果可控且可优化
- 持续改进流程

---

## 上下文架构师的四大核心能力

### 能力一：铺垫背景（Context）

**做什么：**
- 提供必要的前置信息
- 设置合理的上下文边界
- 控制信息密度

**为什么：**
- AI 需要背景信息才能准确理解
- 过少或过多的背景都会影响质量

---

### 能力二：展示范例（ICL）

**做什么：**
- 提供高质量的输入-输出对
- 覆盖不同情况
- 确保范例的正确性

**为什么：**
- 模型擅长模仿模式
- 范例比描述更准确
- 稳定输出质量

---

### 能力三：规划步骤（CoT）

**做什么：**
- 将复杂任务拆解为步骤
- 强制模型展示推理过程
- 每一步都要有明确目标

**为什么：**
- 给模型足够的"计算空间"
- 减少逻辑错误
- 提高可解释性

---

### 能力四：明确角色（Persona）

**做什么：**
- 定义精准的身份
- 界定能力边界
- 设定语言风格

**为什么：**
- 收缩解空间
- 提高准确性
- 稳定输出风格

---

## 快速检查清单

在使用 AI 之前，问自己这 4 个问题：

### ✅ 问题 1：背景信息够吗？

- [ ] AI 需要知道什么背景？
- [ ] 哪些信息是必要的？
- [ ] 哪些信息是冗余的？

---

### ✅ 问题 2：需要范例吗？

- [ ] 期望的输出格式是什么？
- [ ] 有没有相关的示例可以提供？
- [ ] 提供几个范例最合适？

---

### ✅ 问题 3：需要步骤吗？

- [ ] 这需要逻辑推理吗？
- [ ] 是否应该让模型展示思考过程？
- [ ] 任务可以拆分为哪些步骤？

---

### ✅ 问题 4：角色明确吗？

- [ ] AI 应该扮演什么角色？
- [ ] 这个角色的专业领域是什么？
- [ ] 期望的说话风格是什么？

---

## 实战优化流程

```
Step 1: 需求分析
明确你想让 AI 做什么，期望的输出是什么

    ↓

Step 2: 策略选择
根据任务类型选择合适的策略
- 需要 ICL？
- 需要 CoT？
- 需要拆解任务？

    ↓

Step 3: 上下文构建
按照四大要素组织 Prompt
- 身份设定
- 背景信息
- 范例展示
- 任务指令

    ↓

Step 4: 测试与迭代
- 检查输出质量
- 找到问题所在
- 针对性优化

    ↓

Step 5: 保存最佳实践
记录成功的 Prompt 模式，供复用
```
