# RT-Thread 文档阅读笔记（第一阶段：Basic）

## 0. 阅读策略（先文档，后源码）
我的主路线：**先吃透 RTOS 设计文档，再进入源码**。

这样做的目标：
1. 先建立模块边界，避免陷入函数细节。
2. 先理解“为什么这样设计”，再看“代码怎么实现”。
3. 形成可面试复述的架构表达。

---

## 1. Basic.md 一页纸核心结论
1. RT-Thread 是可裁剪 RTOS，内核 API 以 `rt_` 前缀统一命名。
2. 调度策略是“**优先级抢占** + **同优先级时间片轮转**”。
3. 定时器分“单次/周期”，执行上下文分“硬定时器/软定时器”。
4. 线程同步核心对象：信号量、互斥量、事件集。
5. 互斥量有优先级继承，用于解决优先级反转。
6. 线程通信核心对象：邮箱、消息队列。
7. 内存管理存在多种策略：小内存算法、SLAB、memheap（可拼接不连续内存）。
8. 设备模型统一管理 I2C/SPI/UART/USB，用户通过统一设备 API 访问外设。
9. 学源码要从启动流程入手，这是理解全局架构的最高性价比入口。
10. 自动初始化机制是关键架构点：`INIT_*_EXPORT` 让“注册”与“调用时机”解耦。

---

## 2. 启动流程（必须能口述）
可归纳为四阶段：
1. 初始化硬件环境。
2. 初始化内核对象（调度器、定时器、IPC 等）。
3. 创建 main 线程，在 main 线程里做后续模块初始化。
4. 初始化定时器线程/空闲线程并启动调度器。

我的理解重点：
- `rt_components_board_init()`：更早执行，偏硬件准备。
- `rt_components_init()`：在 OS 运行后、main 线程上下文里执行，偏系统/应用组件初始化。

---

## 3. 自动初始化机制（架构关键）
核心思想：初始化函数不需要手工逐个调用，而是通过宏注册到指定段，由系统按阶段自动遍历执行。

常用阶段：
1. `INIT_BOARD_EXPORT(fn)`：最早，调度器尚未启动。
2. `INIT_PREV_EXPORT(fn)`：纯软件预初始化。
3. `INIT_DEVICE_EXPORT(fn)`：设备驱动初始化。
4. `INIT_COMPONENT_EXPORT(fn)`：组件初始化（如 FS/LwIP）。
5. `INIT_ENV_EXPORT(fn)`：环境初始化（如挂载文件系统）。
6. `INIT_APP_EXPORT(fn)`：应用初始化。

一句话总结：
**用宏做“声明式注册”，用启动流程做“确定性调度”。**

---

## 4. 内存段口径（面试高频）
- `Code`：代码段。
- `RO-data`：只读常量段。
- `RW-data`：已初始化且非零的全局/静态变量。
- `ZI-data`：未初始化或初始化为 0 的全局/静态变量。

统计关系：
- `RO Size = Code + RO-data`（主要对应 Flash 占用）
- `RW Size = RW-data + ZI-data`（运行时 RAM 占用）
- `ROM Size = Code + RO-data + RW-data`

---

## 5. 宏与编译器属性速记（来自 Basic）
1. `rt_inline`：内联，减少函数调用开销（空间换时间）。
2. `rt_used`：防止符号被优化器误删。
3. `RT_UNUSED`：抑制“未使用”告警（不是“不可使用”）。
4. `rt_weak`：弱符号，提供默认实现并允许用户重写。
5. `ALIGN(n)`：地址对齐。
6. `RT_ALIGN(size, align)`：大小向上对齐到 `align` 的倍数。

---

## 6. 易错点纠正
1. `$Sub$$main/$Super$$main` 是特定工具链机制，不是所有编译环境都一致。
2. `board_init` 和 `components_init` 的职责边界必须分清。
3. “邮箱固定 4 字节”可理解为“一个机器字大小消息槽位”，注意平台位宽语境。
4. 不要把“文档定义”直接等同“所有 BSP 的具体实现细节”。

# RT-Thread 文档阅读笔记（第二阶段：OBJECT）


静态对象会占用 RAM 空间，且不依赖于内存堆管理器。分配静态对象时，所需时间是固定的。动态对象则依赖于内存堆管理器：在运行过程中会请求 RAM 空间；对象被删除后，所占用的 RAM 空间会被释放。这两种方式各有优缺点，可根据实际需求进行选择。
