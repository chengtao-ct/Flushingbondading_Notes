
# Basic.md

1. RT-Thread 内核库也同样提供了此类函数的实现，主要供核心内核使用。不过，为了避免命名冲突，这些函数的名称特意加上了 rt_ 前缀
2. 线程调度算法采用的是**基于优先级的全抢占式多线程调度算法**
3. RT-Thread 支持创建多个具有相同优先级的线程。相同优先级的线程采用**时间片轮转调度算法 (Time Slice Rotation Scheduling)** 进行调度，以确保每个线程运行相同的时间。
4. RT-Thread 定时器提供两种定时机制：

- **单次触发定时器 (One-Shot Timer)** - 启动后仅触发一次定时器事件，随后自动停止。
    
- **周期触发定时器 (Periodic Trigger Timer)** - 周期性地触发定时器事件，直到用户手动停止，否则它将一直运行。
    

根据超时函数执行的上下文环境，RT-Thread 定时器可以设置为 HARD_TIMER（硬件定时器）或 SOFT_TIMER（软件定时器）模

5. 线程同步是通过获取和释放信号量与互斥量来实现的。

**互斥量 (Mutex)** 采用“优先级继承 (priority inheritance)”机制，有效解决了实时系统中常见的“优先级翻转 (priority inversion)”问题。线程同步机制允许线程按照优先级进行等待，或者按照先进先出 (FIFO) 的方式获取信号量/互斥量。

**事件集 (Event sets)** 主要用于线程间的同步，能够实现“一对多”和“多对多”的同步。它支持“或触发（独立同步）”和“与触发（关联同步）”，非常适合线程需要等待多个事件发生的情况。
6. RT-Thread 支持邮箱、消息队列等通信机制。邮箱中的消息长度固定为 4 字节。而消息队列则可以接收不同长度的消息，并将这些消息缓存在其自身的内存空间中
7. 在不同系统资源的情况下，动态内存堆管理模块会分别为内存规模较小的系统提供内存管理算法，为内存规模较大的系统提供 SLAB 内存管理算法
8. 还有一种名为 memheap 的动态内存堆管理方式，适用于那些具有不连续地址结构的系统。使用 memheap，用户可以将多个内存堆“拼接”在一起，使其像一个统一的内存堆那样被使用
9. RT-Thread 将 I2C、SPI、USB、UART 等作为外设设备，并通过统一的机制对这些设备进行管理。它实现了按名称访问的设备管理子系统，用户可通过统一的 API 接口来操作各种硬件设备。在设备驱动程序层面，根据嵌入式系统的特性，可以为不同设备绑定相应的事件。当设备事件发生时，驱动程序会通知上层应用程序。
10. 理解大多数代码通常都是从学习启动过程开始的，笑死我了，太有道理
11. `$Sub$$main` 可以调用一些需要在 main 函数之前执行的函数（此处添加了 RT-Thread 系统初始化函数）。之后，再调用 `$Super$$main` 来执行 `main()` 函数，这样用户就无需手动管理 main() 函数之前的系统初始化过程
12. 这段启动代码大致可以分为四个部分：

13. Initialize hardware related to the system.  
    初始化与系统相关的硬件。
14. Initialize system kernel objects, such as timers, schedulers, and signals.  
    初始化系统内核对象，如定时器、调度程序和信号处理机制。
15. Create the main thread initialize various modules in the main thread one by one.  
    在主线程中依次初始化各个模块。
16. Initialize the timer thread, idle thread, and start the scheduler.  
    初始化计时器线程和空闲线程，并启动调度器

18. 自动初始化设计

he automatic initialization mechanism means that the initialization function does not need to be called by explicit function. It only needs to be declared by macro definition at the function definition, and it will be executed during system startup.  
自动初始化机制意味着无需通过显式函数来调用初始化函数。只需在函数定义处用宏定义进行声明，系统启动时该函数便会自动执行。

For example, calling a macro definition in the serial port driver to inform the function that needs to be called to initialize the system. The code is as follows:  
例如，在串口驱动程序中调用宏定义，从而告知系统需要调用哪个函数来进行初始化。代码如下：

```c
int rt_hw_usart_init(void)  /* Serial port initialization function */
{
     ... ...
     /* Register serial port 1 device */
     rt_hw_serial_register(&serial1, "uart1",
                        RT_DEVICE_FLAG_RDWR | RT_DEVICE_FLAG_INT_RX,
                        uart);
     return 0;
}
INIT_BOARD_EXPORT(rt_hw_usart_init);    /* Use component auto-initialization mechanism */
```

The last part of the sample code `INIT_BOARD_EXPORT(rt_hw_usart_init)` indicates that the automatic initialization function is used. In this way, `rt_hw_usart_init()` function is automatically called by the system, so where is it called?  
示例代码的最后部分 `INIT_BOARD_EXPORT(rt_hw_usart_init)` 表明使用了自动初始化功能。这样一来， `rt_hw_usart_init()` 函数就会由系统自动调用。那么，它具体是在哪里被调用的呢？

In the system startup flowchart, there are two functions: `rt_components_board_init()` and `rt_components_init()`, subsequent functions inside the box with the underlying color represent functions that are automatically initialized, where:  
在系统启动流程图中，有两个功能： `rt_components_board_init()` 和 `rt_components_init()` 。方框内带有底色的功能表示会自动初始化的功能，其中：

1. “board init functions” are all initialization functions declared by `INIT_BOARD_EXPORT(fn)`.  
    “board init functions”指的是所有由 `INIT_BOARD_EXPORT(fn)` 声明的初始化函数。
2. “pre-initialization functions” are all initialization functions declared by `INIT_PREV_EXPORT(fn)`.  
    “预初始化函数”是指所有由 `INIT_PREV_EXPORT(fn)` 声明的初始化函数。
3. “device init functions” are all initialization functions declared by `INIT_DEVICE_EXPORT(fn)`.  
    “device init functions”指的是所有由 `INIT_DEVICE_EXPORT(fn)` 声明的初始化函数。
4. “components init functions” are all initialization functions declared by `INIT_COMPONENT_EXPORT(fn)`.  
    “components init functions”都是由 `INIT_COMPONENT_EXPORT(fn)` 声明的初始化函数。
5. “enviroment init functions” are all initialization functions declared by `INIT_ENV_EXPORT(fn)`.  
    “环境初始化函数”都是由 `INIT_ENV_EXPORT(fn)` 声明的初始化函数。
6. “application init functions” are all initialization functions declared by `INIT_APP_EXPORT(fn)`.  
    “application init functions”指的是所有由 `INIT_APP_EXPORT(fn)` 声明的初始化函数。

The `rt_components_board_init()` function executes earlier, mainly to initialize the relevant hardware environment. When this function is executed, it will traverse the initialization function table declared by `INIT_BOARD_EXPORT(fn)` and call each function.  
`rt_components_board_init()` 函数会提前执行，主要用于初始化相关的硬件环境。该函数执行时，会遍历 `INIT_BOARD_EXPORT(fn)` 声明的初始化函数表，并依次调用每个函数。

The `rt_components_init()` function is called and executed in the main thread created after the operating system is running. At this time, the hardware environment and the operating system have been initialized and the application-related code can be executed. The `rt_components_init()` function will transverse through the remaining few initialization function tables declared by macros.  
`rt_components_init()` 函数是在操作系统运行后创建的主线程中调用并执行的。此时，硬件环境和操作系统都已经初始化完成，可以执行与应用程序相关的代码了。 `rt_components_init()` 函数将会依次遍历那些通过宏声明的其余初始化函数表。

RT-Thread's automatic initialization mechanism uses a custom RTI symbol segment, it puts the function pointer that needs to be initialized at startup into this segment and forms an initialization function table which will be traversed during system startup. It calls the functions in the table to achieve the purpose of automatic initialization.  
RT-Thread 的自动初始化机制使用了一个自定义的 RTI 符号段。该机制将那些需要在启动时被初始化的函数指针放入这个符号段中，从而形成一张初始化函数表。系统启动时会遍历这张表中的函数，从而实现自动初始化的功能。

The macro interface definitions used to implement the automatic initialization function are described in the following table:  
用于实现自动初始化功能的宏接口定义如下表所示：

|Initialization sequence  初始化序列|Macro Interface  宏接口|Description  描述|
|---|---|---|
|1|INIT_BOARD_EXPORT(fn)|Very early initialization, the scheduler has not started yet.  <br>在非常早期的初始化阶段，调度程序尚未启动。|
|2|INIT_PREV_EXPORT(fn)|Mainly used for pure software initialization, functions without too many dependencies  <br>主要用于纯软件初始化，以及那些依赖关系不复杂的函数。|
|3|INIT_DEVICE_EXPORT(fn)|Peripheral driver initialization related, such as network card devices  <br>与外围设备驱动程序初始化相关的内容，例如网卡设备|
|4|INIT_COMPONENT_EXPORT(fn)|Component initialization, such as file system or LWIP  <br>组件初始化，例如文件系统或 LWIP|
|5|INIT_ENV_EXPORT(fn)|System environment initialization, such as mounting file systems  <br>系统环境初始化，例如挂载文件系统|
|6|INIT_APP_EXPORT(fn)|Application initialization, such as application GUI  <br>应用程序初始化，例如应用程序的图形用户界面|

Initialization function actively declares through these macro interfaces, such as `INIT_BOARD_EXPORT(rt_hw_usart_init)`, the linker will automatically collect all the declared initialization functions, placed in the RTI symbol segment, the symbol segment is located in the RO segment of the memory distribution. All functions in this RTI symbol segment are automatically called when the system is initialized.  
初始化函数通过这些宏接口（如 `INIT_BOARD_EXPORT(rt_hw_usart_init)` ）进行声明。链接器会自动收集所有被声明的初始化函数，并将它们放入 RTI 符号段中。该符号段位于内存分配中的 RO 段。系统初始化时，会自动调用 RTI 符号段中的所有函数。

20. 宏定义，**编译器特性（Compiler Attributes）、代码优化、链接器行为以及内存对齐**

### 1. rt_inline (内联函数)

- **宏定义**：#define rt_inline static __inline
    
- **原理解析**：
    
    - 普通的函数调用涉及 CPU 压栈（保存现场）、跳转到函数地址、执行代码、出栈（恢复现场）和返回。如果一个函数只有一两行代码且被频繁调用，这种调用的时间开销甚至比执行代码本身还长。
        
    - inline 告诉编译器：“请不要做函数跳转，直接把这段代码**复制粘贴**到调用它的地方”。这就是**空间换时间**，提高了执行速度。
        
    - static 限制了该函数只在当前 .c 文件可见，避免了多个文件同时内联同一个函数导致的“重复定义”错误。
        

### 2. rt_used (防止被优化器删除)

- **宏定义**：#define rt_used __attribute__((used))
    
- **原理解析**：
    
    - 现在的编译器（如 GCC, ARM Compiler 6）非常聪明。如果它发现一个函数或变量在代码中**没有任何人调用它**，它就会在优化时把它删掉（死代码消除，Dead Code Elimination），以节省内存。
        
    - 但在 RT-Thread 中，有些函数是靠**底层汇编、硬件中断或特定的段表（Section）**来隐式调用的（例如自动初始化组件 INIT_BOARD_EXPORT）。编译器查不出调用关系，就会误删。
        
    - __attribute__((used)) 就是给这个函数发一张“免死金牌”，告诉编译器：“别管有没有人调用，**强制把它留在最终的固件里**”。
        

### 3. RT_UNUSED (消除未使用警告) —— **🚨 原文翻译有误**

- **宏定义**：#define RT_UNUSED __attribute__((unused))
    
- **纠错**：原文中文翻译“表示某个函数或变量不可被使用”是完全错误的！英语原文是 "may not be used"（可能不会被使用）。
    
- **原理解析**：
    
    - 在写回调函数或实现特定接口时，往往必须按照规定的参数列表写代码。比如 void my_timer_callback(void *parameter)。但如果你的代码里根本不需要用到 parameter 这个变量，编译器就会烦人地弹出一个 Warning：“变量定义了却未使用”。
        
    - 加上 RT_UNUSED，就是告诉编译器：“我知道我没用它，这是我故意的，**请闭嘴，不要报 Warning**”。这对于保持代码整洁（Zero Warnings）非常重要。
        

### 4. rt_weak (弱符号 / 弱函数)

- **宏定义**：#define rt_weak __weak(在 GCC 中通常是**attribute**((weak))`)
    
- **原理解析**：
    
    - C 语言默认不允许存在两个名字相同的函数。
        
    - __weak 修饰的函数是一个**“备胎”**。RT-Thread 内核或驱动库通常会用 rt_weak 提供一个默认的空函数或基础实现（例如硬件初始化 rt_hw_board_init）。
        
    - 如果开发者在自己的代码里写了一个**同名但没有 rt_weak 修饰**的强函数，链接器就会使用开发者写的版本，而丢弃内核的“备胎”版本，且不会报错。
        
    - **应用场景**：为系统提供默认行为，同时允许用户自由重写（Override），非常适合底层框架设计。
        

### 5. ALIGN(n) (内存地址对齐)

- **宏定义**：#define ALIGN(n) __attribute__((aligned(n)))
    
- **原理解析**：
    
    - 这控制的是**变量在内存中存放的“起始地址”**。
        
    - 32位的 CPU（比如 ARM Cortex-M）最喜欢从 4 的倍数或 8 的倍数的地址去读写数据。如果地址不对齐（比如从地址 0x03 读取一个 32 位整数），CPU 可能需要读两次内存才能拼出这个数，甚至直接触发总线错误（HardFault 死机）。
        
    - 使用 ALIGN(8) 修饰一个结构体或数组，就是强行命令编译器：“给它分配内存时，起始地址必须是 8 的倍数”。
        

### 6. RT_ALIGN(size, align) (向上取整对齐算法)

- **宏定义**：#define RT_ALIGN(size, align) (((size) + (align) - 1) & ~((align) - 1))
    
- **原理解析**：
    
    - 与第 5 点不同，第 5 点管的是“地址”，这个宏管的是**“大小 (Size)”**。
        
    - 这是一个非常经典的位运算数学魔法，用于**向上取 align 的整数倍**（要求 align 必须是 2 的 n 次方，比如 4, 8, 16）。
        
    - **算法拆解（以计算 RT_ALIGN(13, 4) 为例）**：
        
        1. 目标：把 13 变成最接近且大于等于 13 的 4 的倍数（也就是 16）。
            
        2. align - 1 = 4 - 1 = 3（二进制 0011）。
            
        3. size + (align - 1) = 13 + 3 = 16。这一步保证了即便 size 刚好跨过一个倍数的一点点，也会被推到下一个倍数的区间里。
            
        4. ~((align) - 1) = ~3 = 二进制 1111...1100。
            
        5. 最后做按位与（&）操作：16 & 1111...1100。作用是**强制把最低的两位清零**，从而保证结果绝对是 4 的倍数。最终得出 16。
            
    - **应用场景**：给线程分配栈空间，或者分配内存块时，如果用户申请 13 个字节，系统为了硬件安全，强制分配给它 16 个字节。
