

##  内存管理功能特性

- 内存分配的时间必须是可预测的。在一般的内存管理算法中，系统需要找到一块与待存储数据大小相匹配的空闲内存区域，然后将数据存储到该区域内。不过，找到这样的空闲内存区域所需的时间是不可预测的。对于实时系统来说，这种情况是不可接受的。实时系统要求内存分配过程必须在可预测的时间内完成，否则，实时任务对外部事件的响应将会变得不可预测。

- 由于内存会不断地被分配和释放，整个内存区域就会变得支离破碎。这意味着，内存不再是一个连续的大块区域，而是由许多小块内存组成。这样一来，代码所能请求使用的最大连续内存块大小也就受到了限制。对于通用系统来说，可以通过重启系统来解决这个问题（比如每月或每几个月重启一次）。不过，对于那些需要全年无休地在野外持续运行的嵌入式系统来说，这种解决方案是不可行的。

首先是针对小内存块的分配管理机制（即小型内存管理算法）
第二点是针对大内存块的分配管理机制（即内存块管理算法）
第三点是针对多个内存堆的分配管理机制（即内存堆管理算法)



### 小型内存管理算法


### ① 小内存管理算法 (Small Memory Algorithm)

- **底层结构**：它将整块连续内存初始化为一个大块，每个分割出来的内存块前都带有一个**管理数据头（Data Head）** 。
    
- **链表指针操作**：所有的内存块（无论是已分配的 `used` 还是空闲的）都通过一个双向链表（Doubly Linked List）连接 。系统维持一个空闲链表指针 `lfree`，始终指向当前可用的空闲块 。
    
- **执行流程（以 `rt_malloc` 为例）**：
    
    1. 从 `lfree` 寻找第一个匹配尺寸的空闲块 。
        
    2. 如果该块太大（比如需要 64 字节，当前块有 128 字节），算法会执行**指针拆分** 。
        
    3. 将前半部分标记为 `used = 1` 并返回给用户；后半部分（剩余 52 字节）重新计算数据头，留在 `lfree` 双向链表中 。

1. 魔术值：这是一个可变数值（也被称为“魔术数字”）。其初始值为 0x1ea0，这个数值实际上与英文单词“heap”相关。该数值的作用是标记某个内存块为用于内存管理的数据块。该数值不仅用于标识某个数据块属于内存管理范畴，同时还具有内存保护功能：如果该区域的值被修改了，那就意味着该内存块被非法篡改了（通常只有内存管理程序才有权操作该内存区域）。


 2.  (USED)已使用：指示当前内存块是否已被分配


pool_ptr：小型内存对象的地址。如果该内存块的最低位为 1，则表示该内存块被使用；否则，该内存块未被使用。当该内存块被使用时，就可以通过计算来快速获取与该小型内存对象相关的各种信息


如下图所示，空闲内存指针 lfree 最初指向一个 32 字节大小的内存块。当用户线程需要分配 64 字节的内存空间时，由于 lfree 所指向的内存块只有 32 字节，无法满足需求，因此内存管理器会继续搜索下一个可用内存块。当找到一个 128 字节大小的内存块后，该内存块就符合分配要求。由于这个内存块较大，内存分配器会将其分割成多个部分。其中 52 字节的内存块仍会保留在 lfree 链表中。下表展示了分配 64 字节内存后的情况。



至于释放内存的过程，则是相反的。分配器会检查相邻的内存块是否为空闲。如果这些内存块都是空闲的，那么分配器就会把它们合并成一个大的空闲内存块。



####  板坯管理算法

RT-Thread 的 slab 分配器是一种针对嵌入式系统的优化内存分配算法。该算法基于 DragonFly BSD 的创始人 Matthew Dillon 所开发的 slab 分配器实现。而最原始的 slab 算法，则是 Jeff Bonwick 为 Solaris 操作系统设计的、高效的内核内存分配算法。


每个区域的大小在 32K 到 128K 字节之间。在堆初始化时，内存分配器会根据堆的大小自动调整各区域的规模。系统中，每个区域最多可容纳 72 个对象，每次最多可分配 16K 的内存。如果需要分配超过 16K 的内存，系统会直接从页面分配器中获取内存。每个区域所分配的内存块大小是固定的。那些能够分配相同大小内存块的区域会被链接在一起，形成链表。这 72 个区域对应的链表则被存储在一个数组中（zone_array[]）


- **底层结构**：这是一种**基于页面和区域（Zone）的内存池缓存算法** 。
    
- **链表指针操作**：系统拥有一个固定大小的 `zone` 数组（区域阵列），每个 `zone` 维护一个相同大小（如 32 字节、64 字节）的空闲块链表 。
    
- **执行流程**：
    
    1. **分配**：当申请 32 字节时，直接根据哈希或索引映射到 32 字节的 `zone` 链表，如果链表不空，直接从头部“摘除”一个块返回 。时间复杂度接近 $O(1)$。
        
    2. **释放**：根据指针计算出它属于哪个 `zone`，直接插回该 `zone` 的空闲链表 。


**(1) 内存分配**


假设分配了 32 字节的内存，那么内存分配器会首先根据这个 32 字节的数值，从区域数组的链表头部找到对应的区域节点。如果该链表为空，那么就需要为该页面分配一个新的区域，并从该区域中获取第一个空闲的内存块。如果链表不为空，那么该区域链表中的第一个节点必定包含一个空闲的内存块（否则该节点就不会被添加到链表中），此时就可以使用这个空闲内存块。如果在分配完内存后，该区域内的所有空闲内存块都被用完了，那么内存分配器就需要将该区域节点从链表中删除。


**(2)内存释放**

分配器需要找到存储该内存块所在的区域节点，然后将该内存块添加到该区域的空闲内存块链表中。如果该区域的空闲内存块链表表明该区域的所有内存块都已被释放，那么该区域就完全处于空闲状态。当区域链表中空闲区域的数量达到一定数值时，系统会将这些完全空闲的区域释放给页面分配器。



### 内存堆管理算法

Memheap 的工作机制如下图所示。首先，将多个内存块添加到 memheap_item 链表中。内存块的分配是从默认内存堆开始的。如果默认内存堆无法满足分配需求，则会查询 memheap_item 链表，尝试从其他内存堆中分配内存块。这一过程对用户来说是透明的，用户只看到一个内存堆而已。


### ③ memheap 管理算法 (memheap Algorithm)

- **底层结构**：它的核心是 `struct rt_memheap` 和 `struct rt_memheap_item` 控制块 。它不改变上述两种算法的内部细节，而是充当一个“胶水层” 。
    
- **链表指针操作**：它通过一个单向或双向链表将多块**物理地址不连续**的堆内存块（`memheap_item`）串联起来 。
    
- **执行流程**：
    
    1. 用户调用 `rt_malloc` 时，默认先从第一个默认堆分配 。
        
    2. 如果失败，则顺着 `memheap_item` 链表去遍历其它的内存堆，尝试在新的非连续堆上分配 。



### 共同点（The Standard）

- **硬件位置**：在 RT-Thread 中，它们默认管理的都是从 **ZI 段结束（`the end of ZI segment`）到内存尾部**的这块连续动态区域 。
    
- **多线程安全**：因为涉及全局链表的操作，为了防止多线程同时写产生竞争条件（Race Condition），内部必须考虑**互斥机制** 。因此文档中明确警告：**绝对不能在中断服务例程（ISR）中调用 `rt_malloc`/ `rt_free`**，否则可能导致当前线程被挂起（Suspend），破坏实时性 。
    

### 差异点与设计初衷（Why Author Wrote This）

- **小内存算法（追求低开销）**：针对资源紧缺（小于 2MB）的 MCU 。它的管理头开销极小，但缺点是随着不断拆分、释放，容易产生**内存碎片**，且搜索空闲块的时间具有不确定性 。
    
- **SLAB 算法（追求确定性与高性能）**：针对内存丰富的系统 。在实时系统（RTOS）中，“时间确定性”大于一切（即必须在可预测的时间内完成分配） 。SLAB 通过免去搜索过程、直接定位桶（Zone）的方式，消除了时间不确定性，且基本不产生内部碎片 。
    
- **memheap 算法（解决硬件碎片化）**：现代很多 MCU 的 RAM 是不连续的（例如内部 SRAM 64KB + 外部扩展 SDRAM 8MB）。传统的单堆无法跨越地址空洞，而 memheap 就像“胶水”一样把它们缝合，让用户感觉在使用一块巨大的连续内存

### 创建和销毁

`rt_mp_create` 和 `rt_mp_delete` 实现了**动态内存池的生命周期管理**：它们负责从系统动态堆（Heap）中申请空间并将其切割、组织成一组**固定大小、分配速度极快（常数时间）且绝无物理碎片**的内存块，并在不需要时予以完整回收



RT-Thread 为了不浪费额外的内存去存链表节点，采用了一个极其鸡贼（聪明）的办法：**直接利用空闲内存块的前 4 个字节（32位系统）来存放下一个空闲块的地址**。

1. `block_ptr + offset * ...`：这只是一个纯地址计算（游标偏移） 。
    
2. `(rt_uint8_t )`：意思是“**这是一个指针的指针**”，或者说“这是一个格子，里面装的是别人的地址”。
    
3. 最左边的 `*`：解引用，表示我们要强行往这个 4 字节的格子里写数据。
    
4. 右边的值：就是下一个块的绝对首地址 。 这样，整个内存池在初始化后，不需要分配任何额外的链表节点，就形成了一条完美的**隐式单向链表**

**零内存开销链表（Zero-overhead List）**： RT-Thread 巧妙利用了“**既然块是空闲的，里面的内存就没人用**”的特点，把前 4 字节当成了 `next` 指针 。当用户申请走这块内存后，前 4 字节又变成了用户的有效数据区 。这种将同一块物理内存在不同生命周期赋予不同含义的技巧，在资源紧张的嵌入式开发中非常高明。


```c
#ifdef RT_USING_HEAP  /* 只有当系统开启了动态堆管理器时，才能使用动态创建内存池的功能 */
/**
 * @brief 动态创建内存池对象并从堆中分配物理存储空间
 * @param name 内存池的名字，用于内核对象管理
 * @param block_count 内存池中包含的内存块总数
 * @param block_size 每个内存块的有效用户数据大小（字节）
 * @return 成功则返回内存池控制块句柄，失败则返回 RT_NULL
 */
rt_mp_t rt_mp_create(const char *name,
                     rt_size_t   block_count,
                     rt_size_t   block_size)
{
    rt_uint8_t *block_ptr;
    struct rt_mempool *mp;
    rt_size_t offset;

    /* 临界检查：确保当前不在中断服务例程（ISR）中，因为 rt_malloc 会阻塞或引起挂起 */
    RT_DEBUG_NOT_IN_INTERRUPT;

    /* 参数有效性校验：名字不能为空，数量和大小必须大于 0 */
    RT_ASSERT(name != RT_NULL);
    RT_ASSERT(block_count > 0 && block_size > 0);

    /* 【核心操作：对象分配】从系统内核对象管理器中，分配一个内核对象，类类别为内存池 */
    mp = (struct rt_mempool *)rt_object_allocate(RT_Object_Class_MemPool, name);
    if (mp == RT_NULL)
        return RT_NULL; /* 如果内存堆满了，分配控制块失败，直接返回 */

    /* 【对齐处理】将用户要求的块大小向上对齐到 CPU 的对齐边界（如 4字节或8字节） */
    block_size     = RT_ALIGN(block_size, RT_ALIGN_SIZE);
    mp->block_size = block_size;
    
    /* 核心公式计算：总物理大小 = (用户有效大小 + 单向链表指针大小) * 块总数 */
    mp->size       = (block_size + sizeof(rt_uint8_t *)) * block_count;

    /* 【物理空间分配】调用 rt_malloc 从堆空间切出整块动态连续内存给内存池 */
    mp->start_address = rt_malloc((block_size + sizeof(rt_uint8_t *)) *
                                  block_count);
    if (mp->start_address == RT_NULL)
    {
        /* 容错处理：如果物理内存不够，必须把前面刚刚创建成功的内核控制块释放掉，防止对象泄漏 */
        rt_object_delete(&(mp->parent));
        return RT_NULL;
    }

    /* 计数器初始化 */
    mp->block_total_count = block_count;
    mp->block_free_count  = mp->block_total_count;

    /* 初始化用于存放由于“没有空闲内存块”而挂起等待的线程双向链表 */
    rt_list_init(&(mp->suspend_thread));

    /* 【高能指针操作：初始化空闲单向链表】将大块内存切割并用指针串联 */
    block_ptr = (rt_uint8_t *)mp->start_address;
    for (offset = 0; offset < mp->block_total_count; offset ++)
    {
        /* * 晦涩指针拆解：
         * 1. block_ptr + offset * (block_size + 指针大小) 定位到当前内存块的起始绝对地址。
         * 2. *(rt_uint8_t **)(...) 将这个地址强制转换成一个“存放指针的格子”。
         * 3. 赋值号右边：下一块内存的绝对地址。
         * 核心作用：在当前块的最开始 4 字节里，存入下一个可用块的地址。
         */
        *(rt_uint8_t **)(block_ptr + offset * (block_size + sizeof(rt_uint8_t *)))
            = block_ptr + (offset + 1) * (block_size + sizeof(rt_uint8_t *));
    }

    /* 最后一个内存块没有下一块了，将它头部的指针格子清空（指向 RT_NULL），标志链表结束 */
    *(rt_uint8_t **)(block_ptr + (offset - 1) * (block_size + sizeof(rt_uint8_t *)))
        = RT_NULL;

    /* 将空闲链表头指针指向第一个内存块的入口 */
    mp->block_list = block_ptr;
    
    /* 初始化多核/多线程安全自旋锁 */
    rt_spin_lock_init(&(mp->spinlock));

    return mp; /* 返回构建完毕的内存池对象句柄 */
}
RTM_EXPORT(rt_mp_create); /* 导出符号，使得此函数在动态模块中也能被调用 */

/**
 * @brief 销毁动态内存池并释放其占用的所有资源
 * @param mp 内存池句柄
 * @return RT_EOK
 */
rt_err_t rt_mp_delete(rt_mp_t mp)
{
    rt_base_t level;

    /* 临界检查：中断中绝不允许删除内存池 */
    RT_DEBUG_NOT_IN_INTERRUPT;

    /* 参数安全性校验 */
    RT_ASSERT(mp != RT_NULL);
    RT_ASSERT(rt_object_get_type(&mp->parent) == RT_Object_Class_MemPool);
    RT_ASSERT(rt_object_is_systemobject(&mp->parent) == RT_FALSE); /* 静态初始化的对象不能用 delete 释放 */

    /* 【安全锁机制】保存中断状态并获取自旋锁，防止此时有其他线程分配或释放 */
    level = rt_spin_lock_irqsave(&(mp->spinlock));
    
    /* * 唤醒等待线程：
     * 如果此时有线程因为申请不到该池的内存正在挂起等待，内存池都要删除了，
     * 必须把它们全部强行唤醒，并赋予 RT_ERROR 错误码，告知它们“别等了，池子没了”。
     */
    rt_susp_list_resume_all(&mp->suspend_thread, RT_ERROR);

    /* 恢复中断状态，解锁 */
    rt_spin_unlock_irqrestore(&(mp->spinlock), level);

    /* 【释放物理空间】把 rt_malloc 申请出来的整块内存池大块归还给堆管理器 */
    rt_free(mp->start_address);

    /* 【内核对象注销】将内存池从内核对象容器中删除并释放控制块本身 */
    rt_object_delete(&(mp->parent));

    return RT_EOK;
}
RTM_EXPORT(rt_mp_delete);
```



### 线程获取API


这两段函数的核心作用是：**从指定的内存池中以 $O(1)$ 的常数时间极其高效地分配或释放固定大小的内存块** ；当内存池空了时，它允许申请线程**进行有期限或永久性的阻塞挂起**，直到有其他线程释放内存并将其唤醒


```c
/**
 * @brief 从指定的内存池中分配一个内存块
 * @param mp 内存池对象句柄
 * @param time 最大等待时间（滴答定时器 Tick 数）。0 表示不等待，负数可能表示永久等待
 * @return 成功返回可用的内存块地址，失败返回 RT_NULL
 */
void *rt_mp_alloc(rt_mp_t mp, rt_int32_t time)
{
    rt_uint8_t *block_ptr;
    rt_base_t level;
    struct rt_thread *thread;
    rt_uint32_t before_sleep = 0;

    /* 关键参数校验 */
    RT_ASSERT(mp != RT_NULL);

    /* 获取当前正在运行的线程控制块指针 */
    thread = rt_thread_self();

    /* 获取自旋锁并关中断，进入临界区，保证多核/多线程安全 */
    level = rt_spin_lock_irqsave(&(mp->spinlock));

    /* 【经典RTOS设计】使用 while 循环检查空闲块数量。用 while 是为了防止“惊群效应”或假唤醒 */
    while (mp->block_free_count == 0)
    {
        /* 如果没有空闲块，且用户设置 time == 0（不等待），则直接报错退出 */
        if (time == 0)
        {
            /* 退出临界区，恢复中断 */
            rt_spin_unlock_irqrestore(&(mp->spinlock), level);

            /* 设置线程错误码为超时 */
            rt_set_errno(-RT_ETIMEOUT);

            return RT_NULL;
        }

        /* 既然要等待，说明线程可能会挂起，因此绝不允许在中断上下文中执行 */
        RT_DEBUG_NOT_IN_INTERRUPT;

        /* 初始化当前线程的错误状态为 EOK */
        thread->error = RT_EOK;

        /* 【核心操作：挂起线程】将当前线程从就绪队列移除，挂起到内存池的 suspend_thread 阻塞链表上 */
        rt_thread_suspend_to_list(thread, &mp->suspend_thread, RT_IPC_FLAG_FIFO, RT_UNINTERRUPTIBLE);

        /* 如果设置了有限的超时时间（time > 0） */
        if (time > 0)
        {
            rt_tick_t time_tick = time;
            /* 记录投入睡眠前的系统滴答计数值，用于后续精确计算扣减时间 */
            before_sleep = rt_tick_get();

            /* 设置并启动该线程内置的线程定时器 */
            rt_timer_control(&(thread->thread_timer),
                             RT_TIMER_CTRL_SET_TIME,
                             &time_tick);
            rt_timer_start(&(thread->thread_timer));
        }

        /* 释放自旋锁并开中断。此时，线程虽然已经标记为挂起，但必须要开中断才能触发调度 */
        rt_spin_unlock_irqrestore(&(mp->spinlock), level);

        /* 【触发调度】把 CPU 让给其他就绪线程。当前线程在这里被切走，卡死在此处，直到被唤醒 */
        rt_schedule();

        /* -------- 线程被唤醒后，从此处开始继续执行 -------- */

        /* 检查醒来后的错误码，如果是因为内存池被删除（RT_ERROR）或定时器超时，直接返回 NULL */
        if (thread->error != RT_EOK)
            return RT_NULL;

        /* 如果是有限时间等待，醒来后需要扣除已经过去的时间，更新剩余等待时间 */
        if (time > 0)
        {
            time -= rt_tick_get() - before_sleep;
            if (time < 0)
                time = 0; /* 防止减过头变成负数 */
        }
        
        /* 重新进入 while 循环前，必须再次加锁关中断，保护 block_free_count */
        level = rt_spin_lock_irqsave(&(mp->spinlock));
    }

    /* 【核心操作：$O(1)$ 摘除空闲块】能走到这里，说明 mp->block_free_count > 0 */
    mp->block_free_count--;

    /* 1. 让 block_ptr 指向当前空闲链表的头块 */
    block_ptr = mp->block_list;
    RT_ASSERT(block_ptr != RT_NULL);

    /* 2. 晦涩指针操作：把链表头更新为当前块里面保存的“下一块的地址” */
    mp->block_list = *(rt_uint8_t **)block_ptr;

    /* 3. 偷梁换柱：当前块已经变成已分配块了。
     * 它的最前4个字节原本存放的是下一个空闲块地址，现在没用了，
     * 作者强行把它们改写为“当前所属内存池控制块 mp 的地址”！
     * 这为后面的 rt_mp_free 埋下了极巧妙的伏笔。
     */
    *(rt_uint8_t **)block_ptr = (rt_uint8_t *)mp;

    /* 退出临界区 */
    rt_spin_unlock_irqrestore(&(mp->spinlock), level);

    /* 调用用户注册的分配钩子函数 */
    RT_OBJECT_HOOK_CALL(rt_mp_alloc_hook,
                        (mp, (rt_uint8_t *)(block_ptr + sizeof(rt_uint8_t *))));

    /* 【重要】绝对不能把块的头 4 字节（存mp地址的格子）返回给用户！
     * 必须跳过这 4 字节，返回后面真正的有效数据区。
     */
    return (rt_uint8_t *)(block_ptr + sizeof(rt_uint8_t *));
}
RTM_EXPORT(rt_mp_alloc);

/**
 * @brief 释放一个内存块回到对应的内存池
 * @param block 要释放的内存块的用户数据区指针（即 rt_mp_alloc 返回的地址）
 */
void rt_mp_free(void *block)
{
    rt_uint8_t **block_ptr;
    struct rt_mempool *mp;
    rt_base_t level;

    /* 零指针安全校验 */
    if (block == RT_NULL) return;

    /* 【高能指针逆向操作】
     * 1. 传进来的 block 是用户数据区。
     * 2. (rt_uint8_t *)block - sizeof(rt_uint8_t *) 往前挪 4 字节，定位到该块真正的起始绝对地址！
     * 3. 强转为指针的指针 block_ptr。
     * 4. 读出里面的值：正是分配时强行存进去的内存池控制块地址 mp！这就不需要用户手动传 mp 句柄了。
     */
    block_ptr = (rt_uint8_t **)((rt_uint8_t *)block - sizeof(rt_uint8_t *));
    mp        = (struct rt_mempool *)*block_ptr;

    /* 调用用户注册的释放钩子函数 */
    RT_OBJECT_HOOK_CALL(rt_mp_free_hook, (mp, block));

    /* 加锁关中断进入临界区 */
    level = rt_spin_lock_irqsave(&(mp->spinlock));

    /* 空闲块计数器递增 */
    mp->block_free_count ++;

    /* 【核心操作：将块插回链表头部】
     * 1. 让当前被释放块的头 4 字节，指向当前内存池的整个空闲链表头。
     * 2. 让内存池的空闲链表头，指向当前刚刚释放的这个块。
     */
    *block_ptr = mp->block_list;
    mp->block_list = (rt_uint8_t *)block_ptr;

    /* 【唤醒阻塞线程】
     * 检查是否有线程在挂起链表上等内存。如果有，将最前面的那个线程出队并唤醒。
     * 如果成功唤醒了一个线程，返回真，进入 if 内部。
     */
    if (rt_susp_list_dequeue(&mp->suspend_thread, RT_EOK))
    {
        /* 既然唤醒了更高优先级的线程，必须先解锁，然后立刻触发一次调度，让被唤醒的线程马上去抢内存 */
        rt_spin_unlock_irqrestore(&(mp->spinlock), level);

        /* 执行调度 */
        rt_schedule();

        return;
    }
    
    /* 如果没有线程在等内存，直接解锁退出 */
    rt_spin_unlock_irqrestore(&(mp->spinlock), level);
}
RTM_EXPORT(rt_mp_free);
```

- **无感知的内聚设计（Encapsulation via Offset）**： RT-Thread 通过往前偏移 4 字节隐藏系统元数据的技巧，在 C 语言日常开发（如定制简易内存分配器、RPC 通信协议解析）中非常实用。它让 API 极其精简（用户释放时只需提供指针本身），把复杂性完全内聚在系统底层 。
    
- **IPC（进程间通信）的通用框架**：
    
    如果你仔细观察 `rt_mp_alloc` 中线程挂起（`rt_thread_suspend_to_list`）、开启定时器（`rt_timer_start`）、触发调度（`rt_schedule`）的组合拳，你会发现这和 RT-Thread 的**信号量（Semaphore）、队列（Queue）、邮箱（Mailbox）的获取（Take/Receive）函数底层逻辑一模一样**。掌握了这一段，整个 RT-Thread 的 IPC 阻塞唤醒机制你也就通了一大半！
