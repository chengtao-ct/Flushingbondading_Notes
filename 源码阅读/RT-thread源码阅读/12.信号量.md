
#### 定义和概览

信号的本质其实是一种“软中断”，其作用是通知某个线程：有异步事件发生了。它被用来在各个线程之间传递异常信息或处理紧急情况。线程无需做任何操作来等待信号的到达。实际上，线程根本不知道信号何时会到来。各个线程可以通过互相调用 `rt_thread_kill()` 来发送软中断信号

- 第一个是一种类似中断的处理程序。对于那些需要被处理的信号，线程可以指定相应的函数来进行处理。
- 第二种做法是忽略该信号，不采取任何行动，就好像它从未发生过一样。
- 第三点是保留系统的默认值，以便用于信号处理




#### 强行跳转现场实现函数

 ##### `_signal_deliver`
 - 底层实现逻辑，真正开始调度和操纵底层硬件寄存器上下文，把信号“强行灌入”目标线程的幕后黑手


`_signal_deliver` 是 RT-Thread 异步信号的“强行执法官”**；它负责针对目标线程的不同状态（阻塞、自己跑、别人跑、多核跑），通过**强行唤醒线程、发送核间硬件中断或直接在堆栈中伪造寄存器上下文的手法，确保目标线程能够在第一时间跳转去执行信号处理函数

```c
static void _signal_deliver(rt_thread_t tid)
{
    rt_base_t level;

    /* 1. 全局上锁：保护线程状态及信号位图 */
    level = rt_spin_lock_irqsave(&_thread_signal_lock);

    /* 2. 双重过滤：再次判定目标线程是否真的有关心并且挂起的未决信号，没有就直接返回 */
    if (!(tid->sig_pending & tid->sig_mask))
    {
        rt_spin_unlock_irqrestore(&_thread_signal_lock, level);
        return;
    }

    /* =========================================================================
     * 【情况一】：目标线程此刻正在“挂起/阻塞”状态（例如正在睡眠或等信号量）
     * ========================================================================= */
    if ((RT_SCHED_CTX(tid).stat & RT_THREAD_SUSPEND_MASK) == RT_THREAD_SUSPEND_MASK)
    {
        /* 强行“劫持”并唤醒线程去处理信号 */
#ifdef RT_USING_SMART
        rt_thread_wakeup(tid); // 混合微内核架构下的唤醒
#else
        rt_thread_resume(tid); // 强行将线程从阻塞链表中拉出来，投入就绪队列
#endif
        /* 给线程状态打上特殊钢印：SIGNAL（需要处理信号）和 SIGNAL_PENDING（有未决信号） */
        RT_SCHED_CTX(tid).stat |= (RT_THREAD_STAT_SIGNAL | RT_THREAD_STAT_SIGNAL_PENDING);

        rt_spin_unlock_irqrestore(&_thread_signal_lock, level);

        /* 主动触发调度：既然目标线程已经被强行唤醒了，立刻让调度器重新评估该让谁上 CPU */
        rt_schedule();
    }
    /* =========================================================================
     * 【情况二】：目标线程此刻是“就绪/运行”状态 (READY / RUNNING)
     * ========================================================================= */
    else
    {
        /* 子情况 2.1：当前线程在给自己发信号（即：信号的接收者就是我自己） */
        if (tid == rt_thread_self())
        {
            /* 状态打上 SIGNAL 标记 */
            RT_SCHED_CTX(tid).stat |= RT_THREAD_STAT_SIGNAL;

            rt_spin_unlock_irqrestore(&_thread_signal_lock, level);

            /* 【原地处决】：如果当前没有嵌套在硬件中断里，立刻原地调用 handler 执行信号动作！ */
            if (rt_interrupt_get_nest() == 0)
            {
                rt_thread_handle_sig(RT_TRUE);
            }
        }
        /* 子情况 2.2：我正在向“另外一个处于就绪态的线程”投递信号 */
        else if (!((RT_SCHED_CTX(tid).stat & RT_THREAD_STAT_SIGNAL_MASK) & RT_THREAD_STAT_SIGNAL))
        {
            /* 先行注入信号状态标记 */
            RT_SCHED_CTX(tid).stat |= (RT_THREAD_STAT_SIGNAL | RT_THREAD_STAT_SIGNAL_PENDING);

#ifdef RT_USING_SMP
            /* 【2.2.A - 多核 SMP 场景】：
             * 如果目标线程正在另一个 CPU 核心上欢快地运行，我们不能直接改它的寄存器！
             * 必须通过硬件发送 IPI（核间中断），利用硬件中断强行把那个 CPU 核打断，
             * 那个核在退出中断时，就会自动切去处理这个信号。
             */
            {
                int cpu_id;

                cpu_id = RT_SCHED_CTX(tid).oncpu; // 查出目标线程在哪个 CPU 上
                if ((cpu_id != RT_CPU_DETACHED) && (cpu_id != rt_cpu_get_id()))
                {
                    rt_uint32_t cpu_mask;

                    cpu_mask = RT_CPU_MASK ^ (1 << cpu_id);
                    rt_hw_ipi_send(RT_SCHEDULE_IPI, cpu_mask); // 发送核间调度中断
                }
            }
#else
            /* 【2.2.B - 单核极其硬核场景：制造切片上下文（筑栈魔术）】 
             * 目标线程是就绪的，但当前没在跑（躺在就绪队列里，sp 存的是它上次被切走时的全套寄存器现场）。
             * 我们的目标：当它下一次被调度上 CPU 的瞬间，不让它回到原本被打断的代码，而是先去执行信号处理函数！
             */
            RT_SCHED_CTX(tid).stat &= ~RT_THREAD_STAT_SIGNAL_PENDING;
            
            /* 1. 用 sig_ret 备份线程原本真正的栈指针 sp (里面存着原业务代码现场) */
            tid->sig_ret = tid->sp;
            
            /* 2. 伪造一个崭新的硬件中断栈帧（压入栈中）！
             * 调用 rt_hw_stack_init 构造一个虚假的上下文，将 PC（程序计数器）强行指向 _signal_entry，
             * 同时把新的栈顶传回给 tid->sp。
             * (void *)((char *)tid->sig_ret - 32) 是为了在当前栈中空出一段安全隔离区，防止栈踩踏。
             */
            tid->sp = rt_hw_stack_init((void *)_signal_entry, RT_NULL,
                                       (void *)((char *)tid->sig_ret - 32), RT_NULL);
#endif /* RT_USING_SMP */

            rt_spin_unlock_irqrestore(&_thread_signal_lock, level);
            LOG_D("signal stack pointer @ 0x%08x", tid->sp);

            /* 重新调度：确保状态被修改的目标线程能尽快获得执行 */
            rt_schedule();
        }
        else
        {
            /* 目标线程已经在处理信号中了，不再重复处理，安全退出 */
            rt_spin_unlock_irqrestore(&_thread_signal_lock, level);
        }
    }
}
```


当调度器下一次选中这个线程，去读取它的 `tid->sp` 并执行汇编中断退出（如 ARM 的 `bx lr` 或 `RFE`）时，CPU 会以为自己是在恢复一个正常的中断。但由于数据被我们篡改了，CPU 弹栈出来的 **PC 指针直接指向了 `_signal_entry`**！ 于是，线程一复苏，就莫名其妙地跑去执行信号处理函数了。当信号函数执行完，它会再从 `tid->sig_ret` 里把当年真正的业务代码现场恢复回来，接着走。这种**在线程浑然不知的情况下强行更改其执行流**的手段，就叫异步软件中断。


- **数据所有权与跨核协议（核间解耦）：** 在 SMP（多核）代码块中，RT-Thread 展现了极强的多核体系架构功底。**多核 RTOS 编程的铁律是：谁的核，谁负责。** 永远不要试图在本地核心上强行修改另一个核心的微架构状态，通过 IPI（核间中断）将需求发送给对方核心，由对方核心在其本地中断上下文中自理，这是多核系统实现高内聚、低耦合、避免死锁的最高原则。
    
- **硬件机制的软件复刻：** 这段代码再次证明，操作系统的本质就是用软件去扩充硬件。硬件有中断向量表，RT-Thread 就用 `sig_vectors` 指针数组做软件信号向量表；硬件有 NVIC 中断控制器，RT-Thread 就用 `sig_mask` 和 `sig_pending` 位图做软件控制器。这种将硬件原理提炼并映射到软件层面的设计，是每一位资深嵌入式架构师都必须具备的“软硬件合一”的通透思维。


#### 下载信号量


##### `rt_signal_install`

`rt_signal_install` 的核心作用是**为当前线程的指定异步信号绑定特定的处理函数（回调）**；它在底层通过“懒加载”机制按需为线程分配信号向量表内存，并在安全临界区内完成函数指针的覆写与置换。


```c
rt_sighandler_t rt_signal_install(int signo, rt_sighandler_t handler)
{
    rt_base_t level;
    rt_sighandler_t old = RT_NULL;
    rt_thread_t tid = rt_thread_self(); // 获取当前正在运行、试图安装信号的线程句柄

    /* 1. 防御性合法性校验：确保信号编号 signo 在 0 ~ RT_SIG_MAX 之间 */
    if (!sig_valid(signo)) return SIG_ERR;

    /* 2. 第一次上锁：检查当前线程是否已经拥有“信号向量表” 
     * _thread_signal_lock 是保护全系统所有线程信号操作的全局自旋锁 */
    level = rt_spin_lock_irqsave(&_thread_signal_lock);
    
    /* 【核心黑科技：延迟加载/懒加载模式】
     * 并非每个线程都需要用信号。为了省内存，线程刚创建时 sig_vectors 指针是空的（RT_NULL）。
     * 只有当线程第一次调用 install 试图安装信号时，系统才发现它没有向量表，需要现场去分配。
     */
    if (tid->sig_vectors == RT_NULL)
    {
        /* 暂时解开自旋锁并开中断 
         * 为什么？因为后面的分配函数 rt_thread_alloc_sig 内部涉及内核堆内存分配(malloc)，
         * 可能会发生阻塞或耗时较长，绝对不能在拿着自旋锁、关闭中断的情况下执行！ */
        rt_spin_unlock_irqrestore(&_thread_signal_lock, level);

        /* 现场为该线程分配一块存放信号处理函数指针的数组内存（向量表） */
        rt_thread_alloc_sig(tid);

        /* 内存分配完毕，重新关中断并锁死自旋锁，继续往下走 */
        level = rt_spin_lock_irqsave(&_thread_signal_lock);
    }

    /* 3. 向量表就绪后，开始安全覆写注册函数 */
    if (tid->sig_vectors)
    {
        /* 暂存并备份旧的处理函数指针，作为返回值还给用户 */
        old = tid->sig_vectors[signo];

        /* 情况 A：如果用户传入 SIG_IGN，表示“忽略”该信号，清空指针 */
        if (handler == SIG_IGN) tid->sig_vectors[signo] = RT_NULL;
        
        /* 情况 B：如果用户传入 SIG_DFL，表示“恢复系统默认处理”，挂载内核自带的默认函数 */
        else if (handler == SIG_DFL) tid->sig_vectors[signo] = _signal_default_handler;
        
        /* 情况 C：普通情况，直接将用户自定义的函数指针（回调函数）填入对应的向量表中 */
        else tid->sig_vectors[signo] = handler;
    }
    
    /* 4. 退出临界区，恢复中断状态 */
    rt_spin_unlock_irqrestore(&_thread_signal_lock, level);

    /* 返回原先的旧处理函数指针（方便用户在处理完后能恢复现场） */
    return old;
}
```


**什么是信号机制？（软件中断的模拟）：** 在单片机裸机开发中，当外部引脚发生电平跳变，硬件会触发中断，CPU 强行停下当前 `main` 循环，跳去执行 `ISR` 中断服务程序，执行完再回 `main`。 **信号机制，就是操作系统在线程层面上完整复制了这一套硬件中断行为：** 当 A 线程调用 `rt_thread_kill(tid, signo)` 向 B 线程发送信号时，B 线程只要还没屏蔽该信号，无论它此刻在执行什么业务代码，内核都会在它下一次进入就绪态时，**强行修改它的运行上下文（覆写 PC 指针）**，逼迫它去调用通过 `rt_signal_install` 安装的这个 `handler` 函数，执行完后，再让它回到被信号打断的那一行代码继续走


- **原子操作中的“中途呼吸（Unlock Window）”：** 在编写底层驱动或多线程高并发系统时，常会遇到“检查某个条件，不满足则创建资源”的场景。切记：**凡是耗时的操作（数据拷贝、内存分配、打印日志、等待外设），坚决不要放在紧锁的中断临界区里。** 像 RT-Thread 这样，先解锁，把耗时的事做完，再重新上锁复查条件，是让你的嵌入式软件支持高并发、降低硬件中断响应延迟（Interrupt Latency）的至关重要的 C 语言编程修养。
    
- **置换旧指针的设计模式（Swap Pattern）：** 函数最后返回了 `old` 指针。这种设计在 Linux API 中也极为常见。它给应用层留了一条完美的退路：你可以临时接管一个信号（安装你的 handler 捕获它），处理完你的特殊逻辑后，你可以再次调用 install 把原先的 `old` 指针传回去，无缝恢复系统原有的执行生态。这是一种极度安全、不破坏前人劳动成果的面向对象包装策略。



#### 信号量的标志位的设置


这两个函数是 RT-Thread 异步信号（Signal）机制中控制“开关闸门”的经典实现：`rt_signal_mask`（信号阻塞/屏蔽）与 `rt_signal_unmask`（信号解除阻塞/恢复）。

它们通过对 C 语言中极其高效的**位运算（Bitwise Operations）**，完美模拟了硬件 CPU 中“屏蔽中断（Mask Interrupt）”和“使能中断（Enable Interrupt）”的行为。


```c
/**
 * @brief    此函数用于阻塞（屏蔽）指定的异步信号。
 * @note     一旦信号被屏蔽，即使其他线程调用 rt_thread_kill() 发送该信号，
 * 当前线程也会假装没看见，信号会被挂起（Pending），直到解除屏蔽。
 */
void rt_signal_mask(int signo)
{
    rt_base_t level;
    rt_thread_t tid = rt_thread_self(); // 获取当前正在执行的线程句柄

    /* 1. 全局上锁：进入信号机制的临界区，保护全系统线程的信号状态 */
    level = rt_spin_lock_irqsave(&_thread_signal_lock);

    /* 2. 【核心黑科技：位清零】
     * sig_mask(signo) 宏将信号编号转换为对应的位掩码（例如信号3 变为 00001000）。
     * 然后按位取反（~）变成 11110111，再与 tid->sig_mask 进行按位与（&=）操作。
     * 作用：将该信号对应的 Bit 位强行清零（0 表示该通道被锁死，信号无法通过）。
     */
    tid->sig_mask &= ~sig_mask(signo);

    /* 3. 释放锁并恢复中断 */
    rt_spin_unlock_irqrestore(&_thread_signal_lock, level);
}

/**
 * @brief    此函数用于解除（恢复）指定异步信号的阻塞。
 * @note     解除阻塞后，如果该信号在之前被屏蔽期间曾经投递过（处于 Pending 状态），
 * 线程会立刻停下手中的活，转去执行对应的信号处理函数。
 */
void rt_signal_unmask(int signo)
{
    rt_base_t level;
    rt_thread_t tid = rt_thread_self(); // 获取当前线程句柄

    /* 1. 全局上锁：保护线程信号位图状态 */
    level = rt_spin_lock_irqsave(&_thread_signal_lock);

    /* 2. 【核心黑科技：位置 1】
     * 使用按位或（|=）操作，将该信号对应的 Bit 位强行置为 1（1 表示通道大开，允许通行）。
     */
    tid->sig_mask |= sig_mask(signo);

    /* 3. 【检查债务】：看看在闭关锁国（屏蔽）期间，有没有积压的“未决消息（Pending）”？
     * tid->sig_mask & tid->sig_pending 
     * 作用：如果某个信号既处于“允许通行(mask为1)”状态，又处于“正在卡关排队(pending为1)”状态，
     * 说明有积压的异步信号需要立刻处理！
     */
    if (tid->sig_mask & tid->sig_pending)
    {
        /* 4A. 既然要处理积压信号，首先解开自旋锁。
         * 为什么要在 _signal_deliver 之前解锁？因为信号处理函数（用户回调）可能非常耗时，
         * 绝对不能死死攥着临界区全局锁去执行用户的未知业务代码，否则全系统信号机制都会瘫痪。
         */
        rt_spin_unlock_irqrestore(&_thread_signal_lock, level);
        
        /* 5. 递交/投递信号：强行改变当前线程的栈帧流，强迫其立刻跳转执行信号处理 handler */
        _signal_deliver(tid);
    }
    else
    {
        /* 4B. 闸门打开了，但期间并没有人发过这封信，平安无事，直接解锁退出 */
        rt_spin_unlock_irqrestore(&_thread_signal_lock, level);
    }
}
```

- **`&= ~` 与 `|=` 的经典组合**：
    
    - **关闸**：`mask &= ~掩码`，用于将位图中特定的一位精确定位清零，其他位不受影响。
        
    - **开闸**：`mask |= 掩码`，用于将位图中特定的一位精确定位置 1，其他位保持原样。
        
- **清算交叉判定 (`mask & pending`)**：这是解开阻塞时的核心逻辑。`sig_pending` 是一个“记录员位图”，如果信号在被屏蔽时送达，系统无法执行它，就会把 `sig_pending` 中对应的 Bit 位置 1（即：记账挂起）。当执行 `unmask` 开闸时，通过两者按位与，就能在 1 个 CPU 周期内查出“是否有刚开闸的通道正有数据在排队”。

**软件中断的“不丢弃与延期执行”机制**： 这与硬件中断（如 MCU 的 NVIC）的运作方式如出一辙。当你在单片机里把某个外设中断关闭（Mask）了，期间外设触发了硬件中断，NVIC 内部的 Pending 寄存器依然会被置位。一旦你在程序里重新打开该中断，CPU 会**瞬间跳进中断服务函数**。

`rt_signal_unmask` 完美复制了这一硬件行为。当你由于某段代码处于紧急状态（不能被打断）而 `mask` 了某个信号，处理完后一调 `unmask`，`_signal_deliver(tid)` 就会像硬件中断控制器一样，硬生生把当前线程的汇编 PC 指针（程序计数器）和 SP 指向的栈帧强行篡改，瞬间去执行信号的回调函数。





#### 信号主动同步等待接口

**而这个函数提供的是一种**“主动同步阻塞”机制（类似于 POSIX 标准中的 `sigwait`）：线程在此主动停下，死等某一个或某一组信号的发生，并在拿到信号携带的数据后继续向下运行。

- `rt_signal_wait`

```c
int rt_signal_wait(const rt_sigset_t *set, rt_siginfo_t *si, rt_int32_t timeout)
{
    int ret = RT_EOK;
    rt_base_t level;
    rt_thread_t tid = rt_thread_self(); // 获取当前主动等信号的线程控制块
    struct siginfo_node *si_node = RT_NULL, *si_prev = RT_NULL;

    /* 1. 上下文环境自检：死等信号的动作只能在线程上下文中执行，严禁在中断里调用 */
    RT_DEBUG_IN_THREAD_CONTEXT;

    /* 2. 入参防御性校验：检查信号集和接收缓冲区是否合法 */
    if (set == NULL || *set == 0 || si == NULL )
    {
        ret = -RT_EINVAL;
        goto __done_return;
    }

    /* 清空接收缓冲区，防止残留垃圾数据造成逻辑污染 */
    memset(si, 0x0, sizeof(rt_siginfo_t));

    /* 3. 进入全局信号临界区：关中断，上自旋锁 */
    level = rt_spin_lock_irqsave(&_thread_signal_lock);

    /* 【快路径检查（已经有信在手）】：
     * tid->sig_pending 是目前已经送达但还没处理的信号位图。
     * *set 是用户想等的信号位图。通过按位与（&），如果结果不为0，
     * 说明不用睡了，用户想等的信号早就处于 Pending（未决状态）在排队了！直接跳转到 __done 领信。
     */
    if (tid->sig_pending & *set) goto __done;

    /* 如果没信，且用户是个急性子（timeout==0），直接报错超时退出 */
    if (timeout == 0)
    {
        ret = -RT_ETIMEOUT;
        goto __done_int;
    }

    /* 4. 【慢路径：进入休眠】 
     * 代码执行到这里，说明没有现成的信号。线程必须交出 CPU 开始排队睡觉。
     * 强行挂起当前线程：这里传入 RT_UNINTERRUPTIBLE（不可中断），
     * 意味着在接下来睡觉期间，它不接受常规的异步信号弹栈打断，必须一心一意在这里等它想要的信或者超时！
     */
    rt_thread_suspend_with_flag(tid, RT_UNINTERRUPTIBLE);
    
    /* 核心状态位修改：在线程的状态字段上追加“正在死等信号(SIGNAL_WAIT)”的钢印 */
    RT_SCHED_CTX(tid).stat |= RT_THREAD_STAT_SIGNAL_WAIT;

    /* 5. 启动线程自带的超时“闹钟” */
    if (timeout != RT_WAITING_FOREVER)
    {
        rt_tick_t timeout_tick = timeout;
        rt_timer_control(&(tid->thread_timer), RT_TIMER_CTRL_SET_TIME, &timeout_tick);
        rt_timer_start(&(tid->thread_timer));
    }
    
    /* 6. 暂时解开全局锁，并立刻触发调度器！
     * 当前线程正式在后台睡着，CPU 切换去跑别的就绪任务。
     */
    rt_spin_unlock_irqrestore(&_thread_signal_lock, level);
    rt_schedule(); // <--- 线程在这里断点停滞，直到由于“等到了信”或“闹钟响了”被唤醒

    /* =========================================================================
     * 【唤醒复苏现场】：线程醒来了，立刻重新关中断上锁，保护信号状态
     * ========================================================================= */
    level = rt_spin_lock_irqsave(&_thread_signal_lock);

    /* 擦除“死等信号”的状态位标记 */
    RT_SCHED_CTX(tid).stat &= ~RT_THREAD_STAT_SIGNAL_WAIT;

    /* 7. 检查醒来的原因：如果是被自带的闹钟超时唤醒的 */
    if (tid->error == -RT_ETIMEOUT)
    {
        tid->error = RT_EOK; // 清除线程内部的错误记账
        rt_spin_unlock_irqrestore(&_thread_signal_lock, level);

        ret = -RT_ETIMEOUT; // 返回给应用层特定的超时错误码
        goto __done_return;
    }

__done:
    /* =========================================================================
     * 8. 【清算领信阶段】：代码来到这里，说明手里一定抓到信号了（不管是本来就有的还是刚等到的）
     * 目标：从线程附带的“信号信息单向链表 (si_list)”里，掏出第一封符合用户要求的信。
     * ========================================================================= */
    si_node = (struct siginfo_node *)tid->si_list; // 拿到链表头
    while (si_node)
    {
        int signo;

        signo = si_node->si.si_signo; // 获取当前节点的信号编号
        
        /* 检查当前节点的信号是否在用户等待的信号集(*set)中 */
        if (sig_mask(signo) & *set)
        {
            /* 找到了！将信号节点里的完整信息结构体(rt_siginfo_t)拷贝给用户的接收缓冲区 */
            *si = si_node->si;

            LOG_D("sigwait: %d sig raised!", signo);
            
            /* 【硬核链表断开与重连】：将这个被领走的节点从单向链表中摘除 */
            if (si_prev) 
                si_prev->list.next = si_node->list.next; // 如果不是表头，前节点的 next 越过当前节点
            else
            {
                /* 如果被领走的就是第一个节点（表头变动） */
                struct siginfo_node *node_next;

                if (si_node->list.next)
                {
                    // 链表还有后继，更新后继节点为新的链表头 tid->si_list
                    node_next = (void *)rt_slist_entry(si_node->list.next, struct siginfo_node, list);
                    tid->si_list = node_next;
                }
                else
                {
                    // 链表空了，头指针彻底置空
                    tid->si_list = RT_NULL;
                }
            }

            /* 9. 销毁遗迹：既然这封信被应用层完整接走了，Pending 位图中对应的 Bit 位清零 */
            tid->sig_pending &= ~sig_mask(signo);
            
            /* 将存放信件信息节点的物理内存还给内核小内存池 (Memory Pool)，防止内存泄漏 */
            rt_mp_free(si_node);
            break; // 成功领到一封，功德圆满，跳出遍历
        }

        /* 链表滑动更新指针：当前节点不是用户想要的，继续往后找 */
        si_prev = si_node;
        if (si_node->list.next)
        {
            si_node = (void *)rt_slist_entry(si_node->list.next, struct siginfo_node, list);
        }
        else
        {
            si_node = RT_NULL; // 到达链表末尾，没找到
        }
     }

__done_int:
    /* 释放全局信号锁并恢复中断 */
    rt_spin_unlock_irqrestore(&_thread_signal_lock, level);

__done_return:
    return ret;
}
```


- **阶段一：状态判定与睡眠（第 3 ~ 7 步）**
    
    - **快路径检查**：`tid->sig_pending & *set`。利用位运算在 1 个时钟周期内判定是否有现成满足条件的信号已经在排队。如果有，根本不需要进入昂贵的 `rt_schedule()`，直接跳到阶段二领信，这是内核极致性能的体现。
        
    - **不可中断休眠**：调用 `rt_thread_suspend_with_flag(..., RT_UNINTERRUPTIBLE)`。既然线程在这里调用了显式的 `wait` 接口，在它等的目标信号或超时到来前，系统给它打上不可中断标记，防止外部其他琐碎异步信号打断这里的同步等待语义。
        
- **阶段二：单向链表检索与垃圾回收（第 8 ~ 9 步）**
    
    - 当线程醒来且没有超时，它开始遍历 `tid->si_list` 链表。这个链表里面存放的是内核通过小内存池（`rt_mp_alloc`）动态申请并串起来的信件本体（包含 signo、发送者、附带参数等）。
        
    - **链表摘除与 `rt_mp_free`**：一旦在链表里匹配到符合 `*set` 的节点，通过经典的 `si_prev->list.next = si_node->list.next` 将其从单链表中剔除，并立刻调用 `rt_mp_free(si_node)` 释放这块节点内存。

- **异步通信向同步通信的工程妥协：**
    
    异步信号机制（通过 `rt_signal_install` 注册回调）虽然响应极其迅速，类似于硬件中断，但在业务逻辑开发中它存在一个痛点：**会打断正常的控制流**。如果你的线程正在进行极其紧密的数学计算或文件写操作，突然中断跳开去执行回调，很容易引发重入风险（比如死锁、全局变量被踩）。
    
    `rt_signal_wait` 提供了一种**把异步事件拉直成同步流**的手段：线程主动在安全、确定的代码段停下来，像取快递一样等信号。当它醒来时，上下文是完全连贯可控的。这种由异步向同步转换的思想，是编写高安全、高确定性嵌入式应用时的核心架构技巧。
    
- **未决队列与小内存池（Memory Pool Fixed Allocation）：**
    
    当外部通过 `rt_thread_kill` 频繁向一个线程发送信号时，如果这个线程正在闭关（`mask` 屏蔽或者没有来得及收），这些信号的信息（`rt_siginfo_t`）必须在内核中找个地方存起来，系统会在内核里把它们串成 `si_list` 链表。
    
    既然要动态存消息，就必须要分配内存节点（`struct siginfo_node`）。在实时操作系统中，为了绝对禁止动态堆分配（`malloc`）带来的时间不确定性和碎片化，RT-Thread 为此专门划分了一个固定尺寸的**小内存池（Memory Pool）**。所有的信号节点申请（`rt_mp_alloc`）和这里的释放（`rt_mp_free`）都是在固定块内存池中进行的，耗时为绝对的 $O(1)$，完美保证了实时系统的底线。



#### 生命周期终点站与资源打扫人



```c
void rt_thread_handle_sig(rt_bool_t clean_state)
{
    rt_base_t level;
    rt_thread_t tid = rt_thread_self(); // 信号是在“当前执行的线程”上下文里被处理的
    struct siginfo_node *si_node;

    /* 1. 锁住全局信号自旋锁，并关闭全局中断 */
    level = rt_spin_lock_irqsave(&_thread_signal_lock);
    
    /* 2. 双重过滤判定：如果当前有未决信号，且没有被屏蔽 */
    if (tid->sig_pending & tid->sig_mask)
    {
        /* 【关键防御】：如果线程此时正处于“同步死等信号(SIGNAL_WAIT)”状态，
         * 说明有专门的 rt_signal_wait() 函数在安全地处理它，本函数绝不能插手（防止二次重复处理） */
        if (!(RT_SCHED_CTX(tid).stat & RT_THREAD_STAT_SIGNAL_WAIT))
        {
            /* 3. 【核心循环】：只要还有未决信号在排队，就逐个拉出来“原地正法” */
            while (tid->sig_pending & tid->sig_mask)
            {
                int signo, error;
                rt_sighandler_t handler;

                /* 掏出挂在当前线程 TCB 上的信号信息链表(si_list)的头节点 */
                si_node = (struct siginfo_node *)tid->si_list;
                if (!si_node) break;

                /* 【链表摘除】：把这个即将被处理的节点从单向链表中移出 */
                if (si_node->list.next == RT_NULL)
                    tid->si_list = RT_NULL; // 后面没节点了，头指针直接置空
                else
                    tid->si_list = (void *)rt_slist_entry(si_node->list.next, struct siginfo_node, list);

                /* 4. 提取信号的核心元数据 */
                signo   = si_node->si.si_signo;     // 拿到信号编号
                handler = tid->sig_vectors[signo];  // 丛向量表中查出对应的用户回调函数
                tid->sig_pending &= ~sig_mask(signo); // 位清零：表明该信号已经出队并正在被处理

                /* 5. 【极其高明：临时解开锁窗】
                 * 在调用户的自定义处理函数 handler 之前，必须先恢复中断，解开自旋锁！
                 * 理由：用户的 handler 逻辑未知，可能会极度耗时、打印日志甚至阻塞，绝不能带着全局锁执行！ */
                rt_spin_unlock_irqrestore(&_thread_signal_lock, level);

                LOG_D("handle signal: %d, handler 0x%08x", signo, handler);
                
                /* 6. 执行用户的信号处理回调函数（业务逻辑在这里真正被激活） */
                if (handler) handler(signo);

                /* 7. 用户的 handler 执行完了，重新加锁关闭中断，继续回临界区打扫战场 */
                level = rt_spin_lock_irqsave(&_thread_signal_lock);
                
                /* 【核心机制：软件中断惊醒标志】
                 * 强制将错误码强行改成 -RT_EINTR。
                 * 作用：告诉那些刚刚在等信号量/等消息队列而被信号震醒的阻塞函数：“你的休眠被软件中断打断了！” */
                error = -RT_EINTR;

                /* 将刚刚处理完的信号节点内存彻底还给小内存池(Memory Pool) */
                rt_mp_free(si_node); 
                
                /* 覆写 TCB 里的错误记账 */
                tid->error = error;
            } // 循环回去，看还有没有其他未屏蔽的信号排队，直到全部处理完

            /* 8. 退出循环，根据指示决定是否清除 TCB 的 SIGNAL 处理状态 */
            if (clean_state == RT_TRUE)
            {
                RT_SCHED_CTX(tid).stat &= ~RT_THREAD_STAT_SIGNAL;
            }
            else
            {
                rt_spin_unlock_irqrestore(&_thread_signal_lock, level);
                return;
            }
        }
    }
    rt_spin_unlock_irqrestore(&_thread_signal_lock, level);
}

/**
 * @brief    为指定的线程动态分配“信号向量表”内存（即函数指针数组）。
 */
void rt_thread_alloc_sig(rt_thread_t tid)
{
    int index;
    rt_bool_t need_free = RT_FALSE;
    rt_base_t level;
    rt_sighandler_t *vectors;

    /* 1. 在锁外面，大张旗鼓地调用堆内存分配函数申请 RT_SIG_MAX 个处理函数指针的空间 */
    vectors = (rt_sighandler_t *)RT_KERNEL_MALLOC(sizeof(rt_sighandler_t) * RT_SIG_MAX);
    RT_ASSERT(vectors != RT_NULL);

    /* 2. 初始化：默认全部挂载系统自带的默认信号处理程序 */
    for (index = 0; index < RT_SIG_MAX; index ++)
    {
        vectors[index] = _signal_default_handler;
    }

    /* 3. 内存准备好了，现在进入临界区，开始核对所有权（严密核对双重判定） */
    level = rt_spin_lock_irqsave(&_thread_signal_lock);

    if (tid->sig_vectors == RT_NULL)
    {
        /* 此时没有竞争，完美绑定给线程 */
        tid->sig_vectors = vectors;
    }
    else
    {
        /* 【高并发防御】：发现在我刚刚脱衣服去申请堆内存的空档里，另一个 CPU 核心上的线程
         * 抢先一步帮这个 tid 申请并安装好了向量表！那我刚刚申请的这块就多余了，必须标记释放 */
        need_free = RT_TRUE;
    }

    rt_spin_unlock_irqrestore(&_thread_signal_lock, level);

    /* 4. 如果发现竞争失败了（多余了），在锁外面默默地把它 free 掉 */
    if (need_free)
    {
        rt_free(vectors);
    }
}

/**
 * @brief    当某个线程要消亡退出时（如调用 rt_thread_detach 或自动退出），
 * 此函数负责将该线程名下残存的所有信号资源搜刮干净，防碎片、防内存泄漏。
 */
void rt_thread_free_sig(rt_thread_t tid)
{
    rt_base_t level;
    struct siginfo_node *si_node;
    rt_sighandler_t *sig_vectors;

    /* 1. 关中断上锁，迅速把指针“偷走”并断开连接，将临界区时间压缩到最短 */
    level = rt_spin_lock_irqsave(&_thread_signal_lock);
    
    si_node = (struct siginfo_node *)tid->si_list;
    tid->si_list = RT_NULL; // 斩断挂起的未处理信号链表

    sig_vectors = tid->sig_vectors;
    tid->sig_vectors = RT_NULL; // 斩断向量表
    
    rt_spin_unlock_irqrestore(&_thread_signal_lock, level);

    /* 2. 【安全窗口清理】：在没有任何自旋锁的自由状态下，开始漫长的单向链表内存释放 */
    if (si_node)
    {
        struct rt_slist_node *node;
        struct rt_slist_node *node_to_free;

        LOG_D("free signal info list");
        node = &(si_node->list);
        do
        {
            node_to_free = node;
            node = node->next;
            /* 借助单链表宏还原出节点本体，一个个拍死（还给小内存池） */
            si_node = rt_slist_entry(node_to_free, struct siginfo_node, list);
            rt_mp_free(si_node);
        } while (node); // 直到链表末尾，一条不留
    }

    /* 3. 如果当年申请过向量表内存，彻底从内核堆（Heap）里释放它 */
    if (sig_vectors)
    {
        RT_KERNEL_FREE(sig_vectors);
    }
}
```

- **全异步环境下的“死锁防御机制” (Double-Check Locking & Free Window)：** 在 `rt_thread_alloc_sig` 中，作者展示了操作系统处理多核/中断高并发时最经典的“双重判定锁”技巧。 **为什么要先在锁外面 malloc ，进锁检查完后又可能在锁外面 free 呢？** 因为 `malloc` 是一件非常耗时、且其内部可能也会去争夺内存堆锁的操作。如果我们在 `_thread_signal_lock` 被锁死的情况下无脑去调 `malloc`，高并发下全系统的信号收发和软中断会瞬间全部被卡死在这个核心上。
    
    “在无锁时高高兴兴申请，在有锁时严密核对所有权，多余了就在无锁时默默销毁”，这体现了极其高超的系统级原子操作驾驭能力。
    
- **为什么要将线程的错误码强行改成 `-RT_EINTR` ？** 在 `rt_thread_handle_sig` 内部有这样一行：`tid->error = -RT_EINTR;`。这是类 UNIX/Linux 操作系统中极其重要的“被信号中断的系统调用（Interrupted System Call）”原理。 假设你的线程原本正阻塞在 `rt_sem_take` 或者 `rt_mq_recv` 上睡觉，突然外面来了一个信号。上一课我们讲到，`_signal_deliver` 会暴力篡改堆栈把你的线程叫醒来处理信号。当你的信号处理函数 `handler` 执行完毕后，你回到刚才的代码，如果操作系统不作为，你的 `rt_sem_take` 就会糊里糊涂地继续往下走（但你实际根本没抢到信号量！）。 RT-Thread 强行灌入 `-RT_EINTR`（内部打断错误），当你的正常业务流醒来后检查 `if (rt_thread_self()->error == -RT_EINTR)`，就会恍然大悟：“原来我是被信号吵醒的，而不是真正拿到了资源！”从而主动退回，重新排队。



#### 传递信号入口


虽然它的名字叫 `kill`（借鉴了 POSIX 标准），但它的真实目的**绝不是无脑杀死线程**，而是**向指定线程投递一个软件中断（信号）**。

我先为您在源码上编写硬核的底层注释，重点剖析其中“合并同类信号”**和**“内存池分配”的巧妙设计。

```c
int rt_thread_kill(rt_thread_t tid, int sig)
{
    siginfo_t si;
    rt_base_t level;
    struct siginfo_node *si_node;

    /* 1. 防御性入参校验 */
    RT_ASSERT(tid != RT_NULL);
    if (!sig_valid(sig)) return -RT_EINVAL; // 确保信号编号合法

    LOG_I("send signal: %d", sig);
    
    /* 2. 在栈上打包信号的基本元数据包 (siginfo_t) */
    si.si_signo = sig;                  // 记录信号编号
    si.si_code  = SI_USER;              // 标记该信号是由用户线程发出的，而非系统内核异常触发
    si.si_value.sival_ptr = RT_NULL;    // 附带的额外参数（此处为空）

    /* 3. 进入全局信号临界区：关中断，上自旋锁 */
    level = rt_spin_lock_irqsave(&_thread_signal_lock);
    
    /* =========================================================================
     * 【策略一：合并同类项机制】
     * tid->sig_pending 是目标线程当前“已经送达但还没来得及处理”的信号位图。
     * 通过判定 (tid->sig_pending & sig_mask(sig))，如果对应 Bit 位已经是 1，
     * 说明在这个信号处理完之前，同样编号的信号曾经送达过一次！
     * ========================================================================= */
    if (tid->sig_pending & sig_mask(sig))
    {
        struct rt_list_node *node;
        struct siginfo_node  *entry;

        /* 获取目标线程未处理信号单向链表(si_list)的表头 */
        si_node = (struct siginfo_node *)tid->si_list;
        if (si_node)
            node = (struct rt_slist_node *)&si_node->list;
        else
            node = RT_NULL;

        /* 遍历链表，找到之前发过的那个同编号信号，用最新的数据直接覆盖(memcpy)它！
         * 作用：防止同一种未决信号在链表中无限制堆积，实现“只记账，不堆砌”的非实时信号标准。 */
        for (; (node) != RT_NULL; node = node->next)
        {
            entry = rt_slist_entry(node, struct siginfo_node, list);
            if (entry->si.si_signo == sig)
            {
                /* 覆盖旧数据 */
                memcpy(&(entry->si), &si, sizeof(siginfo_t));
                
                /* 既然更新了原有的信号数据，说明不需要再申请新内存，直接解锁提前返回成功！ */
                rt_spin_unlock_irqrestore(&_thread_signal_lock, level);
                return 0; 
            }
        }
    }
    
    /* 如果这是一个全新的信号（此前没有在排队的同类信号），释放自旋锁，去准备新节点 */
    rt_spin_unlock_irqrestore(&_thread_signal_lock, level);

    /* =========================================================================
     * 【策略二：全新信号投递——小内存池分配】
     * ========================================================================= */
    /* 从内核专门预留的固定块小内存池(_siginfo_pool)中，申请一个空闲节点。
     * 耗时是绝对确定的 O(1)，绝不产生堆碎片，非常安全。 */
    si_node = (struct siginfo_node *) rt_mp_alloc(_siginfo_pool, 0);
    if (si_node)
    {
        rt_slist_init(&(si_node->list));
        memcpy(&(si_node->si), &si, sizeof(siginfo_t)); // 将信号数据拷入节点中

        /* 重新关中断上锁，准备挂入目标线程名下 */
        level = rt_spin_lock_irqsave(&_thread_signal_lock);

        if (tid->si_list)
        {
            /* 如果目标线程原本就有其他信号在排队，用单链表追加接口挂到队伍末尾 */
            struct siginfo_node *si_list;

            si_list = (struct siginfo_node *)tid->si_list;
            rt_slist_append(&(si_list->list), &(si_node->list));
        }
        else
        {
            /* 如果队列是空的，当前节点直接作为链表头 */
            tid->si_list = si_node;
        }

        /* 【重要账本置位】：在目标线程的 Pending 位图中，把当前信号对应的 Bit 位置 1！
         * 告诉系统：“该线程背上了这个未决信号债务，醒来时必须清算！” */
        tid->sig_pending |= sig_mask(sig);

        rt_spin_unlock_irqrestore(&_thread_signal_lock, level);
    }
    else
    {
        /* 极端情况：系统里的未决信号太多，把内存池榨干了，报错返回 */
        LOG_E("The allocation of signal info node failed.");
        return -RT_EEMPTY;
    }

    /* 4. 终极激活：调用我们在上一节深度拆解过的硬核分发器，
     * 强行去篡改目标线程的运行栈，或者跨核发送硬件 IPI 中断，强迫其去执行信号流！ */
    _signal_deliver(tid);

    return RT_EOK;
}
```


- **同类去重处理 (The Duplicate Filter)**： 进入函数并获取自旋锁后，首先检查 `tid->sig_pending` 标志位。如果对应信号位已经是 1，说明上一次投递的同类信号还没被目标线程消费。此时，它会顺着单向链表找到那个历史节点，用最新的参数直接覆盖它，然后**直接解锁退出**（省去了后续昂贵的内存分配和分发流程）。
    
- **小内存池原子装填 (O(1) Memory Injection)**： 如果通过了去重过滤器，说明是一个新信号。系统使用 `rt_mp_alloc` 快速在专属信号内存池里切出一个小方块（节点），重新加锁后，将其追加到 `tid->si_list` 链表的尾部（若是首次则作为头）。并用位运算 `|= sig_mask(sig)` 将 Pending 位图盖上章。
    
- **软件中断递交 (Trigger Delivery)**： 最后一行调用 `_signal_deliver(tid)`。通过上节课的内容我们知道，这行代码会硬生生篡改目标线程的 SP 和 PC，或者跨 CPU 核发送硬件中断，让信号“见缝插针”地执行。


- **非可靠信号（Unreliable Signal）的工程妥协**： 这段代码完美展现了标准 POSIX 信号中“非可靠信号”**（信号值在 1~31 之间）的内核实现真相。 在复杂的实时系统中，如果 A 线程以极快的速度（比如在 1ms 内）连续给 B 线程发了 10 次“常规数据刷新信号”，而 B 线程由于正在执行高优先级任务处于闭关状态。 **如果内核选择老老实实为这 10 次发送都申请内存排队，那么内核的信号小内存池在几毫秒内就会被彻底榨干并瘫痪崩溃。** RT-Thread 的做法是：信号只代表“发生过”，不代表“发生了几次”。只要前面的信号还没被处理，后面的信号直接覆盖前面的数据，Pending 位图始终只有 1 个 Bit 记录。这种**“同类信号合并”设计，在用最小的内存代价满足异步通知需求的同时，展现了极强的系统防御力。
    
- **缩小锁粒度的“断开视窗”（Lock Window Break）**： 请再次高度赞赏 RT-Thread 架构师对锁的克制： 在第一阶段检查完去重后，它**主动释放了自旋锁**（`rt_spin_unlock_irqrestore`），然后去调用了 `rt_mp_alloc`。分配成功后，在挂载单链表前，**又重新上锁**。 因为 `rt_mp_alloc` 内部需要去检索内存池的位图和控制块，会消耗数个甚至数十个 CPU 周期。如果全程死死攥着临界区全局自旋锁 `_thread_signal_lock` 去做内存分配，那么多核系统下其他所有 CPU 核心只要遇到信号操作就只能全部空转（Spin）等待，极大地恶化了全系统的硬实时响应指标。
