**ai**: ## 【问题诊断】

这份笔记的核心价值在于：**它不是在讲"代码怎么写"，而是在讲"模块边界怎么划"**。很多项目的维护灾难，根源就是接口定义模糊——调用者不知道该传什么，被调用者不知道该返回什么，出了问题互相甩锅。

---

## 【根本原因分析】

### 为什么接口书写如此重要？

嵌入式系统的复杂度往往不在于单个算法，而在于**模块间的耦合关系**。一个模糊的接口会引发连锁反应：

```
参数范围不明 → 调用者传边界值 → 被调用者未处理 → 现场偶发死机
    ↓
排查时发现：是"0"代表"停止"还是"0"代表"无效"？没人说得清
```

笔记中的16条原则，本质上是在回答三个问题：
1. **谁在调用？**（调用者身份、运行上下文）
2. **能传什么？**（参数范围、单位、边界）
3. **失败了怎么办？**（错误处理、安全回退）

---

## 【解决方案】

### 一、全局约定是什么？

**全局约定**是整个项目的"宪法"——在写任何接口之前，先定义好所有模块共同遵守的规则。它解决的是"重复约定"和"隐式假设"的问题。

### 二、工程实例：电机控制系统

假设我们有一个运动控制系统，架构如下：

```
┌─────────────────────────────────────────────────┐
│                   Application                    │
│              (状态机、业务逻辑)                    │
├─────────────────────────────────────────────────┤
│                   Motion 层                      │
│         (运动规划、速度曲线、坐标变换)              │
├─────────────────────────────────────────────────┤
│                   Motor 层                       │
│         (PWM驱动、编码器读取、电流采样)            │
├─────────────────────────────────────────────────┤
│                   HAL 层                         │
│              (TIM、GPIO、DMA)                    │
└─────────────────────────────────────────────────┘
```

```markdown
# 全局约定 v1.0

## 1. 轮序与方向定义
- 轮序编号：左前=0，右前=1，左后=2，右后=3
- 方向符号：正值=前进/顺时针，负值=后退/逆时针

## 2. 单位约定
| 物理量 | 单位 | 数据类型 | 说明 |
| :--- | :--- | :--- | :--- |
| 速度 | mm/s | int16_t | 范围 [-5000, 5000] |
| 角速度 | mrad/s | int16_t | 范围 [-31415, 31415] |
| 占空比 | 千分比 | int16_t | 范围 [-1000, 1000]，0=停止 |
| 位置脉冲 | count | int32_t | 编码器原始计数 |

## 3. 运行上下文规则
| 上下文 | 允许的操作 | 禁止的操作 |
| :--- | :--- | :--- |
| ISR | 置标志、采样、读取寄存器 | printf、malloc、HAL_Delay、控制计算 |
| 主循环 | 所有API调用 | 长时间阻塞 |
| RTOS任务 | 所有API调用 + 阻塞等待 | 在临界区内调用阻塞API |

- 所有API返回 `bool` 或错误码枚举
- 非法参数：立即返回失败，不修改任何状态
- 硬件故障：触发安全回调，执行 StopAll()
- 通信超时：重试3次后上报错误

## 5. 初始化顺序
```
SystemClock_Config() → GPIO_Init() → TIM_Init() → Motor_Init() → Motion_Init() → Application_Init()
```

## 6. 安全优先级
1. EStop() - 紧急停止，可在任意上下文调用
2. StopAll() - 正常停止，仅主循环调用
3. 其他控制接口
```

## 【大师的工程建议】

### 1. 接口文档先于代码

```
错误做法：写完代码 → 补注释 → 被迫改接口 → 文档过期
正确做法：写接口契约 → 评审 → 写代码 → 契约即文档
```

### 2. 用静态断言约束参数

```c
// 在接口实现中强制执行契约
bool Motor_SetDuty(uint8_t wheel_id, int16_t duty) {
    // 契约：wheel_id ∈ [0, 3]
    if (wheel_id >= WHEEL_COUNT) {
        return false;  // 非法参数，不修改状态
    }
    // 契约：duty ∈ [-1000, 1000]
    duty = CLAMP(duty, -1000, 1000);
    
    // 执行硬件操作
    __HAL_TIM_SET_COMPARE(&htim_motor, channel[wheel_id], 
                          duty_to_ccr(duty));
    return true;
}
```

### 3. ISR中的铁律

```c
// ❌ 错误：在ISR中做控制计算
void TIMx_IRQHandler(void) {
    speed = calculate_speed();      // 计算耗时不确定
    duty = pid_compute(speed);      // 浮点运算，可能阻塞
    Motor_SetDuty(0, duty);         // 可能触发其他中断
}

// ✅ 正确：ISR只采样，主循环做计算
volatile int32_t encoder_delta;
void TIMx_IRQHandler(void) {
    encoder_delta = TIM2->CNT - last_cnt;  // 仅读取寄存器
    last_cnt = TIM2->CNT;
    flag_control_tick = true;              // 置标志
}

void MainLoop(void) {
    while (1) {
        if (flag_control_tick) {
            flag_control_tick = false;
            speed = calculate_speed(encoder_delta);  // 主循环计算
            duty = pid_compute(speed);
            Motor_SetDuty(0, duty);
        }
    }
}
```

**ai**: ## 【问题诊断】

这三个问题触及了电机控制的核心概念混淆：**控制量 vs 被控量**、**位置 vs 速度**、**原始数据 vs 物理量**。这些概念不清，PID调参时就会陷入"改了参数没效果"的困境。

---

## 【根本原因分析】

### 三个概念的本质区别

| 概念 | 本质 | 角色 | 单位示例 |
| :--- | :--- | :--- | :--- |
| **占空比** | PWM输出的电压比例 | **控制量（输入）** | 千分比 [-1000, 1000] |
| **速度** | 电机的运动快慢 | **被控量（输出）** | mm/s 或 mrad/s |
| **角速度** | 旋转速度（自转） | **被控量（输出）** | rad/s 或 mrad/s |
| **偏转角** | 旋转位置（姿态） | **状态量（位置）** | rad 或 deg |
| **位置脉冲** | 编码器原始计数 | **测量值（原始）** | count（无量纲） |

---

## 【解决方案】

### 一、占空比 ≠ 速度

这是最常见的误解。用一个生活类比：

```
占空比 = 油门踏板的位置（你踩多深）
速度   = 车实际跑多快（结果）

踩油门50% → 平路可能跑60km/h → 上坡可能只跑30km/h
```

**工程场景**：

```c
// ❌ 错误理解：占空比直接等于速度
Motor_SetDuty(0, 500);  // 以为"速度=500mm/s"
// 实际：上坡时速度变慢，下坡时速度变快

// ✅ 正确理解：占空比是PID的输出，速度是PID的目标
int16_t target_speed = 500;  // 目标速度 mm/s
int16_t current_speed = Motor_GetSpeed(0);  // 实际速度
int16_t error = target_speed - current_speed;
int16_t duty = PID_Compute(error);  // PID计算占空比
Motor_SetDuty(0, duty);  // 占空比是控制手段
```

**关系图**：

```
          ┌──────────────┐
          │   目标速度    │ (你想要的)
          │  500 mm/s    │
          └──────┬───────┘
                 │
                 ▼
          ┌──────────────┐
          │   PID控制器   │
          │  (误差计算)   │
          └──────┬───────┘
                 │
                 ▼
          ┌──────────────┐
          │    占空比     │ ← 控制量（PID输出）
          │    +600      │
          └──────┬───────┘
                 │
                 ▼
          ┌──────────────┐
          │   电机+负载   │
          │  (物理系统)   │
          └──────┬───────┘
                 │
                 ▼
          ┌──────────────┐
          │   实际速度    │ ← 被控量（反馈）
          │   480 mm/s   │
          └──────────────┘
```

### 二、角速度 ≠ 偏转角

这两个是**速度 vs 位置**的关系，就像"车速"和"里程"。

| 概念 | 含义 | 单位 | 典型场景 |
| :--- | :--- | :--- | :--- |
| **角速度** | 旋转快慢（转多快） | rad/s | 电机转速、陀螺仪输出 |
| **偏转角** | 旋转位置（转多少） | rad | 舵机角度、机器人姿态 |

**工程场景**：

```c
// 场景1：舵机控制 → 用偏转角
Servo_SetAngle(90);  // 转到90度位置（位置控制）

// 场景2：电机调速 → 用角速度
Motor_SetAngularSpeed(100);  // 以100rad/s旋转（速度控制）

// 场景3：机器人转向 → 两者都用
// 角速度控制"转多快"，偏转角控制"转多少度后停"
void Robot_Turn(int16_t angular_speed, int16_t target_angle) {
    while (current_angle < target_angle) {
        Motor_SetAngularSpeed(angular_speed);  // 控制转速
        current_angle += angular_speed * dt;   // 积分得到角度
    }
    Motor_Stop();
}
```

**数学关系**：

$$\theta(t) = \int_0^t \omega(\tau) \, d\tau$$

$$\omega = \frac{d\theta}{dt}$$

- 角速度 $\omega$ 是偏转角 $\theta$ 的导数
- 偏转角是角速度的积分

### 三、位置脉冲是什么？

**位置脉冲是编码器的原始输出**，是"未加工的原材料"。

**编码器工作原理**：

```
电机轴旋转 → 编码器光盘转动 → 光电传感器输出脉冲
                              ↓
                         TIM硬件计数
                              ↓
                         CNT寄存器值 = 位置脉冲
```

**为什么用"脉冲"而不是"角度"？**

```c
// 编码器参数：1000线，4倍频 → 每转4000脉冲
#define ENCODER_PPR  4000  // Pulses Per Revolution

// 原始数据
int32_t pulse_count = TIM2->CNT;  // 位置脉冲，单位：count

// 转换为角度（需要知道编码器参数）
float angle_deg = (pulse_count * 360.0f) / ENCODER_PPR;
float angle_rad = (pulse_count * 2 * PI) / ENCODER_PPR;

// 转换为速度（需要采样时间）
int32_t pulse_delta = pulse_count - last_pulse_count;
float speed_rad_s = (pulse_delta * 2 * PI) / (ENCODER_PPR * dt);
```

**为什么全局约定里保留"位置脉冲"？**

| 方式 | 优点 | 缺点 |
| :--- | :--- | :--- |
| **用脉冲** | 无精度损失、无浮点运算、硬件直接读取 | 不直观、依赖编码器参数 |
| **用角度** | 直观、物理意义明确 | 浮点运算开销、精度损失 |

**工程建议**：

```c
// ✅ 底层接口用原始脉冲（高效、无精度损失）
int32_t Motor_GetEncoderPulse(uint8_t wheel_id);

// ✅ 应用层接口用物理单位（直观、易理解）
int16_t Motor_GetSpeed(uint8_t wheel_id);  // mm/s

// 内部转换
int16_t Motor_GetSpeed(uint8_t wheel_id) {
    int32_t pulse = Motor_GetEncoderPulse(wheel_id);
    int32_t delta = pulse - last_pulse[wheel_id];
    last_pulse[wheel_id] = pulse;
    // 转换：脉冲 → mm/s（需要轮子周长、减速比等参数）
    return (int16_t)((delta * WHEEL_CIRCUMFERENCE) / (ENCODER_PPR * CONTROL_PERIOD));
}
```

### 问题一：控制周期是什么？

**控制周期不是"计算出来的时钟"，而是你人为设计的"心跳节奏"。**

---

## 【解决方案】

### 一、控制周期的本质

用一个生活类比：

```
控制周期 = 医生查房的间隔时间

每1小时查一次房 → 发现问题慢，但医生轻松
每1分钟查一次房 → 发现问题快，但医生累死
```

**在电机控制中**：

```
┌─────────────────────────────────────────────────────────────┐
│                      时间轴                                  │
```

**为什么选1ms？**

| 控制周期 | 控制频率 | 适用场景 | CPU负载 |
| :--- | :--- | :--- | :--- |
| 10ms | 100Hz | 慢速系统（温度控制、液位） | 低 |
| 5ms | 200Hz | 一般运动控制 | 中 |
| **1ms** | **1kHz** | **电机速度控制（常用）** | **中高** |
| 0.1ms | 10kHz | 高性能伺服、FOC | 高 |

**选择依据**：

```c
// 经验法则：控制频率 ≥ 10倍系统带宽
// 电机机械时间常数约10ms → 带宽约100Hz → 控制频率≥1kHz → 周期≤1ms

// 你的TIM配置（假设84MHz时钟）：
// PSC = 83, ARR = 999
// 中断周期 = (PSC+1) × (ARR+1) / f_clock = 84 × 1000 / 84MHz = 1ms
```

### 二、副作用是什么？

**副作用 = 函数执行时，除了返回值之外，对外部世界产生的任何影响。**

---

#### 副作用的四种类型

| 类型 | 示例 | 风险等级 |
| :--- | :--- | :--- |
| **修改硬件寄存器** | `TIMx->CCR = duty` | ⚠️ 中（时序敏感） |
| **修改全局变量** | `motor_state = RUNNING` | ⚠️ 中（并发问题） |
| **调用其他函数** | `HAL_GPIO_WritePin(...)` | ⚠️ 高（可能阻塞） |
| **无副作用（纯函数）** | `int add(a,b) { return a+b; }` | ✅ 安全 |

---

#### Motor_SetDuty 的副作用分析

```c
bool Motor_SetDuty(uint8_t wheel_id, int16_t duty) {
    // 参数检查（无副作用）
    if (wheel_id >= WHEEL_COUNT) return false;
    
    // 限幅（无副作用）
    duty = CLAMP(duty, -1000, 1000);
    
    // ====== 副作用开始 ======
    
    // 副作用1：修改TIM寄存器 → PWM输出立即改变
    __HAL_TIM_SET_COMPARE(&htim_motor, channel[wheel_id], duty_to_ccr(duty));
    
    // 副作用2：修改全局状态（如果有）
    motor_duty[wheel_id] = duty;
    
    // 副作用3：记录日志（如果有）
    // log_printf("Motor %d duty set to %d", wheel_id, duty);  // 可能阻塞！
    
    // ====== 副作用结束 ======
    
    return true;
}
```

#### 为什么要在接口文档中声明副作用？

**场景1：ISR中调用**

```c
// ❌ 灾难：副作用包含阻塞操作
void TIMx_IRQHandler(void) {
    Motor_SetDuty(0, 500);  // 如果内部有printf，ISR会卡死
}

// ✅ 安全：副作用明确声明"仅修改寄存器"
void TIMx_IRQHandler(void) {
    Motor_SetDuty(0, 500);  // 文档保证：无阻塞，ISR安全
}
```

**场景2：调试时**

```c

```

**场景3：单元测试**

**ai**: ## 【问题诊断】

这是一个非常经典的概念混淆：**PWM频率 ≠ 控制频率**。这两个"频率"完全不是一回事，但经常被搞混。

---

## 【根本原因分析】

### 两个频率的本质区别

| 概念 | 本质 | 谁决定 | 典型值 |
| :--- | :--- | :--- | :--- |
| **PWM频率** | 载波频率，一个PWM波形的周期 | TIM的ARR/PSC | 20kHz（50μs周期） |
| **控制频率** | PID计算频率，多久调整一次占空比 | 另一个定时器中断 | 1kHz（1ms周期） |

---

## 【解决方案】

### 一、图解两者的关系

```
PWM频率 20kHz（周期50μs）—— 硬件层面，TIM自动生成
├─ 波形1: ████████░░░░░░░░░░░░  (50μs)
├─ 波形2: ████████░░░░░░░░░░░░  (50μs)
├─ 波形3: ████████░░░░░░░░░░░░  (50μs)
├─ ...（中间省略18个波形）
├─ 波形20: ████████░░░░░░░░░░░░  (50μs)
│
│  ↑ 20个PWM波形 = 20 × 50μs = 1ms
│
控制周期 1ms —— 软件层面，PID计算一次
│
├─ 第1次PID：计算 duty = 500 → 更新CCR寄存器
│   └─ 接下来20个PWM波形都用这个duty
│
├─ 第2次PID：计算 duty = 520 → 更新CCR寄存器
│   └─ 接下来20个PWM波形都用新的duty
│
└─ ...
```

**关键理解**：

```
PWM频率 20kHz → 每秒产生20000个PWM波形
控制频率 1kHz → 每秒调整20000次占空比？错！每秒只调整1000次！

每调整一次占空比，PWM硬件自动跑20个周期（1ms / 50μs = 20）
```

### 二、硬件配置对比

```c
// ========== PWM配置（TIM3，输出PWM波形）==========
// 目标：20kHz PWM频率
void MX_TIM3_PWM_Init(void) {
    htim3.Instance = TIM3;
    htim3.Init.Prescaler = 0;        // 不分频，84MHz
    htim3.Init.CounterMode = TIM_COUNTERMODE_UP;
    htim3.Init.Period = 4199;        // 84MHz / 4200 = 20kHz
    // PWM周期 = 1/20kHz = 50μs
    HAL_TIM_PWM_Init(&htim3);
    
    // 启动PWM输出（不需要中断，硬件自动跑）
    HAL_TIM_PWM_Start(&htim3, TIM_CHANNEL_1);
}

// ========== 控制周期配置（TIM6，触发PID计算）==========
// 目标：1kHz控制频率
void MX_TIM6_Control_Init(void) {
    htim6.Instance = TIM6;
    htim6.Init.Prescaler = 83;       // 84MHz / 84 = 1MHz
    htim6.Init.CounterMode = TIM_COUNTERMODE_UP;
    htim6.Init.Period = 999;         // 1MHz / 1000 = 1kHz
    // 控制周期 = 1/1kHz = 1ms
    HAL_TIM_Base_Init(&htim6);
    
    // 启动控制中断
    __HAL_TIM_CLEAR_FLAG(&htim6, TIM_FLAG_UPDATE);
    HAL_TIM_Base_Start_IT(&htim6);
}

// ========== 控制中断（每1ms执行一次）==========
void TIM6_DAC_IRQHandler(void) {
    if (__HAL_TIM_GET_FLAG(&htim6, TIM_FLAG_UPDATE)) {
        __HAL_TIM_CLEAR_FLAG(&htim6, TIM_FLAG_UPDATE);
        
        // 每1ms执行一次PID计算
        int16_t target_speed = 500;
        int16_t current_speed = Motor_GetSpeed(0);
        int16_t duty = PID_Compute(target_speed, current_speed);
        
        // 更新占空比 → 接下来20个PWM周期都用这个值
        Motor_SetDuty(0, duty);
    }
}
```
