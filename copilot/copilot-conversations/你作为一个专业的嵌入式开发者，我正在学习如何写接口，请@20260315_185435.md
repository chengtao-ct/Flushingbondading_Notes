## 一、接口文档的核心要素

参考 FreeRTOS 和 Zephyr 的 API 文档风格，一个合格的接口文档必须包含：

### 1. 函数头注释模板

```c
/**
 * @brief  设置电机目标速度
 * @note   此函数非阻塞，仅更新目标值，实际控制由Motion_Update()在主循环执行
 * @param  speed_rpm: 目标转速，单位RPM，范围 [-3000, 3000]，正转/负反转
 * @param  accel: 加速度限制，单位 RPM/s，范围 [100, 10000]，0表示使用默认值
 * @retval 0: 成功入队
 * @retval -EINVAL: 参数越界
 * @retval -EBUSY: 上次指令尚未处理完毕
 * @warning不可在ISR中调用（内部使用互斥锁）
 * @see Motion_Update(), Motor_GetSpeed()
 */
int32_t Motion_SetSpeed(int32_t speed_rpm, uint16_t accel);
```

## 二、接口契约表格式

参考 Zephyr 的 syscall 文档风格，用表格形式明确契约：

| 字段 | 说明 |
| --- | --- |
| 函数名 | Motion_SetSpeed |
| 职责 | 设置运动层目标速度，不直接操作硬件 |
| 调用者 | 应用层 / 上位机指令解析模块 |
| 调用时机 | 收到速度指令后立即调用 |
| 运行上下文 | 主循环（Thread），禁止ISR |
| 参数约束 | speed_rpm ∈ [-3000,3000]，accel ∈ [100,10000] |
| 返回语义 | 0=成功，负值=错误码（见errno.h） |
| 失败处理 | 返回错误码，不改变当前运动状态 |
| 副作用 | 更新 MotionCtx.target_speed |
| 线程安全 | 是（内部有互斥保护） |

## 三、分层接口设计原则

参考 Linux 驱动分层模型：

```
┌─────────────────────────────────────┐
│  Application Layer│  ← 业务逻辑，不碰硬件
├─────────────────────────────────────┤
│  Middleware Layer   (Motion/PID)   │  ← 算法封装，调Driver API
├─────────────────────────────────────┤
│  Driver Layer       (Motor)        │  ← 硬件抽象，封装TIM/GPIO
├─────────────────────────────────────┤
│  HAL Layer          (STM32 HAL)    │  ← 厂商提供，不动
└─────────────────────────────────────┘
```

**核心规则**：
- 上层只依赖下层接口，不跨层调用
- Driver 层接口参数用**物理单位**（RPM、mA），不用寄存器值
- Middleware 层接口参数用**业务语义**（速度档位、位置坐标）

## 四、错误码设计规范

参考 Linux errno 风格，定义项目专用错误码：

```c
/* project_errno.h */
#define ERR_MOTOR_PARAM    (-1)   /* 参数越界 */
#define ERR_MOTOR_TIMEOUT  (-2)   /* 通信超时 */
#define ERR_MOTOR_FAULT    (-3)   /* 硬件故障 */
#define ERR_MOTION_BUSY    (-4)   /* 状态机忙 */
#define ERR_MOTION_STATE   (-5)   /* 当前状态不允许此操作 */
```

**规则**：
- 成功用 `0` 或正值（如实际读取的字节数）
- 错误用负值，便于 `if (ret < 0)` 统一判断
- 每个模块有独立错误码区间，便于定位

## 五、ISR 接口特殊规范

参考 FreeRTOS 的 `FromISR` 后缀约定：

```c
/**
 * @brief  编码器采样（ISR专用）
 * @note   必须在TIM更新中断中调用，仅做数据采集
 * @param  ctx: 运动上下文指针
 * @retval 无
 * @warning 此函数为ISR版本，不可在主循环调用
 */
void Motion_EncoderSample_ISR(MotionCtx_t *ctx);
```

**ISR 接口规则**：
- 函数名加 `_ISR` 或 `FromISR` 后缀
- 禁止使用阻塞操作（延时、互斥锁、动态内存）
- 只做采样/置标志，不做控制计算
- 如需通知主循环，使用无锁队列或原子变量

## 六、安全接口设计

参考 IEC 61508 功能安全标准：

```c
/**
 * @brief  紧急停止（最高优先级）
 * @note   可在任意上下文调用（ISR/主循环/错误处理）
 * @param  none
 * @retval 无
 * @warning 此函数会关闭所有PWM输出，不可恢复
 * @post系统进入SAFE_STOP状态，需Reset才能重新启动
 */
void Motor_EStop(void);
```

**安全接口规则**：
- 独立于正常控制流程
- 可在任意上下文安全调用
- 不依赖任何动态资源（内存、队列）
- 执行后系统进入确定的安全状态

## 七、接口评审检查清单

在 Code Review 时，逐项检查：

- [ ] 函数名是否表达"做什么"而非"怎么做"
- [ ] 参数是否有明确的单位和范围约束
- [ ] 返回值是否覆盖所有可能的执行结果
- [ ] 是否标注了运行上下文要求（ISR/Thread）
- [ ] 是否声明了线程安全性
- [ ] 失败时是否有确定的行为（不改变状态/回退安全值）
- [ ] 是否有对应的测试用例（输入-输出-副作用）

## 一、接口文档的核心结构

一份合格的接口文档，必须包含以下模块：

```
1. 模块概述
2. 数据结构定义
3. 接口契约表
4. 错误码定义
5. 使用示例
6. 注意事项与限制
```

---

## 二、每个接口必须写清楚的要素

| 要素 | 说明 | 示例 |
| --- | --- | --- |
| 函数名 | 做什么，不是怎么做 | `UART_Send` 而非 `UART_WriteRegister` |
| 职责 | 一句话说明功能 | 发送数据到UART总线 |
| 参数 | 名称 + 类型 + 单位 + 范围 | `timeout_ms: 超时时间，单位ms，范围[1, 10000]` |
| 返回值 | 成功/失败的语义 | `>=0: 发送字节数，<0: 错误码` |
| 运行上下文 | ISR / Thread / 两者皆可 | 仅限主循环，禁止ISR |
| 线程安全 | 是否可重入 | 是（内部有锁）/ 否 |
| 副作用 | 改变了什么全局状态 | 修改 `ctx->state` |
| 失败行为 | 失败后系统状态 | 不改变当前状态，返回错误码 |

---

## 三、参数描述三要素

每个参数必须写清楚：

```c
/**
 * @param data: 发送数据缓冲区指针，不可为NULL
 * @param len:  数据长度，单位字节，范围 [1, 256]
 * @param timeout_ms: 超时时间，单位ms，范围 [1, 10000]，0=无限等待
 */
```

**三要素**：
1. **单位**：ms、字节、RPM、百分比
2. **范围**：明确上下限
3. **边界**：NULL是否允许、0是否有特殊含义

---

## 四、返回值设计规范

### 推荐模式

```c
/* 成功返回0或正值，失败返回负值 */
int32_t UART_Send(const uint8_t *data, uint16_t len, uint32_t timeout_ms);

/* 返回值语义 */
#define SUCCESS         (0)     /* 成功 */
#define ERR_INVALID_PARAM (-1)  /* 参数无效 */
#define ERR_TIMEOUT     (-2)    /* 超时 */
#define ERR_BUSY        (-3)    /* 资源忙 */
```

### 规则

- 成功用 `0` 或正值（如实际操作的字节数）
- 错误用负值，便于 `if (ret < 0)` 统一判断
- 每个模块错误码独立，便于定位

---

## 五、运行上下文声明

必须在文档中明确：

| 上下文 | 说明 | 标注方式 |
| --- | --- | --- |
| Thread Only | 仅主循环/任务中调用 | `@warning 禁止在ISR中调用` |
| ISR Only | 仅中断中调用 | 函数名加 `_ISR` 后缀 |
| Both | 两者皆可 | `@note 线程安全，可在任意上下文调用` |

---

## 六、线程安全声明

必须明确回答：

- 多任务同时调用是否安全？
- 内部是否有互斥保护？
- 是否操作全局/静态变量？

```c
/**
 * @note 线程安全：内部使用互斥锁，可多任务调用
 * @note 非线程安全：调用者需自行加锁
 */
```

---

## 七、常见陷阱

| 陷阱 | 问题 | 正确做法 |
| --- | --- | --- |
| 返回值语义混乱 | 有时返回布尔，有时返回错误码 | 统一用 `int32_t`，负值为错误 |
| 参数单位不明 | `duty` 是百分比还是千分比？ | 必须写明单位 |
| 上下文未声明 | ISR中调用了阻塞函数 | 明确标注运行上下文 |
| 副作用未说明 | 调用后改变了全局状态 | 在文档中声明副作用 |
| 错误处理未定义 | 失败后系统状态不确定 | 明确失败后的行为 |
