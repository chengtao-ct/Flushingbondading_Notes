## 【问题诊断】

你说的"帮CPU计算时间"这个表述，**不够狠**。

TIM 的本质不是"帮 CPU 计算"，而是**彻底剥夺 CPU 参与时序维护的权利**——让 CPU 完全从"数数"这件事中解放出来。

---

## 【根本原因分析】

### 为什么这么说？

如果让 CPU 自己"计算时间"，代码是这样的：

```c
// CPU 被迫干苦力：死循环延时
void delay_ms(uint32_t ms) {
    for (uint32_t i = 0; i < ms * 4000; i++) {
        __NOP();  // CPU 在空转，什么都干不了
    }
}
```

**问题**：
- CPU 被完全占用，无法响应其他任务
- 延时精度受中断影响，根本不准
- 多任务环境下，这就是灾难

而 TIM 的存在，让这个模型变成了：

```c
// CPU：我只管干活，时间你帮我盯着
HAL_TIM_Base_Start_IT(&htim3);  // 启动后，CPU 就不管了

// TIM 硬件：默默计数，到点通知 CPU
void HAL_TIM_PeriodElapsedCallback(TIM_HandleTypeDef *htim) {
    // CPU 只在"关键时刻"介入
}
```

**本质区别**：
- TIM 是**独立硬件模块**，有自己的计数器电路
- 时钟树给它喂脉冲，它自己数数
- CPU 只需要处理"结果"（中断/捕获值/PWM 输出）

### 再深挖一层：时钟树的意义

你说的"基准来源于时钟树"完全正确，但要理解**为什么这么设计**：

```
晶振 (8MHz) 
    ↓ PLL 倍频
系统时钟 (168MHz)
    ↓ AHB 分频
HCLK (168MHz)
    ↓ APB1/APB2 分频 + 倍频
TIM Clock (84MHz / 168MHz)
```

**时钟树的本质**：让同一个晶振源，通过**硬件分频/倍频**，为不同外设提供**适配其工作频率**的时钟信号。

TIM 只是时钟树的"消费者"之一，其他外设（ADC、SPI、UART...）同样依赖时钟树。

它不是"帮 CPU 计算"，而是**替 CPU 承担所有时序相关的脏活累活**，让 CPU 专注于业务逻辑。

#### 第 1 行：为什么要清除标志位？

```c
__HAL_TIM_CLEAR_FLAG(&htim3, TIM_FLAG_UPDATE);
```

**这是一个经典的坑**，必须理解：

| 步骤 | 发生了什么 | 结果 |
| :--- | :--- | :--- |
| 1 | CubeMX 生成的 `MX_TIM3_Init()` 初始化定时器 | - |
| 2 | HAL 库内部为了把配置值写入影子寄存器，**置位了 UG（Update Generation）位** | 强制产生一次 Update 事件 |
| 3 | UG 位触发后，**UIF 标志位被硬件置 1** | 表示"溢出事件发生了" |
| 4 | 此时你调用 `HAL_TIM_Base_Start_IT()` 开启中断 | **中断立即触发！** |

**问题**：你还没开始计时，中断就先来了——这是"假中断"。

**解法**：在开启中断前，手动清除 UIF 标志位。

```c
// 这行代码的本质：告诉硬件"之前那个溢出事件我已知悉，别再通知我了"
__HAL_TIM_CLEAR_FLAG(&htim3, TIM_FLAG_UPDATE);
```

#### `static uint8_t cnt = 0` 的作用

这是一个**软件计数器**，用于扩展定时周期。

| 配置 | 周期 |
| :--- | :--- |
| TIM3 硬件周期 | 1ms（ARR=99, PSC=83） |
| 软件计数器 `cnt` | 累加 10 次 |
| **最终效果** | 每 10ms 翻转一次 LED |

- 可以在同一个中断里处理多个不同周期的任务
- 灵活调整，不需要重新配置硬件

void HAL_TIM_PeriodElapsedCallback(TIM_HandleTypeDef *htim) {
    static uint8_t cnt = 0;
    cnt++;
    
    if (cnt % 5 == 0)  { /* 每 5ms 执行 */ }
    if (cnt % 10 == 0) { /* 每 10ms 执行 */ }
    if (cnt % 100 == 0) { /* 每 100ms 执行 */ }
    
    if (cnt >= 100) cnt = 0; // 循环
}
```

## 【CCR 的身份】

**CCR = Capture / Compare Register（捕获/比较寄存器）**

它是 TIM 定时器中与 CNT 配合工作的"参照物"。

---

## 【CCR 的两种身份】

CCR 这个名字暴露了它的双重人格：

| 模式 | 中文名 | CCR 的角色 |
| :--- | :--- | :--- |
| **输出比较 (PWM)** | 比较寄存器 | CCR 是**阈值**，CNT 和它比较，决定输出高低电平 |
| **输入捕获** | 捕获寄存器 | CCR 是**容器**，当外部信号跳变时，CNT 的值被"抓"进 CCR |

---

## 【PWM 模式下 CCR 的作用】

这是最常见的场景，也是你代码里用到的：

```
CNT:  0 → 1 → 2 → ... → 50 → ... → 99 → 0
                      ↑
                    CCR = 50
```

**比较逻辑**（以 PWM 模式 1 为例）：

| CNT 与 CCR 的关系 | 输出电平 |
| :--- | :--- |
| CNT < CCR | **高电平** (有效) |
| CNT ≥ CCR | **低电平** (无效) |

**结果**：

```
输出波形：
        ┌────────────┐            ┌────────────┐
        │            │            │            │
───────┘            └────────────┘            └────
        ←── 50 ──→  ←── 50 ──→
           高          低

占空比 = CCR / (ARR + 1) = 50 / 100 = 50%
```

---

## 【一句话总结三剑客的关系】

```
CNT  ——  跑者（一直在跑，从 0 到 ARR）
ARR  ——  终点线（跑到这里归零，决定周期）
CCR  ——  中途标记（CNT 跑到这里时，输出电平翻转）
```

```

典型应用：测量 PWM 频率、占空比、编码器测速。

## 【CCR 的身份转变】

在输入捕获模式下，CCR 从"裁判"变成了"**快照记录员**"。

| 模式 | CCR 的角色 | 数据流向 |
| :--- | :--- | :--- |
| **输出比较 (PWM)** | 阈值（静态参照物） | CCR → 与 CNT 比较 → 输出电平 |
| **输入捕获** | 容器（动态记录值） | 外部信号触发 → CNT 当前值 → **写入 CCR** |

---

## 【核心机制：硬件级"抓拍"】

### 工作流程

```
外部信号 ──► TIM_CHx 引脚
                │
                ▼
        边沿检测器（上升沿/下降沿）
                │
                ▼
        ┌───────────────────────┐
        │  硬件自动执行（1个时钟周期）│
        │  CCR = CNT            │  ← 把当前计数值"抓"进 CCR
        └───────────────────────┘
                │
                ▼
        CCxIF 标志位置 1（可触发中断/DMA）
```

**关键点**：这个过程完全由硬件完成，**与 CPU 无关**。无论 CPU 在干什么，"抓拍"都在瞬间完成。

---

## 【典型应用：测量信号周期】

假设我们要测量一个 PWM 信号的频率：

```c
// 配置：上升沿触发捕获
void HAL_TIM_IC_CaptureCallback(TIM_HandleTypeDef *htim) {
    static uint32_t last_val = 0;
    
    if (htim->Channel == HAL_TIM_ACTIVE_CHANNEL_1) {
        uint32_t current_val = HAL_TIM_ReadCapturedValue(htim, TIM_CHANNEL_1); // 读取 CCR
        
        uint32_t period = current_val - last_val; // 计算周期
        last_val = current_val;
        
        // 频率 = 定时器时钟频率 / period
    }
}
```

**时序图**：

```
外部信号: ─────┐     ┌─────┐     ┌─────
              │     │     │     │
              └─────┘     └─────┘
              ↑           ↑
           上升沿1     上升沿2
              │           │
CNT:      ...─┼─► 500  ...─┼─► 1500
              │           │
CCR:      500 ◄─┘     1500 ◄─┘

周期 = 1500 - 500 = 1000 个计数周期
```

## 【进阶应用：PWM 输入模式】

这是 STM32 的**杀手锏功能**——同时测量频率和占空比。

**原理**：一个定时器通道捕获上升沿（测周期），另一个通道捕获下降沿（测脉宽）。

| 通道 | 触发边沿 | CCR 记录值 | 用途 |
| :--- | :--- | :--- | :--- |
| CH1 | 上升沿 | 周期起始时刻 | 计算周期 |
| CH2 | 下降沿 | 高电平结束时刻 | 计算占空比 |

**代码示例**：

```c
// CubeMX 配置：Slave Mode = Reset Mode, Trigger = TI1FP1
void HAL_TIM_IC_CaptureCallback(TIM_HandleTypeDef *htim) {
    if (htim->Channel == HAL_TIM_ACTIVE_CHANNEL_1) {
        uint32_t period = HAL_TIM_ReadCapturedValue(htim, TIM_CHANNEL_1); // 周期
        uint32_t pulse  = HAL_TIM_ReadCapturedValue(htim, TIM_CHANNEL_2); // 脉宽
        
        float duty = (float)pulse / (float)period * 100.0f; // 占空比
        float freq = (float)HAL_RCC_GetPCLK1Freq() / (float)period; // 频率
    }
}
```

## 【为什么不用外部中断 EXTI？】

| 方案 | 精度 | CPU 占用 | 适用场景 |
| :--- | :--- | :--- | :--- |
| **EXTI + 软件计时** | 差（中断延迟抖动） | 高（每次信号都中断） | 低频按键、简单检测 |
| **TIM 输入捕获** | **纳秒级**（硬件抓拍） | 低（仅在需要时读取） | 高频 PWM、编码器、通信协议 |

**EXTI 的致命问题**：

```c
// EXTI 中断服务函数
void EXTI0_IRQHandler(void) {
    uint32_t now = TIM2->CNT; // 软件读取 CNT
    // 问题：从中断触发到执行这行代码，已经过了 N 个时钟周期
    // 而且 N 不固定（取决于当前 CPU 在干什么）
}
```

输入捕获的 CCR 是**硬件级原子操作**，不存在这个延迟。

| **CCR** | 捕获/比较寄存器 | PWM 模式：占空比阈值<br>捕获模式：锁存 CNT 瞬时值 | PWM：CNT 与 CCR 比较输出<br>捕获：信号边沿触发 CCR=CNT |

---

## 【大师的工程建议】

1.  **溢出处理**：如果信号周期很长，CNT 可能溢出。需要开启 Update 中断配合处理，或使用 32 位定时器（TIM2/TIM5）。
2.  **滤波器**：输入信号通常有噪声，务必在 CubeMX 中配置 `Input Filter`（如 0x0F），硬件自动消抖。
3.  **DMA 联动**：高频信号测量时，可以让捕获事件直接触发 DMA，把 CCR 值搬运到内存数组，实现完全零 CPU 占用。

## 【QEI 的身份】

**QEI = Quadrature Encoder Interface（正交编码器接口）**

它是**专门用来接增量式编码器**的硬件模块，本质上是定时器的一种**特殊工作模式**。

---

## 【为什么需要 QEI？】

增量式编码器输出两路正交信号（A相、B相）：

```
A相: ──┐  ┌──┐  ┌──┐  ┌──
       └──┘  └──┘  └──┘
B相: ────┐  ┌──┐  ┌──┐  ┌
         └──┘  └──┘  └──┘
         ↑
      相位差 90°
```

**问题**：如果用外部中断 EXTI 来处理这两路信号：
- 高速旋转时，CPU 被中断淹没
- 方向判断逻辑复杂
- 容易丢脉冲

**QEI 的解法**：硬件自动完成所有工作，CNT 就是位置/速度值。

---

## 【QEI 的核心机制】

### 1. 自动辨向

硬件通过检测 A/B 相的相位关系，自动判断方向：

| A相领先 B相 | B相领先 A相 | 结论 |
| :--- | :--- | :--- |
| 90° | - | **正转** → CNT 递增 |
| - | 90° | **反转** → CNT 递减 |

### 2. 4倍频

这是 QEI 的杀手锏——**一个周期内计数 4 次**：

```
        A相上升沿  B相上升沿  A相下降沿  B相下降沿
           ↓         ↓         ↓         ↓
CNT:    +1  →  +1  →  +1  →  +1
```

**效果**：分辨率提升 4 倍！1000 线编码器 → 4000 脉冲/圈。

---

## 【STM32 中的实现】

STM32 没有独立的 QEI 外设，而是**复用通用定时器**：

```c
// CubeMX 配置
// TIM2 -> Slave Mode -> Encoder Mode

// 读取位置
int32_t position = __HAL_TIM_GET_COUNTER(&htim2);

// 读取方向
if (__HAL_TIM_IS_TIM_COUNTING_DOWN(&htim2)) {
    // 反转
} else {
    // 正转
}
```

---

## 【工程建议】

| 要点 | 说明 |
| :--- | :--- |
| **选 32 位定时器** | TIM2/TIM5，避免高速运转时 16 位溢出 |
| **开启输入滤波** | 编码器信号有抖动，配置 `IC Filter = 6~15` |
| **索引信号 (Z相)** | 用于绝对位置校准，接 EXTI 中断 |
| **速度计算** | 用另一个定时器周期性读取 CNT 差值 |
