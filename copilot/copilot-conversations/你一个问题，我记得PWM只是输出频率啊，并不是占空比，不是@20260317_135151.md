## 【根本原因分析】

### PWM 的两个核心参数

| 参数 | 决定因素 | 物理意义 |
|:---|:---|:---|
| **频率** | ARR + PSC | 载波周期，决定电流纹波、噪音 |
| **占空比** | CCR | 高电平时间占比，决定**平均电压** |

### 它们的关系

```
PWM频率 = f_CLK / ((PSC+1) × (ARR+1))    ← 固定不变
占空比 = CCR / (ARR+1)                   ← 动态调节
```

**CCR 确实就是占空比的控制者！**

---

## 【核心澄清】

### 你说的"PWM只是输出频率"——只对了一半

- **ARR** 决定了 PWM 的周期（频率）
- **CCR** 决定了在这个周期内，高电平持续多久

```

CCR=200: ████░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░  占空比 20%
```

## 【大师的工程建议】

### 频率 vs 占空比的作用

| 你调什么 | 影响什么 |
|:---|:---|
| **调 ARR/PSC（频率）** | 电机噪音、电流纹波、驱动器发热 |
| **调 CCR（占空比）** | 电机转速、扭矩 |

### 为什么频率要固定？

- 电机控制常用 **10kHz~20kHz**
- 太低 → 电机有啸叫声
- 太高 → 开关损耗大，驱动器发热

**频率是"背景设定"，占空比才是"实时控制量"。**

> **PWM 频率（ARR）是舞台，占空比（CCR）是演员。**

---

## 【问题诊断】

你在纠结：**PID 的 $T_s$（采样周期）从哪来？**

---

## 【根本原因分析】

### 两种方案的本质区别

| 方案 | 时间来源 | $T_s$ 特性 |
|:---|:---|:---|
| **A. 固定时间** | 硬件定时器中断 | 恒定不变 |
| **B. 实际时间** | `HAL_GetTick()` 或计时器 | 每次都变 |

### 为什么这很重要？

看 PID 的核心公式：

$$Output = K_p \cdot e + K_i \cdot \sum(e \cdot T_s) + K_d \cdot \frac{\Delta e}{T_s}$$

**问题来了：**

- **积分项** $\sum(e \cdot T_s)$：如果 $T_s$ 变化，积分就乱了
- **微分项** $\frac{\Delta e}{T_s}$：如果 $T_s$ 变化，微分就抖了

---

## 【解决方案】

### 工业标准答案：**固定时间基准**

```c
// ✅ 正确做法：硬件定时器中断
void HAL_TIM_PeriodElapsedCallback(TIM_HandleTypeDef *htim)
{
    if (htim == &htim3)  // 10ms 固定周期
    {
        // Ts 永远是 0.01s，不需要测量
        float output = PID_Compute(&motor_pid, target, measure);
        Motor_SetPWM(output);
    }
}
```

### 为什么不用实际时间？

```c
// ❌ 错误示范：测量实际时间
void PID_Task(void)
{
    static uint32_t last_tick = 0;
    uint32_t now = HAL_GetTick();
    float Ts = (now - last_tick) / 1000.0f;  // 每次都不同！
    last_tick = now;
    
    // 问题1：Ts 不稳定 → 积分项漂移
    // 问题2：Ts 不稳定 → 微分项抖动
    // 问题3：如果任务被抢占，Ts 可能突然变大 → 输出异常
}
```

---

## 【深度剖析】

### 如果 $T_s$ 不固定会发生什么？

**场景：积分项失控**

```
理想情况（Ts = 10ms 固定）：
误差 e = 5, 累加 100 次 → 积分 = 5 × 100 × 0.01 = 5.0

实际情况（Ts 随机变化）：
第1次: e=5, Ts=8ms  → 累加 0.04
第2次: e=5, Ts=12ms → 累加 0.06
第3次: e=5, Ts=15ms → 累加 0.075  ← 被高优先级任务抢占了
...
结果：积分项完全不可预测！
```

**场景：微分项爆炸**

```
理想情况（Ts = 10ms 固定）：
Δe = 2 → 微分 = 2 / 0.01 = 200

实际情况（Ts 突然变小）：
Δe = 2, Ts = 1ms → 微分 = 2 / 0.001 = 2000  ← 炸了！
```

---

## 【什么时候用实际时间？】

只有一种情况：**非实时系统**（如 Linux 用户态程序）

## 【大师的工程建议】

### 嵌入式控制的铁律

```
┌─────────────────────────────────────────┐
│  PID 必须运行在硬件定时器中断中          │
│  Ts 必须是编译期确定的常数               │
│  不要在运行时测量 Ts                    │
└─────────────────────────────────────────┘
```

### 代码模板

```c
// pid.h
#define PID_SAMPLE_TIME_MS  10    // 固定！写死！
#define PID_SAMPLE_TIME_S   0.01f

// pid.c - 初始化时把 Ts 融入 Ki 和 Kd
void PID_Init(PID_TypeDef *pid, float Kp, float Ki, float Kd)
{
    pid->Kp = Kp;
    pid->Ki = Ki * PID_SAMPLE_TIME_S;  // 预乘 Ts
    pid->Kd = Kd / PID_SAMPLE_TIME_S;  // 预除 Ts
}

// 这样 PID_Compute 里根本不需要 Ts 变量
float PID_Compute(PID_TypeDef *pid, float error)
{
    // 积分：直接累加，Ts 已经融入 Ki
    pid->integral += error;
    
    // 微分：直接除，Ts 已经融入 Kd
    float derivative = error - pid->last_error;
    
    return pid->Kp * error + 
           pid->Ki * pid->integral + 
           pid->Kd * derivative;
}
```

> 
> 就像你的心脏不需要你告诉它"该跳了"——它自己按固定节奏跳。PID 也是一样。
