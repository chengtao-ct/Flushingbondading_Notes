## 【根本原因分析】

### 1. 时间戳 `state_enter_time` 的核心用途

这个时间戳是**非阻塞超时机制的基石**。看你的笔记里已经写了超时检测代码，但可能没串起来：

```c
void check_timeout(void) {
    uint32_t elapsed = get_system_tick() - fsm.state_enter_time;  // 关键！
    // ...
}
```

**设计意图**：
- 记录"进入当前状态的那一刻"的系统tick
- 每次主循环轮询时，用 `当前tick - 进入tick` 计算已停留时间
- 超过配置阈值 → 主动触发 `EV_TIMEOUT` 事件

**为什么不用 `delay()`？**
```c
// ❌ 阻塞式：CPU 在这里死等，其他任务全停
void state_running(void) {
    motor_start();
    delay_ms(5000);  // 5秒内无法响应任何事件！
    motor_stop();
}

// ✅ 非阻塞式：时间戳 + 轮询
void state_running(void) {
    motor_start();
    // 立即返回，让主循环继续跑
    // 超时检测在 check_timeout() 中异步完成
}
```

### 2. 遍历效率问题：从 O(n) 到 O(1)

你的直觉是对的——线性遍历在状态/事件数量爆炸时会成为瓶颈。但嵌入式场景下，优化方案有讲究：

#### 方案A：二维数组直接索引（推荐）

```c
// 状态转移表：[状态][事件] → 转移项索引
// 用 -1 表示"无效转移"
static const int8_t transition_index[STATE_MAX][EV_MAX] = {
    //           EV_NONE  EV_START  EV_STOP  EV_TIMEOUT  EV_ERROR
    /* IDLE     */ { -1,     0,       -1,      -1,        1      },
    /* RUNNING  */ { -1,    -1,        2,       3,        4      },
    /* STOPPING */ { -1,    -1,       -1,       5,       -1      },
    /* ERROR    */ { -1,     6,       -1,      -1,       -1      },
};

void fsm_engine_fast(FSM_Event event) {
    int8_t idx = transition_index[fsm.current_state][event];
    
    if (idx < 0) return;  // 无效转移，直接忽略
    
    const FSM_Transition *t = &transition_table[idx];
    if (t->action) t->action();
    
    fsm.current_state = t->next_state;
    fsm.state_enter_time = get_system_tick();
}
```

**代价**：空间换时间。假设 20 个状态 × 15 个事件 = 300 字节的索引表（int8_t）。对于 STM32F407 的 192KB RAM，这点开销完全可接受。

#### 方案B：分层状态机（HSM）

当状态超过 30 个时，考虑拆分为**父子状态机**：
```
顶层状态机：IDLE / RUNNING / ERROR
    └── RUNNING 子状态机：FORWARD / BACKWARD / TURN_LEFT / TURN_RIGHT
```

每次只需在当前层级的小表中遍历，复杂度大幅降低。

### 3. 指针 `*` 的"身份识别"

你的困惑太真实了——C语言的 `*` 确实是"符号重载"的典型：

| 上下文 | 含义 | 示例 |
|--------|------|------|
| **声明时** | 指针类型声明符 | `int *p;` → p 是指向 int 的指针 |
| **使用时** | 解引用运算符 | `*p = 10;` → 往 p 指向的地址写 10 |
| **表达式中** | 乘法运算符 | `a * b` → a 乘以 b |

**快速判断口诀**：
- 看左边是不是类型名（`int`、`char`、`void`）→ 声明指针
- 看左边是不是变量名或 `*` → 解引用或乘法
- 看右边是不是变量名 → 乘法

```c
int a = 5, b = 3;
int *p = &a;    // 声明：p 是 int*
int c = *p * b; // 解引用 *p (得5) 乘以 b (得3) = 15
```

这段代码展示了**软硬分离架构**的核心思想：通过条件编译，让同一套业务逻辑（FSM）在 PC 和硬件板上跑两套完全不同的底层实现。

## 【根本原因分析】

### 为什么需要桩？

嵌入式开发的痛点：

| 场景 | 问题 |
|------|------|
| **硬件未就绪** | PCB 还在打样，但软件必须先写 |
| **调试困难** | 电机一转，万用表、示波器全得上，排查一个问题半小时起步 |
| **风险高** | 代码有 Bug，电机可能直接飞车、烧驱动 |
| **迭代慢** | 改一行代码 → 编译 → 烧录 → 复位 → 观察，循环 5 分钟 |

桩的本质：**用假的底层实现，骗过上层业务逻辑，让它在 PC 上跑起来**。

---

## 【桩的具体作用】

### 1. 逻辑验证，脱离硬件

```c
// PC 上跑这个
void act_motor_start(void) {
    printf("[MOCK] Motor START\n");  // 假装启动了电机
}
```

你在键盘敲 `s`，FSM 收到 `EV_START_BTN`，状态从 `IDLE` 跳到 `RUNNING`，打印 `[MOCK] Motor START`。

**验证了什么？**
- 状态转移表是否正确
- 事件分发逻辑是否通顺
- 边界条件是否处理到位

**代价**：0 硬件，0 烧录，秒级反馈。

### 2. 安全沙盒

```c
// 真实硬件上跑这个
void act_motor_start(void) {
    HAL_TIM_PWM_Start(&htim3, TIM_CHANNEL_1);  // 真的启动电机
}
```

如果 FSM 逻辑有 Bug（比如状态卡死、事件丢失），在 PC 上只是打印乱一点，在硬件上可能就是电机狂转停不下来。

**桩测试 = 上真机前的安全网**。

### 3. 问题定位的二分法

当硬件上出现诡异 Bug 时：

```
PC 桩测试通过 → 业务逻辑没问题 → 查硬件/驱动
PC 桩测试失败 → 业务逻辑有问题 → 在 PC 上调，不用管硬件
```

## 【工程级理解】

把这段代码看作一个**契约**：

```c
// 契约：无论 PC 还是硬件，act_motor_start() 这个函数签名必须一致
void act_motor_start(void);  // 无参数，无返回值

// PC 履约方式：打印日志
// 硬件履约方式：操作寄存器
```

这就是**硬件抽象层（HAL）**的雏形。上层只认契约，不关心底层怎么实现。
