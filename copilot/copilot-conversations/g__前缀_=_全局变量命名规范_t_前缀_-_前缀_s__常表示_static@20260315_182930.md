## 一、作用域前缀（你问的核心）

| 前缀 | 含义 | 示例 | 说明 |
|------|------|------|------|
| `g_` | Global 全局变量 | `g_systemTick` | 跨文件可见，慎用 |

## 【问题诊断】

**幂等性**：同一个操作执行一次和执行多次，对系统状态产生的影响是相同的。

在嵌入式系统中，幂等保护的核心目的是：**防止重复操作带来的副作用**——比如重复初始化、重复申请内存、重复发送命令、重复进入临界区。

---

## 【根本原因分析】

为什么嵌入式系统特别需要幂等保护？

| 场景 | 非幂等的风险 |
|------|-------------|
| 外设初始化 | 重复配置寄存器导致时序错乱、时钟树异常 |
| 内存申请 | 重复 `malloc` 导致内存泄漏 |
| 中断使能/禁用 | 嵌套调用导致中断状态不一致 |
| 状态机切换 | 重复触发进入/退出动作 |
| 通信协议 | 重复发送命令导致对端状态异常 |
| RTOS资源 | 重复获取信号量导致死锁 |

### 场景一：外设初始化（你笔记中的场景）

```c
/* 模式切换请求 - 带幂等保护 */
typedef struct {
    e_ModeType pendingMode;     /* 待执行模式 */
    e_ModeType activeMode;      /* 当前生效模式 */
    bool       b_requestPending; /* 是否有待处理请求 */
} ModeManager_t;

static ModeManager_t s_modeMgr = {
    .pendingMode     = MODE_IDLE,
    .activeMode      = MODE_IDLE,
    .b_requestPending = false
};

/**
 * @brief  请求切换模式（幂等设计）
 * @return true: 请求被接受; false: 无效请求
 */
bool Mode_Request(e_ModeType newMode)
{
    /* 幂等保护 Case 1: 已有相同请求在等待 */
    if (s_modeMgr.b_requestPending && 
        s_modeMgr.pendingMode == newMode)
    {
        return true;  /* 重复请求，直接OK，不重复写 */
    }
    
    /* 幂等保护 Case 2: 无pending，且已在目标模式 */
    if (!s_modeMgr.b_requestPending && 
        s_modeMgr.activeMode == newMode)
    {
        return true;  /* 已在此模式，OK no-op */
    }
    
    /* 真正的模式切换请求 */
    s_modeMgr.pendingMode      = newMode;
    s_modeMgr.b_requestPending = true;
    
    return true;
}
```

{

        __disable_irq();  /* 只在第一次真正关中断 */

    s_criticalNesting++;

{
    if (s_criticalNesting > 0)

        s_criticalNesting--;
        if (s_criticalNesting == 0)
        {
            __enable_irq();  /* 只在最后一次真正开中断 */
        }
    }
    /* else: 幂等保护，多余的Exit被忽略 */
}

## 【大师的工程建议】

### 1. 幂等 vs 非幂等，如何选择？

| 设计策略 | 适用场景 | 风险 |
|----------|----------|------|
| **幂等保护** | 初始化、状态切换、资源申请 | 可能隐藏逻辑错误 |
| **断言报错** | 调试阶段、关键路径 | 发布版本需移除 |
| **返回错误码** | 需要调用者感知的场景 | 增加调用复杂度 |

**我的建议**：调试阶段用断言，发布版本用幂等保护 + 日志记录。

```c
void Peripheral_Init(void)
{
    if (s_isInitialized)
    {
        DEBUG_LOG("WARN: Duplicate init detected");  /* 调试时可见 */
        return;  /* 发布时安全 */
    }
    /* ... */
}
```

### 2. 幂等保护的代价

- **状态变量开销**：每个需要保护的模块需要一个 `bool` 或计数器
- **逻辑复杂度**：增加代码分支，需要覆盖测试
- **隐藏Bug**：如果重复调用本身就是逻辑错误，幂等保护会掩盖问题

### 3. 什么时候不需要幂等？

```c
/* 不需要幂等：每次调用都有意义 */
void LED_Toggle(void)
{
    HAL_GPIO_TogglePin(LED_GPIO_Port, LED_Pin);
}

/* 不需要幂等：数据操作 */
void UART_SendByte(uint8_t data)
{
    /* 每次发送都是有效操作 */
}
```

## 优化版本

### 循环控制关键字对比

| 关键字 | 作用范围 | 执行效果 | 典型场景 |
|--------|----------|----------|----------|
| `continue` | 本次迭代 | 跳过剩余代码，进入下一次循环判断 | 过滤无效数据 |
| `break` | 整个循环 | 立即跳出循环，执行循环后代码 | 找到目标后提前退出 |
| `return` | 整个函数 | 立即退出函数，返回调用者 | 错误处理、提前返回结果 |

---

### 代码示例对比

```c
void Process_SensorData(uint8_t *pData, uint8_t len)
{
    for (uint8_t i = 0; i < len; i++)
    {
        /* continue: 跳过无效数据，处理下一个 */
        if (pData[i] == 0xFF)  /* 无效标记 */
        {
            continue;  /* 跳过本次，i++ 继续循环 */
        }
        
        /* break: 发现错误，停止处理 */
        if (pData[i] == 0x00)  /* 错误标记 */
        {
            break;  /* 跳出整个for循环 */
        }
        
        /* return: 致命错误，直接退出函数 */
        if (pData[i] == 0xFE)  /* 致命错误 */
        {
            return;  /* 退出整个函数 */
        }
        
        /* 正常处理 */
        UART_Send(pData[i]);
    }
    
    /* break会跳到这里，return不会 */
    LED_Toggle();  /* 处理完成指示 */
}
```

### 核心原理：Name Mangling（名字修饰）

| 编译器 | 函数 `void print(int a)` 的符号名 | 机制 |
|--------|-----------------------------------|------|
| C (gcc) | `print` | 无修饰，符号即函数名 |
| C++ (g++) | `_Z5printi` | 编码函数名+参数类型 |

**为什么 C++ 要修饰？**
- 支持函数重载：`print(int)` 和 `print(float)` 可共存
- 支持命名空间：`NS::func()` 需要编码命名空间信息
- 支持类成员：`Class::method()` 需要编码类名

### 解决方案：`extern "C"` 禁用修饰

```c
/* uart.h - 通用头文件 */
#ifdef __cplusplus
extern "C" {
#endif

void uart_init(void);
void uart_send(uint8_t data);
uint8_t uart_receive(void);

#ifdef __cplusplus
}
#endif
```

```
C++ 编译器看到 extern "C" 后:
  uart_init → uart_init (禁用修饰)

链接器查找: uart_init ✓ 匹配成功
```

### 大师的工程建议

**1. 头文件模板（推荐写法）**

```c
/* my_driver.h */
#ifndef MY_DRIVER_H
#define MY_DRIVER_H

#ifdef __cplusplus
extern "C" {
#endif

/* ========== 公开接口 ========== */
void Driver_Init(void);
int  Driver_Read(uint8_t *buf, uint32_t len);
void Driver_DeInit(void);

#ifdef __cplusplus
}
#endif

#endif /* MY_DRIVER_H */
```

**2. 常见错误**

```cpp
/* 错误: C++ 类成员函数不能用 extern "C" */
extern "C" void MyClass::method(void);  // 编译错误！

/* 正确: 静态成员或全局函数可以 */
extern "C" void callback_wrapper(void);

class MyClass {
    static void callback_wrapper(void);  // OK
};
```

### 笔记精简版

```
extern "C" 的作用：禁用 C++ 的 Name Mangling

原理:
- C 编译器: 函数符号 = 函数名（无修饰）
- C++ 编译器: 函数符号 = 函数名 + 参数类型编码（支持重载）

问题: C++ 调用 C 库时，符号名不匹配导致链接失败
解决: 用 extern "C" 告诉 C++ 编译器按 C 规则生成符号

头文件模板:
#ifdef __cplusplus
extern "C" {
#endif
/* 函数声明 */
#ifdef __cplusplus
}
#endif
```
