# 内存池

> [!info] 核心概念
> **内存池** 是一种预先分配一大块连续内存，然后按固定大小切分管理的机制。它的核心目的是**避免内存碎片**，提高分配效率。

## 🎯 通俗理解：停车场的智慧

想象你经营一个停车场：

| 方式 | 类比 | 特点 |
| ---- | ---- | ---- |
| 动态分配 | 随意停车 | 车辆大小不一，停得乱七八糟，空间浪费严重 |
| 内存池 | 划定车位 | 每个车位固定大小，整齐划一，空间利用率高 |

**内存池就像划定好固定车位的停车场**：
- 车位大小固定 → 内存块大小固定
- 车位数量固定 → 内存块数量固定
- 车来了直接停 → 分配速度快（O(1)）
- 车走了车位还在 → 不会产生碎片

## 🔧 FreeRTOS 中的内存管理

FreeRTOS 本身**没有直接提供**名为"内存池"的 API，但它提供了多种堆管理方案，其中 `heap_4.c` 最常用：

### FreeRTOS 堆管理方案对比

| 方案 | 特点 | 适用场景 |
| ---- | ---- | ---- |
| [[heap_1]] | 只分配不释放 | 极简单系统 |
| [[heap_2]] | 可释放但不合并碎片 | 已废弃 |
| [[heap_3]] | 封装标准库 malloc | 不推荐 |
| [[heap_4]] | 可释放且合并碎片 | **最常用** |
| [[heap_5]] | 多段不连续内存 | 外部 SRAM 场景 |

### 如何在 FreeRTOS 中实现内存池效果？

```c
/* 方法一：静态分配（真正的内存池思想） */
#define POOL_BLOCK_SIZE  64    // 每个块 64 字节
#define POOL_BLOCK_COUNT 10    // 共 10 个块

static uint8_t mem_pool[POOL_BLOCK_SIZE * POOL_BLOCK_COUNT];  // 预分配内存池
static uint8_t block_used[POOL_BLOCK_COUNT];  // 使用标记

/* 从内存池申请一块 */
void* pool_alloc(void)
{
    for (int i = 0; i < POOL_BLOCK_COUNT; i++)
    {
        if (block_used[i] == 0)  // 找到空闲块
        {
            block_used[i] = 1;
            return &mem_pool[i * POOL_BLOCK_SIZE];  // 返回块地址
        }
    }
    return NULL;  // 池已满
}

/* 释放回内存池 */
void pool_free(void *ptr)
{
    int index = ((uint8_t*)ptr - mem_pool) / POOL_BLOCK_SIZE;
    if (index >= 0 && index < POOL_BLOCK_COUNT)
    {
        block_used[index] = 0;  // 标记为空闲
    }
}
```

```c
/* 方法二：使用 FreeRTOS 的 pvPortMalloc */
void task_example(void *pvParameters)
{
    /* 注意：heap_4 方案下，这会从 FreeRTOS 管理的堆中分配 */
    uint8_t *buffer = (uint8_t*)pvPortMalloc(128);
    
    if (buffer != NULL)
    {
        /* 使用内存 */
        memset(buffer, 0, 128);
        
        /* 用完必须释放 */
        vPortFree(buffer);
    }
}
```

## ⚡ 软硬结合：STM32F407 上的实际表现

### 内存布局（以 STM32F407 为例）

```
┌─────────────────────────────────────┐ 0x20000000
│         SRAM1 (112 KB)              │ ← 主内存，存放堆、栈、全局变量
├─────────────────────────────────────┤
│         SRAM2 (16 KB)               │ ← 可用于特定外设
├─────────────────────────────────────┤
│         SRAM3 (64 KB)               │ ← CCM 内存，DMA 无法访问！
└─────────────────────────────────────┘ 0x20030000
```

### 如何验证内存池在工作？

```c
/* 在 Keil/IAR 中查看内存使用情况 */
extern uint32_t _estack;      // 栈顶
extern uint32_t _Min_Heap_Size;  // 堆大小

/* FreeRTOS 任务栈溢出检测 */
void vApplicationStackOverflowHook(TaskHandle_t xTask, char *pcTaskName)
{
    /* 如果跑到这里，说明某个任务栈爆了！ */
    while(1);  // 死循环，方便调试器定位
}
```

**用调试器观察**：
1. 打开 Memory 窗口，输入 `0x20000000`
2. 观察内存池区域的数据变化
3. 申请时看到数据被写入，释放后数据还在（只是标记为可用）

---

> [!warning] 避坑经验
> 1. **CCM 内存陷阱**：STM32F407 的 CCM RAM（64KB）不能被 DMA 访问！如果把内存池放在 CCM，再用 DMA 搬运会直接卡死。
> 2. **忘记释放**：内存池虽然减少了碎片，但忘记释放仍会导致"池满"，后续申请全部失败。
> 3. **线程安全问题**：多任务环境下，内存池的分配/释放操作必须加 [[互斥锁]] 或关中断保护。

# ARM Cortex-M4

> [!abstract] 核心定位
> ARM Cortex-M4 是 ARMv7-M 架构的 32 位 RISC 处理器内核，专为**数字信号控制（DSC）** 场景设计。相比 Cortex-M3，它新增了 **[[FPU]]（浮点运算单元）** 和 **[[DSP指令集]]**，是 STM32F4 系列的核心引擎。

---

## 一、物理架构层：硬件设计哲学

### 1.1 哈佛架构与三总线系统

早期单片机（如 8051）采用冯·诺依曼架构，指令和数据共享一条总线，如同单行道——取指和取数据必须排队。

Cortex-M4 采用**哈佛架构**，物理上分离了三条独立总线：

| 总线 | 连接目标 | 职责 |
| ---- | ---- | ---- |
| I-Code 总线 | [[Flash]] (0x0800 0000) | 专用于取指令 |
| D-Code 总线 | Flash + [[SRAM]] | 读写常量与变量 |
| System 总线 | 外设寄存器 | 访问 GPIO、USART 等 |

**物理效果**：CPU 可在单周期内同时取指和读写数据，这是 168MHz 高频运行的硬件基础。

### 1.2 三级流水线

M4 将指令执行拆分为三个独立阶段：

```
取指 → 译码 → 执行
```

三个阶段由独立硬件电路并行工作：
- 执行第 N 条指令时
- 译码第 N+1 条指令
- 同时取指第 N+2 条指令

> [!tip] 分支预测
> 遇到 `if-else` 分支时，M4 内置分支预测电路，提前加载最可能执行的代码路径。

### 1.3 两大硬件加速器

这是 M4 区别于 M3 的核心差异：

| 加速器 | 功能 | 性能提升 |
| ------ | ---- | -------- |
| [[FPU]] | 单精度浮点运算硬件 | `float` 乘法从 ~200 周期降至 **1 周期** |
| [[MAC单元]] | 乘加运算 | `A += B * C` 在 **1 周期**内完成 |

**典型应用场景**：
- 电机控制（PID 算法）
- 音频处理（FIR/IIR 滤波）
- 传感器融合（卡尔曼滤波）

## 二、神经中枢层：中断与异常系统

### 2.1 NVIC（嵌套向量中断控制器）

[[NVIC]] 是紧贴 CPU 内核的中断管理硬件：

- 支持最多 **256 个中断源**
- 支持 **256 级可编程优先级**
- 硬件自动处理中断嵌套

### 2.2 硬件级现场保护

这是 Cortex-M 系列的革命性设计：

**传统 CPU**：中断发生时，需软件手动 `PUSH` 寄存器保护现场。

**Cortex-M4**：硬件自动将 8 个核心寄存器压栈：

```
R0, R1, R2, R3, R12, LR, PC, xPSR
```

- 耗时仅 **12 个时钟周期**
- C 语言中断服务函数可直接编写，无需汇编包装

### 2.3 咬尾中断

当连续中断发生时：

```
中断A结束 → 传统方式：恢复现场 → 再次保护现场 → 中断B开始
         → 咬尾方式：直接跳转执行中断B（省去出栈/入栈开销）
```

**效果**：连续中断响应延迟减少 **10+ 个时钟周期**。

### 2.4 双栈指针机制

M4 内核拥有两个物理栈指针：

| 栈指针 | 用途 | 使用场景 |
| ------ | ---- | -------- |
| [[MSP]]（主栈指针） | 内核栈 | 操作系统代码、中断服务函数 |
| [[PSP]]（进程栈指针） | 用户栈 | 用户任务 |

**RTOS 意义**：
- 硬件层面隔离内核与用户任务
- 任务栈溢出不会破坏内核栈
- 系统仍能响应中断并报错

## 三、代码逻辑层：寄存器与调用约定

### 3.1 寄存器组

M4 内核包含 16 个 32 位通用寄存器：

| 寄存器 | 别名 | 用途 |
| ------ | ---- | ---- |
| R0-R3 | - | 函数参数传递 / 返回值 |
| R4-R11 | - | 局部变量（需被调用者保护） |
| R12 | IP | 过程调用临时寄存器 |
| R13 | SP | 栈指针（MSP/PSP） |
| R14 | LR | 链接寄存器（返回地址） |
| R15 | PC | 程序计数器 |

### 3.2 AAPCS 调用约定

[[AAPCS]]（ARM 架构过程调用标准）规定：

```c
/* 前 4 个参数通过寄存器传递（极速） */
void func(int a, int b, int c, int d);

/* 第 5 个参数必须通过栈传递（较慢） */
void func(int a, int b, int c, int d, int e);  // e 走 RAM
```

> [!warning] 编码规范
> 高质量嵌入式代码应控制函数参数不超过 4 个，避免栈操作开销。

## 四、深水区：高级调试与系统机制

### 4.1 HardFault 现场勘探

当程序崩溃进入 `HardFault_Handler` 时，硬件已自动将案发时刻的寄存器压栈。

**定位死机代码的方法**：

```c
// 在 HardFault_Handler 中读取压栈的 PC 值
void HardFault_Handler(void)
{
    uint32_t *stack_ptr;
    __asm volatile ("MRS %0, MSP\n" : "=r"(stack_ptr));
    
    uint32_t pc = stack_ptr[6];  // PC 是第 7 个被压入的
    // 用 pc 值在 .map 文件中查找对应的 C 代码行
    while(1);
}
```

### 4.2 排他性访问指令

解决多任务/中断环境下的数据竞争：

| 指令 | 功能 |
| ---- | ---- |
| `LDREX` | 排他性加载，标记地址 |
| `STREX` | 排他性存储，检测标记是否被清除 |

**原理**：配合 [[Exclusive Monitor]] 硬件，实现无锁原子操作。

```c
// FreeRTOS 中互斥锁的底层实现原理
while(1) {
    if (LDREX(&lock) == 0 && STREX(&lock, 1) == 0) {
        break;  // 成功获取锁
    }
}
```

### 4.3 MPU（内存保护单元）

[[MPU]] 提供硬件级内存访问控制：

- 将内存划分为多个区域
- 设置每个区域的访问权限（读/写/执行）
- 越界访问触发 `MemManage_Handler`

**应用场景**：
- 隔离不可信的第三方代码
- 保护关键数据区
- 实现安全关键系统

### 4.4 Bit-Banding（位带操作）

[[Bit-Banding]] 是 Cortex-M4 的"空间换时间"魔法：

**原理**：将 1MB 的 SRAM 区域的每个 bit，映射到 32MB 的"位带别名区"。

```c
// 传统方式：读-改-写（3 步，可能被打断）
GPIOA->ODR |= (1 << 5);

// 位带方式：单周期原子操作
*(volatile uint32_t*)(0x42000000 + (bit_offset * 4)) = 1;
```

**优势**：
- 单周期完成
- 原子操作，不可打断
- 适合高速 I/O 翻转（软件模拟 I2C/SPI）

### 4.5 Boot 流程详解

从上电到 `main()` 执行的完整过程：

```
┌─────────────────────────────────────────────────┐
│ 1. 硬件从 0x0800 0000 读取 MSP 初始值           │
│    → CPU 获得栈指针，可以运行 C 代码            │
├─────────────────────────────────────────────────┤
│ 2. 硬件从 0x0800 0004 读取 Reset_Handler 地址   │
│    → 跳转执行启动代码                           │
├─────────────────────────────────────────────────┤
│ 3. 初始化 .data 段：Flash → RAM                 │
│    → 已初始化全局变量就位                       │
├─────────────────────────────────────────────────┤
│ 4. 清零 .bss 段                                 │
│    → 未初始化全局变量清零                       │
├─────────────────────────────────────────────────┤
│ 5. 调用 SystemInit()（时钟初始化）              │
├─────────────────────────────────────────────────┤
│ 6. 调用 __libc_init_array（C++ 构造函数）       │
├─────────────────────────────────────────────────┤
│ 7. BL main → 进入用户代码                       │
└─────────────────────────────────────────────────┘
```
