## 二、从技术实现维度分类

| 类型 | 技术原理 | 代表器件 | 易失性 | 典型应用 |
|------|----------|----------|--------|----------|
| **SRAM** | 锁存器（6个晶体管存1bit） | CPU Cache | 易失 | L1/L2/L3缓存 |
| **DRAM** | 电容充放电（1晶体管+1电容） | DDR4/DDR5内存条 | 易失 | 主内存 |
| **NOR Flash** | 浮栅晶体管，随机读取快 | 嵌入式Flash | 非易失 | MCU程序存储、BIOS |
| **NAND Flash** | 浮栅晶体管，页读写，密度高 | SSD、eMMC、U盘 | 非易失 | 大容量存储 |
| **MRAM** | 磁阻效应 | 新型非易失RAM | 非易失 | 车载、工业（新兴） |
| **HDD** | 磁盘磁头机械读写 | 机械硬盘 | 非易失 | 冷数据存储 |

## 一、先给结论：寄存器的统一定义

**寄存器 = 能够在一个时钟周期内完成读/写的存储单元**

无论是CPU内部的R0、PC、LR，还是外设的GPIO->ODR，本质上都是**D触发器**的集合。区别只在于：
- **谁在访问它**（CPU核心 vs 外设逻辑）
- **地址是否可见**（有地址映射 vs 无地址映射）
- **功能是什么**（数据暂存 vs 硬件控制）

## 二、硬件层面：寄存器是如何构成的？

### 2.1 基本单元：D触发器

```mermaid
flowchart LR
    subgraph DFF["D触发器 (1-bit寄存器)"]
        D["D输入"] -->|上升沿锁存| Q["Q输出"]
        CLK["时钟CLK"] --> DFF
        RST["复位"] --> DFF
    end
    
    style DFF fill:#e1f5fe
```

**工作原理**：
- 每个时钟上升沿，D端的值被"锁存"到Q端
- 在下一个时钟上升沿到来之前，Q端保持不变
- 这就是"寄存"的含义——**锁存并保持状态**

### 2.2 N位寄存器 = N个D触发器并联

```mermaid
flowchart LR
    D0["D[0]"] --> FF0["DFF"] --> Q0["Q[0]"]
    D1["D[1]"] --> FF1["DFF"] --> Q1["Q[1]"]
    D2["D[2]"] --> FF2["DFF"] --> Q2["Q[2]"]
    D3["D[3]"] --> FF3["DFF"] --> Q3["Q[3]"]
    
    CLK["CLK"] --> FF0
    CLK --> FF1
    CLK --> FF2
    CLK --> FF3
    
    subgraph "4位寄存器"
        FF0
        FF1
        FF2
        FF3
    end
```

**关键点**：一个32位寄存器，就是32个D触发器共享同一个时钟信号。

## 三、寄存器的三大分类（这才是你困惑的根源）

```mermaid

    PERI --> DATA["数据寄存器<br/>DR/TDR/RDR等"]
    
    INTERN --> PIPE["流水线寄存器<br/>IF/ID/EX阶段锁存"]

    style INTERN fill:#9c27b0,color:#fff
```

### 3.1 CPU内部寄存器（程序员可见）

这是你写汇编时直接操作的寄存器：

| 寄存器 | 全称 | 作用 | 硬件位置 |
|--------|------|------|----------|
| R0-R12 | 通用寄存器 | 存放临时数据、函数参数、返回值 | CPU核心内部 |
| SP (R13) | 栈指针 | 指向当前栈顶 | CPU核心内部 |
| LR (R14) | 链接寄存器 | 存放函数返回地址 | CPU核心内部 |
| PC (R15) | 程序计数器 | 存放下一条指令地址 | CPU核心内部 |
| xPSR | 程序状态寄存器 | N/Z/C/V标志 + 异常状态 | CPU核心内部 |

**特点**：
- **无地址**：不占用内存地址空间，通过指令名访问（`MOV R0, #1`）
- **最快**：CPU直接访问，无总线延迟
- **数量有限**：ARM架构只有16个（R0-R15）

### 3.2 外设寄存器（MMIO - Memory Mapped I/O）

这是你写C代码时通过指针访问的寄存器：

```c
#define GPIOA_ODR  (*(volatile uint32_t *)0x40020014)
GPIOA_ODR = 0x01;  // 写寄存器
```

**特点**：
- **有地址**：映射到内存地址空间
- **通过总线访问**：CPU → AHB/APB总线 → 外设
- **有副作用**：读写可能触发硬件动作（如写DR触发UART发送）

### 3.3 内部状态寄存器（程序员不可见）

这是CPU流水线内部的寄存器，你永远看不到：

```mermaid
flowchart LR
    subgraph Pipeline["CPU流水线"]
        IF["取指 IF"] --> IF_ID["IF/ID寄存器"]
        IF_ID --> ID["译码 ID"]
        ID --> ID_EX["ID/EX寄存器"]
        ID_EX --> EX["执行 EX"]
        EX --> EX_MEM["EX/MEM寄存器"]
        EX_MEM --> MEM["访存 MEM"]
        MEM --> MEM_WB["MEM/WB寄存器"]
        MEM_WB --> WB["写回 WB"]
    end
    
    style IF_ID fill:#ff9800
    style ID_EX fill:#ff9800
    style EX_MEM fill:#ff9800
    style MEM_WB fill:#ff9800
```

**作用**：锁存每个流水阶段的中间结果，防止数据冒险。

## 四、为什么需要寄存器？三个核心作用

```mermaid
flowchart TB
    subgraph 作用1["作用1: 速度缓冲"]
        MEM2["内存 (50-100ns)"] -->|"太慢"| CPU2["CPU (1ns)"]
        CPU2 -->|"需要"| REG2["寄存器 (1ns)"]
        REG2 -->|"快速暂存"| CPU2
    end
    
    subgraph 作用2["作用2: 硬件控制接口"]
        SW["软件"] -->|"写控制寄存器"| HW["硬件模块"]
        HW -->|"读状态寄存器"| SW
    end
    
    subgraph 作用3["作用3: 状态保持"]
        CLK3["时钟周期1"] --> STATE["状态寄存器"]
        STATE --> CLK3_2["时钟周期2"]
        CLK3_2 --> STATE
    end
```

### 作用1：速度缓冲（解决CPU与内存的速度差距）

CPU运算速度是纳秒级，内存访问是几十纳秒级。没有寄存器，CPU每执行一条指令都要等内存。

**寄存器就是CPU的"工作台"**：
- 内存是"仓库"（大但远）
- Cache是"货架"（中距离）
- 寄存器是"工作台"（小但就在手边）

### 作用2：硬件控制接口（软件操控硬件的唯一手段）

```mermaid
flowchart LR
    subgraph 软件侧
        CODE["C代码<br/>GPIO->ODR = 0x01"]
    end
    
    subgraph 硬件侧
        REG["ODR寄存器<br/>(D触发器阵列)"]
        LOGIC["输出逻辑电路"]
        PIN["GPIO引脚"]
    end
    
    CODE -->|"STR指令"| REG
    REG --> LOGIC
    LOGIC --> PIN
    
    style REG fill:#4caf50
```

**本质**：写寄存器 = 改变D触发器的值 = 改变硬件电路的状态

### 作用3：状态保持（时序电路的基础）

组合逻辑电路没有记忆，输出只取决于当前输入。**加上寄存器，就变成了时序电路**，可以"记住"之前的状态。

```
组合逻辑：Output = f(Input)
时序逻辑：Output = f(Input, CurrentState)
         CurrentState = g(PreviousState, Input)
```

## 五、深入理解：外设寄存器的硬件实现

以GPIO输出数据寄存器（ODR）为例：

```mermaid
flowchart TB
    subgraph CPU写操作
        ADDR["地址总线 0x40020014"] --> DEC["地址译码器"]
        WDATA["写数据总线 0x00000001"] --> MUX["数据选择器"]
        WR["写使能信号"] --> MUX
        DEC -->|"选中"| MUX
        MUX -->|"更新"| DFF["D触发器阵列<br/>(ODR寄存器)"]
    end
    
    subgraph 硬件动作
        DFF -->|"Q输出"| OUT_LOGIC["输出驱动逻辑"]
        OUT_LOGIC -->|"驱动"| PIN["GPIO引脚 PA0"]
    end
    
    subgraph CPU读操作
        DFF -->|"Q输出"| RD_MUX["读数据选择器"]
        ADDR --> RD_DEC["地址译码器"]
        RD_DEC --> RD_MUX
        RD_MUX -->|"返回"| RDATA["读数据总线"]
    end
    
    style DFF fill:#4caf50
```

**关键理解**：
1. **地址译码**：地址总线上的值决定"选中"哪个寄存器
2. **写使能**：只有WR信号有效时，D触发器才更新
3. **读路径**：D触发器的Q端直接连到读数据总线

## 六、LR寄存器的特殊角色：栈帧管理的核心

你提到了LR，这是CPU寄存器中比较特殊的一个：

```mermaid
flowchart TB
    subgraph 函数调用过程
        CALL["BL func<br/>(Branch with Link)"] -->|"PC+4 → LR"| LR["LR寄存器"]
        LR -->|"保存到栈"| STACK["内存栈区"]
        RET["BX LR<br/>(返回)"] -->|"LR → PC"| PC["PC寄存器"]
    end
    
    subgraph 栈帧结构
        direction TB
        STACK_TOP["栈顶"]
        STACK_TOP --> OLD_LR["保存的LR"]
        OLD_LR --> OLD_R4["保存的R4"]
        OLD_R4 --> LOCAL_VAR["局部变量"]
    end
```

**LR的本质**：
- 它就是一个普通的32位寄存器（D触发器阵列）
- 特殊之处在于**CPU硬件会自动操作它**：
  - 执行`BL`指令时，硬件自动把`PC+4`写入LR
  - 执行`BX LR`时，硬件自动把LR的值写入PC

## 八、大师的工程建议

1. **区分"程序员可见"和"程序员不可见"**：
   - 可见：R0-R15、外设寄存器
   - 不可见：流水线寄存器、ALU临时寄存器

2. **理解volatile的必要性**：
   ```c
   // 外设寄存器必须加volatile
   #define GPIO_ODR (*(volatile uint32_t *)0x40020014)
   ```
   因为外设寄存器的值可能被硬件改变，编译器不能优化掉对它的读写。

3. **寄存器访问的时序问题**：
   - CPU寄存器：1个时钟周期
   - 外设寄存器：需要经过总线，可能有等待周期
   - 这就是为什么`__NOP()`循环比`GPIO->ODR`操作快

4. **推荐深入方向**：
   - D触发器的门级电路（RS锁存器 → D锁存器 → D触发器）
   - 寄存器堆的读写端口设计
   - APB/AHB总线协议中的寄存器访问时序

## 六、链接脚本：内存布局的"设计图纸"

```ld
/* STM32F407 链接脚本示例 */
MEMORY
{
    FLASH (rx)  : ORIGIN = 0x08000000, LENGTH = 512K
    SRAM (rwx)  : ORIGIN = 0x20000000, LENGTH = 128K
    CCM (rw)    : ORIGIN = 0x10000000, LENGTH = 64K
}

SECTIONS
{
    .isr_vector : { . = ALIGN(4); KEEP(*(.isr_vector)) } > FLASH
    .text       : { *(.text*) } > FLASH
    .rodata     : { *(.rodata*) } > FLASH
    
    .data : {
        _sdata = .;           /* SRAM起始 */
        *(.data*)
        _edata = .;           /* SRAM结束 */
    } > SRAM AT > FLASH       /* 运行在SRAM，存储在Flash */
    _sidata = LOADADDR(.data);/* Flash中的位置 */
    
    .bss : {
        _sbss = .;
        *(.bss*)
        _ebss = .;
    } > SRAM
}
```

## 二、汇编的基本语法结构

### 2.1 汇编指令的通用格式

```
[标号:] 操作码 操作数1, 操作数2, ... [; 注释]
```

```asm
loop:           ; 标号（地址标记）
    MOV  R0, #10    ; 操作码 MOV，操作数 R0 和 #10
    ADD  R0, R0, #1 ; R0 = R0 + 1
    B    loop       ; 跳转到loop标号
```

| 操作数类型 | 示例 | 含义 |

| 寄存器偏移 | `[R0, #4]` | R0+4地址处的值 |

### 3.1 数据传送指令

```asm
MOV  R0, #10         ; R0 = 10 (立即数传送)
MOV  R1, R0          ; R1 = R0 (寄存器传送)

LDR  R0, [R1]        ; R0 = *R1 (从内存加载)
LDR  R0, [R1, #4]    ; R0 = *(R1+4) (带偏移加载)
LDR  R0, =0x12345678 ; 伪指令，加载32位立即数

STR  R0, [R1]        ; *R1 = R0 (存储到内存)
STR  R0, [R1, #4]!   ; R1=R1+4, *R1=R0 (预索引)
STR  R0, [R1], #4    ; *R1=R0, R1=R1+4 (后索引)

PUSH {R4-R7, LR}     ; 压栈，SP自动减
POP  {R4-R7, PC}     ; 出栈，SP自动增
```

### 3.2 算术与逻辑指令

```asm
ADD  R0, R1, R2      ; R0 = R1 + R2
ADD  R0, R0, #1      ; R0 = R0 + 1
SUB  R0, R1, R2      ; R0 = R1 - R2

AND  R0, R1, R2      ; R0 = R1 & R2
ORR  R0, R1, R2      ; R0 = R1 | R2
EOR  R0, R1, R2      ; R0 = R1 ^ R2
BIC  R0, R1, R2      ; R0 = R1 & (~R2) 位清除

LSL  R0, R1, #2      ; R0 = R1 << 2 (左移)
LSR  R0, R1, #2      ; R0 = R1 >> 2 (逻辑右移)
```

### 3.3 比较与跳转指令

```asm
CMP  R0, #10         ; 比较 R0 和 10，设置标志位
BEQ  equal           ; 如果相等(Z=1)，跳转到equal
BNE  not_equal       ; 如果不等(Z=0)，跳转到not_equal
BLT  less_than       ; 如果小于(N!=V)，跳转
BGT  greater         ; 如果大于(Z=0且N=V)，跳转

B    loop            ; 无条件跳转
BL   function        ; 带链接跳转 (LR = 返回地址)
BX   LR              ; 跳转到LR (函数返回)
```

## 五、C语言与汇编的对应关系

### 5.1 函数调用

**C代码**：
```c
int add(int a, int b) {
    return a + b;
}

int main(void) {
    int result = add(1, 2);
    return 0;
}
```

**对应汇编**：
```asm
; main函数
main:
    PUSH {LR}           ; 保存返回地址
    MOV  R0, #1         ; 参数1 → R0
    MOV  R1, #2         ; 参数2 → R1
    BL   add            ; 调用add，LR = 返回地址
    ; R0现在包含返回值
    MOV  R0, #0         ; return 0
    POP  {PC}           ; 返回

; add函数
add:
    ADD  R0, R0, R1     ; R0 = R0 + R1
    BX   LR             ; 返回
```

### 5.2 循环结构

**C代码**：
```c
for (int i = 0; i < 10; i++) {
    sum += i;
}
```

**对应汇编**：
```asm
    MOV  R0, #0         ; sum = 0
    MOV  R1, #0         ; i = 0
loop:
    CMP  R1, #10        ; i < 10 ?
    BGE  end_loop       ; 如果 i >= 10，退出
    ADD  R0, R0, R1     ; sum += i
    ADD  R1, R1, #1     ; i++
    B    loop           ; 继续循环
end_loop:
```

### 5.3 条件判断

**C代码**：
```c
if (x > 0) {
    y = 1;
} else {
    y = -1;
}
```

**对应汇编**：
```asm
    CMP  R0, #0         ; x > 0 ?
    BLE  else_branch    ; 如果 x <= 0，跳转
    MOV  R1, #1         ; y = 1
    B    end_if
else_branch:
    MOV  R1, #-1        ; y = -1
end_if:
```

### 5.4 指针与数组

**C代码**：
```c

int x = *p;      // x = arr[0]
int y = p[2];    // y = arr[2]
```

**对应汇编**：
```asm
    LDR  R0, =arr       ; R0 = arr的地址
    LDR  R1, [R0]       ; R1 = *R0 = arr[0]
    LDR  R2, [R0, #8]   ; R2 = *(R0+8) = arr[2]
                        ; 偏移8是因为int是4字节
```

### 详细说明

| 启动模式 | 向量表位置 | 地址 | 触发条件 |
|---------|-----------|------|----------|
| **从Flash启动** | Flash | 0x08000000 | BOOT0=0（默认） |

### 为什么有时需要重定位到SRAM？

```c
// 1. 运行时动态修改中断向量
// 2. Flash访问速度慢（有等待周期）
// 3. 某些应用需要在线升级固件

// 重定位向量表到SRAM
#define VTOR   (*(volatile uint32_t *)0xE000ED08)

void relocate_vector_table(void) {
    // 1. 把Flash中的向量表复制到SRAM
    memcpy((void *)0x20000000, (void *)0x08000000, 0x200);
    
    // 2. 设置VTOR指向新位置
    VTOR = 0x20000000;
}
```

### 3.4 完整代码示例

```c
/**
 * @brief  向量表重定位示例
 * @note   适用于STM32/Cortex-M系列
 */

#include <string.h>

#define SCB_VTOR   (*(volatile uint32_t *)0xE000ED08)

// 向量表大小 (取决于中断数量)
// STM32F407有82个外部中断 + 16个系统异常 = 98个向量
// 向量表大小 = 98 * 4 = 392字节，向上对齐到512字节
#define VECTOR_TABLE_SIZE  512

// 在SRAM中分配向量表空间 (必须对齐)
__attribute__((aligned(128))) 
static uint32_t vector_table_ram[VECTOR_TABLE_SIZE / 4];

/**
 * @brief  将向量表重定位到SRAM
 * @param  app_offset  App在Flash中的偏移量
 */
void relocate_vector_table_to_sram(uint32_t app_offset) {
    uint32_t *src_vectors = (uint32_t *)(0x08000000 + app_offset);
    uint32_t vector_count = VECTOR_TABLE_SIZE / 4;
    
    // 1. 复制向量表到SRAM
    memcpy(vector_table_ram, src_vectors, VECTOR_TABLE_SIZE);
    
    // 2. 设置VTOR指向SRAM中的向量表
    SCB_VTOR = (uint32_t)vector_table_ram;
    
    // 3. 确保写入生效
    __DSB();
    __ISB();
}

/**
 * @brief  动态修改某个中断的处理函数
 * @param  irq_num   中断号 (0 = IRQ0)
 * @param  handler   处理函数指针
 */
void set_irq_handler(uint8_t irq_num, void (*handler)(void)) {
    // IRQ0对应向量表偏移16 (前16个是系统异常)
    uint32_t vector_index = irq_num + 16;
    
    // 修改SRAM中的向量表
    vector_table_ram[vector_index] = (uint32_t)handler;
    
    // 确保写入生效
    __DSB();
}
```

```asm
    B   reset_handler           ; 0x00: Reset

    B   svc_handler             ; 0x08: Software interrupt
```

```c
    uint32_t base_addr;
    __asm__ volatile ("MRC p15, 0, %0, c12, c0, 0" : "=r"(base_addr));

}
```

## 七、Bootloader与App的向量表切换

这是嵌入式开发中最常见的向量表应用场景：

```mermaid
sequenceDiagram
    participant CPU
    participant VTOR
    participant Flash
    participant SRAM
    
    Note over CPU: 系统复位
    CPU->>Flash: 从0x08000000取初始SP和PC
    CPU->>VTOR: VTOR = 0x08000000 (默认)
    
    Note over CPU: Bootloader运行
    CPU->>Flash: 检查是否有App
    Flash-->>CPU: App存在于0x08008000
    
    Note over CPU: 跳转到App
    CPU->>VTOR: VTOR = 0x08008000
    CPU->>Flash: 从App向量表取SP和PC
    CPU->>CPU: 跳转到App的Reset_Handler
    
    Note over CPU: App运行
```

### 完整Bootloader跳转代码

```c
/**
 * @brief  Bootloader跳转到App
 * @param  app_addr  App的起始地址 (如0x08008000)
 */
void jump_to_app(uint32_t app_addr) {
    // 1. 检查App是否存在
    uint32_t app_stack = *(uint32_t *)app_addr;
    if (app_stack == 0xFFFFFFFF) {
        return;  // Flash未编程
    }
    
    // 2. 关闭所有中断
    __disable_irq();
    
    // 3. 关闭SysTick
    SysTick->CTRL = 0;
    SysTick->LOAD = 0;
    SysTick->VAL = 0;
    
    // 4. 清除所有中断挂起位
    for (int i = 0; i < 8; i++) {
        NVIC->ICER[i] = 0xFFFFFFFF;
        NVIC->ICPR[i] = 0xFFFFFFFF;
    }
    
    // 5. 设置VTOR指向App的向量表
    SCB->VTOR = app_addr;
    
    // 6. 设置主栈指针 (MSP)
    __set_MSP(app_stack);
    
    // 7. 跳转到App的Reset_Handler
    uint32_t app_entry = *(uint32_t *)(app_addr + 4);
    void (*app_reset_handler)(void) = (void (*)(void))app_entry;
    
    // 8. 确保所有操作完成
    __DSB();
    __ISB();
    
    // 9. 跳转!
    app_reset_handler();
    
    // 永远不应该执行到这里
    while(1);
}
```

# VTOR寄存器和向量表

> [!abstract]
> 向量表是CPU响应中断/异常的"电话簿"，存放各处理程序的入口地址。VTOR寄存器用于重定位向量表位置，是Bootloader跳转、动态中断配置的关键机制。不同架构（Cortex-M/A、RISC-V、Xtensa）的实现方式存在差异。

### 4.4 完整代码示例

```c
/**
 * @brief  向量表重定位示例
 * @note   适用于STM32/Cortex-M系列
 */

#include <string.h>

#define SCB_VTOR   (*(volatile uint32_t *)0xE000ED08)

// 向量表大小 (取决于中断数量)
// STM32F407有82个外部中断 + 16个系统异常 = 98个向量
// 向量表大小 = 98 * 4 = 392字节，向上对齐到512字节
#define VECTOR_TABLE_SIZE  512

// 在SRAM中分配向量表空间 (必须对齐)
__attribute__((aligned(128))) 
static uint32_t vector_table_ram[VECTOR_TABLE_SIZE / 4];

/**
 * @brief  将向量表重定位到SRAM
 * @param  app_offset  App在Flash中的偏移量
 */
void relocate_vector_table_to_sram(uint32_t app_offset) {
    uint32_t *src_vectors = (uint32_t *)(0x08000000 + app_offset);
    uint32_t vector_count = VECTOR_TABLE_SIZE / 4;
    
    // 1. 复制向量表到SRAM
    memcpy(vector_table_ram, src_vectors, VECTOR_TABLE_SIZE);
    
    // 2. 设置VTOR指向SRAM中的向量表
    SCB_VTOR = (uint32_t)vector_table_ram;
    
    // 3. 确保写入生效
    __DSB();
    __ISB();
}

/**
 * @brief  动态修改某个中断的处理函数
 * @param  irq_num   中断号 (0 = IRQ0)
 * @param  handler   处理函数指针
 */
void set_irq_handler(uint8_t irq_num, void (*handler)(void)) {
    // IRQ0对应向量表偏移16 (前16个是系统异常)
    uint32_t vector_index = irq_num + 16;
    
    // 修改SRAM中的向量表
    vector_table_ram[vector_index] = (uint32_t)handler;
    
    // 确保写入生效
    __DSB();
}
```

```asm
    B   prefetch_abort_handler  ; 0x0C: Prefetch abort
```

- 外部中断通过**IRQ入口**统一处理，需要软件查询中断控制器

### 5.3 VBAR寄存器

```c
// Cortex-A VBAR (Vector Base Address Register)
// 每个异常级别有自己的VBAR

// 设置异常向量表基地址
void set_vbar(uint32_t base_addr) {
    // VBAR必须32字节对齐
    __asm__ volatile ("MCR p15, 0, %0, c12, c0, 0" : : "r"(base_addr));
}

// 读取当前VBAR
uint32_t get_vbar(void) {
    uint32_t base_addr;
    __asm__ volatile ("MRC p15, 0, %0, c12, c0, 0" : "=r"(base_addr));
    return base_addr;
}
```

## 八、Bootloader与App的向量表切换

这是嵌入式开发中最常见的向量表应用场景：

```mermaid
sequenceDiagram
    participant CPU
    participant VTOR
    participant Flash
    participant SRAM
    
    Note over CPU: 系统复位
    CPU->>Flash: 从0x08000000取初始SP和PC
    CPU->>VTOR: VTOR = 0x08000000 (默认)
    
    Note over CPU: Bootloader运行
    CPU->>Flash: 检查是否有App
    Flash-->>CPU: App存在于0x08008000
    
    Note over CPU: 跳转到App
    CPU->>VTOR: VTOR = 0x08008000
    CPU->>Flash: 从App向量表取SP和PC
    CPU->>CPU: 跳转到App的Reset_Handler
    
    Note over CPU: App运行
```

### 完整Bootloader跳转代码

```c
/**
 * @brief  Bootloader跳转到App
 * @param  app_addr  App的起始地址 (如0x08008000)
 */
void jump_to_app(uint32_t app_addr) {
    // 1. 检查App是否存在
    uint32_t app_stack = *(uint32_t *)app_addr;
    if (app_stack == 0xFFFFFFFF) {
        return;  // Flash未编程
    }
    
    // 2. 关闭所有中断
    __disable_irq();
    
    // 3. 关闭SysTick
    SysTick->CTRL = 0;
    SysTick->LOAD = 0;
    SysTick->VAL = 0;
    
    // 4. 清除所有中断挂起位
    for (int i = 0; i < 8; i++) {
        NVIC->ICER[i] = 0xFFFFFFFF;
        NVIC->ICPR[i] = 0xFFFFFFFF;
    }
    
    // 5. 设置VTOR指向App的向量表
    SCB->VTOR = app_addr;
    
    // 6. 设置主栈指针 (MSP)
    __set_MSP(app_stack);
    
    // 7. 跳转到App的Reset_Handler
    uint32_t app_entry = *(uint32_t *)(app_addr + 4);
    void (*app_reset_handler)(void) = (void (*)(void))app_entry;
    
    // 8. 确保所有操作完成
    __DSB();
    __ISB();
    
    // 9. 跳转!
    app_reset_handler();
    
    // 永远不应该执行到这里
    while(1);
}
```

## 十、大师的工程建议

### 10.1 常见陷阱

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| **跳转后HardFault** | VTOR未更新 | 跳转前设置VTOR |
| **中断不响应** | 向量表未对齐 | 确保128字节对齐 |
| **App无法运行** | MSP未设置 | 跳转前设置MSP |
| **中断跑错函数** | 向量表索引错误 | 检查IRQ号与向量索引对应关系 |

### 10.2 调试技巧

```c
// 调试时打印向量表信息
void debug_vector_table(void) {
    uint32_t vtor = SCB->VTOR;
    printf("VTOR = 0x%08X\n", vtor);
    printf("Initial SP = 0x%08X\n", *(uint32_t *)vtor);
    printf("Reset Handler = 0x%08X\n", *(uint32_t *)(vtor + 4));
    printf("HardFault Handler = 0x%08X\n", *(uint32_t *)(vtor + 0x0C));
}
```
