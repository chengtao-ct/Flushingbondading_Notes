# 编码器模块：从脉冲计数到速度反馈

> [!abstract] 实战场景
> 编码器模块负责把电机转动变成可计算的位移和速度。它是 [[7.速度环]] 的反馈来源，也是判断 [[2.电机模块]] 方向是否正确的关键证据。

> [!note] 快速结论
> - 霍尔编码器通过脉冲数量判断转动角度，通过 A/B 相相位差判断方向。
> - 本项目原思路使用 GPIO + EXTI 计数，而不是直接使用 TIM 编码器模式。
> - 速度估算要同时关心单位、采样窗口、低速归零、换向毛刺和原子读取。
> - 闭环之前先开环验证：电机正方向、编码器正方向、速度正方向必须一致。

## 硬件认知

![](assets/3.编码器模块/{E5D26EF8-7745-43C5-A518-7E191F4C0027}.png)

**图意：** 霍尔编码器通过磁环和霍尔传感器产生脉冲。随着电机转动，A/B 两相信号交替变化。

**工程结论：**
- 脉冲数量对应转过的角度。
- A/B 相谁超前谁滞后，对应旋转方向。
- 编码器装在电机轴还是轮轴，会直接影响换算公式里的减速比。

> [!warning] 易错点：线数、边沿数、减速比不要混在一起
> `编码器线数` 是硬件每圈产生的基础脉冲数；`单边单沿/单边双沿/双边双沿` 是软件计数倍率；`减速比` 决定电机轴转多少圈才等于轮子转一圈。三者必须分清楚。

## 软件链路

```text
A/B 相 GPIO
  -> AFIO/EXTI 映射
  -> 上升沿/下降沿触发
  -> 中断内判断方向并更新 position
  -> 周期任务读取 position 差值
  -> pulse -> angle / speed
  -> [[7.速度环]]
```

**工程结论：** 中断里只做计数和方向判断，速度换算放在周期任务里。这样可以降低中断耗时，也便于做原子读取和滤波。

## IO 重映射和 EXTI

![](assets/3.编码器模块/943d974d2f5d2483a1da6c07ca8530d4.jpg)

![](assets/3.编码器模块/2cfd2142a52f9a5610bfb0214e01cb37.jpg)

**图意：** 这两张图对应 GPIO 复用/重映射相关知识，尤其是部分 JTAG 引脚需要释放成普通 GPIO 后才能作为编码器输入。

**工程结论：** 如果编码器接在复用调试口上，要先确认是否需要关闭 JTAG 或保留 SWD。否则代码看起来配置正确，但引脚根本没有按预期工作。

![](assets/3.编码器模块/c2795cdcbfb0d775cbf614a9052e5ca1.jpg)

![](assets/3.编码器模块/a35317a3caedd5375c37e1f9594aa277.jpg)

**图意：** EXTI 线路从 GPIO 端口映射到外部中断线，再进入 NVIC 和中断服务函数。

**工程结论：**
- EXTI 需要同时配置 AFIO、EXTI line、触发边沿和 NVIC。
- `EXTI15_10_IRQn` 这种共享中断必须在 ISR 里判断具体 pending bit。
- `Cmd` 一般就是开关，配置完还要真正 enable。

### EXTI 计数骨架

```c
static volatile int32_t g_encoder_pos = 0;

void EXTIx_IRQHandler(void)
{
    if (EXTI_GetITStatus(ENC_A_EXTI_LINE) == SET) {
        if (GPIO_ReadInputDataBit(ENC_B_PORT, ENC_B_PIN)) {
            g_encoder_pos++;
        } else {
            g_encoder_pos--;
        }
        EXTI_ClearITPendingBit(ENC_A_EXTI_LINE);
    }
}
```

> [!warning] 易错点：中断里不要做复杂速度计算
> EXTI 触发频率会随转速上升。中断里做浮点、滤波或串口打印，很容易把系统节奏拖乱。中断只更新计数，主循环周期任务再读取。

## 角度换算

```text
轮子角度 =
    编码器计数
    / 编码器线数
    / 软件计数倍率
    / 减速比
    * 360 deg
```

**工程结论：** 公式最重要的是单位闭环。你要能说清楚当前变量到底是“电机轴脉冲数”“电机轴角度”“轮轴角度”还是“车轮线速度”。

## 速度算法

### M 法测速

![](assets/3.编码器模块/{61521A94-F04C-44FC-8F9E-E7F0B5CAE7E5}.png)

**图意：** M 法是在固定时间窗口内统计脉冲数，用脉冲数量除以采样时间得到速度。

**工程结论：** M 法适合中高速。窗口越长，误差越小但响应越慢；窗口越短，响应快但低速抖动明显。

![](assets/3.编码器模块/{79BD016A-8487-486B-8A5B-30C1350811D1}.png)

**图意：** M 法误差来自编码器边沿和采样窗口不同步，脉冲边界可能多算或少算。

**工程结论：** 原文提到误差约为 `2/M`，可以理解为窗口边界最多带来少量边沿误差。`M` 越大，相对误差越小，但系统延迟也变大。

### T 法测速

![](assets/3.编码器模块/{44CFFEB9-0CDA-47AD-AD20-8E564AE85D71}.png)

**图意：** T 法测量相邻脉冲之间的时间，用周期反推速度。

**工程结论：** T 法适合低速，能在脉冲少的时候保持分辨率；但一旦没有新脉冲，速度更新会延迟，容易出现“不归零”问题。

![](assets/3.编码器模块/{582B1596-64E5-4F78-8119-172EA4400E94}.png)

**图意：** 低速或停止时，如果只在 EXTI 到来时更新速度，旧速度会滞留。

**工程结论：** 需要用当前时间和上一次脉冲时间比较，超过阈值后主动衰减或归零。原文的 `T = max(T0 - T1, now - T0)` 就是在补这个延时问题。

![](assets/3.编码器模块/{584D243A-1B87-40CC-8096-6B8ABEFEF303}.png)

**图意：** 换向、毛刺、中断并发会让速度计算出现负数、溢出或短脉冲异常。

**工程结论：**
- 读取多字节计数时要考虑原子性，可以短暂关中断或使用临界区。
- 换向点可以强制速度过零，避免极窄脉冲带来虚假大速度。
- 毛刺需要硬件滤波、软件死区或最小脉宽判断。

## 选择策略

| 方法 | 适合场景 | 主要问题 |
| --- | --- | --- |
| M 法 | 中高速、周期控制 | 低速分辨率差 |
| T 法 | 低速、脉冲稀疏 | 停止时速度不自动归零 |
| M/T 混合 | 宽速度范围 | 实现复杂 |
| TIM 编码器模式 | 硬件资源允许时 | 需要占用定时器和对应引脚 |

> [!tip] 工程取舍
> 复刻阶段用 EXTI 能帮助理解编码器细节；工程稳定后可以评估 TIM 编码器模式，让硬件帮你完成正反计数。

## 调试和排错

| 现象 | 优先怀疑 | 验证动作 |
| --- | --- | --- |
| 计数一直不变 | GPIO/EXTI/AFIO 映射错误 | 手转电机，看 EXTI pending 或翻转 LED |
| 方向反了 | A/B 相判断反、左右电机镜像 | 开环正转，观察 position 正负 |
| 速度不归零 | T 法只在脉冲中断更新 | 加超时归零逻辑 |
| 速度有尖峰 | 毛刺、换向窄脉冲、原子读取失败 | 加最小周期限制和临界区 |
| 接入速度环后发散 | 编码器方向和电机方向不一致 | 回到 [[2.电机模块]] 开环验证 |
