# 面试提问：平衡小车项目模拟面试

> [!abstract] 实战场景
> 基于平衡小车项目（`bal` → `CT_bal_car` → `HAL-CAR` 三版演进）的完整模拟面试记录。共 8 轮 18 题，覆盖项目概述、系统架构、控制链路、传感器原理、中断安全、外设配置、项目复盘和 RTOS 意识。每题包含：问题、标准回答、易错点和面试话术。

> [!note] 快速结论
> - 面试官最关心的三件事：**你做了什么**（技术深度）、**你怎么做的**（工程方法）、**你学到了什么**（复盘能力）。
> - 回答任何技术问题，先讲"是什么"，再讲"为什么这样设计"，最后讲"如果出问题怎么办"。
> - 不要把复刻项目说成完全原创，但必须说清楚自己的贡献点（分层架构、安全设计、M-T 测速等）。

## 第一轮：项目概述与动机

### Q1：请用 2 分钟介绍这个项目。你为什么选择做平衡车？

**标准回答结构：**

> 这是一个基于 STM32F103C8T6 的两轮自平衡车项目。硬件上用 MPU6050 做姿态反馈，左右编码器做轮速反馈，TB6612 驱动电机，USART3 接收遥控串口命令。软件上我设计了 BSP → Device → Service → Algorithm → Control → App 六层架构，实现了四级级联 PID 平衡控制——速度环、角度环、角速度环、转向环串级，最后输出左右轮目标速度，经电机速度 PID 转成 PWM 占空比驱动电机。
>
> 项目分两个阶段：第一阶段复刻铁头山羊的标准库工程，理解控制链路；第二阶段用 CubeMX + HAL 库重构，做分层优化和安全设计。
>
> 选平衡车的原因：它是裸机开发中少数能同时检验多外设协同、闭环控制和安全保护意识的项目，涉及 I2C、USART、ADC、TIM、EXTI、PWM、PID，综合性强。

> [!warning] 易错点：说错硬件细节
> 你的项目里**没有蓝牙模块**，遥控走的是 USART3 串口。面试中说错硬件细节很扣分，考官会怀疑你是不是真的做过。

> [!warning] 易错点："裸机"用反了
> 你的项目就是裸机开发（没有 RTOS）。正确说法是"在没有操作系统的情况下，裸机开发可以很好地检验……"，而不是"在没有裸机的情况下"。

### Q2：三个版本之间的关系和学习路径是什么？

```text
bal（铁头山羊参考工程）
  → 功能最全：校准、自动起立、RGB 灯、完整测试套件
  → 作用：参考实现，看"好的长什么样"

CT_bal_car（第一版复刻）
  → 用标准外设库复刻 bal 的核心控制链路
  → 作用：理解传感器→控制→电机的闭环

HAL-CAR（第二版重构）
  → 用 CubeMX + HAL 重构，加入六层架构和安全设计
  → 作用：工程化能力提升
```

**工程结论：** 三版的关系是"参考 → 学习 → 改进"。面试时要讲清楚每一步你学到了什么、改进了什么。

## 第二轮：系统架构

### Q3：HAL-CAR 是怎么分层的？为什么要把 Algorithm 和 Control 分开？

**六层架构职责：**

| 层级 | 目录 | 职责 | 关键规则 |
|------|------|------|---------|
| BSP | `BSP/` | 封装 HAL API：UART、GPIO、PWM、ADC、I2C、EXTI、时间基准 | 可以调用 HAL API |
| Device | `Device/` | 板级器件驱动：MPU6050、TB6612、编码器电机 | 调用 BSP，不直接写业务逻辑 |
| Service | `Service/` | 将硬件数据整理为业务状态：姿态、轮速、电池、按键、命令 | 调用 Device |
| Algorithm | `Algorithm/` | 纯数学工具，当前主要是 PID | **不允许调用 HAL 外设 API** |
| Control | `Control/` | 平衡控制器、电机速度控制器 | 消费 Service 输出和 Algorithm 工具 |
| App | `App/` | 输入处理、状态机、周期调度、安全状态切换 | 负责调度和运行许可 |

**Algorithm 和 Control 分开的原因：**

```text
Algorithm 层放的是纯数学工具（PID）
  → 不知道自己在控制平衡车
  → 不知道 MPU6050 是什么
  → 不 include 任何 HAL 头文件
  → 可以在 PC 上编译运行、写单元测试

Control 层是业务控制器（平衡控制器、电机速度控制器）
  → 知道自己在控制什么
  → 调用 Service 获取传感器数据
  → 调用 Algorithm 的 PID 做计算
  → 和具体硬件业务绑定

边界：Algorithm 可以移植到任何平台编译；Control 和具体硬件业务绑定。
```

**工程结论：** 分层的本质不是"代码放不同目录"，而是"依赖方向必须单向"。上层可以调下层，下层不能调上层，Algorithm 不能知道硬件的存在。

### Q4：`app_input.c` 直接 include 了 `bsp_uart3.h`，这违反分层规则吗？

> [!warning] 面试陷阱题
> "不违反吧"不是回答。面试官问这个问题是在考验你对分层规则的思考和权衡能力。

**标准回答：**

> 严格来说是违反了，App 层不应该直接调 BSP。原因是 `app_input.c` 需要注册 UART3 的字节接收回调，而 `command_service` 是处理完整命令的，中间的"字节接收→命令拼接"环节没有对应的 Service 层封装。这是一个务实妥协。更理想的做法是在 `command_service` 层暴露一个注册接口，把 BSP 回调注册收敛到 Service 内部，App 层只调 Service。

## 第三轮：控制链路

### Q5：从遥控输入到电机 PWM 输出，描述完整的控制链路

**四级级联 PID + 电机速度环完整信号流：**

```text
遥控 speed_cmd ──→ [速度环PID] ──→ theta_ref
                       ↑                    ↓
                  轮速+姿态修正        [角度环PID] ──→ theta_dot_ref
                                                          ↓
遥控 turn_cmd ──→ [转向PID] → omega_diff    [角速度环PID] ──→ 动力学换算 → omega_ref
                     ↓                                              ↓
            ═══════════════════════════════════════════════════════════
            左轮目标 = omega_ref + omega_diff    右轮目标 = omega_ref - omega_diff
                          ↓                                    ↓
                   [左电机PID] ──→ V ──→ duty          [右电机PID] ──→ V ──→ duty
                          ↓                                    ↓
                      TB6612 左电机                        TB6612 右电机
```

**第一级：速度环（Velocity PID）**

| 项 | 内容 |
|---|---|
| 输入 | 目标车速 `speed_cmd` vs 实际车速 `car_speed` |
| 车速计算 | `car_speed = wheel_speed + pitch_rate × (body_length + wheel_radius)` |
| 输出 | 目标加速度 → `atan(a_ref / g)` → 目标倾角 `theta_ref` |
| 物理含义 | "我要跑这么快，那我应该往前倾斜多少度" |
| 参数 | Kp=10, Ki=1, Kd=0 |

> [!tip] 关键细节
> 车体速度不是简单等于轮速。因为车倾斜时，轮子和地面接触点的速度不等于质心速度，所以要用 `pitch_rate × (lp + rw)` 做修正。

**第二级：角度环（Angle PID）**

| 项 | 内容 |
|---|---|
| 输入 | 目标倾角 `theta_ref` vs 实际倾角 `pitch`（IMU 互补滤波输出） |
| 输出 | 目标角速度 `theta_dot_ref` |
| 物理含义 | "我应该倾斜到这个角度，那我的角速度应该设为多少" |
| 参数 | Kp=4, Ki=0, Kd=0 |

**第三级：角速度环（Rate PID）**

| 项 | 内容 |
|---|---|
| 输入 | 目标角速度 `theta_dot_ref` vs 实际角速度 `gx`（陀螺仪 X 轴） |
| 输出 | 目标角加速度 → 倒立摆动力学方程换算 → 基础轮速参考 `omega_ref` |
| 动力学方程 | `a_horizontal = (lp × alpha_dd - g × sin(pitch)) / cos(pitch)`，积分得 `omega_ref` |
| 物理含义 | "为了达到这个角速度，轮子应该转多快" |
| 参数 | Kp=10, Ki=10, Kd=0 |

> [!tip] 关键细节
> 这里不是直接把 PID 输出当轮速，而是通过**倒立摆动力学方程**做了物理换算。这是区别于"简单调参党"的核心。

**第四级：转向环（Turn PID）——与上面三级并行**

| 项 | 内容 |
|---|---|
| 输入 | 目标转向角速度 `turn_cmd` vs 实际 yaw 角速度 `gz` |
| 输出 | 左右轮差速 `omega_diff` |
| 物理含义 | "我要转这么快，那左右轮应该差多少速度" |
| 参数 | Kp=1, Ki=0, Kd=0 |

**电机速度环（Motor Speed PID）——每个轮一个**

| 项 | 内容 |
|---|---|
| 输入 | 目标轮速 vs 实际轮速（编码器 M-T 测速） |
| 输出 | 目标电机电压 `V_target` → `duty = V_target / V_battery × 100%` |
| 物理含义 | "轮子应该转这么快，给电机加多少电压" |
| 参数 | Kp=0.5, Ki=7, Kd=0 |

> [!tip] 电池电压前馈补偿
> 电池电量低时，同样的 duty 产生的电机电压不同，所以用 `V_target / V_battery` 换算，保证控制效果一致。

**面试话术：**

> 我的平衡控制器采用四级级联 PID。最外层是速度环，根据目标车速和实际车速的误差，通过 `atan(a/g)` 计算出车体应该倾斜的目标角度。第二层是角度环，根据目标倾角和 IMU 实际 pitch 的误差，输出目标角速度。第三层是角速度环，根据目标角速度和陀螺仪反馈，通过倒立摆动力学方程换算出轮子的基础角速度参考。转向环是独立并行的，输出左右轮差速。最后做左右轮混合，送到各自的电机速度 PID，PID 输出目标电压后除以电池电压得到 PWM 占空比，驱动 TB6612。

## 第四轮：传感器与姿态估计

### Q6：互补滤波的原理

**为什么不能只用一种传感器？**

```text
单用陀螺仪积分：
  → 短期精度高（响应快，不怕振动）
  → 有零偏漂移（bias drift），积分不断累积误差
  → 时间一长角度就飘了

单用加速度计：
  → 长期均值准确（重力方向不漂）
  → 车运动时产生额外加速度（刹车、颠簸、振动）
  → 混入重力信号，导致角度抖动
```

**互补滤波公式：**

```c
pitch = alpha * (pitch + gx * dt) + (1.0f - alpha) * pitch_accel;
//       ↑ 信任陀螺仪的短期变化     ↑ 信任加速度计的长期绝对参考
```

- `alpha` 接近 1 → 更信任陀螺仪（平滑，但会漂）
- `alpha` 接近 0 → 更信任加速度计（不漂，但抖）

**项目参数：** `alpha = 0.95238`，对应时间常数：

```text
tau = alpha * dt / (1 - alpha) = 0.95238 * 0.005 / 0.04762 ≈ 0.1s
```

含义：**100ms 以内的快速变化用陀螺仪，100ms 以上的长期趋势用加速度计修正。**

**为什么不直接用卡尔曼滤波？**

> 卡尔曼滤波理论上更优，但计算量大，需要调协方差矩阵。互补滤波在平衡车场景下效果足够好，实现简单，Cortex-M3 没有浮点单元，互补滤波的计算开销远小于卡尔曼。

> [!warning] 易错点：积分起点
> 正确做法是 `pitch = alpha * (pitch + gx*dt) + ...`，积分起点是**上次融合结果**（已修正过的，漂移不累积）。
>
> 错误做法是单独维护 `pitch_g = pitch_g + gx*dt` 再融合，漂移会持续累积。

**面试话术：**

> 互补滤波的核心是取两者优点：陀螺仪短期可信，给它 95% 权重保证快速响应；加速度计长期可信，给它 5% 权重慢慢修正漂移。alpha=0.95238 对应约 100ms 的时间常数，意思是 100ms 以内的高频变化由陀螺仪主导，100ms 以上的低频漂移由加速度计拉回。相比卡尔曼滤波，互补滤波计算量小，在无 FPU 的 Cortex-M3 上更实际。

### Q7：M 法与 T 法测速

**M 法（频率测量法）：**

```text
速度 = 固定周期内的脉冲数 / 周期时间

高速时好：脉冲多，分辨率高
低速时差：5ms 内可能只有 0-1 个脉冲，分辨率极差
```

**T 法（周期测量法）：**

```text
速度 = 1 / 相邻两个脉冲之间的时间间隔

低速时好：间隔长，测量精度高
高速时差：间隔太短，定时器分辨率不够，中断太频繁
```

**为什么不能只用一种？**

| 速度 | 只用 M 法 | 只用 T 法 |
|------|----------|----------|
| 高速 | 分辨率高，好 | 中断太频繁，漏脉冲 |
| 低速 | 5ms 可能 0 个脉冲 | 精度高，好 |
| 静止 | 永远是 0，好 | 没有脉冲，无法计算 |

**代码中的切换逻辑：**

```text
每个 5ms 周期：
  1. 从编码器拿快照（脉冲数 + 最近边沿时间）
  2. 如果本周期脉冲数 >= 阈值 → M 法：omega = 脉冲数 / 周期时间
  3. 如果本周期脉冲数 < 阈值，但有最近边沿 → T 法：omega = 1 / 边沿间隔
  4. 如果超过停止超时没有新边沿 → 强制归零
  5. 最终结果做最大速度限幅，抑制异常尖峰
```

**面试话术：**

> 编码器测速我用 M-T 混合。M 法在固定周期内数脉冲数，高速时分辨率好；T 法测量相邻脉冲的时间间隔，低速时精度高。每个 5ms 控制周期里，先看脉冲数够不够，够就用 M 法，不够就 fallback 到 T 法用最近边沿间隔计算，超时没有新边沿就强制归零。

## 第五轮：中断、实时性与安全设计

### Q8：I2C 读取卡死会发生什么？怎么防护？

**当前系统的行为：**

```text
HAL_I2C_Mem_Read() 带 10ms 超时
  → 如果 I2C 总线挂死，最多阻塞 10ms
    → 平衡控制周期 5ms 被挤占
      → app_state 检测到任务超时 → 进入 FAULT → 电机停止
```

**当前系统不会永久卡死**（HAL I2C 有超时 + 任务超时检测兜底），但存在问题：
- 10ms 阻塞远超 5ms 控制周期，会丢掉控制周期
- 连续 I2C 超时会导致系统反复进出 FAULT，电机输出抖动
- I2C 总线挂死后不会自动恢复

**改进方案：**

| 方案 | 说明 | 复杂度 |
|------|------|--------|
| I2C 总线恢复 | 通信失败时重新初始化 I2C 外设，连续失败 N 次进 FAULT | 低 |
| DMA 异步读取 | I2C 不阻塞控制循环，用"最新有效数据"做控制，数据过期进 FAULT | 中 |
| IWDG 看门狗 | 软件挂死时触发芯片复位，电机引脚复位后默认低电平 | 低 |

**面试话术：**

> 当前 I2C 有 HAL 层超时保护，加上 5ms 任务超时检测，不会永久卡死。但 I2C 阻塞会挤占控制周期。更理想的方案是改用 DMA 异步读取，控制循环只消费最新快照，数据过期则进入 FAULT。同时应该加 IWDG 作为终极兜底，防止整个软件挂死时电机失控。

### Q9：从按键启动到失控倒地的完整状态转换

```text
上电
  ↓
[INIT] HAL_Init(), CubeMX 外设初始化, app_state_init(), app_input_init()
  ↓ 全部成功
[READY] 电机保持停止，motor_speed_controller_stop() 反复调用
  ↓ 用户单击按键
[RUNNING] 刷新调度时间戳，复位所有 PID
  │  每 5ms：轮速更新 → 平衡控制 → 设置电机目标
  │  每 10ms：电机速度控制 → 更新 PWM duty
  │
  ├─ pitch > 80° ────────────────→ [FAULT]
  ├─ IMU 数据无效 ───────────────→ [FAULT]
  ├─ 平衡任务超过 8ms ───────────→ [FAULT]
  ├─ 电机任务超过 15ms ──────────→ [FAULT]
  ├─ 电池无效或低压 ─────────────→ [FAULT]
  ├─ TB6612 驱动更新失败 ────────→ [FAULT]
  └─ 用户单击按键 ──────────────→ [READY]
  
[FAULT] balance_controller_stop() + motor_speed_controller_stop()
  │  PWM 归零，STBY 拉低，电机完全停止
  │  不会自动恢复
  └─ 用户长按按键 ──────────────→ [READY]
```

**所有 FAULT 触发条件：**

| 触发条件 | 检测位置 | 检测机制 |
|---------|---------|---------|
| pitch > 80° | balance_controller | 倾倒阈值判断 |
| IMU 数据无效 | balance_controller | imu_attitude_service 返回无效 |
| 平衡任务超过 8ms | app_state | 时间戳差值检测 |
| 电机任务超过 15ms | app_state | 时间戳差值检测 |
| 电池无效或低压 | motor_speed_controller | battery_service 返回异常 |
| TB6612 驱动更新失败 | motor_speed_controller | set_duty 返回错误 |

**面试话术：**

> 上电进入 INIT，初始化完成后进入 READY，电机保持停止。用户单击按键进入 RUNNING，启动 5ms 平衡环和 10ms 电机环。任何异常——倾倒、IMU 失效、任务超时、电池异常——都会进入 FAULT，FAULT 里同时停止平衡控制器和电机控制器，电机 duty 归零、STBY 拉低。FAULT 不会自动恢复，必须长按按键清除故障后才回到 READY。

### Q10：`__disable_irq()` 的作用、问题和替代方案

**作用：**

```c
// 主循环中读取编码器快照
__disable_irq();                         // PRIMASK=1，全局屏蔽所有可屏蔽中断
snapshot.count = encoder->count;         // 这几行不会被任何中断打断
snapshot.direction = encoder->direction;
snapshot.last_edge_us = encoder->last_edge_us;
__enable_irq();                          // PRIMASK=0，恢复中断
```

不这样做的话，主循环读到一半 count，编码器 ISR 进来改了 count 和 direction，拿到的就是**半个旧值半个新值**，数据不一致。

**潜在问题：**

```text
问题1：屏蔽了所有中断
  → UART、ADC、TIM 全部被屏蔽
  → 临界区代码耗时过长会影响其他实时任务

问题2：嵌套调用出问题
  → A 函数 disable_irq 后调用 B 函数
  → B 也 disable_irq 然后 enable_irq
  → B 的 enable_irq 会在 A 的临界区还没结束时恢复中断
```

**更好的替代方案：**

| 方案 | 说明 |
|------|------|
| 保存/恢复 PRIMASK | `primask = __get_PRIMASK(); __disable_irq(); ... __set_PRIMASK(primask);` 支持嵌套 |
| BASEPRI 优先级屏蔽 | `__set_BASEPRI(阈值);` 只屏蔽低于某优先级的中断，更高优先级仍响应 |
| 双缓冲/无锁设计 | ISR 写缓冲 A，主循环读缓冲 B，用标志位切换，根本不需要关中断 |

**面试话术：**

> 我用 `__disable_irq()` 做编码器快照的原子复制，防止主循环读到半更新的数据。它会全局屏蔽中断，影响其他实时任务响应。更好的做法是用 `__get_PRIMASK()` / `__set_PRIMASK()` 保存恢复中断状态，支持嵌套调用；或者用 BASEPRI 只屏蔽特定优先级以下的中断，让高优先级控制中断不受影响。

## 第六轮：外设配置与调试方法

### Q11：PWM 频率计算

**计算公式：**

```text
PWM频率 = 定时器时钟 / ((PSC + 1) × (ARR + 1))
```

项目参数（TIM1，挂在 APB2）：

```text
时钟 = 72MHz
PSC = 0   → 分频系数 = 1
ARR = 3599 → 计数周期 = 3600

PWM频率 = 72,000,000 / (1 × 3600) = 20kHz
```

**改成 10kHz：**

```text
方法：ARR 改成 7199
  10kHz = 72,000,000 / (1 × 7200)

优先改 ARR 而不是 PSC：
  → ARR 越大，duty 分辨率越高
  → ARR=7199 时 duty 分辨率 = 1/7200 ≈ 0.014%
```

> [!warning] 易错点：PSC 和 ARR 都要 +1
> 定时器分频系数是 `PSC+1`，计数周期是 `ARR+1`。忘记 +1 是最常见的计算错误。

### Q12：ADC 电池电压换算中 8.4V 的来源

**硬件分压电路：**

```text
电池(0~8.4V)
  ├── R1 ──┬── 接 ADC 引脚 (0~3.3V)
  │        │
  │       R2
  │        │
  └────────┴── GND

分压比 = R2 / (R1 + R2)
```

如果设计成 8.4V 对应 ADC 满量程（3.3V），则：

```text
ADC 原始值 = 0~4095 (12位)
电池电压 = ADC_raw / 4095 × 8.4V
```

**8.4V 就是"ADC 满量程时对应的电池电压"，由硬件分压电阻的比值决定。**

**如果实际满电是 8.2V 不是 8.4V，采样结果仍然准确**——因为 8.4V 是满量程参考点，不是假设电池一定是 8.4V。只要分压电阻的比值和公式一致，任何实际电压都能正确换算。

**真正会出错的情况：** 硬件分压比变了（比如换了电阻），但公式里的 8.4V 没跟着改 → 所有采样结果都有固定比例偏差 → 低压保护阈值不准。

> [!warning] 易错点
> `8.4V` 不是 ADC 参考电压。`4095` 对应的是 ADC 输入脚接近 `Vref(3.3V)`，不是电池直接等于 `8.4V`。只有在分压电路把 `8.4V` 缩放到 `3.3V` 时，才能写成 `raw / 4095 × 8.4`。

### Q13：CubeMX 生成代码与用户代码的边界

**CubeMX 的保护机制：**

```c
/* USER CODE BEGIN xxx */

// 你自己写的代码放这里，CubeMX 重新生成时不会被覆盖

/* USER CODE END xxx */
```

**项目中的划分：**

```text
CubeMX 负责生成的（不要手动改）：
  ├── Core/Src/main.c 里的 MX_GPIO_Init(), MX_USART2_Init() 等
  ├── Core/Src/stm32f1xx_hal_msp.c

你自己写的（完全不受 CubeMX 影响）：
```

## 第七轮：项目复盘与改进能力

### Q14：CT_bal_car 的三大问题以及 HAL-CAR 的解决方案

**问题一：没有分层，模块职责混乱**

```text
CT_bal_car：
  app_control.c 直接调用 I2C 读 MPU6050
  app_motor.c   直接操作 TIM 寄存器写 PWM
  app_rc.c      直接在 ISR 里 strcpy
  → 控制逻辑和硬件操作混在一起，换芯片就要重写全部

HAL-CAR 解决：
  BSP 封装 HAL，Device 封装 BSP，Service 整理业务状态
  balance_controller 不知道 I2C 是什么，只知道调 imu_attitude_service
  → 换芯片只需要重写 BSP 层，上层不用动
```

**问题二：几乎没有错误处理和安全保护**

```text
CT_bal_car：
  MPU6050 读取失败 → 没有检查，用垃圾数据做控制
  I2C 卡死         → 没有超时，整个系统挂死
  UART 接收        → 64 字节缓冲区没有边界保护
  倾倒             → 没有倾倒检测，倒了电机还在转

HAL-CAR 解决：
  MPU6050 失败 → IMU 标记无效 → 平衡控制器拒绝输出 → FAULT
  HAL I2C 自带超时 → 不会永久阻塞
  command_service 有缓冲区长度保护 + 命令超时归零
  pitch > 80° → 倾倒检测 → FAULT → 电机停止
```

**问题三：PID 实现重复且不规范**

```text
CT_bal_car：
  user/app_pid.c 一套 PID（简单版）
  my_lib/pid.c   另一套 PID（功能更全）
  → 两套并存，PID 内部用 GetUs() 获取时间，算法层依赖硬件定时器

HAL-CAR 解决：
  只保留一套 Algorithm/pid_controller.c
  PID 不调用任何 HAL 函数，时间由外部传入 dt_s
  → 可以在 PC 上编译测试，和硬件完全无关
```

**面试话术：**

> CT_bal_car 有三个核心问题。第一是没有分层，控制逻辑直接操作寄存器，换芯片就要全部重写。HAL-CAR 用六层架构隔离了硬件依赖。第二是几乎没有错误处理，传感器失效时用垃圾数据做控制，也没有倾倒保护。HAL-CAR 做了完整的错误传播链和状态机安全设计。第三是 PID 重复且依赖硬件，HAL-CAR 只保留一套纯算法 PID，不依赖任何 HAL 头文件。

### Q15：拿到板子后的验证顺序（十步法）

> [!note] 核心原则
> **先验证硬件事实，再做软件闭环。每一步只引入一个变量，失败时能精确定位是硬件还是软件问题。**

```text
第一步：串口能打印
  → 看到输出：芯片能运行、时钟对、串口引脚对、下载器正常
  → 串口不通，后面所有调试都没法做

第二步：LED 和按键
  → GPIO 输入输出基本功能

第三步：电池电压
  → 万用表量实际电压，对比串口输出
  → 验证分压比和公式里的 8.4V 是否一致

第四步：MPU6050 原始数据
  → 静止时 az ≈ 1g，晃动时 gx/gy/gz 明显变化

第五步：IMU 姿态方向（极其关键）
  → 往前倾斜 → pitch 应增大（或减小，关键是一致）
  → 方向反了，控制输出会让车加速倒下而不是回正
  → 如果方向反了 → 在 IMU 层取反，不要改控制器

第六步：编码器方向
  → 正转 omega 为正，反转 omega 为负

第七步：电机开环测试（架空）
  → 左电机正转时，左编码器读数为正 → 方向一致
  → 这两个必须配对，否则闭环必定发散

第八步：电机速度闭环（架空）
  → 给小目标速度，轮子平稳达到并稳定

第九步：手持闭环（绝对不放地上）
  → 前倾时轮子往前转（追重心）
  → 方向反了立刻按键停止

第十步：落地闭环调参
  → 小 duty 限幅开始（±30%），逐步放开
```

> [!warning] 易错点：跳步验证
> 跳过任何一步，后面的失败原因都无法定位。最常见的错误是跳过第五步和第七步直接闭环——pitch 方向反了电机就猛转，编码器方向不对速度环就发散。

**面试话术：**

> 拿到板子后我不会直接闭环。第一步先确认串口能打印，然后逐个验证电池电压、IMU 方向、编码器方向、电机和编码器方向是否配对。这些都是硬件事实，必须在开环下确认。确认完毕后做架空电机速度闭环，然后手持验证平衡方向，最后才落地调参。核心原则是每一步只引入一个变量，失败时能精确定位原因。

## 第八轮：调试思维与 RTOS 意识

### Q16：小车往一个方向漂移，怎么系统性排查？

**从外到内，逐层排除：**

```text
小车往一个方向漂移
  │
  ├─ 静止 pitch ≠ 0 ？ ──→ 是 → IMU 校准问题（gyro bias / pitch bias）
  │                         否 → 继续
  │
  ├─ 左右轮速对称吗？ ──→ 否 → 电机摩擦力或编码器不对称
  │                       是 → 继续
  │
  ├─ theta_ref 在漂移吗？ ──→ 是 → 看实际轮速
  │                            轮速真在跑？→ 轮速反馈方向反了
  │                            轮速没跑？→ 编码器有噪声偏移
  │                          否 → 继续
  │
  ├─ 静止时编码器 = 0？ ──→ 否 → 加死区，|omega| < 阈值时归零
  │                        是 → 继续
  │
  └─ 角度环/角速度环参数问题
     → 降低 Kp，观察漂移是否减弱
     → 减弱了 → 参数整定问题
```

**需要看的关键数据：**

| 数据 | 判断方法 | 指向问题 |
|------|---------|---------|
| 静止时 pitch | 是否接近 0 | gyro bias 没校准 |
| 左右轮速对称性 | 架空给相同目标速度 | 电机/编码器不对称 |
| theta_ref 是否漂移 | 串口持续观察 | 速度环反馈方向或噪声 |
| 编码器静止读数 | 是否为 0 | 编码器噪声偏移，需加死区 |

**面试话术：**

> 我会从传感器源头开始排查。先看静止时 pitch 是否为 0，不为 0 说明 IMU 校准有问题。然后看左右轮速是否对称，不对称说明电机或编码器有差异。再看速度环输出的 theta_ref 是否在持续漂移，如果是，看实际轮速是不是真的在跑——如果没跑但反馈说在跑，就是编码器方向或噪声问题。核心思路是从传感器到控制器逐层排除，每一步只关注一个变量。

### Q17：裸机调度 vs RTOS 调度，迁移时最担心什么？

**最大区别：**

| | 裸机 | RTOS |
|---|---|---|
| 调度方式 | 顺序轮询 + 时间戳门控 | 优先级抢占调度 |
| 实时性 | 无法保证（前面任务超时就影响后面） | 高优先级任务保证响应时间 |
| 共享数据 | 不需要加锁（单线程） | **必须加锁或用消息队列** |
| 阻塞操作 | 一个阻塞全卡死 | 只影响当前线程 |

**迁移时最担心的三个问题：**

**问题一：共享数据的并发访问（最致命）**

```text
裸机时：get_state() 直接读全局变量，没问题
RTOS 时：多个线程可能同时读写
  → 平衡线程读到一半被电机线程抢占，数据不一致
  → __disable_irq() 不够用了，需要 mutex 或双缓冲
```

**问题二：HAL 回调在中断上下文中执行**

```text
裸机时：HAL_GPIO_EXTI_Callback 在 ISR 里直接更新编码器状态，快且简单
RTOS 时：ISR 里不能等 mutex、不能打印
  → ISR 应该只发信号量/事件标志
  → 实际处理在线程上下文完成
```

**问题三：printf 阻塞**

```text
裸机时：printf 用阻塞 UART 发送，无所谓
RTOS 时：在平衡线程里 printf，阻塞期间 5ms 周期被破坏
  → 解决：printf 放最低优先级 telemetry 线程，用消息队列传数据
```

**迁移策略（HLD 中已预留）：**

```text
线程划分：
  imu_thread:      优先级最高，5ms，读 MPU6050 更新姿态
  control_thread:  优先级高，5ms，平衡控制
  motor_thread:    优先级中高，10ms，电机速度控制
  input_thread:    优先级低，按键和命令
  telemetry_thread:优先级最低，串口输出

IPC 设计：
  命令输入 → 消息队列
  共享状态 → 互斥量或临界区
  故障切换 → 事件标志

好消息：Algorithm 和 Control 层不依赖调度方式，迁移只需改 App 层
```

**面试话术：**

> 裸机和 RTOS 最大的区别是调度方式——裸机是顺序轮询，RTOS 是优先级抢占。迁移时我最担心的是共享数据的并发访问问题。裸机下 `get_state()` 直接读全局变量没问题，RTOS 下多个线程可能同时读写，需要加 mutex 或用双缓冲。另外 ISR 里不能做阻塞操作，迁移后应该改为 ISR 只发信号量，实际处理放到线程上下文。好消息是我的算法层和 Control 层本身不依赖调度方式，迁移只需要改 App 层的调度模型。

| 忽略安全设计 | 没提 FAULT 状态和电机停机路径 | 安全设计是加分项，必须提 |
| 不知道自己代码的问题 | 被问到弱点时说不出来 | 主动说"没有看门狗"、"没有遥测输出" |
| 回答太短或太长 | 2 分钟的项目介绍说了 10 秒或 10 分钟 | 练习 2 分钟版本 |
