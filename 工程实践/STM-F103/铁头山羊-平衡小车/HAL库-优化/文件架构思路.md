# HAL-CAR 六层架构复盘

> [!abstract] 实战场景
> HAL-CAR 是从 CT_bal_car（标准外设库）重构到 CubeMX + HAL 的两轮自平衡车工程。最终代码按 BSP → Device → Service → Algorithm → Control → App 六层组织，共 48 个源文件、约 6000 行自定义代码。本文档基于**实际代码**复盘每一层的职责、依赖关系、数据流和初始化链路，不再停留于规划阶段。

> [!note] 快速结论
> - 分层的核心不是"文件放不同目录"，而是**依赖方向必须单向**：上层可以调下层，下层不能调上层。
> - Algorithm 层（PID）不 include 任何 HAL 头文件，可以在 PC 上编译运行——这是分层正确性的试金石。
> - 每个模块遵循统一模式：`init()` 初始化 → `update()` 周期处理 → `get_state()` 暴露快照。上层只通过快照结构体读取状态，不直接碰模块内部变量。
> - 实际代码中有两处已知的分层妥协：`app_input.c` 直接 include 了 `bsp_uart3.h`（注册回调）；`wheel_speed_service.c` 直接 include 了 `bsp_time.h`（跳过 Device 层读时间）。

## 总体架构

### 分层总览

```text
┌─────────────────────────────────────────────┐
│  App        app_state, app_input            │  调度、状态机、输入分发
├─────────────────────────────────────────────┤
│  Control    balance_controller,             │  平衡控制、电机速度控制
│             motor_speed_controller          │
├─────────────────────────────────────────────┤
│  Algorithm  pid_controller                  │  纯数学工具（不碰 HAL）
├─────────────────────────────────────────────┤
│  Service    imu_attitude, wheel_speed,      │  将硬件数据整理为业务状态
│             battery, button, command,       │
│             imu_calibration                 │
├─────────────────────────────────────────────┤
│  Device     mpu6050, encoder_motor,         │  板级器件驱动
│             motor_driver_tb6612             │
├─────────────────────────────────────────────┤
│  BSP        bsp_uart, bsp_uart3, bsp_i2c,  │  MCU 外设封装
│             bsp_pwm, bsp_gpio, bsp_button,  │
│             bsp_encoder_exti, bsp_time,     │
│             bsp_adc                         │
├─────────────────────────────────────────────┤
│  Core/HAL   CubeMX 生成的初始化和 HAL 驱动   │
└─────────────────────────────────────────────┘
```

### 分层职责与文件清单

| 层级 | 文件数 | 职责 | 关键规则 |
|------|:------:|------|---------|
| **BSP** | 9 对 (.h+.c) | 封装 HAL API，对外设做最小功能封装 | 可以调用 HAL API 和 CubeMX 生成的句柄 |
| **Device** | 3 对 | 封装具体芯片的通信协议和功能语义 | 调用 BSP，不直接调 HAL |
| **Service** | 6 对 | 将硬件数据整理为系统可用的业务状态 | 调用 Device，必要时调用 BSP |
| **Algorithm** | 1 对 | 纯数学算法，平台无关 | **不允许 include HAL 头文件** |
| **Control** | 2 对 | 控制策略，消费 Service 和 Algorithm | 不直接调用 HAL 外设 API |
| **App** | 2 对 | 初始化编排、状态机、周期调度、安全 | 负责调度和运行许可 |

### 实际依赖图

```text
main.c
 ├──→ app_input
 │      ├──→ app_state          (App)
 │      ├──→ command_service    (Service)
 │      ├──→ button_service     (Service)
 │      └──→ bsp_uart3          (BSP) ← 分层妥协：注册回调
 │
 └──→ app_state
        ├──→ balance_controller (Control)
        │      ├──→ pid_controller            (Algorithm)
        │      ├──→ imu_attitude_service      (Service)
        │      ├──→ wheel_speed_service       (Service)
        │      └──→ motor_speed_controller    (Control)
        │             ├──→ pid_controller            (Algorithm)
        │             ├──→ wheel_speed_service       (Service)
        │             ├──→ battery_service           (Service)
        │             └──→ motor_driver_tb6612       (Device)
        │                    ├──→ bsp_pwm   (BSP)
        │                    ├──→ bsp_gpio  (BSP)
        │                    └──→ main.h    (Core)
        ├──→ motor_speed_controller (Control)
        ├──→ wheel_speed_service    (Service)
        │      ├──→ encoder_motor   (Device)
        │      │      └──→ bsp_encoder_exti (BSP)
        │      └──→ bsp_time        (BSP) ← 分层妥协：直接读时间
        └──→ bsp_time               (BSP)
```

> [!warning] 两处分层妥协
> 1. `app_input.c` 直接 include `bsp_uart3.h`——因为 UART3 字节接收回调注册没有对应的 Service 封装。更理想的做法是在 `command_service` 内部暴露注册接口。
> 2. `wheel_speed_service.c` 直接 include `bsp_time.h`——因为需要微秒级时间戳做 M-T 测速，Device 层的 `encoder_motor` 不提供时间信息。更理想的做法是在 `encoder_motor` 快照中包含时间戳。

## 初始化链路

> [!note] 初始化原则
> 先准备底层输入输出，再启动依赖这些输入输出的控制逻辑。初始化链路中任何一步失败都不应该让电机输出。

### main.c 实际执行顺序

```text
main()
 │
 │── HAL_Init()
 │── SystemClock_Config()                // HSE 8MHz → PLL ×9 → 72MHz
 │
 │── CubeMX 外设初始化（按 .ioc 配置顺序）
 │    MX_GPIO_Init()                     // 所有 GPIO：电机、LED、按键、编码器
 │    MX_USART2_UART_Init()             // 调试串口，921600 baud
 │    MX_TIM1_Init()                     // 左电机 PWM，20kHz
 │    MX_TIM4_Init()                     // 右电机 PWM，20kHz
 │    MX_ADC1_Init()                     // 电池 ADC，TIM3 触发
 │    MX_TIM3_Init()                     // ADC 触发定时器，100Hz
 │    MX_I2C1_Init()                     // MPU6050 I2C，PB8/PB9
 │    MX_USART3_UART_Init()             // 遥控串口，9600 baud
 │    MX_TIM2_Init()                     // 系统微秒时间基准
 │
 │── bsp_time_init()                     // 启动 TIM2 中断，开始计时
 │── bsp_uart2_send_string("[BOOT]")     // 串口打印启动信息
 │
 │── app_state_init()                    // 控制链路初始化（见下方展开）
 │── app_input_init()                    // 输入链路初始化（见下方展开）
 │
 └── while(1)
      ├── app_input_update()             // 轮询按键 + 解析命令
      └── app_state_update()             // 状态机 + 周期控制任务
```

### app_state_init() 展开链路

```text
app_state_init()
 │
 ├── balance_controller_init()
 │    │
 │    ├── imu_attitude_service_init()
 │    │    └── mpu6050_init()
 │    │         ├── bsp_i2c_init()            // 检查 I2C 句柄
 │    │         ├── WHO_AM_I 校验 (0x68)
 │    │         ├── 软件复位 (0x6B = 0x80)
 │    │         └── 寄存器配置
 │    │              0x6B=0x01  唤醒，陀螺仪时钟
 │    │              0x19=0x04  采样分频 200Hz
 │    │              0x1A=0x02  DLPF 94Hz
 │    │              0x1B=0x18  gyro ±2000dps
 │    │              0x1C=0x00  accel ±2g
 │    │
 │    └── motor_speed_controller_init()
 │         ├── wheel_speed_service_init()
 │         │    └── encoder_motor_init()
 │         │         ├── bsp_encoder_exti_init() (LEFT)    // PB14 EXTI
 │         │         └── bsp_encoder_exti_init() (RIGHT)   // PB3  EXTI
 │         │
 │         ├── battery_service_init()
 │         │    └── bsp_adc_init()
 │         │         ├── ADC 校准
 │         │         └── 启动 TIM3 触发转换
 │         │
 │         └── motor_driver_tb6612_init()
 │              ├── STBY = LOW（电机默认关闭）
 │              ├── IN1/IN2 全部 LOW
 │              └── bsp_pwm_stop()（PWM 默认不输出）
 │
 ├── balance_controller_stop()           // 复位所有 PID，清零目标
 ├── motor_speed_controller_stop()       // 复位电机 PID，关闭 PWM
 └── s_app_state = APP_STATE_READY       // 进入 READY 状态
```

### app_input_init() 展开链路

```text
app_input_init()
 │
 ├── command_service_init()              // 清空命令缓冲区，复位超时
 ├── bsp_uart3_set_rx_callback()         // 注册：USART3 收到字节 → command_service_receive_byte
 ├── bsp_uart3_start_receive_it()        // 启动 USART3 中断接收
 │
 └── button_service_init()
      └── PA11，低电平有效，默认消抖参数
```

**工程结论：** 初始化链路体现了"自底向上"原则——先让每个底层模块就绪，再组装上层。`app_state_init()` 返回后系统处于 READY 状态，电机保持停止，直到用户按按键才进入 RUNNING。

## BSP 层

> [!note] BSP 层设计原则
> BSP 只做一件事：**把 HAL API 的复杂性藏在简单函数后面**。它不知道 MPU6050 是什么，不知道 TB6612 是什么，只知道"往某个 UART 发数据""设置某个 TIM 的 PWM 占空比"。

### 模块清单

| 模块 | 文件 | 职责 | 关键 API |
|------|------|------|---------|
| bsp_uart | .h/.c | USART2 阻塞发送 + printf 重定向 | `send_data()`, `send_string()` |
| bsp_uart3 | .h/.c | USART3 中断接收 + 注册回调 | `start_receive_it()`, `set_rx_callback()`, `send_data()` |
| bsp_i2c | .h/.c | I2C1 内存读写 + 7 位地址转换 | `mem_read()`, `mem_write()`, `is_device_ready()` |
| bsp_pwm | .h/.c | TIM PWM 启停 + 占空比设置 | `start()`, `stop()`, `set_duty_percent()` |
| bsp_gpio | .h/.c | GPIO 读写翻转 | `write()`, `read()`, `toggle()` |
| bsp_button | .h/.c | GPIO 按键电平读取 | `is_pressed()` |
| bsp_encoder_exti | .h/.c | EXTI 编码器边沿计数 + 方向 + 毛刺过滤 + 快照 | `on_exti()`, `get_snapshot()`, `reset_count()` |
| bsp_time | .h/.c | TIM2 微秒时间基准 + 溢出累积 | `get_us()`, `elapsed_us()`, `on_tim_period_elapsed()` |
| bsp_adc | .h/.c | ADC1 校准 + TIM3 触发转换 + 原始值保存 | `init()`, `get_battery_raw()`, `on_conversion_complete()` |

### BSP 中断回调注册关系

```text
CubeMX 生成的 ISR              BSP 回调函数                    上层消费者
─────────────────              ────────────                    ──────────
EXTI3_IRQHandler      ──→      HAL_GPIO_EXTI_Callback   ──→   bsp_encoder_exti_on_exti(RIGHT)
EXTI15_10_IRQHandler  ──→      HAL_GPIO_EXTI_Callback   ──→   bsp_encoder_exti_on_exti(LEFT)
ADC1_2_IRQHandler     ──→      HAL_ADC_ConvCpltCallback ──→   bsp_adc_on_conversion_complete()
TIM2_IRQHandler       ──→      HAL_TIM_PeriodElapsedCallback ──→ bsp_time_on_tim_period_elapsed()
USART3_IRQHandler     ──→      HAL_UART_RxCpltCallback  ──→   bsp_uart3 回调 ──→ command_service_receive_byte()
```

**工程结论：** 所有 HAL 回调都在 BSP 层内部分发，上层只通过 BSP 提供的函数获取数据。`bsp_encoder_exti` 是 BSP 中最复杂的模块（348 行），因为它需要处理双边沿检测、方向判断、毛刺过滤和原子快照。

## Device 层

> [!note] Device 层设计原则
> Device 层知道"某个芯片怎么用"，但不知道"系统要拿它做什么"。它把 BSP 提供的原始外设能力组合成芯片级语义操作。

### 模块清单

| 模块 | 文件 | 职责 | 依赖 BSP |
|------|------|------|---------|
| mpu6050 | .h/.c (296 行) | MPU6050 初始化、寄存器读写、原始数据读取、单位换算 | bsp_i2c |
| encoder_motor | .h/.c (125 行) | 编码器方向修正（左轮 +1、右轮 -1）、快照透传 | bsp_encoder_exti |
| motor_driver_tb6612 | .h/.c (204 行) | TB6612 方向控制 + PWM duty 输出 + STBY 使能 | bsp_pwm, bsp_gpio |

### MPU6050 关键配置

```text
设备地址：0x68（7 位）
加速度：±2g        → 灵敏度 16384 LSB/g
陀螺仪：±2000 dps  → 灵敏度 16.4 LSB/(deg/s)
采样率：200Hz（SMPLRT_DIV = 4）
DLPF：94Hz（CONFIG = 0x02）
```

### TB6612 方向逻辑

```text
duty > 0（正转）：IN1=1, IN2=0
duty < 0（反转）：IN1=0, IN2=1
duty = 0（刹车）：IN1=0, IN2=0
STBY=1 使能，STBY=0 休眠（所有输出关闭）
```

### encoder_motor 方向修正

```text
为什么需要方向修正？
  左编码器和右编码器安装方向镜像对称
  同样"正转"，左编码器 count++，右编码器 count--
  → Device 层乘以 sign 系数（LEFT=+1, RIGHT=-1），让上层统一理解
```

**工程结论：** Device 层的核心价值是**隐藏芯片细节**。上层只需要调 `mpu6050_read_data()` 拿到 `g` 和 `deg/s`，不需要知道 I2C 地址 0x68、寄存器 0x3B-0x48、高低字节拼接这些事。

## Service 层

> [!note] Service 层设计原则
> Service 层把 Device 输出的原始硬件数据整理成**系统可用的业务状态**。它的输入是硬件事实，输出是业务语义。

### 模块清单

| 模块 | 文件 | 职责 | 依赖 | 更新周期 |
|------|------|------|------|---------|
| imu_attitude_service | .h/.c (210 行) | 互补滤波姿态估计（pitch/roll/yaw） | mpu6050 | 5ms |
| imu_calibration_service | .h/.c (157 行) | 陀螺仪零偏 + pitch 零偏校准 | mpu6050, imu_attitude_service | 采集阶段 |
| wheel_speed_service | .h/.c (310 行) | M-T 混合测速，输出 rad/s 和 m/s | encoder_motor, bsp_time | 5ms |
| battery_service | .h/.c (102 行) | ADC → 电池电压换算 + 低压判断 | bsp_adc | 10ms |
| button_service | .h/.c (277 行) | 消抖 + 多击 + 长按检测状态机 | bsp_button | 主循环每次 |
| command_service | .h/.c (268 行) | UART 字节组包 + move 命令解析 + 超时归零 | — | 主循环每次 |

### 各模块输入输出

```text
imu_attitude_service
  输入：ax(g) ay(g) az(g) gx(dps) gy(dps) gz(dps)（来自 mpu6050）
  处理：互补滤波（alpha=0.95238，约 100ms 时间常数）
  输出：pitch(deg) roll(deg) yaw(deg) gx(dps) gz(dps) valid 标志
  ⚠️ 内部用 deg，Control 层需要时再转 rad

wheel_speed_service
  输入：编码器快照（delta_count + 边沿周期）（来自 encoder_motor）
  处理：M-T 混合测速（delta_count≥3 用 M 法，否则用 T 法，超时归零）
  输出：左右轮 omega(rad/s)、linear_speed(m/s)、angle(rad)
  硬件常数：22 counts/rev，减速比 30613/1500≈20.41，轮半径 0.032m

battery_service
  输入：ADC 原始值（来自 bsp_adc）
  处理：voltage = raw / 4095 × 8.4
  输出：voltage_v, valid, low(<6.0V)

button_service
  输入：GPIO 电平（来自 bsp_button）
  处理：四步状态机（抖动检测 → 消抖确认 → 稳定判断 → 事件产生）
  输出：PRESSED / RELEASED / CLICK(n) / LONG_PRESS_START / LONG_PRESS_REPEAT

command_service
  输入：USART3 字节流（中断中调用 receive_byte）
  处理：组包成行 → 解析 "move <turn> <speed>" → 限幅 [-100,100]
  输出：turn_cmd, speed_cmd, 超时 500ms 自动归零
```

### wheel_speed_service M-T 混合测速逻辑

```text
每个 5ms 周期：
  1. 从 encoder_motor 拿快照（delta_count + last_edge_us + edge_period_us）
  2. delta_count >= 3？
     → 是：M 法，omega = delta_count × 换算系数 / period_s
     → 否：有边沿数据？
          → 是：T 法，omega = 换算系数 / edge_period_s
          → 否：超过 200ms 没新边沿？
               → 是：强制归零
               → 否：保持上次值
  3. omega 限幅 ±80 rad/s
```

> [!warning] 易错点：M 法和 T 法的切换边界
> 阈值 `delta_count >= 3` 是经验值。如果编码器线数很少或控制周期很长，这个阈值可能需要调整。M 法在低速时分辨率差（5ms 内可能 0 个脉冲），T 法在高速时中断太频繁——切换边界就是两者都"还行"的区域。

## Algorithm 层

> [!note] Algorithm 层设计原则
> Algorithm 层是**纯数学工具库**。它不知道自己在控制平衡车，不知道 MPU6050 是什么，不知道 PWM 是什么。它唯一的工作是：给定 setpoint、feedback 和 dt，计算 output。

### 模块清单

| 模块 | 文件 | 职责 | HAL 依赖 |
|------|------|------|---------|
| pid_controller | .h/.c (218 行) | 位置式 PID + 积分限幅 + 输出限幅 + reset | **无（只 include stdint.h）** |

### PID API 设计

```c
typedef struct {
    float kp, ki, kd;
    float integral;
    float integral_limit;
    float last_error;
    float output_min, output_max;
    uint8_t initialized;
} PidController_t;

// 初始化：设定参数和限幅
void pid_controller_init(PidController_t* pid, ...);

// 更新：传入 setpoint、feedback、dt_s，返回 output
float pid_controller_update(PidController_t* pid, float setpoint, float feedback, float dt_s);

// 复位：清零积分和上次误差
void pid_controller_reset(PidController_t* pid);
```

**分层正确性验证：**

```text
pid_controller.h 的 #include 列表：
  └── <stdint.h>    ← 只有标准 C 类型，没有任何 HAL 头文件

可以在 PC 上编译：
  gcc -c pid_controller.c -o pid_controller.o    ← 直接通过
```

**工程结论：** Algorithm 层是分层架构的"试金石"。如果 PID 模块里出现了 `HAL_GetTick()` 或 `TIM2->CNT`，说明分层失败了。时间必须由外部通过 `dt_s` 参数传入，PID 自己不获取时间。

## Control 层

> [!note] Control 层设计原则
> Control 层知道自己在控制什么（平衡车），但不知道硬件怎么操作。它消费 Service 输出的业务状态，调用 Algorithm 做计算，输出控制目标。

### 模块清单

| 模块 | 文件 | 职责 | 依赖 | 更新周期 |
|------|------|------|------|---------|
| balance_controller | .h/.c (302 行) | 四级级联 PID：速度→角度→角速度→转向 | pid_controller, imu_attitude_service, wheel_speed_service, motor_speed_controller | 5ms |
| motor_speed_controller | .h/.c (261 行) | 左右电机速度 PID + 电池电压补偿 | pid_controller, wheel_speed_service, battery_service, motor_driver_tb6612 | 10ms |

### balance_controller 控制链路

```text
输入：
  speed_cmd, turn_cmd（来自 command_service，经 app_state 传递）
  pitch(deg), gx(dps), gz(dps)（来自 imu_attitude_service）
  left_omega, right_omega, car_speed（来自 wheel_speed_service）

控制链路：
  speed_cmd → 车速目标(m/s)
  speed_error → [速度环 PID] → target_accel → atan(accel/g) → theta_ref
  theta_ref - pitch → [角度环 PID] → theta_dot_ref
  theta_dot_ref - gx → [角速度环 PID] → target_ddalpha → 动力学换算 → omega_ref
  turn_cmd → [转向 PID] → omega_diff
  左轮目标 = -omega_ref + omega_diff
  右轮目标 = -omega_ref - omega_diff

输出：
  左右轮目标角速度 → 传给 motor_speed_controller
```

### motor_speed_controller 电压补偿

```text
输入：
  left_target_omega, right_target_omega（来自 balance_controller）
  left_omega, right_omega（来自 wheel_speed_service）
  battery_voltage_v（来自 battery_service）

链路：
  omega_error → [PID] → target_voltage_v
  duty_percent = target_voltage_v / battery_voltage_v × 100
  duty 限幅 ±100%

安全：
  电池无效 → 不输出
  电池低压(<6.0V) → 不输出
  TB6612 更新失败 → 不输出
```

### PID 参数表

| PID | Kp | Ki | Kd | 输出限幅 | 积分限幅 | 含义 |
|-----|:--:|:--:|:--:|---------|---------|------|
| 速度环 | 10 | 1 | 0 | ±10° (theta_ref) | — | 车速误差→目标倾角 |
| 角度环 | 4 | 0 | 0 | — | — | 倾角误差→目标角速度 |
| 角速度环 | 20 | 20 | 0 | — | — | 角速度误差→轮速参考 |
| 转向环 | 1 | 0 | 0 | — | — | yaw 角速度误差→差速 |
| 左电机 | 0.5 | 7 | 0 | ±8.2V | 10 | 轮速误差→目标电压 |
| 右电机 | 0.5 | 7 | 0 | ±8.2V | 10 | 轮速误差→目标电压 |

**工程结论：** Control 层的设计让每个控制器只关心自己的输入输出，不直接读传感器寄存器也不直接写 PWM。如果要换电机驱动芯片（比如从 TB6612 换成 DRV8833），只需要改 Device 层和 BSP 层，Control 层的 PID 逻辑完全不变。

## App 层

> [!note] App 层设计原则
> App 层做三件事：**编排初始化**、**运行调度**、**管理状态**。它不做控制计算，不做传感器处理，只决定"什么时候做什么"。

### 模块清单

| 模块 | 文件 | 职责 |
|------|------|------|
| app_state | .h/.c (246 行) | 状态机（INIT/READY/RUNNING/FAULT）+ 5ms/10ms 周期调度 + 超时检测 |
| app_input | .h/.c (166 行) | 按键事件分发 + UART 命令传递 |

### 状态机

```text
INIT ──(init 成功)──→ READY
INIT ──(init 失败)──→ FAULT

READY ──(按键单击)──→ RUNNING
RUNNING ──(按键单击)──→ READY
RUNNING ──(任意故障)──→ FAULT
FAULT ──(按键长按)──→ READY
```

**各状态行为：**

| 状态 | 电机 | 控制环 | 说明 |
|------|------|--------|------|
| READY | 停止（反复调 stop） | 不运行 | 等待用户启动 |
| RUNNING | 正常输出 | 5ms 平衡 + 10ms 电机 | 正常工作 |
| FAULT | 强制停止 | 强制停止 | 不会自动恢复 |

### FAULT 触发条件

| 触发条件 | 检测位置 | 阈值 |
|---------|---------|------|
| 车体倾倒 | balance_controller | pitch > 80° |
| IMU 数据无效 | balance_controller | mpu6050 读取失败 |
| 平衡任务超时 | app_state | > 8ms（预期 5ms） |
| 电机任务超时 | app_state | > 15ms（预期 10ms） |
| 电池异常 | motor_speed_controller | 无效或 < 6.0V |

### app_input 按键映射

| 事件 | 条件 | 动作 |
|------|------|------|
| 单击 | READY 状态 | → RUNNING（启动控制） |
| 单击 | RUNNING 状态 | → READY（停止控制） |
| 长按 | FAULT 状态 | → READY（清除故障） |
| 双击 | 任意状态 | 预留：IMU 校准（未接入） |

### 周期调度

```text
主循环每次：
  app_input_update()
    → button_service_update()     // 消抖 + 事件检测
    → command_service_update()    // 命令解析 + 超时检查

app_state_update()：
  RUNNING 状态下：
    每 5ms：
      → wheel_speed_service_update()     // 更新轮速
      → balance_controller_update()      // 平衡控制
          → imu_attitude_service_update() // 更新姿态
          → PID 链路
          → motor_speed_controller_set_target()  // 设置轮速目标
    每 10ms：
      → motor_speed_controller_update()  // 电机速度控制
          → battery_service_update()      // 更新电池电压
          → 电机 PID → duty → TB6612
```

## 运行时数据流

### 5ms 平衡控制链路

```text
app_state_update() [每 5ms]
  │
  ├── wheel_speed_service_update()
  │    └── encoder_motor_get_snapshot()
  │         └── bsp_encoder_exti_get_snapshot()  ← ISR 已更新 count/direction/edge
  │    └── M-T 混合测速 → left_omega_rad_s, right_omega_rad_s, car_speed_m_s
  │
  └── balance_controller_update(dt_s)
       │
       ├── imu_attitude_service_update()
       │    └── mpu6050_read_data()
       │         └── bsp_i2c1_mem_read()  ← 阻塞 I2C，约 1-2ms
       │    └── 互补滤波 → pitch_deg, gx_dps, gz_dps
       │
       ├── 读取 wheel_speed_service_get_state()
       │
       ├── 速度环 PID → theta_ref
       ├── 角度环 PID → theta_dot_ref
       ├── 角速度环 PID + 动力学换算 → omega_ref
       ├── 转向 PID → omega_diff
       ├── 左右轮混合 → left_target, right_target
       │
       └── motor_speed_controller_set_target(left_target, right_target)
```

### 10ms 电机速度控制链路

```text
app_state_update() [每 10ms]
  │
  └── motor_speed_controller_update(dt_s)
       │
       ├── wheel_speed_service_get_state()  → left_omega, right_omega
       ├── battery_service_update()
       │    └── bsp_adc_get_battery_raw()   ← ISR 已更新
       │    └── voltage = raw / 4095 × 8.4
       │
       ├── 左右电机 PID → target_voltage_v
       ├── duty = target_voltage / battery_voltage × 100
       ├── duty 限幅 ±100%
       │
       └── motor_driver_tb6612_set_duty()
            ├── 方向 GPIO 设置
            └── bsp_pwm_set_duty_percent()   ← 写 TIM CCR 寄存器
```

### 中断驱动链路

```text
EXTI3 / EXTI15_10（编码器边沿）
  → HAL_GPIO_EXTI_Callback()
  → bsp_encoder_exti_on_exti()
  → 更新 count、direction、edge_period_us、last_edge_us

TIM2 溢出（每 65.536ms）
  → HAL_TIM_PeriodElapsedCallback()
  → bsp_time_on_tim_period_elapsed()
  → s_overflow_us += 65536

ADC1 转换完成（TIM3 触发，约 100Hz）
  → HAL_ADC_ConvCpltCallback()
  → bsp_adc_on_conversion_complete()
  → 保存 raw 值

USART3 字节接收
  → HAL_UART_RxCpltCallback()
  → bsp_uart3 回调
  → command_service_receive_byte()
  → 组装命令行，遇到 '\n' 设置 pending 标志
  → 自动重开下一次接收
```

**工程结论：** 整个系统的数据流是**中断写入 → 主循环读取 → 控制计算 → 主循环输出**。中断只做最少的"取走数据"工作，所有计算和决策都在主循环中完成。

## 分层规则与实际遵守情况

### 六条分层规则

| 规则 | 说明 | 实际遵守 |
|------|------|:--------:|
| BSP 可以调用 HAL API | BSP 是 HAL 的唯一消费者 | ✅ |
| Device 调用 BSP，不直接调 HAL | Device 通过 BSP 间接使用外设 | ✅ |
| Service 调用 Device，必要时调 BSP | Service 消费 Device 输出 | ⚠️ wheel_speed_service 直接调 bsp_time |
| Algorithm 不允许调 HAL | PID 不碰任何硬件 | ✅ 只 include stdint.h |
| Control 消费 Service 和 Algorithm | 控制器不直接操作硬件 | ✅ |
| App 负责调度和状态管理 | App 不做计算 | ⚠️ app_input 直接调 bsp_uart3 |

### 值得记录的设计权衡

**权衡一：app_input 直接 include bsp_uart3.h**

```text
原因：UART3 字节接收回调需要注册到 BSP 层
     command_service 只处理完整命令，中间缺一层"字节接收服务"
理想方案：command_service 内部暴露注册接口，app_input 只调 Service
当前方案：务实妥协，影响有限
```

**权衡二：wheel_speed_service 直接 include bsp_time.h**

```text
原因：M-T 测速需要微秒级时间戳
     encoder_motor（Device 层）的快照不包含时间信息
理想方案：在 encoder_motor_get_snapshot() 中包含时间戳字段
当前方案：直接调 bsp_time_get_us()，跳过了 Device 层
```

## 与 CT_bal_car 的对比复盘

> [!tip] 复盘视角
> 以下对比不是"谁好谁坏"，而是"从旧工程到新工程，哪些设计决策改变了、为什么改变"。

### 架构对比

| 维度 | CT_bal_car | HAL-CAR |
|------|-----------|---------|
| **分层** | user/ + my_lib/ 两层，职责混杂 | BSP → App 六层，单向依赖 |
| **PID 依赖** | PID 内部调 `GetUs()`，依赖硬件定时器 | PID 接受外部 `dt_s`，纯算法 |
| **错误传播** | MPU6050 失败无检查，用垃圾数据 | IMU 无效 → 控制拒绝输出 → FAULT |
| **安全** | 无倾倒保护、无任务超时 | 倾倒 80° + 任务超时 + 电池保护 |
| **测速** | 纯 T 法（边沿间隔） | M-T 混合（高速 M 法 + 低速 T 法） |
| **电机输出** | 直接操作 TIM 寄存器 | TB6612 Device 层 + bsp_pwm 封装 |
| **命令** | ISR 里 strcpy，无缓冲区保护 | 三段缓冲 + 长度保护 + 超时归零 |

### 典型拆分：app_mpu6050.c → 三个模块

```text
CT_bal_car 的 app_mpu6050.c（一个文件 240 行）：
  ├── I2C GPIO 初始化               → BSP/bsp_i2c.c
  ├── I2C 外设初始化                 → CubeMX 生成
  ├── MPU6050 寄存器读写             → Device/mpu6050.c
  ├── 原始数据读取 + 单位换算        → Device/mpu6050.c
  └── 互补滤波 + pitch/roll/yaw     → Service/imu_attitude_service.c

HAL-CAR 拆成三个文件，每个文件职责单一：
  BSP/bsp_i2c.c        (155 行)  只管 I2C 收发
  Device/mpu6050.c      (296 行)  只管 MPU6050 芯片操作
  Service/imu_attitude_service.c (210 行)  只管姿态解算
```

### 典型拆分：app_pwm.c + app_motor.c → 三个模块

```text
CT_bal_car：
  app_pwm.c (248 行)  方向 GPIO + PWM + STBY
  app_motor.c (95 行)  电机 PID + 电池补偿

HAL-CAR：
  BSP/bsp_pwm.c              (70 行)   只管 TIM PWM 占空比
  Device/motor_driver_tb6612.c (204 行)  只管 TB6612 芯片操作
  Control/motor_speed_controller.c (261 行) 只管电机速度 PID + 电池补偿
```

**工程结论：** 拆分的收益不是代码量变化，而是**改一个地方不会影响其他地方**。换电机驱动芯片只改 Device 层，调 PID 参数只改 Control 层，改 I2C 引脚只改 CubeMX + BSP 层。

## 调试和排错

| 现象 | 优先怀疑 | 验证动作 |
|------|---------|---------|
| 串口无输出 | 时钟配置、USART2 引脚、波特率 | 检查 .ioc 中 USART2 配置 |
| MPU6050 WHO_AM_I 不对 | I2C 引脚重映射、上拉电阻 | 用 `bsp_i2c1_is_device_ready()` 扫描 |
| pitch 方向反了 | 安装方向和代码假设不一致 | 手动前倾，看 pitch 正负；在 IMU 层取反 |
| 编码器不计数 | EXTI 配置、Pin 定义 | 在 `HAL_GPIO_EXTI_Callback` 打断点 |
| 电机不转 | STBY 引脚、方向 GPIO、PWM 输出 | 先开环给固定 duty 测试 |
| 左右轮速不对称 | 编码器方向修正 sign 系数 | 架空单转一轮，确认 omega 符号 |
| 控制周期不稳定 | I2C 阻塞挤占时间 | 串口打印 `dt_s` 观察抖动 |

> [!tip] 分层排错法
> 出问题时从底层开始排除：先确认硬件（万用表/示波器），再看 BSP（串口打印 raw 值），再看 Device（单位换算对不对），再看 Service（滤波/测速结果合理吗），最后才看 Control（PID 参数）。不要一上来就怀疑 PID 参数。
