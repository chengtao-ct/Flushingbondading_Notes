核心思想一句话：**旧工程是按功能文件组织，新工程要按职责层级组织。**

也就是说，旧工程里一个文件经常同时做了“外设初始化 + 数据处理 + 算法 + 业务决策”；HAL 重构时要把它拆开。

## 分层对应关系

| 新 HAL 分层    | 负责什么                                       | 对应旧标准库文件                                                                      |
| ----------- | ------------------------------------------ | ----------------------------------------------------------------------------- |
| BSP 层       | 只管 STM32 外设本身，比如 GPIO、UART、I2C、ADC、TIM、PWM | app_usart2.c、app_pwm.c、app_battry.c、app_mpu6050.c、app_encoder.c、delay.c、i2c.c |
| Device 层    | 只管具体硬件器件怎么用，比如 MPU6050、TB6612、电机驱动、编码器采集   | app_mpu6050.c、app_pwm.c、app_encoder.c                                         |
| Service 层   | 把硬件数据整理成系统可用状态，比如姿态、电池状态、轮速、遥控命令           | app_mpu6050.c、app_battry.c、app_encoder.c、app_rc.c、app_button.c                |
| Algorithm 层 | 纯算法，不碰 HAL，比如 PID、平衡控制、电机速度控制、限幅           | app_pid.c、app_control.c、app_motor.c                                           |
| App 层       | 初始化顺序、任务调度、状态机、安全保护                        | main.c、app_button.c、部分 app_control.c、部分 app_motor.c                           |

## 关键拆分例子

### 1. app_mpu6050.c

旧工程里它同时做了：

`I2C GPIO 初始化 I2C 外设初始化 MPU6050 寄存器读写 原始数据读取 单位换算 互补滤波 pitch/roll/yaw 保存`

HAL 新架构应该拆成：

`BSP/bsp_i2c.* 只负责 HAL I2C 基础发送接收 Device/mpu6050_driver.* 负责 MPU6050 寄存器读写、读取原始 ax/gx 数据 Service/imu_service.* 负责单位换算、零偏修正、互补滤波、输出 pitch/gyro`

### 2. app_pwm.c + app_motor.c

旧工程里：

`app_pwm.c 负责方向 GPIO、PWM、STBY app_motor.c 负责电机 PID、电池电压补偿、调用 PWM 输出`

新架构应该拆成：

`BSP/bsp_pwm.* 只负责 HAL_TIM_PWM_Start、设置 CCR Device/motor_driver_tb6612.* 负责 TB6612 的 STBY、方向、duty 输出 Algorithm/motor_controller.* 负责目标轮速 -> 电机 PID -> duty`

### 3. app_control.c

旧工程里它直接读：

`app_mpu6050_getpitch() app_encoder_getspeed_l() app_motor_setomega_l()`

这就是强耦合。

新架构里它变成：

`Algorithm/balance_controller.*`

输入：

`SensorSnapshot + ControlCommand + dt`

输出：

`WheelSpeedTarget`

它不允许直接调用 MPU6050、编码器、电机、HAL。

### 4. app_rc.c

旧工程里 USART3 中断接收和命令解析在一起。

新架构拆成：

`BSP/bsp_uart.* 负责 UART 接收 Service/command_service.* 负责缓冲区、命令解析、遥控超时、目标速度更新`

## 你现在应该怎么理解

你现在生成了 CubeMX 工程，接下来不是马上写平衡算法，而是先建立这个骨架：

`App/ BSP/ Device/ Service/ Algorithm/`

然后按这个顺序做：




为什么要分两层？

因为 PWM 是 STM32 外设能力，属于 BSP：

`TIM1 CH1 输出 PWM TIM4 CH1 输出 PWM`

而 TB6612 是具体电机驱动芯片，属于 Device：

`AIN1/AIN2/BIN1/BIN2/STBY 正转、反转、刹车、休眠`




BSP 层：bsp_encoder_exti
只做最底层：
- 初始化 EXTI 由 CubeMX 完成
- 在 HAL_GPIO_EXTI_Callback() 里分发 pin
- 读取 A/B 电平
- 更新 raw count

Device 层：encoder_motor
只关心左右轮：
- left_count
- right_count
- get_count()
- get_delta_count()
- reset_count()

Service 层：wheel_speed_service
以后再做：
- delta_count 转 rpm
- delta_count 转 rad/s
- 滤波
- 编码器异常检测
