# 工程经验：HAL-CAR 模块开发记录

> [!abstract] 实战场景
> 本文档记录 HAL-CAR 六层架构中每个模块的**实际设计决策和踩坑经验**。不讲原理（原理链接到标准库笔记），只记录"做了什么选择、为什么这样做、遇到什么问题"。如果你在做类似的分层重构，或者需要向面试官解释某个设计决策的动机，可以从这里找到参考。

> [!note] 快速结论
> - 电机初始化的目标不是让电机动，而是**保证电机绝对不乱动**。STBY 默认 LOW，IN1/IN2 默认 LOW，PWM 默认不输出。
> - 速度不是编码器中断里算的，而是**周期任务里算的**。中断只做"取走数据"，计算和决策都在主循环完成。
> - PID 不自己获取时间——`dt_s` 由外部传入。这是 Algorithm 层不碰 HAL 的关键设计。
> - 姿态层内部统一用 `deg` 和 `deg/s`，Control 层需要弧度时自己转换。单位边界在层与层之间明确约定。

## 电机驱动（TB6612）

### 核心设计决策：上电默认不动

```text
motor_driver_tb6612_init() 做的事：
  1. STBY = LOW          → 电机驱动芯片休眠，所有输出关闭
  2. IN1/IN2 全部 = LOW  → 即使 STBY 被意外拉高，电机也不会转
  3. bsp_pwm_stop()      → PWM 不输出，CCR = 0

真正让电机动，必须：
  1. motor_driver_tb6612_cmd(1)   → STBY = HIGH，芯片使能
  2. motor_driver_tb6612_set_duty(left, right) → 设置方向和 duty
```

> [!warning] 踩坑：初始化顺序
> CubeMX 的 `MX_GPIO_Init()` 会把所有配置为输出的 GPIO 设为 LOW，但这不等同于"安全的电机状态"。如果 `MX_TIM1_Init()` 在 `MX_GPIO_Init()` 之后启动了 PWM，而 STBY 引脚因为上拉或浮空变成了 HIGH，电机可能在初始化过程中短暂转动。所以 TB6612 的 init 必须显式地先把所有引脚设为安全状态。

### 方向逻辑

```text
duty > 0（正转）：IN1=1, IN2=0  → 电机正转
duty < 0（反转）：IN1=0, IN2=1  → 电机反转
duty = 0（刹车）：IN1=0, IN2=0  → 电机短路制动

左电机硬件映射：TIM1_CH1 (PA8) PWM, PA9=IN1, PA10=IN2
右电机硬件映射：TIM4_CH1 (PB6) PWM, PB5=IN1, PB7=IN2
STBY：PA1

duty 限幅 ±100%，在 motor_driver_tb6612_set_duty() 内部完成
```

### 电压补偿设计

> [!tip] 为什么 duty 用电压除以电压来算
> 电机 PID 输出的单位是**目标电压（V）**，不是直接给 duty。因为同一 duty 在满电和快没电时产生的电机电压不同。用 `target_voltage / battery_voltage × 100%` 换算，可以让控制效果在电量变化时保持一致。这个设计思路来自 [[7.速度环]] 和 bal 参考工程。

## 编码器与轮速（EXTI + M-T 测速）

### 设计决策：中断只计数，周期任务算速度

```text
编码器 ISR（bsp_encoder_exti_on_exti）做的事：
  1. 判断是左还是右（Pin 匹配）
  2. 读 A/B 相电平，判断方向
  3. 毛刺过滤（边沿间隔 < 100us 的忽略）
  4. count++ 或 count--
  5. 记录边沿时间戳 last_edge_us 和边沿周期 edge_period_us

ISR 不做的事：
  ✗ 不算速度
  ✗ 不调 PID
  ✗ 不打印
  ✗ 不做任何浮点运算
```

**工程结论：** 中断越短越好。中断里算速度需要除法（浮点），Cortex-M3 没有 FPU，一个除法就是几十个时钟周期。把速度计算放到 5ms 周期任务里，中断只做整数加减，系统更稳定。

### M-T 混合测速的阈值选择

```text
M 法（固定周期数脉冲）：
  delta_count >= 3 → 用 M 法
  阈值 3 的来源：编码器线数 22 counts/rev，减速比 20.41
  5ms 内 3 个脉冲约对应 1.2 rad/s，低于这个速度 M 法分辨率太差

T 法（相邻脉冲间隔）：
  delta_count < 3 但有边沿数据 → 用 T 法
  用 edge_period_us（上次边沿到这次边沿的时间）计算速度

停止检测：
  超过 200ms 没有新边沿 → 强制归零
  200ms 的选择：车轮静止后最慢速度约 0.3 rad/s，对应边沿间隔约 150ms
  留 50ms 余量

最大速度限幅：
  ±80 rad/s，超过的认为是异常尖峰
```

> [!warning] 踩坑：reset 后 T 法残留数据
> `bsp_encoder_exti_reset_count()` 最初只清了 count，没有清 direction / last_edge_us / edge_period_us / has_edge / glitch_count。reset 后 T 法会用到旧的边沿周期数据，导致速度跳变。后来补上了：reset 时把对应侧所有快照字段清零。

### 编码器方向修正（encoder_motor Device 层）

```text
为什么需要方向修正？
  左编码器和右编码器物理安装方向镜像对称
  同样正转：左 count++，右 count--

  encoder_motor 的做法：
    left_sign  = +1
    right_sign = -1
    output_count = bsp_count × sign
    output_direction = bsp_direction × sign

  上层拿到的数据：正转都是正数，反转都是负数
```

### 三层编码器链路

```text
BSP 层：bsp_encoder_exti
  EXTI 边沿计数 + 方向检测 + 毛刺过滤 + 时间戳 + 快照
  → 输出：raw_count, direction, edge_period_us

Device 层：encoder_motor
  左右轮 sign 修正 + 快照透传
  → 输出：signed_count, signed_direction, signed_snapshot

Service 层：wheel_speed_service
  M-T 混合测速 + 单位换算
  → 输出：omega(rad/s), linear_speed(m/s), angle(rad)
```

**工程结论：** 三层各管一件事。BSP 不知道"轮子"的概念，只知道"某个 Pin 的 EXTI"；Device 不知道"速度"的概念，只知道"左轮右轮方向修正"；Service 才把硬件事实变成业务数据。

## PID 控制器（Algorithm 层）

### 设计决策：为什么选位置式 PID

| 对比 | 位置式 | 增量式 |
|------|--------|--------|
| 输出含义 | 绝对控制量 | 控制量的增量 |
| 理解难度 | 直观 | 需要理解差分 |
| 输出限幅 | 直接 clamp | 需要累积限幅 |
| 积分项 | 天然存在，需要 anti-windup | 不显式存在 |
| 切换冲击 | 需要手动 reset | 天然无冲击 |
| 适合场景 | 平衡车 duty 输出、温度控制 | 伺服电机、步进 |

> 第一版选位置式：更容易理解，输出可以直接对应 duty_percent，限幅简单。

### 设计决策：dt_s 外部传入

```text
CT_bal_car 的做法（错误）：
  pid_compute() 内部调 GetUs() 获取时间
  → PID 依赖硬件定时器
  → 算法层和硬件耦合
  → 无法在 PC 上编译测试

HAL-CAR 的做法（正确）：
  pid_controller_update(pid, setpoint, feedback, dt_s)
  → dt_s 由调用者传入
  → PID 不知道时间从哪来
  → 可以用固定 dt_s 在 PC 上跑单元测试

验证：
  pid_controller.h 只 include <stdint.h>
  gcc -c pid_controller.c  ← 直接通过
```

### 两个限幅保护

```text
1. integral 限幅（anti-windup）
   |integral| > integral_limit 时 clamp
   → 防止积分项无限累积
   → 比如电机 PID 积分限幅 = 10，对应最大积分电压约 10V

2. output 限幅
   output > output_max 时 clamp 到 output_max
   output < output_min 时 clamp 到 output_min
   → 电机 PID 输出限幅 ±8.2V，对应满量程 duty
   → 平衡控制器角度环输出限幅 ±10°，不允许要求太大倾角
```

> [!tip] 两个限幅的区别
> integral 限幅是**限制积分增长**，防止 windup；output 限幅是**限制最终输出**，保护执行器。两者缺一不可。只有 output 限幅没有 integral 限幅 → 积分持续累积，取消限幅后输出会猛烈跳回。

## MPU6050 驱动（Device 层）

### 寄存器配置清单

```text
地址  值    含义                          选择原因
─────────────────────────────────────────────────────────────
0x6B  0x80  软件复位                      初始化前先复位，确保干净状态
0x6B  0x01  唤醒 + 陀螺仪时钟             用陀螺仪时钟比用内部振荡器精度好
0x19  0x04  SMPLRT_DIV = 4                采样率 = 1kHz/(1+4) = 200Hz
0x1A  0x02  DLPF = 94Hz                   滤掉高频噪声，保留运动信号
0x1B  0x18  GYRO_CONFIG ±2000 dps        平衡车可能快速倾倒，需要大量程
0x1C  0x00  ACCEL_CONFIG ±2g             静态重力 1g，±2g 足够且精度最高
```

### 踩坑：0x1D 寄存器

> [!warning] 旧工程写了 0x1D
> bal 的旧工程还配置了 `0x1D` 寄存器。但 MPU6050 数据手册中 `0x1D` 不是标准核心配置寄存器，更接近 MPU6500/MPU9250 系列的加速度滤波。对 MPU6050 写这个寄存器可能无效或产生未定义行为。HAL-CAR 中去掉了 `0x1D` 的配置。

### 灵敏度系数

```text
加速度 ±2g       → 16384 LSB/g      → accel_g = raw / 16384.0f
陀螺仪 ±2000dps  → 16.4 LSB/(deg/s) → gyro_dps = raw / 16.4f
温度              → raw / 340.0 + 36.53

⚠️ 量程配置变了，系数必须同步变
    比如改成 ±4g，系数就变成 8192 LSB/g
```

### WHO_AM_I 校验

```text
mpu6050_init() 第一步：读 0x75 寄存器，期望返回 0x68
  → 不对：I2C 通信有问题（地址错、引脚错、上拉缺失）
  → 对：继续配置

如果不校验直接配置，后面的所有数据都可能是垃圾
```

## 姿态解算（imu_attitude_service）

> [!note] 原理不在这里重复
> 互补滤波的完整公式推导和物理直觉见 [[4.MPU6050]]。这里只记录 HAL-CAR 中的设计决策。

### alpha 选值和时间常数

```text
alpha = 0.95238
dt = 5ms = 0.005s
时间常数 Tf = alpha × dt / (1 - alpha) ≈ 0.1s

含义：
  < 10Hz（慢变化）→ 加速度计主导（修正漂移）
  > 10Hz（快变化）→ 陀螺仪主导（响应快，不受振动影响）

为什么不用更小的 alpha（更信任加速度计）？
  → 电机振动频率在 20-200Hz，会被加速度计吸收
  → alpha 太小 → pitch 会随电机振动抖动 → 控制不稳定

为什么不用更大的 alpha（更信任陀螺仪）？
  → 陀螺仪零偏漂移修正太慢
  → 长时间运行后 pitch 会慢慢偏离真实值
```

### ±180° 跳变处理

```text
问题：pitch 从 179° 变到 -179°，数学上只差 2°，但数值跳了 358°
     互补滤波会把 358° 的差值放大到输出

处理：
  在融合前先修正 pitch 和 pitch_accel 之间的角度差
  if (pitch - pitch_accel > 180)  pitch -= 360;
  if (pitch_accel - pitch > 180)  pitch += 360;

  → 把 179° 和 -179° 理解为"差 2°"而不是"差 358°"
```

### deg vs rad 的分层决策

```text
imu_attitude_service 内部全部用 deg 和 deg/s
  → 输出：pitch(deg), roll(deg), yaw(deg), gx(dps), gz(dps)

balance_controller 在控制计算时转成 rad 和 rad/s
  → pitch_rad = pitch_deg × PI / 180.0f
  → gx_rad_s = gx_dps × PI / 180.0f

为什么不在 IMU 层直接输出 rad？
  → IMU 层不应该知道控制算法用什么单位
  → deg 是更直觉的角度单位，调试时串口打印 deg 更直观
  → 转换是 Control 层的责任，不是 Service 层的
```

### 首次更新初始化

```text
第一次 update 时 pitch 没有有效初值
  → 用加速度计角度 pitch_accel 直接初始化
  → 避免从 0° 缓慢收敛到真实角度的过渡期

s_first_update 标志位控制，只在第一次有效
```

## 平衡控制器（balance_controller）

> [!note] 原理不在这里重复
> 级联 PID 的完整控制理论和物理推导见 [[6.倒立摆]]。这里只记录工程实现中的关键决策。

### 级联链路 8 步

```text
1. 读取 pitch(deg) gx(dps) gz(dps)    ← imu_attitude_service
2. 读取 car_speed(m/s)                 ← wheel_speed_service
3. 转换单位 deg → rad
4. speed_error → [速度环 PID] → target_accel → atan(accel/g) → theta_ref
5. theta_error → [角度环 PID] → theta_dot_ref
6. theta_dot_error → [角速度环 PID] → ddalpha_ref → 动力学换算 → omega_ref
7. turn_error → [转向 PID] → omega_diff
8. 左右混合：left = -omega_ref + omega_diff, right = -omega_ref - omega_diff
```

### 动力学方程的工程简化

```text
完整方程：a_horizontal = (lp × ddalpha - g × sin(alpha)) / cos(alpha)

工程简化：
  → 平衡车正常工作在 ±15° 以内
  → sin(alpha) ≈ alpha（小角度近似）
  → cos(alpha) ≈ 1
  → 简化后：a_horizontal ≈ lp × ddalpha - g × alpha

但代码中仍保留了 sin/cos 计算，因为：
  → 超过 30° 时近似误差已经明显
  → Cortex-M3 的 sinf/cosf 虽然慢，但 5ms 周期够用
  → 保留完整公式对后续大角度场景更安全
```

### 左右轮混合的符号问题

```text
left  = -omega_ref + omega_diff
right = -omega_ref - omega_diff

为什么 omega_ref 前面有负号？
  → 和电机安装方向有关
  → 实际符号需要在实车验证时确认
  → 如果方向反了，在 Control 层改符号，不要改 Device 层

omega_diff 的正负决定转向：
  → omega_diff > 0 → 左轮加速、右轮减速 → 左转
  → omega_diff < 0 → 右轮加速、左轮减速 → 右转
```

### 倾倒保护

```text
pitch > 80° 时：
  → balance_controller 内部状态切换为 FAULT
  → 清零左右轮目标
  → 通知上层（app_state 也会进入 FAULT）

为什么是 80°？
  → 车体已经完全倒下，继续输出只会让电机空转或猛转
  → 80° 足够大，不会误触发（正常平衡在 ±10° 以内）
  → 但又不会等到 90°（sin/cos 计算会有精度问题）
```

## 电机速度控制器（motor_speed_controller）

### 完整链路

```text
1. wheel_speed_service_get_state()     → left_omega, right_omega
2. 读取 left_target, right_target      ← balance_controller 设置的
3. 左右分别做 PID：
     omega_error → [PID] → target_voltage_v
4. battery_service_update()            → battery_voltage_v
5. 检查电池：无效或 < 6.0V → 停止输出
6. duty = target_voltage_v / battery_voltage_v × 100
7. duty 限幅 ±100%
8. motor_driver_tb6612_set_duty(left_duty, right_duty)
```

### PID 参数选择

```text
Kp = 0.5, Ki = 7.0, Kd = 0.0

为什么 Ki 这么大？
  → 电机速度闭环需要快速消除稳态误差
  → 机械摩擦力是常数干扰，需要积分项来抵消
  → Kd = 0 因为速度反馈本身已经有噪声，微分项会放大噪声

输出范围 ±8.2V
  → 对应电池满电 8.4V 时约 97.6% duty
  → 留一点余量，不跑到 100%

积分限幅 = 10
  → 防止电机堵转时积分无限累积
  → 解除堵转后不会瞬间输出最大电压
```

## 状态机（app_state）

### 四状态设计

```text
INIT    → 系统正在初始化
READY   → 初始化完成，电机停止，等待启动
RUNNING → 控制环运行
FAULT   → 故障，电机强制停止

为什么不用更少的状态？
  → 去掉 INIT：需要区分"正在初始化"和"初始化完成等待启动"
  → 去掉 FAULT：故障后如果直接回 READY，用户可能不知道出过错
  → 去掉 READY：上电就 RUNNING？太危险，电机可能意外转动

为什么 FAULT 不自动恢复？
  → 故障原因可能是硬件问题（IMU 断线、电池没电）
  → 自动恢复可能导致反复进出 FAULT，电机输出抖动
  → 要求用户长按按键确认"我知道出问题了"才恢复
```

### 超时阈值

```text
平衡控制周期：预期 5ms，超时阈值 8ms（60% 余量）
电机控制周期：预期 10ms，超时阈值 15ms（50% 余量）

为什么有余量？
  → 主循环中 app_input_update() 也占用时间
  → I2C 阻塞读取 MPU6050 约需 1-2ms
  → 串口中断偶尔会抢占
  → 余量太小会导致正常工作时误触发 FAULT

为什么不用更宽松的阈值？
  → 阈值太松 → 真正的超时无法及时检测
  → 控制周期不稳定会直接影响 PID 效果
```

## 按键服务（button_service）

### 三阶段消抖状态机

```text
阶段一：抖动检测
  raw_pressed != stable_pressed
  → 记录时间戳和 pending 状态
  → 进入消抖等待

阶段二：消抖确认
  等待 debounce_ms（默认 10ms）后再读一次
  → raw_pressed == pending_pressed → 确认状态变化
  → 稳定后产生 PRESSED 或 RELEASED 事件

阶段三：长按和点击判断
  stable_pressed == 1 且超过 long_press_ms → LONG_PRESS_START
  每 long_repeat_ms → LONG_PRESS_REPEAT
  stable_pressed == 0 且 click_count > 0 且超时 → CLICK(n)
```

### 为什么 CLICK 不是松开时立刻产生

```text
因为要支持双击判断：

第一次松开 → click_count = 1 → 不立刻产生事件
等待 200ms（click_interval_ms）
  → 没有第二次按下 → 确认是单击 → CLICK(1)
  → 200ms 内又按下松开 → click_count = 2 → 确认是双击 → CLICK(2)

RELEASED 和 CLICK 是两个不同事件：
  RELEASED：物理动作，"手指离开了"
  CLICK：语义动作，"确认是几连击"
```

### 事件分发链路

```text
button_service_update() → 返回 ButtonEvent_t
  ↓
app_input_update() 解释事件：
  CLICK(1) + READY    → app_state_start()    → RUNNING
  CLICK(1) + RUNNING  → app_state_stop()     → READY
  LONG_PRESS + FAULT  → app_state_clear_fault() → READY
  CLICK(2) + 任意     → 预留：IMU 校准（未接入）
```

## 命令服务（command_service）

### 设计决策：为什么第一版用单字节中断接收

```text
可选方案：
  1. 轮询接收      → 实时性差，可能丢字节
  2. 单字节中断     → 实时性好，复杂度适中 ✗ 当前选择
  3. DMA + IDLE    → 最优方案，但学习复杂度高

选择单字节中断的原因：
  → 先把"命令闭环"跑通
  → 9600 波特率下每字节约 1ms，中断频率很低
  → DMA + IDLE 留给后续优化
```

### 超时归零的安全意义

```text
超时时间：500ms

为什么需要超时？
  → 如果遥控器断连或串口线脱落
  → 没有超时：最后一次 move 命令会一直生效
  → 小车会持续按最后的目标速度运动 → 失控

超时后的行为：
  speed_cmd = 0
  turn_cmd = 0
  → 速度环的目标车速归零
  → 小车逐渐减速停下（不会急停，因为是通过 PID 缓慢归零）
```

### 缓冲区保护

```text
接收缓冲区：64 字节
保护措施：
  → 累计字节数 < 64 时才写入
  → 超过 64 字节的行直接丢弃（不解析）
  → 遇到 '\n' 才认为一行结束

CT_bal_car 的做法（有风险）：
  → 64 字节缓冲区但没有边界检查
  → 长命令可能越界覆盖其他变量

HAL-CAR 的改进：
  → 增加 overflow 检测
  → 超长行直接丢弃并等待下一个 '\n'
```

### 命令数据流

```text
USART3 中断收到 1 字节
  → bsp_uart3 回调
  → command_service_receive_byte(byte)
      → 放入 rx_line_buffer
      → 遇到 '\n' → 标记 pending = 1

主循环：
  command_service_update(now_ms)
    → pending == 1？
      → 复制到解析缓冲区（关中断保护）
      → sscanf 解析 "move %d %d"
      → 限幅 [-100, 100]
      → 更新 turn_cmd, speed_cmd
      → 更新 last_command_ms
    → now_ms - last_command_ms > 500？
      → turn_cmd = 0, speed_cmd = 0
```

## 时间基准（bsp_time）

### TIM2 溢出累积方案

```text
CubeMX 配置：
  Prescaler = 71  → 72MHz / 72 = 1MHz 计数频率
  Period = 65535   → 每 65536us 溢出一次

溢出处理：
  TIM2 中断 → s_overflow_us += 65536

读取当前时间：
  time_us = s_overflow_us + TIM2->CNT

覆盖范围：
  uint32_t 最大值 ≈ 4294 秒 ≈ 71.6 分钟
  → 对平衡车调试完全够用
```

### 溢出与计数器读数的竞争条件

```text
潜在问题：
  读 s_overflow_us 时是 100000
  读 TIM2->CNT 之前发生了溢出
  s_overflow_us 变成 165536
  读到的 TIM2->CNT 是一个很小的值
  → 结果 = 165536 + 小值 ≈ 165600（跳了一次溢出）

解决：
  读两次 overflow，中间读 CNT
  如果两次 overflow 不同，说明读 CNT 时发生了溢出，重新读

  do {
      ovf1 = s_overflow_us;
      cnt  = TIM2->CNT;
      ovf2 = s_overflow_us;
  } while (ovf1 != ovf2);

  time_us = ovf1 + cnt;
```

> [!warning] 踩坑：不要用 HAL_GetTick() 做控制计时
> `HAL_GetTick()` 精度只有 1ms。PID 的 dt 如果用 1ms 精度，5ms 周期的 dt 可能是 4ms 或 6ms，误差 20%。这对积分项和微分项影响很大。bsp_time 提供的微秒级计时可以让 dt 精确到 us 级别。

## 调度和周期

### 周期选择

```text
平衡控制：5ms（200Hz）
  → 平衡车倾倒速度很快，200Hz 是最低可接受频率
  → 更低：控制跟不上车体运动，无法稳定
  → 更高：CPU 负载增加，I2C 阻塞时间占比变大

电机速度控制：10ms（100Hz）
  → 电机机械响应比车体倾倒慢得多
  → 100Hz 足够
  → 不需要和平衡环同频，减少 CPU 负载

ADC 电池采样：TIM3 触发约 100Hz
  → 电池电压变化缓慢，100Hz 采样远超需求
  → 但硬件自动触发不占 CPU，无所谓

按键扫描：主循环每次（约几十 kHz）
  → 消抖由 button_service 内部管理
  → 扫描频率越高，消抖延迟越低
```

### dt_s 实际传入的设计动机

```text
CT_bal_car 的做法：
  PERIODIC(5) 宏控制周期
  PID 内部用 GetUs() 算 dt
  → dt 是实际间隔，但不透明，外部不知道 PID 用了多少时间

HAL-CAR 的做法：
  app_state 计算 dt_s = (current_us - last_us) / 1000000.0f
  传给 balance_controller_update(dt_s)
  传给 motor_speed_controller_update(dt_s)
  → dt_s 是明确的函数参数
  → 可以在串口打印 dt_s 观察调度是否稳定
  → PID 不需要自己获取时间
```

**工程结论：** `dt_s` 从隐式变成显式，是算法层脱离硬件依赖的关键一步。

## 调试和排错

| 现象 | 优先怀疑 | 验证动作 | 相关模块 |
|------|---------|---------|---------|
| 串口无输出 | 时钟/波特率/接线 | 检查 .ioc USART2 配置 | bsp_uart |
| MPU6050 WHO_AM_I ≠ 0x68 | I2C 引脚/上拉/地址 | `is_device_ready()` 扫描 | bsp_i2c, mpu6050 |
| pitch 静止时不为 0 | gyro bias 未校准 | 静止采样看 pitch 趋势 | imu_attitude_service |
| pitch 方向和直觉相反 | 安装方向和代码假设不一致 | 手动前倾，看 pitch 正负 | imu_attitude_service |
| 编码器不计数 | EXTI 配置/Pin 定义 | 在 HAL_GPIO_EXTI_Callback 打断点 | bsp_encoder_exti |
| 编码器方向反 | sign 系数不对 | 架空单转一轮，看 omega 正负 | encoder_motor |
| 电机不转 | STBY/方向 GPIO/PWM | 先开环给固定 duty 测试 | motor_driver_tb6612 |
| 左右轮速不对称 | sign 系数或减速比 | 架空给相同目标速度 | encoder_motor, wheel_speed_service |
| 控制周期抖动 | I2C 阻塞挤占 | 串口打印 dt_s 观察 | bsp_i2c, app_state |
| 小车往一个方向漂 | IMU 校准/轮速对称/编码器噪声 | 见 [[面试提问]] Q16 | 多个模块 |

> [!tip] 分层排错法
> 出问题时从底层开始排除：硬件（万用表/示波器）→ BSP（串口打印 raw 值）→ Device（单位换算对不对）→ Service（滤波/测速结果合理吗）→ Control（PID 参数）。不要一上来就怀疑 PID 参数，80% 的问题出在传感器方向或单位换算上。
