# 2. CH552G CDC 问题和经验

## 2.1 CH552G 不是普通串口

现象：

```text
CH552G 插电脑时像 COM 口。
ESP32-S3 侧却不是用 UART 直接通信。
```

判断：

```text
CH552G 在这个项目里是 USB CDC-ACM Device。
ESP32-S3 必须作为 USB Host，通过 CDC-ACM Host driver 打开它。
```

层级关系：

```text
ESP32-S3 USB Host Library
    -> CDC-ACM Host Driver
        -> CH552G CDC Device
```

结论：

```text
电脑上的“串口体验”只是 CDC 抽象出来的效果。
底层仍然是 USB 枚举、接口打开、端点传输。
```

## 2.2 CH552G 启动和枚举比较慢

现象：

```text
ESP32-S3 能枚举到 CH552G。
但是 cdc_acm_host_open / ch552_cdc_open 失败。
后来把超时时间拉长后成功。
```

关键配置：

```c
#define APP_CDC_OPEN_TIMEOUT_MS         15000
```

原因：

```text
CH552G 是 MCU。
它上电后要先启动固件，再初始化 USB Device，再等待 Host 枚举。
如果 ESP32-S3 侧等待时间太短，就会认为设备不存在。
```

操作：

```text
1. APP_CDC_OPEN_TIMEOUT_MS 从 5000 改成 15000
2. 重新 build / flash / monitor
3. 再测试 PING / STATUS / L1 ON / L1 OFF
```

结论：

```text
CDC open 失败不一定是协议错。
在 MCU 型 USB Device 上，启动时序很重要。
```

## 2.3 文本命令协议更适合早期调试

当前 CH552G 指令：

```text
PING
STATUS
L1 ON
L1 OFF
L2 ON
L2 OFF
L3 ON
L3 OFF
L4 ON
L4 OFF
L5 ON
L5 OFF
ALL ON
ALL OFF
```

经验：

```text
文本协议比二进制协议更适合第一阶段。
电脑串口助手、VOFA+、ESP32-S3 都可以直接发同一套命令。
```

注意：

```text
响应统一使用 \r\n 结束。
这样 ESP32-S3 可以按行读取，串口工具也容易观察。
```

结论：

```text
第一阶段协议目标不是“高级”，而是“可观察、可验证、可复现”。
```

## 2.4 UART1 桥接要停用

现象：

```text
CH552G 原始 CDC 示例可能带 CDC 转 UART 的逻辑。
但是本项目需要 P1.6 / P1.7 控制 L3 / L4。
```

判断：

```text
如果继续启用 UART1，会占用 P1.6 / P1.7。
这会和五路灯控 GPIO 冲突。
```

结论：

```text
当前 CH552G 固件只做 USB 命令设备。
不再做 CDC 转物理 UART1。
```

# 3. ESP32-S3 USB Host 问题和经验

## 3.1 Host 主动管理总线

核心理解：

```text
Host 主动枚举和访问 Device。
Device 不能主动说“我在这里”。
```

USB Host 基本顺序：

```text
设备插入
    -> Root port 检测
    -> Reset
    -> 读取 Device Descriptor
    -> 读取 Configuration Descriptor
    -> 设置 Configuration
    -> 类驱动 open
    -> 数据传输
```

结论：

```text
只要还没有成功枚举，就不要讨论 CDC 命令、UVC 图像、HTTP 上传。
```

## 3.2 有电不等于能枚举

现象：

```text
设备有灯亮。
电脑/开发板有插入声音。
但是 ESP32-S3 日志没有 USB NEW_DEV event。
```

判断：

```text
有电只说明 VBUS 存在。
枚举需要 D+ / D- 数据链路也正常。
```

正确判断依据：

```text
USB NEW_DEV event
VID / PID
Device descriptor
Configuration descriptor
```

结论：

```text
不要用“灯亮了”判断 USB 链路正常。
必须用枚举日志判断。
```

## 3.3 Root port reset failed 的含义

典型日志：

```text
E (...) HUB: Root port reset failed
```

可能原因：

```text
1. Host 口没有稳定 5V
2. USB 线只有供电没有数据
3. 设备接错口
4. Hub 上行口没有接到 ESP32-S3 Host
5. 设备刚上电，还没准备好
```

操作顺序：

```text
1. 测 Host 口 5V
2. 换一根明确支持数据的 USB 线
3. 直插 Camera 测试
4. 直插 CH552G 测试
5. 再接空 Hub
6. 最后接 Hub + 下游设备
```

结论：

```text
Root port reset failed 是底层链路问题。
不要直接怀疑业务代码。
```

## 3.4 UVC descriptor 太大时要调 menuconfig

现象：

```text
Configuration descriptor larger than control transfer max length
CHECK_SHORT_CONFIG_DESC FAILED
```

判断：

```text
UVC Camera 的 Configuration Descriptor 比普通 USB 设备大。
ESP32-S3 默认控制传输大小不够。
```

操作：

```text
idf.py menuconfig
    -> Component config
        -> USB-OTG
            -> Largest size of transfers to/from default endpoints
```

项目中使用过的经验值：

```text
4096
```

结论：

```text
UVC Camera 不是“小 USB 设备”。
它的描述符更复杂，默认 USB Host 参数可能不够。
```

# 4. FE1.1s Hub 接线和供电经验

## 4.1 Hub 接口角色

这块 FE1.1s Hub 板子的接口要分清楚。

```text
PC Type-C 口
    -> Hub 上行数据口
    -> 接 ESP32-S3 USB HOST

DC5V Type-C 口
    -> 只负责 5V 供电
    -> 接充电宝 / 5V 电源

USB1 ~ USB4
    -> Hub 下行设备口
    -> 接 Camera / CH552G 等 USB Device
```

正确连接：

```text
ESP32-S3 USB HOST -> Hub PC 口
Hub DC5V          -> 5V 电源 / 充电宝
Hub USB1          -> USB Camera
Hub USB2          -> CH552G
ESP32-S3 USB-UART -> 电脑 monitor
ESP32-S3 USB_DEV  -> 电脑 / 供电
```

结论：

```text
PC 口不是给电脑专用的名字。
在本项目里，它是 Hub 的上行口，要接 ESP32-S3 USB HOST。
```

## 4.2 Hub 有灯亮不代表数据链路正常

现象：

```text
Hub 指示灯亮。
但是 ESP32-S3 没有枚举到下游设备。
```

判断：

```text
Hub 指示灯只能说明供电存在。
不能证明 Hub upstream 数据线连接正确。
```

操作：

```text
1. 空 Hub 接 ESP32-S3 Host，先看能否枚举 Hub 本体
2. 再插 Camera
3. 再插 CH552G
4. 最后同时插 Camera + CH552G
```

结论：

```text
Hub 调试必须分阶段。
不要一开始就把所有设备都插上。
```

# 5. ESP32-S3 Host 供电经验

## 5.1 USB-UART 和 USB_DEV 是两回事

现象：

```text
USB-UART 可以 monitor。
但是 USB Host 口不能枚举设备。
```

判断：

```text
USB-UART 只负责串口日志和烧录。
它不一定给 ESP32-S3 USB HOST 的 VBUS 路径供电。
```

项目里的关键动作：

```text
board_host_power_enable(BOARD_HOST_POWER_USB_DEV)
```

意义：

```text
打开 ESP32-S3 USB HOST 口的 5V 供电路径。
```

结论：

```text
能烧录和 monitor，不代表 Host 口已经能给外设供电。
```

## 5.2 Host 供电问题的判断顺序

推荐顺序：

```text
1. 先测 Host 口 VBUS 是否有 5V
2. 再看设备是否上电
3. 再看 ESP32 是否打印 USB NEW_DEV event
4. 再看 VID / PID
5. 最后才看类驱动 open 是否成功
```

结论：

```text
供电问题、枚举问题、驱动问题、业务协议问题要分层判断。
不能混在一起猜。
```

# 6. UVC Camera 采集经验

## 6.1 UVC Camera 不是普通 USB 输入设备

UVC 层级：

```text
USB Host Library
    -> UVC Host Driver
        -> USB Camera
```

经验：

```text
USB Host 只负责发现设备。
UVC Host Driver 才负责识别视频接口、格式、帧数据。
```

结论：

```text
能枚举到 Camera，只说明 USB 层通了。
能 open stream 并收到 frame，才说明 UVC 层通了。
```

## 6.2 摄像头模式选择

项目中采用：

```text
320x240
MJPEG
约 30 FPS
```

原因：

```text
无 PSRAM 或内存不确定时，低分辨率 MJPEG 更稳。
先保证链路稳定，再考虑更高分辨率。
```

结论：

```text
第一阶段不要追求高分辨率。
先追求稳定采集和完整 JPEG。
```

## 6.3 UVC 统计信息怎么读

重点字段：

```text
frames       -> 收到多少帧
cached       -> 成功写入 latest cache 多少帧
returned     -> 归还 UVC frame 多少帧
fps          -> 实际采集帧率
timeouts     -> 帧等待超时
drops        -> 队列丢帧
errors       -> USB 传输错误
overflow     -> 应用缓存放不下
underflow    -> 数据不足
disconnects  -> 设备断开
```

理想状态：

```text
frames 持续增长
fps 稳定
timeouts=0
drops=0
errors=0
overflow=0
underflow=0
disconnects=0
```

结论：

```text
不要只看“有图”。
要看连续采集时有没有 drops / errors / timeouts。
```

# 7. JPEG cache 和内存经验

## 7.1 不能长期保存 UVC frame 指针

现象：

```text
UVC 回调里可以拿到 frame->data。
但是这个指针属于驱动内部 buffer。
```

判断：

```text
应用层不能长期保存 frame->data 指针。
必须复制到自己的 cache。
```

正确数据流：

```text
UVC callback
    -> 检查 frame
    -> 复制 JPEG 到 latest cache
    -> 记录 len / sequence / timestamp / valid
    -> 归还 frame
```

结论：

```text
UVC frame 是临时数据。
latest JPEG cache 才是应用层可以使用的数据。
```

## 7.2 为什么只保存最近一帧

原因：

```text
ESP32-S3 内存有限。
多帧缓存会快速消耗 RAM。
HTTP 上传、LCD、Wi-Fi、USB Host 都需要内存。
```

第一阶段策略：

```text
只保存 latest JPEG。
不保存历史帧。
不解码 RGB565。
不做实时预览。
```

结论：

```text
latest cache 是第一阶段最稳的方案。
```

## 7.3 静态缓冲不是免费的

现象：

```text
PERIODIC_JPEG_UPLOAD 阶段一切正常。
切换到 INTEGRATION 阶段后 HTTP 上传失败。
错误：ESP_ERR_HTTP_CONNECT / select() timeout。
```

判断：

```text
ESP32-S3 内部 SRAM 约 512KB。
代码里声明了两个大静态数组：

s_latest_frame.data[96KB]       -- BSS 段
s_jpeg_upload_buffer[64KB]      -- BSS 段

这两个数组在 main() 执行之前就已经占着 160KB。
不管你用不用，它都不会释放。
```

ESP32-S3 内存布局：

```text
┌─────────────────────────────┐
│ .text  代码段                 │  编译时确定
├─────────────────────────────┤
│ .bss   未初始化全局变量        │  ← s_latest_frame[96KB] 在这里
│                              │  ← s_jpeg_upload_buffer[64KB] 在这里
│                              │  启动就占，不可释放
├─────────────────────────────┤
│ heap   堆                     │  ← FreeRTOS 任务栈从这里分
│                              │  ← TCP socket 缓冲从这里分
│                              │  ← 剩余空间才是可用的
└─────────────────────────────┘
```

INTEGRATION 比 PERIODIC 多了什么：

```text
CDC-ACM 驱动 + buffer       ~5KB
http_upload_task 栈          8KB
CDC RX line queue           ~512B

这些额外开销把剩余堆从刚好够用变成了不够用。
TCP socket 分配失败 -> sock < 0 -> ESP_ERR_HTTP_CONNECT。
```

解决：

```text
s_latest_frame:      96KB -> 32KB    (实际 JPEG ~9KB, 32KB 有 3.5 倍余量)
s_jpeg_upload:       64KB -> 32KB    (同上)
UVC frame_size:      0(自动54KB) -> 32KB
URB:                 4KB×3 -> 2KB×2
http_upload 栈:      8KB -> 6KB

总计节省约 148KB。
```

常见错误：

```text
错误做法：先预留大缓冲"以防万一"。
正确做法：先测实际数据大小，再定缓冲。
```

验证方法：

```c
ESP_LOGI(TAG, "free heap: %u largest block: %u",
         (unsigned)esp_get_free_heap_size(),
         (unsigned)heap_caps_get_largest_free_block(MALLOC_CAP_DEFAULT));
```

结论：

```text
在嵌入式项目里，每个大数组都要有理由。
BSS 段的静态数组启动就占，不可释放。
多模块融合之前要做内存预算表。
```

# 8. HTTP 上传经验

## 8.1 服务器必须先打开

现象：

```text
ESP32-S3 报 ESP_ERR_HTTP_CONNECT。
```

判断：

```text
这通常不是 Camera 问题。
也不是 CH552G 问题。
而是 HTTP server 没有打开，或者 IP / 端口不可达。
```

电脑端启动服务器：

```powershell
cd C:\Users\tao93\Desktop\5.26Project\ESP32S3_USB_Host
python tools\http_upload_server.py --host 0.0.0.0 --port 8000 --out uploads
```

ESP32-S3 配置：

```c
// 注意: 这个 IP 是 DHCP 分配的，每次可能变
#define APP_HTTP_JPEG_UPLOAD_URL        "http://172.20.150.56:8000/api/upload"
```

结论：

```text
ESP32-S3 是 HTTP Client。
电脑端 server 不开，上传一定失败。
```

## 8.2 Camera FPS 和 HTTP 上传速率不是一回事

现象：

```text
Camera 可以接近 29 FPS。
但是 HTTP 上传不是 29 张/秒。
```

判断：

```text
Camera 持续采集并更新 latest cache。
HTTP 是按周期从 latest cache 抽一张上传。
```

当前集成配置：

```c
#define APP_INTEGRATION_CYCLE_COUNT       12
#define APP_INTEGRATION_INTERVAL_MS       5000
```

含义：

```text
计划上传 12 张。
每 5 秒上传 1 张。
理论速率约 12 张/分钟。
```

结论：

```text
Camera 采集速率是图像输入能力。
HTTP 上传速率是业务抽帧能力。
两者不要混为一谈。
```

## 8.3 HTTP 上传失败时先分层

排查顺序：

```text
1. Wi-Fi 是否连接成功
2. PC server 是否启动
3. ESP32 和 PC 是否在同一网络
4. APP_HTTP_JPEG_UPLOAD_URL 的 IP 是否正确
5. 端口 8000 是否被防火墙拦截
6. latest JPEG 是否 valid / soi / eoi
7. esp_http_client_perform 是否返回 ESP_OK
```

结论：

```text
HTTP 失败不要直接怀疑摄像头。
先确认网络和服务器。
```

## 8.4 IP 地址会变（DHCP 陷阱）

现象：

```text
昨天 HTTP 上传正常。
今天同样的代码、同样的硬件，上传失败。
错误：ESP_ERR_HTTP_CONNECT。
```

判断：

```text
ESP32-S3 和电脑都通过手机热点上网。
手机热点用 DHCP 分配 IP。
IP 每次重连可能变。
project_config.h 里写死了昨天的 IP。
```

验证：

```text
ESP32 monitor 日志搜 got ip，看到当前 IP。
电脑 cmd 运行 ipconfig，看到电脑当前 IP。
两个 IP 必须在同一网段（如都是 172.20.150.x）。
project_config.h 里的 URL 必须和电脑当前 IP 一致。
```

常见错误：

```text
错误做法：在代码里写死 IP，改一次编译一次。
正确做法：开发阶段每次先确认 IP 再编译。
后续可以通过 mDNS 或手机热点固定 IP 解决。
```

结论：

```text
DHCP 环境下 IP 不是永久的。
HTTP 连不上时，第一件事就是确认 IP 有没有变。
```

## 8.5 手机热点的 AP 隔离

现象：

```text
ESP32 和电脑都连着同一个手机热点。
ESP32 got ip 正常。
电脑上网正常。
Python 服务器在跑。
手机浏览器也能访问电脑 8000 端口。
但是 ESP32 仍然 ESP_ERR_HTTP_CONNECT。
```

判断：

```text
手机热点可能开启了 AP 隔离（Client Isolation）。
连同一个热点的设备之间互相不能通信。
各自都能上网，但互相 ping 不通。
```

验证：

```text
1. 电脑 cmd: ping ESP32的IP
   如果不通 -> 确认是 AP 隔离
2. 手机浏览器访问 http://电脑IP:8000/api/test
   如果超时 -> 100% 是 AP 隔离或防火墙
3. 手机浏览器访问正常，但 ESP32 不行
   -> 可能是防火墙只允许局域网但挡了某些子网
```

解决：

```text
方案 1: 手机热点设置里找"允许设备之间互相访问"并开启
方案 2: 换用路由器代替手机热点
方案 3: 电脑和 ESP32 都连路由器，不用手机热点
```

结论：

```text
手机热点不等于路由器。
AP 隔离是手机热点的常见默认行为。
设备都能上网 ≠ 设备之间能互相通信。
```

## 8.6 Windows 防火墙会挡外部 HTTP 请求

现象：

```text
电脑浏览器访问 http://127.0.0.1:8000/api/test 正常（返回 501）。
手机浏览器访问 http://电脑IP:8000/api/test 超时。
```

判断：

```text
127.0.0.1 是本机回环，不受防火墙限制。
外部设备访问需要通过网卡，会被 Windows 防火墙拦截。
Python 服务器绑定的是 0.0.0.0，没有问题。
问题出在 Windows 防火墙入站规则。
```

操作：

```text
临时方案:
    Windows 搜索 "防火墙"
    -> 启用或关闭 Windows Defender 防火墙
```

# 12. ESP32-S3 内存预算和 BSS 陷阱

## 12.1 现象：单模块正常，多模块融合时 HTTP 连不上

现象：

```text
PERIODIC_JPEG_UPLOAD 阶段：Camera + Wi-Fi + HTTP，一切正常。
INTEGRATION 阶段：Camera + CDC + Wi-Fi + HTTP，HTTP 上传失败。
错误：
    E (...) esp-tls: [sock=54] select() timeout
    E (...) transport_base: Failed to open a new connection: 32774
    E (...) HTTP_CLIENT: Connection failed, sock < 0
    W (...) cloud_http: esp_http_client_perform failed: ESP_ERR_HTTP_CONNECT
```

判断：

```text
ESP_ERR_HTTP_CONNECT 意味着 TCP 连接都没建立成功。
不是 HTTP 代码问题，也不是服务器问题。
是 ESP32-S3 没有足够的内存来分配 TCP socket。
```

## 12.2 ESP32-S3 内存全景图

```text
ESP32-S3 内部 SRAM 约 512KB
┌──────────────────────────────────────┐
│ .text (代码段)          ~200-300KB    │  编译时确定，不可修改
├──────────────────────────────────────┤
│ .bss (未初始化全局变量)                │
│   s_latest_frame[96KB]              │  ← 启动就占，不可释放
│   s_jpeg_upload_buffer[64KB]        │  ← 启动就占，不可释放
│   其他全局变量            ~几KB        │
├──────────────────────────────────────┤
│ FreeRTOS 任务栈                       │
│   usb_host_lib_task      4KB         │
│   uvc_driver_task        4KB         │
│   uvc_capture_task       4KB         │
│   http_upload_task       8KB         │
│   cdc_acm 内部任务       ~2KB         │
│   app_main + 系统任务    ~12KB        │
│   小计                   ~34KB        │
├──────────────────────────────────────┤
│ 驱动内部缓冲                          │
│   USB UVC 帧缓冲+URB    ~120KB       │
│   Wi-Fi 驱动             ~60KB        │
│   TCP/IP (lwIP)          ~40KB        │
│   USB Host 库            ~15KB        │
│   CDC-ACM 驱动 + buffer  ~5KB         │
├──────────────────────────────────────┤
│ 剩余可用堆 (heap)                     │
│   TCP socket 需要约 60-80KB           │
│   esp_http_client 需要约 20KB         │
│   其他 malloc            ~??          │
│                                       │
│   实际剩余: ~78KB                      │
│   需求: ~80-100KB                      │
│   结论: 差一点点，刚好不够              │
└──────────────────────────────────────┘
```

## 12.3 哪些地方吃了最多内存

```text
消耗大户：
    s_latest_frame.data          96KB    BSS 静态，启动就占
    s_jpeg_upload_buffer         64KB    BSS 静态，启动就占
    UVC 帧缓冲 (驱动自动分配)    ~108KB   2帧 × 54KB
    Wi-Fi 驱动内部               ~60KB    ESP-IDF 管理，不可调
    TCP/IP 协议栈 (lwIP)         ~40KB    ESP-IDF 管理，不可调
    任务栈合计                   ~34KB    可微调
    ────────────────────────────────
    总计                         ~402KB

ESP32-S3 内部 SRAM             ~512KB
剩余给 heap 的                 ~110KB
TCP socket + HTTP client 需要  ~80-100KB
```

## 12.4 三个缓冲的联动关系

```text
UVC 驱动帧缓冲                 s_latest_frame              s_jpeg_upload_buffer
(驱动的，要还给驱动)             (你自己的缓存)               (上传专用缓存)

frame_callback()                capture task                http_upload_task
  ↓ 收到帧指针                   ↓ memcpy                    ↓ esp_http_client
  ↓ 放入 queue                   ↓ 写入 data[96KB]           ↓ 读取 data[64KB]
  ↓ 立刻返回                     ↓ 归还帧给驱动               ↓ 发送到服务器

当前:  ~54KB × 2                 当前: 96KB (浪费)            当前: 64KB (浪费)
优化后: 32KB × 2                 优化后: 32KB (够用)          优化后: 32KB (够用)
```

每个环节只需要比实际数据大一点就行：

```text
实际 JPEG 大小 (320x240 MJPEG):  ~9KB

UVC 帧缓冲:    54KB → 32KB     3.5 倍余量
s_latest_frame: 96KB → 32KB     3.5 倍余量
s_jpeg_upload:  64KB → 32KB     3.5 倍余量
```

## 12.5 解决：按实际需求分配缓冲

```text
修改项                          文件                    当前值      优化值      节省
─────────────────────────────────────────────────────────────────────────────
UVC frame_size                  camera_uvc_client.c     0(自动)     32KB        ~42KB
UVC number_of_urbs              camera_uvc_client.c     3           2           ~4KB
UVC urb_size                    camera_uvc_client.c     4KB         2KB         ~4KB
s_latest_frame                  camera_uvc_client.c     96KB        32KB        64KB
s_jpeg_upload_buffer            cloud_http_client.c     64KB        32KB        32KB
http_upload_task 栈              project_config.h        8KB         6KB         2KB
─────────────────────────────────────────────────────────────────────────────
总节省                                                              ~148KB
```

优化后：

```text
剩余堆: ~78KB + 148KB = ~226KB
TCP socket 分配绰绰有余。
```

## 12.6 常见错误：在嵌入式项目里滥用大静态数组

```text
错误做法 1:
    觉得"96KB 应该够吧"就直接写 uint8_t buf[96*1024]。
    没有测实际数据大小。

错误做法 2:
    先预留大缓冲"以防万一"。
    等多模块融合时才发现内存不够。

错误做法 3:
    多个模块各自声明大静态数组。
    每个模块觉得"我才占 96KB"，加起来 BSS 段就爆了。

正确做法:
    1. 先测实际数据大小（看日志里的 len=xxx）
    2. 按 3-4 倍余量设缓冲大小
    3. 每加一个模块都打印 esp_get_free_heap_size()
    4. 多模块融合前做内存预算表
```

诊断工具：

```c
// 在关键操作前打印内存状态
ESP_LOGI(TAG, "free heap: %u largest block: %u",
         (unsigned)esp_get_free_heap_size(),
         (unsigned)heap_caps_get_largest_free_block(MALLOC_CAP_DEFAULT));
```

判断标准：

```text
free heap > 100KB            安全
free heap 60-100KB            警告，可能不够 TCP
free heap < 60KB              危险，malloc 随时失败
largest free block < 40KB     即使总量够，也无法分配大块连续内存
```

结论：

```text
嵌入式项目启动前要做内存预算表。
BSS 段的静态数组启动就占，不可释放。
缓冲大小要按实测数据定，不是"感觉够就行"。
```

# 13. 网络环境和 HTTP 连接排查

## 13.1 ESP_ERR_HTTP_CONNECT 不一定是 HTTP 代码问题

现象：

```text
ESP32-S3 报 HTTP 连接失败。
第一反应：是不是 Python 服务器代码有 bug？
```

判断：

```text
ESP_ERR_HTTP_CONNECT = TCP 连接都没建立成功。
HTTP 请求根本没发出去。
Python 服务器代码怎么写的完全无关。
问题在 ESP32 和电脑之间的网络通路。
```

排查分层：

```text
第 1 层: TCP 连接能不能建立？
    -> ESP_ERR_HTTP_CONNECT -> 这层就不通
    -> 原因: 网络/防火墙/内存

第 2 层: HTTP 请求能不能发出？
    -> esp_http_client_perform 返回 ESP_OK -> 这层通了
    -> 原因: URL/方法/Header/Body 配置

第 3 层: 服务器能不能正确处理？
    -> 看 Python 服务器终端有没有输出
    -> 看返回的 status_code

第 4 层: 返回的数据对不对？
    -> 看 response body 里的 JSON
```

结论：

```text
HTTP 报错时，先看错误码是哪一层的。
ESP_ERR_HTTP_CONNECT 是网络层，不是应用层。
不要一上来就改 Python 代码。
```

## 13.2 网络排查五步法

```text
步骤 1: ESP32 Wi-Fi 连上了吗？
    monitor 日志搜 got ip
    没有 -> 先解决 Wi-Fi

步骤 2: 电脑 IP 和 project_config.h 一致吗？
    电脑 cmd 运行 ipconfig
    对比 APP_HTTP_JPEG_UPLOAD_URL 里的 IP
    不一致 -> 更新 URL，重新编译烧录

步骤 3: Python 服务器启动了吗？
    电脑终端有没有 "HTTP server listening"
    没有 -> 先启动服务器

步骤 4: 网络可达吗？（排除 AP 隔离和防火墙）
    手机浏览器（同一热点）访问 http://电脑IP:8000/api/test
    超时 -> AP 隔离或防火墙问题
    正常 -> 网络没问题

步骤 5: ESP32 内存够吗？
    看日志里的 free heap
    < 80KB -> 先优化缓冲
    > 100KB -> 内存不是问题，继续排查
```

结论：

```text
网络问题按层排查：Wi-Fi -> IP -> 服务器 -> 可达性 -> 内存。
不要跳步。
```

## 13.3 常见网络环境坑

```text
坑 1: 手机热点 AP 隔离
    现象: 设备都能上网，但互相 ping 不通
    解决: 热点设置里开启"允许设备互访"，或换路由器

坑 2: DHCP 每次分配不同 IP
    现象: 昨天能用今天不行
    解决: 每次确认 IP，或用 mDNS

坑 3: Windows 防火墙挡外部请求
    现象: localhost 能通，外部不能
    解决: 添加 TCP 8000 入站规则，或临时关闭防火墙

坑 4: Python 服务器绑定了 127.0.0.1
    现象: 服务器在跑，但外部访问不到
    解决: 启动时用 --host 0.0.0.0

坑 5: ESP32 和电脑不在同一网段
    现象: ESP32 是 10.x.x.x，电脑是 172.x.x.x
    解决: 确认两者连的是同一个热点/路由器
```

结论：

```text
HTTP 连接失败时，90% 的情况是网络环境问题，不是代码问题。
先把网络跑通，再怀疑代码。
```


第一阶段核心沉淀：

```text
1. 分阶段验证，一次只引入一个新变量
2. USB 问题从供电开始查，不要跳步
3. 嵌入式项目要做内存预算表，BSS 段不是免费的
4. HTTP 失败先查网络，再查内存，最后看代码
5. 每个缓冲大小要按实测数据定，不是"感觉够就行"
```

这就是本项目第一阶段最值得沉淀的工程方法。
