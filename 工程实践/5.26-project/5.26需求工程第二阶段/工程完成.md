`Phase 2 不追求真云，不追求边缘 AI，不追求复杂 UI。先做一个局域网 PC AI 服务端，把摄像头画面变成人数，把人数稳定变成 5 路灯。`



对，完全可以理解成两个工作面：

**1. ESP32 端代码修改**

ESP32 负责“嵌入式主控”这部分：

- 从 USB UVC 摄像头拿到 latest JPEG
- 通过 Wi-Fi 把 JPEG 发给 PC 服务端
- 调用类似 `POST /api/detect`
- 解析服务端返回的 JSON，例如 `person_count`
- 根据人数执行 `light_policy`
- 通过 CDC 给 CH552G 下发 `L1 ON/OFF` 到 `L5 ON/OFF`
- 做断网、HTTP 失败、JSON 解析失败时的降级处理

也就是说，ESP32 不跑 AI，它只负责采图、上传、解析结果、决策和灯控。

**2. 本地局域网 PC 部署**

PC 负责“云端 AI 服务”这部分：

- 开一个 FastAPI / Flask 服务
- 监听局域网地址，比如 `http://电脑IP:8000`
- 提供 `/api/health` 和 `/api/detect`
- 启动时加载 YOLOv8n 模型
- 收到 ESP32 上传的 JPEG
- 运行人员检测
- 返回 JSON：


**第 1 阶段：PC 云端服务骨架**

目标不是马上 AI，而是先做一个局域网 HTTP 服务。

PC 服务提供两个接口：

```
GET  /api/health
POST /api/detect
```

`/api/health` 用来确认服务活着。

`/api/detect` 先不跑 YOLO，只返回固定人数，例如：

```
{
  "status": "ok",
  "person_count": 1,
  "latency_ms": 3,
  "model": "fake-detector-v1"
}
```

这一步成功的标准是：你用浏览器或 PowerShell 访问 PC 服务，能看到 JSON 返回。

**第 2 阶段：让 ESP32 发到 `/api/detect`**

你现在 ESP32 已经能把 JPEG 发到 `/api/upload`。下一步不是重写网络层，而是把目标接口换成：

```
http://你的电脑IP:8000/api/detect
```

这一步 ESP32 还不用关心 YOLO。它只需要确认：

```
ESP32 -> POST JPEG -> PC
PC -> 返回 person_count
```

成功标准是 PC 终端能打印“收到一张 JPEG”，ESP32 串口能看到 HTTP 200。

**第 3 阶段：ESP32 解析 JSON**

这一步才开始让 ESP32 读返回内容。

ESP32 要从响应里解析：

```
status
person_count
latency_ms
```

然后串口打印：

```
detect result: person_count=1 latency=3ms
```

成功标准是：PC 返回固定 `person_count=1`，ESP32 能稳定解析出来。

**第 4 阶段：接入灯控策略**

假检测结果能解析后，再接 `light_policy`。

例如：

```
person_count=0 -> 全灭
person_count=1 -> L1 ON
person_count=2 -> L1,L2 ON
person_count>=5 -> L1~L5 ON
```

这一步还是不接 YOLO。因为我们先验证“人数结果 -> LED”这条链路。

**第 5 阶段：PC 服务接入 YOLO**

前四步都稳定之后，才把 PC 的 fake detector 换成 YOLOv8n。

也就是：

```
原来：
JPEG -> fake person_count

改成：
JPEG -> YOLOv8n -> person_count
```

这样如果出问题，你马上知道问题在 AI 模型，而不是 HTTP、JSON、ESP32、灯控混在一起。

所以我们的教学顺序是：

```
PC 服务活着
-> PC 假检测
-> ESP32 能 POST
-> ESP32 能解析 JSON
-> LED 能按人数变化
-> 最后接 YOLO
```
