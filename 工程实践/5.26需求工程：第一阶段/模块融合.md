# 5.26 需求工程第一阶段：模块融合

这篇笔记只记录“多模块如何融合起来跑”。

不重复展开 CH552G、摄像头、HTTP、LCD 的单模块细节。那些内容放在对应模块笔记里。

本篇重点：

```text
ESP32-S3 USB HOST
    -> FE1.1s Hub PC 口
        -> USB UVC Camera
        -> CH552G CDC-ACM Device

Camera latest JPEG
    -> HTTP upload / mock detect
    -> CH552G CDC command
    -> L1 ON / L1 OFF
```

---

# 1. 模块融合的目标

目标不是单独证明某个模块能跑。

目标是证明：

```text
ESP32-S3 可以通过一个 USB Host 口，在 Hub 下同时管理两个 USB 设备：
    -> Camera：负责图像获取
    -> CH552G：负责灯光控制
```

第一阶段融合闭环：

```text
Camera through Hub
    -> latest JPEG cache
    -> HTTP upload / mock detection
    -> CH552G CDC command
    -> L1 ON / L1 OFF
```

现象：

```text
摄像头可以持续采集 JPEG
CH552G 可以接收 PING / STATUS / L1 ON / L1 OFF
HTTP 可以上传 latest JPEG
mock detect 可以周期切换 TARGET / EMPTY
L1 可以随检测结果变化
```

结论：

```text
第一阶段已经从“单模块验证”进入“多模块融合验证”。
```

---

# 2. 当前系统拓扑

硬件连接：

```text
电脑 -> ESP32-S3 USB-UART
    -> 用来 monitor / 烧录调试

电脑或电源 -> ESP32-S3 USB_DEV
    -> 给 ESP32-S3 和 USB HOST 供电路径提供 5V

ESP32-S3 USB HOST -> FE1.1s Hub 的 PC Type-C 口
    -> 这是 Hub 的上行数据口

5V 电源或充电宝 -> FE1.1s Hub 的 DC5V 口
    -> 这是 Hub 的独立供电口

Hub USB1 -> USB Camera
Hub USB2 -> CH552G
```

关键理解：

```text
USB-UART：负责日志和烧录
USB_DEV：负责 ESP32-S3 板子的 USB 供电路径
USB HOST：负责枚举外部 USB 设备
Hub PC 口：Hub 的上行口，要接 ESP32-S3 USB HOST
Hub DC5V：Hub 独立供电口，不负责数据
Hub USB1~USB4：下行设备口，接 Camera / CH552G
```

常见误区：

```text
有电 != 能枚举
Hub 灯亮 != 数据链路正常
USB_DIAG 能看到设备 != CDC/UVC 驱动一定能打开设备
```

# 3. 融合前的单模块验证

融合前必须先分层验证。

推荐顺序：

```text
1. ESP32-S3 Host 供电
2. Camera 直插 USB HOST
3. CH552G 直插 USB HOST
4. 空 Hub 接 USB HOST
5. Hub + Camera
6. Hub + CH552G
7. Hub + Camera + CH552G
8. APP_STAGE_INTEGRATION
```

判断标准：

```text
USB_DIAG：证明设备可以被枚举
CDC_DIRECT：证明 CH552G CDC 可以被打开并收发指令
UVC_DIRECT：证明 Camera 可以打开并采集图像
INTEGRATION：证明 Camera + CH552G + HTTP + 灯控可以共存
```

不要一开始就直接接 Hub + Camera + CH552G。

原因：

```text
如果融合阶段失败，很难判断是供电、Hub、枚举、CDC、UVC、HTTP 还是内存问题。
```

# 4. Hub 下多设备枚举逻辑

典型 Camera 枚举日志：

```text
USB NEW_DEV event
USB device descriptor: VID=0x349C PID=0x3307
```

典型 CH552G 枚举日志：

```text
USB NEW_DEV event
USB device descriptor: VID=0x1A86 PID=0x5722
```

现象：

```text
USB_DIAG 能看到 VID/PID
```

判断：

```text
设备已经进入 USB 总线
ESP32-S3 Host、线缆、Hub 数据链路基本成立
```

但还不能说明：

```text
CDC 已经打开
UVC stream 已经打开
可以稳定传输数据
```

下一步：

```text
CH552G -> 进入 CDC_DIRECT 或 INTEGRATION 中的 CDC open
Camera -> 进入 UVC_DIRECT 或 INTEGRATION 中的 camera open
```

# 5. CH552G CDC 灯控链路

CH552G 在本工程中不是普通串口模块。

它是：

```text
USB CDC-ACM Device
    -> ESP32-S3 作为 USB Host 打开它
    -> ESP32-S3 发送文本命令
    -> CH552G 控制 GPIO
```

已验证命令：

```text
PING
STATUS
L1 ON
L1 OFF
ALL ON
ALL OFF
```

典型成功日志：

```text
CH552G PING test passed
CH552G STATUS: STATUS 0 0 0 0 0
CH552G L1 ON test passed
CH552G L1 OFF test passed
```

关键配置：

```c
#define APP_CDC_OPEN_TIMEOUT_MS 15000
```

原因：

```text
CH552G 是 MCU。
上电后要先运行固件，再初始化 USB Device。
如果 ESP32-S3 太快去 open CDC，可能会超时。
```

结论：

```text
APP_CDC_OPEN_TIMEOUT_MS=15000 是当前阶段合理配置。
```

# 6. UVC Camera 图像链路

摄像头在本工程中是标准 UVC 设备。

图像链路：

```text
USB Camera
    -> ESP32-S3 UVC Host driver
    -> frame callback
    -> capture task
    -> latest JPEG cache
    -> meta / copy API
```

`camera_uvc_client` 的职责：

```text
打开摄像头
持续采集帧
检查 JPEG SOI / EOI
更新 latest JPEG cache
提供 latest JPEG meta
提供 latest JPEG copy
统计 frames / fps / drops / errors / underflow
```

它不应该负责：

```text
HTTP 上传
云端 AI
CH552G 灯控
LCD 业务页面
```

推荐摄像头模式：

```text
320x240 MJPEG
```

原因：

```text
ESP32-S3 内存有限
第一阶段只需要证明图像链路，不需要高分辨率预览
```

# 7. latest JPEG cache 的作用

UVC 驱动给出的 frame buffer 不能长期占用。

原因：

```text
frame buffer 属于 UVC 驱动
如果长期不归还，会影响后续 USB 接收
```

所以要做 latest cache：

```text
UVC frame callback
    -> 把 frame 指针送入队列
capture task
    -> 拷贝完整 JPEG 到 s_latest_frame
    -> 归还 frame 给 UVC driver
HTTP / integration
    -> 从 latest cache 读取最新一帧
```

好处：

```text
Camera 采集和 HTTP 上传解耦
灯控逻辑不用直接依赖 UVC driver frame 指针
只保存最近一帧，避免内存无限增长
```

判断 latest JPEG 是否有效：

```text
valid = true
jpeg_soi = true
jpeg_eoi = true
len > 0
```

典型日志：

```text
latest JPEG ready: seq=2 len=7908 timestamp=4900
```

# 8. HTTP 上传链路

HTTP 上传不是 Camera FPS。

两者区别：

```text
Camera FPS：摄像头每秒采集多少帧
HTTP upload rate：每秒或每分钟上传多少张 JPEG
```

当前设计：

```text
Camera 持续采集
latest JPEG cache 持续更新
HTTP task 周期性取最新 JPEG 上传
```

典型成功日志：

```text
POST JPEG http://x.x.x.x:8000/api/upload len=xxxxx seq=xxx timestamp=xxx
status=200 content_length=xxx upload_len=xxxxx seq=xxx ts=xxx
HTTP upload summary: attempts=12 success=12 failed=0
```

如果 HTTP 失败，先分层判断。

现象：

```text
ESP_ERR_HTTP_CONNECT
select() timeout
Connection failed, sock < 0
```

判断：

```text
可能是服务器没开
可能是 IP 错
可能是 Windows 防火墙
可能是手机热点隔离
也可能是 ESP32-S3 内存不足
```

检查顺序：

```text
1. 电脑端 Python server 是否启动
2. ESP32-S3 配置的 URL 是否是电脑真实 IP
3. 电脑本机 curl 是否能访问
4. 同一局域网设备是否能访问电脑 8000 端口
5. Windows 防火墙是否放行 8000
6. ESP32-S3 上传前 heap 是否足够
```

# 9. run_integration_stage 融合逻辑

`run_integration_stage` 是第一阶段融合主入口。

它的目标不是做复杂 AI。

它的目标是证明：

```text
Camera + HTTP + CH552G CDC + mock detect 可以同时运行
```

推荐逻辑：

```text
1. 准备 USB HOST 供电
2. 初始化 USB Host manager
3. 打开 Camera
4. 打开 CH552G CDC
5. 连接 Wi-Fi
6. 启动 UVC continuous capture task
7. 启动 HTTP upload task
8. 主循环读取 latest JPEG meta
9. mock detect 在 TARGET / EMPTY 间切换
10. TARGET -> L1 ON
11. EMPTY -> L1 OFF
12. 最后打印 UVC stats + HTTP stats
```

为什么 HTTP 要独立 task：

```text
HTTP POST 可能阻塞很久
如果主循环同步等待 HTTP，会导致灯控和检测周期一起被卡住
```

错误设计：

```text
integration loop
    -> HTTP upload blocking
    -> mock detect
    -> L1 control
```

推荐设计：

```text
UVC capture task：持续采集
HTTP upload task：周期上传 latest JPEG
integration main loop：周期检测 + 灯控
```

结论：

```text
融合阶段要让慢任务和实时控制任务解耦。
```

# 10. underflow / drops / errors 怎么理解

`UVC frame buffer underflow` 不等于图像失败。

现象：

```text
W camera_uvc_client: UVC frame buffer underflow: 5
```

判断：

```text
UVC driver 的 frame buffer 有一段时间不够用
可能是系统忙、HTTP/Wi-Fi 占用、任务调度延迟导致
```

它不是致命错误。

真正要重点看：

```text
drops
timeouts
errors
overflow
disconnects
invalid_jpeg
```

可以接受的情况：

```text
underflow > 0
但 frames 持续增长
invalid_jpeg = 0
drops = 0
errors = 0
timeouts = 0
disconnects = 0
```

典型统计：

```text
frames=1835 cached=1835 returned=1835 invalid_jpeg=0 fps=27.x drops=0 errors=0 timeouts=0 underflow=128 disconnects=0
```

结论：

```text
underflow 作为 warning 记录，不应该直接作为 integration 失败条件。
```

# 11. 内存问题与缓冲区优化

融合阶段比单模块更吃内存。

原因：

```text
UVC driver buffer
latest JPEG cache
HTTP upload buffer
Wi-Fi / TCP/IP
CDC-ACM driver
FreeRTOS task stack
USB Host library
```

典型问题日志：

```text
heap before HTTP: free=10324 largest=7680
```

判断：

```text
free 只有约 10KB
largest 只有约 7.5KB
HTTP socket / lwIP / esp_http_client 没有足够连续内存
```

判断标准：

```text
free < 80KB
或 largest < 40KB

基本可以认为 HTTP 上传前内存太紧张。
```

当前优化配置：

```c
#define APP_CDC_OPEN_TIMEOUT_MS 15000
#define CAMERA_LATEST_FRAME_MAX_SIZE (48 * 1024)
#define CLOUD_HTTP_JPEG_UPLOAD_BUF_SIZE (32 * 1024)
#define APP_HTTP_UPLOAD_TASK_STACK_SIZE 6144
```

优化思路：

```text
不要盲目开大 buffer
buffer 要按实际 JPEG 大小设置
latest cache 和 upload buffer 都要留余量，但不能浪费几十 KB
```

如果 32KB upload buffer 不够：

```text
出现：latest JPEG is too large for static upload buffer
处理：CLOUD_HTTP_JPEG_UPLOAD_BUF_SIZE -> 48KB
```

如果 latest cache 不够：

```text
出现：buffer_overflows > 0
处理：CAMERA_LATEST_FRAME_MAX_SIZE -> 64KB
```

如果 HTTP task 栈不够：

```text
出现：stack overflow
处理：APP_HTTP_UPLOAD_TASK_STACK_SIZE -> 8192
```

# 13. 推荐调试顺序

以后如果融合阶段出问题，按这个顺序排查。

```text
1. 看供电
    -> ESP32-S3 USB_DEV 是否供电
    -> Hub DC5V 是否供电
    -> Camera / CH552G 是否有电

2. 看枚举
    -> USB NEW_DEV event
    -> Camera VID/PID
    -> CH552G VID/PID

3. 看驱动打开
    -> camera_uvc_open
    -> ch552_cdc_open

4. 看数据链路
    -> latest JPEG ready
    -> CH552G PING / STATUS

5. 看融合逻辑
    -> detect TARGET / EMPTY
    -> L1 ON / L1 OFF

6. 看 HTTP
    -> server 是否开启
    -> URL/IP 是否正确
    -> 防火墙 / 热点隔离
    -> heap before HTTP

7. 看统计
    -> UVC stats
    -> HTTP summary
    -> CDC command result
```

不要跳步。

因为融合问题通常不是一个点，而是多个模块叠加后的结果。
