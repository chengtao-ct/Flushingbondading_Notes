# 5.26 需求工程第一阶段：摄像头模块

这篇笔记只记录 USB UVC 摄像头模块的学习和工程判断。

原始 descriptor 和长日志已经备份，不再放在主笔记里。

本篇重点：

```text
USB Camera
    -> ESP32-S3 USB Host
    -> UVC Host driver
    -> frame callback
    -> capture task
    -> latest JPEG cache
    -> HTTP / integration 使用
```

---

# 1. 摄像头模块的定位

摄像头模块不是整个系统的最终功能。

它在第一阶段的职责是：

```text
稳定打开 UVC 摄像头
持续接收 MJPEG/JPEG 帧
判断 JPEG 是否完整
缓存最近一帧 latest JPEG
给 HTTP 上传和融合逻辑提供图像数据
输出 FPS、drops、errors、underflow 等统计信息
```

它不应该直接负责：

```text
CH552G 灯控
云端 AI 业务判断
LCD 状态面板
run_integration_stage 的检测策略
```

结论：

```text
camera_uvc_client 只做“图像输入层”。
业务逻辑放在 app_state / integration 层。
```

---

# 2. 为什么 USB 摄像头叫 UVC

UVC = USB Video Class。

意思是：

```text
摄像头不是普通串口设备
也不是 HID 设备
它是 USB 标准视频类设备
```

Host 要做的事情：

```text
1. 枚举 USB 设备
2. 读取 descriptor
3. 找到 VideoControl / VideoStreaming interface
4. 选择分辨率、格式、帧率
5. 打开 stream
6. 接收视频帧
```

Windows 上显示为：

```text
USB 视频设备
HD video
Camera class
```

ESP32-S3 上要用：

```text
USB Host driver
    -> 负责 USB 总线和枚举

UVC Host driver
    -> 负责摄像头视频协议
```

---

# 3. Descriptor 应该怎么看

Descriptor 不是要逐字背下来。

它的作用是告诉 Host：

```text
我是什么设备
我有几个 interface
我支持哪些视频格式
我支持哪些分辨率和帧率
我用哪个 endpoint 传输数据
```

本摄像头关键信息：

```text
VID = 0x349C
PID = 0x3307
bNumInterfaces = 5
wTotalLength = 522
bMaxPower = 256mA
```

典型枚举日志：

```text
USB device descriptor: VID=0x349C PID=0x3307 class=0xEF subclass=0x02 protocol=0x01
```

判断：

```text
ESP32-S3 已经看到摄像头
USB Host 基础链路成立
```

但这还不等于：

```text
UVC stream 已经打开
JPEG 帧已经有效
HTTP 可以上传
```

---

# 4. 摄像头支持的分辨率

从 descriptor 和 Windows 工具里看到的 MJPEG 模式：

```text
1280 x 720 @ 20 fps
800  x 480 @ 20 fps
640  x 480 @ 30 fps
480  x 320 @ 30 fps
320  x 240 @ 30 fps
```

第一阶段推荐：

```text
320 x 240 MJPEG
```

原因：

```text
ESP32-S3 内存有限
USB + Wi-Fi + HTTP + CDC 同时运行会吃很多 RAM
第一阶段只需要证明图像链路，不需要高清预览
```

如果后面要做更高分辨率：

```text
先确认是否有 PSRAM
再调整 JPEG cache / upload buffer / UVC frame buffer
最后再考虑 640x480 或更高
```

---

# 5. UVC descriptor 大的问题

曾经遇到的问题：

```text
Configuration descriptor larger than control transfer max length
CHECK_SHORT_CONFIG_DESC FAILED
```

判断：

```text
摄像头 descriptor 比 ESP-IDF 默认控制传输缓冲大
ESP32-S3 还没完整读完配置描述符
```

操作：

```text
idf.py menuconfig
    -> Component config
    -> USB-OTG
    -> Largest size of transfers to/from default endpoints
    -> 调大，例如 4096
```

结论：

```text
UVC 摄像头的 descriptor 往往比较大。
看不到完整 UVC 信息时，不一定是摄像头坏了，可能只是默认 EP0 控制传输缓冲太小。
```

---

# 6. camera_uvc_client 的核心数据流

推荐数据流：

```text
uvc frame callback
    -> 收到 frame 指针
    -> 放入 FreeRTOS queue
    -> callback 尽快返回

capture task
    -> 从 queue 取 frame
    -> 检查 SOI / EOI
    -> 拷贝到 latest JPEG cache
    -> 归还 frame 给 UVC driver

业务层
    -> 读取 latest JPEG meta
    -> 复制 latest JPEG
    -> HTTP 上传或检测
```

为什么 callback 里不要做重活：

```text
callback 属于 USB 视频接收链路
如果在里面做 HTTP / 解码 / 打印大量日志
会拖慢 UVC 接收
容易造成 underflow 或掉帧
```

结论：

```text
callback 只做快动作。
capture task 做缓存。
integration / HTTP task 做业务。
```

---

# 7. JPEG 帧完整性判断

MJPEG 的每一帧本质上是一张 JPEG 图片。

JPEG 开始标记：

```text
FF D8 = SOI = Start Of Image
```

JPEG 结束标记：

```text
FF D9 = EOI = End Of Image
```

注意：

```text
不能只看最后两个字节是不是 FF D9
因为 UVC/MJPEG 帧末尾可能有 padding
FF D9 不一定刚好在最后两个字节
```

正确做法：

```text
检查第一个有效字节是否是 FF D8
扫描整帧是否存在 FF D9
上传前可以裁剪 EOI 后面的 trailing bytes
```

典型 debug：

```text
first=FF D8
has_eoi=1
```

判断：

```text
这是真正的 JPEG 帧，不是随机数据。
```

# 8. latest JPEG cache 的意义

UVC driver 的 frame buffer 必须及时归还。

所以不能让 HTTP 直接长期拿着 UVC frame 指针。

正确做法：

```text
UVC frame
    -> 拷贝到 s_latest_frame
    -> 立刻归还给 UVC driver
```

latest cache 保存内容：

```text
data
len
sequence
timestamp_ms
valid
jpeg_soi
jpeg_eoi
```

使用 latest cache 的好处：

```text
HTTP 上传不会阻塞 UVC driver
integration 可以只看最新一帧，不需要保存所有帧
内存占用可控
```

当前缓存配置：

```c
#define CAMERA_LATEST_FRAME_MAX_SIZE (48 * 1024)
```

判断标准：

```text
如果 buffer_overflows = 0，说明 latest cache 够用
如果 buffer_overflows > 0，需要临时调到 64KB
```

# 9. 连续采集统计怎么看

典型统计日志：

```text
UVC stats: elapsed=10000 ms frames=287 returned=287 fps=28.35 bytes=xxxxx min=2820 max=34796 timeouts=0 drops=0 errors=0 overflow=0 underflow=0 disconnects=0
```

每个字段的意思：

```text
frames：收到的帧数量
returned：归还给 UVC driver 的帧数量
fps：实际采集帧率
bytes：累计接收字节数
min/max：单帧大小范围
timeouts：等待帧超时
drops：驱动或队列丢帧
errors：USB 或 UVC 错误
overflow：latest cache 放不下
underflow：UVC frame buffer 短时间不足
disconnects：摄像头断开
```

理想情况：

```text
frames 持续增长
returned 接近 frames
fps 接近目标帧率
invalid_jpeg = 0
timeouts = 0
drops = 0
errors = 0
overflow = 0
disconnects = 0
```

---

# 10. underflow 怎么理解

现象：

```text
UVC frame buffer underflow: 5
```

判断：

```text
UVC driver 的 frame buffer 曾经不够周转
通常表示系统有一段时间比较忙
```

可能原因：

```text
HTTP 上传阻塞时间长
Wi-Fi / TCP/IP 占用 CPU 和内存
日志打印太多
UVC buffer 数量偏少
任务调度不及时
```

重要结论：

```text
underflow 是 warning
不等于图片坏了
不等于摄像头不能用
不应该单独作为 integration 失败条件
```

真正危险的是：

```text
drops > 0
errors > 0
timeouts > 0
overflow > 0
disconnects > 0
invalid_jpeg > 0
```

---

# 11. UVC buffer 和内存取舍

UVC 采集需要多层 buffer：

```text
UVC driver frame buffer
    -> 驱动接收 USB 数据

latest JPEG cache
    -> 应用层保存最近一张完整 JPEG

HTTP upload buffer
    -> 上传任务复制并发送 JPEG
```

这三个 buffer 会叠加占用内存。

当前阶段的保守优化方向：

```text
latest JPEG cache：48KB
HTTP upload buffer：32KB
HTTP task stack：6144
UVC buffer：先不轻易缩小
```

为什么不一开始就压 UVC buffer：

```text
UVC underflow 已经出现过
如果继续压 UVC buffer，可能让采集更不稳定
```

调参原则：

```text
先保证 Camera 稳定采集
再压缩应用层静态 buffer
最后才考虑 UVC driver buffer
```

---

# 12. Camera FPS 和 HTTP 上传速率

Camera FPS：

```text
摄像头每秒采集多少帧
例如 27~29 FPS
```

HTTP 上传速率：

```text
ESP32-S3 每隔多久上传一张 latest JPEG
例如 5 秒上传 1 张
```

这两个不是一回事。

例子：

```text
Camera 30 FPS 连续采集
HTTP 每 5 秒上传一次
```

含义：

```text
5 秒内摄像头可能采集 150 帧
HTTP 只取其中最新的 1 帧上传
```

结论：

```text
第一阶段不需要每帧都上传。
只需要稳定上传最新一帧，证明图像链路和云端链路可用。
```

---

# 13. 和 HTTP / Integration 的关系

摄像头模块输出的是：

```text
latest JPEG
latest meta
UVC stats
```

HTTP 模块负责：

```text
复制 latest JPEG
裁剪 trailing bytes
POST 到服务器
统计 success / failed
```

Integration 模块负责：

```text
读取 latest meta
执行 mock detect 或云端 detect
根据 TARGET / EMPTY 控制 CH552G
汇总 Camera / HTTP / CDC 状态
```

不要把这些职责混在一起。

原因：

```text
模块职责越混乱，调试时越难判断是 UVC 问题、HTTP 问题还是灯控问题。
```

---

# 14. 推荐调试顺序

摄像头模块出问题时按这个顺序看。

```text
1. 看 USB_DIAG
    -> 是否有 VID=0x349C PID=0x3307

2. 看 descriptor
    -> 是否能读取完整配置描述符
    -> 是否需要调大 USB default endpoint transfer size

3. 看 camera_uvc_open
    -> 是否能打开 UVC stream

4. 看首帧
    -> latest JPEG ready
    -> first=FF D8
    -> has_eoi=1

5. 看连续采集
    -> frames 是否增长
    -> fps 是否稳定
    -> returned 是否接近 frames

6. 看错误统计
    -> drops / errors / timeouts / overflow / disconnects

7. 看 integration 影响
    -> HTTP 上传时 underflow 是否增加
    -> heap before HTTP 是否太低
```

不要只看一条错误日志。

摄像头问题通常要结合：

```text
descriptor
open 结果
帧完整性
UVC stats
系统内存
HTTP 是否阻塞
```

# 15. 当前阶段结论

当前已经完成：

```text
Camera 可以被 ESP32-S3 USB Host 枚举
UVC descriptor 可以读取
320x240 MJPEG 可以打开
连续采集可以稳定运行
JPEG SOI / EOI 可以判断
latest JPEG cache 已建立
HTTP / Integration 可以读取 latest JPEG
```

当前仍需继续关注：

```text
HTTP 上传时的内存余量
underflow 的增长趋势
复杂画面下 JPEG 最大长度
长期运行是否有 drops / errors / disconnects
是否需要 PSRAM 支持更高分辨率
```

下一阶段方向：

```text
1. 继续验证 320x240 MJPEG 长时间稳定采集
2. 保证 latest JPEG 上传时完整
3. 在 LCD 上显示 Camera ONLINE / FPS / Frame len
4. 把 HTTP 返回的云端检测结果接入 integration
5. 根据真实检测结果控制 CH552G 的 L1~L5
```

一句话总结：

```text
摄像头模块第一阶段的核心不是“显示画面”，而是稳定获取最新一张可上传、可检测的 JPEG 图片。
```
