# 工程实战：第一阶段复盘

### 目标

第一阶段的目标不是一上来做一个完整产品，而是先把 RT-Thread 工程的核心闭环跑通。

这个阶段要验证的是：

1. BSP 层能正确打开外设
2. 应用层能通过 RT-Thread 设备模型访问硬件
3. 采样、控制、报警、Shell 调试之间能形成稳定链路
4. 线程之间不直接抢全局变量，而是用 RT-Thread IPC 做解耦
5. 后续接入真实传感器时，不需要推翻整体架构

最开始设定的是温控网关，后面根据开发板资源调整成智能家居环境监测核心版。

当前第一阶段更准确的结论是：

**核心闭环已经完成，真实温湿度闭环等待 SHT30/SHT31 到货后实测。**

不能把它说成完整真实环境检测系统，因为现在温湿度还保留模拟兜底。但是光敏输入、PWM 输出、蜂鸣器报警、Shell 状态查询这条链路已经形成了真实硬件闭环。

### 第一阶段最终闭环状态

当前系统链路：

```text
PF7 光敏传感器
    ↓ ADC3_IN5
board_port
    ↓
sensor_service
    ↓ rt_mq_send()
sample_mq
    ↓ rt_mq_recv()
control_service
    ↓
igw_model 共享状态
    ↓ rt_event_send()
alarm_service
    ↓
PF8 BEEP 蜂鸣器

Shell / MSH
    ↓
igw_status / igw_light / igw_beep / igw_fan
```

当前已经闭合的真实硬件链路：

```text
光照变化 → PF7 ADC采样 → 光照百分比换算 → 消息队列 → 控制线程 → 事件通知 → 报警线程 → PF8蜂鸣器
```

当前已经闭合的执行器链路：

```text
igw_fan / control_service → board_port → rt_pwm_set() → PWM3_CH1 → LED亮度变化
```

当前还没有实测闭合的链路：

```text
SHT30/SHT31 → I2C1 → 温湿度真实采样
```

但是底层设计已经准备好：

```text
sensor_service
    ↓
igw_board_read_sht30()
    ↓
rt_i2c_bus_device_find("i2c1")
    ↓
rt_i2c_transfer()
    ↓
SHT30/SHT31
```

这个判断很重要。工程复盘不能只写“完成了”，要写清楚完成边界。

### 实际完成的模块

#### 1. BSP 外设开启

本阶段打开并验证了这些 RT-Thread 设备：

```text
uart1   → Shell / 日志输出
pin     → 蜂鸣器、GPIO控制
pwm3    → PWM输出，当前用LED模拟风扇
adc3    → PF7 光敏传感器采样
i2c1    → SHT30/SHT31 预留接口，PB8/PB9 软件I2C
```

BSP 层的关键不是只开一个宏，而是三层同时成立：

```text
RT_USING_xxx
    ↓
BSP_USING_xxx
    ↓
HAL_xxx_MODULE_ENABLED + MSP 初始化
```

以 ADC 为例：

```text
RT_USING_ADC
BSP_USING_ADC3
HAL_ADC_MODULE_ENABLED
HAL_ADC_MspInit() 里打开 ADC3 时钟和 GPIOF PF7 模拟输入
```

以软件 I2C 为例：

```text
RT_USING_I2C / RT_USING_I2C_BITOPS
BSP_USING_I2C1
PB8 = SCL
PB9 = SDA
```

软件 I2C 不需要 HAL_I2C_MspInit，因为它本质上是 GPIO 模拟，由 RT-Thread 的 bitops I2C 驱动控制引脚。

---

#### 2. board_port 板级适配层

这层是本阶段最关键的工程分层。

业务层不应该到处写：

```c
rt_device_find("adc3");
rt_adc_read(adc, 5);
rt_pin_write(GET_PIN(F, 8), PIN_HIGH);
```

这些都是板级细节。

所以统一封装成：

```c
igw_board_set_fan_pwm();
igw_board_read_light_raw();
igw_board_read_light_percent();
igw_board_set_beep();
igw_board_read_sht30();
```

这样后面如果 PWM 从 TIM3 换到 TIM4，或者光敏从 PF7 换到别的 ADC 通道，只改 board_port，不改 control_service / sensor_service / alarm_service。

这就是 RTOS 工程里 BSP 层的价值：

```text
应用层关心业务含义
board_port 关心板子资源
drivers 关心 RT-Thread 设备注册
HAL/MSP 关心芯片底层时钟和引脚
```

---

#### 3. sensor_service 采样线程

sensor_service 的职责是周期性产生一帧环境采样数据。

现在采样结构已经从单一温度扩展为：

```c
typedef struct
{
    rt_int32_t temperature_x10;
    rt_int32_t humidity_x10;

    rt_uint32_t light_raw;
    rt_uint8_t light_percent;

    rt_uint32_t sequence;
} igw_sample_t;
```

当前策略：

```text
SHT30 读取成功 → 使用真实温湿度
SHT30 读取失败 → 使用模拟温湿度兜底
PF7 光敏       → 使用真实 ADC 数据
```

这个设计很实用。

如果强依赖 SHT30：

```text
SHT30 没接好 → sensor_service 失败 → control_service 没数据 → alarm_service 不工作
```

现在的降级策略是：

```text
温湿度传感器失败
    ↓
保留模拟温湿度
    ↓
光照、PWM、报警主链路继续运行
```

嵌入式工程里，系统不要因为一个非核心外设暂时不可用就整体趴下。

---

#### 4. control_service 控制线程

control_service 从消息队列接收最新采样。

```text
rt_mq_recv(sample_mq, &sample, sizeof(sample), RT_WAITING_FOREVER)
```

这个线程不主动轮询传感器，也不关心传感器来自 ADC 还是 I2C。

它只关心：

```text
有一帧 sample 到了
根据模式计算 pwm
更新共享模型
输出 PWM
发送控制更新事件
```

当前控制模式：

```text
auto   → 根据温度和目标温度计算 PWM
manual → 使用 shell 设置的 manual_pwm
safe   → PWM 最大输出
```

这一步体现了 RTOS 的优势。

裸机里常见写法是一个 while(1) 里到处 if 判断、读传感器、算控制、刷显示、查报警。

RT-Thread 里更合理的是：

```text
采样线程有采样节奏
控制线程等待消息
报警线程等待事件
Shell 线程修改参数
```

每个模块有自己的职责和阻塞点。

---

#### 5. alarm_service 报警线程

alarm_service 不直接轮询传感器，而是等待 control_service 发出的事件：

```text
control_service
    ↓ rt_event_send(IGW_EVENT_CONTROL_UPDATED)
alarm_service
    ↓ rt_event_recv()
```

报警判断当前包括：

```text
高温报警    IGW_ALARM_HIGH_TEMP
低光照报警  IGW_ALARM_LOW_LIGHT
```

低光照闭环：

```text
遮住光敏
    ↓
light_percent <= IGW_LOW_LIGHT_PERCENT
    ↓
alarm_mask |= IGW_ALARM_LOW_LIGHT
    ↓
igw_board_set_beep(RT_TRUE)
    ↓
PF8 蜂鸣器响
```

报警解除：

```text
光照恢复
    ↓
alarm_mask = 0
    ↓
igw_board_set_beep(RT_FALSE)
    ↓
蜂鸣器停
```

这里用 event 的原因是：报警线程不需要一直 while 轮询 model。

它只在控制状态更新时醒来判断一次。

#### 6. Shell 调试命令

第一阶段 Shell 命令非常重要，因为它让每个硬件能力都能单点验证。

当前命令包括：

```text
igw_status              查看系统状态
igw_target <temp>       修改目标温度
igw_mode auto/manual/safe
igw_pwm <0-1000>        设置手动 PWM
igw_fan <0-1000>        直接测试 PWM 输出
igw_light               读取光敏 raw 和 percent
igw_beep <0|1>          测试蜂鸣器
igw_sht30               测试 SHT30/SHT31 温湿度读取
```

这里的经验是：

**新硬件不要一上来塞进业务线程。**

先做 Shell 命令：

```text
单点测试硬件
确认设备名
确认引脚
确认方向
确认电平有效性
确认换算公式
```

再接入业务链路。

### RT-Thread API 复盘

#### rt_thread_create

```c
rt_thread_create(name, entry, parameter, stack_size, priority, tick);
```

核心参数：

```text
name        线程名，RT_NAME_MAX 有长度限制
entry       线程入口函数
parameter   传给线程的参数
stack_size  线程栈大小
priority    优先级，数字越小优先级越高
tick        同优先级时间片
```

本项目里：

```text
sensor_service 负责采样
control_service 负责控制
alarm_service 负责报警
```

线程拆分不是为了炫技，而是为了让不同功能有独立节奏。

---

#### rt_mq_send / rt_mq_recv

```c
rt_err_t rt_mq_send(rt_mq_t mq, const void *buffer, rt_size_t size);
```

消息队列用于传递采样帧。

```text
sensor_service
    ↓ rt_mq_send()
control_service
    ↓ rt_mq_recv()
```

优点：

1. 采样线程只负责发送数据，不关心谁处理
2. 控制线程可以阻塞等待数据，不需要空转轮询
3. 消息是拷贝传递，小结构体不需要共享内存
4. 短时间处理慢一点，队列可以缓冲
5. 后续扩展日志、显示、通信时边界更清楚

---

#### rt_mutex

mutex 用于保护共享状态 model。

```text
Shell 命令修改模型参数
control 线程更新模型状态
alarm 线程读取模型快照
```

如果没有 mutex，多线程同时读写会出现状态不一致。

本阶段的理解是：

```text
消息队列适合传递一帧数据
互斥量适合保护共享状态
事件适合通知状态变化
```

可以记成：

```text
rt_mq    = 传递数据
rt_mutex = 保护共享状态
rt_event = 通知变化
```

---

#### rt_event_send / rt_event_recv

事件用于通知报警线程：控制状态已经更新。

```text
新采样到了
控制状态更新了
报警线程醒来判断
```

事件不是传递大块数据的，它只传递“某件事发生了”。

本项目里数据在 model 里，事件只负责通知 alarm_service 该去读快照。

---

#### rt_device_find / 设备 API

RT-Thread 的设备模型让应用层不直接碰 HAL。

本阶段用到：

```c
rt_device_find()
rt_pwm_set()
rt_pwm_enable()
rt_adc_read()
rt_adc_enable()
rt_i2c_bus_device_find()
rt_i2c_transfer()
rt_pin_mode()
rt_pin_write()
```

这是和裸机差异最大的地方。

裸机写法经常是：

```c
HAL_ADC_Start();
HAL_ADC_GetValue();
HAL_GPIO_WritePin();
HAL_TIM_PWM_Start();
```

RT-Thread 工程更推荐：

```text
应用层 → board_port → RT-Thread device API → BSP/HAL
```

### BSP 层经验

开一个外设不是只点一个开关。

通用六问：

```text
1. RT_USING_xxx 开了吗？
2. BSP_USING_xxx 开了吗？
3. HAL_xxx_MODULE_ENABLED 开了吗？
4. MSP 初始化写了吗？
5. list device 能看到设备吗？
6. 应用层有没有通过 device API 使用，而不是直接碰 HAL？
```

PWM 的坑：

```text
只开 RT_USING_PWM 不够
还要 BSP_USING_PWM3 / BSP_USING_PWM3_CH1
还要 HAL_TIM_MODULE_ENABLED
还要 HAL_TIM_Base_MspInit 和 HAL_TIM_MspPostInit
```

ADC 的坑：

```text
PF7 是光敏输入
按 STM32F407 映射为 ADC3_IN5
要打开 ADC3 时钟
PF7 要配置为模拟输入
rt_adc_read() 当前驱动是软件触发 + 轮询等待
不是中断采样，也不是 DMA
```

软件 I2C 的坑：

```text
打开 RT_USING_I2C 还不够
必须定义至少一个 BSP_USING_I2Cx
否则 drv_soft_i2c.c 会报：Please define at least one BSP_USING_I2Cx
```

### 调试踩坑

#### 1. board_port 初始化提前 return

一开始 PWM 初始化成功后直接：

```c
return RT_EOK;
```

导致后面的 ADC 和 BEEP 初始化根本没有执行。

结果是：

```text
igw_light → read light failed
```

经验：

```text
板级初始化函数不要在第一个外设成功后就返回
应该所有外设初始化完成后，最后统一 return RT_EOK
```

---

#### 2. rt_device_find 参数写错

错误写法：

```c
rt_device_find(LIGHT_ADC_CHANNEL);
```

正确写法：

```c
rt_device_find("adc3");
```

`rt_device_find()` 找的是设备名，不是通道号。

ADC 的设备和通道要分开理解：

```text
设备名：adc3
通道号：5
```

---

#### 3. 光敏方向不能猜

实测结果：

```text
亮度高  → light_raw 低
亮度低  → light_raw 高
```

所以光敏电路是反相的。

不能直接：

```c
percent = raw * 100 / 4095;
```

应该根据实测方向做反向换算：

```c
light_percent = (LIGHT_ADC_DARK_RAW - raw) * 100 /
                (LIGHT_ADC_DARK_RAW - LIGHT_ADC_BRIGHT_RAW);
```

这件事的经验是：

**模拟量一定要实测方向，不能只凭原理想象。**

---

#### 4. SHT30 不能跳过 CRC

SHT30/SHT31 每两个数据字节后面有一个 CRC。

读取格式：

```text
Temp MSB
Temp LSB
Temp CRC
Humi MSB
Humi LSB
Humi CRC
```

如果不校验 CRC，I2C 抖动、接线错误、地址错误都可能被误认为真实数据。

CRC 参数：

```text
初始值：0xFF
多项式：0x31
```

换算公式：

```c
raw_temp = (buf[0] << 8) | buf[1];
raw_humi = (buf[3] << 8) | buf[4];

*temp_x10 = -450 + (1750 * raw_temp) / 65535;
*humi_x10 = (1000 * raw_humi) / 65535;
```

对应真实公式：

```text
T  = -45 + 175 * raw_temp / 65535
RH = 100 * raw_humi / 65535
```

注意用 x10 定点数，避免在 MCU 上引入不必要的浮点运算。

### 裸机 vs RTOS 的理解

裸机常见结构：

```text
while(1)
{
    读传感器
    算控制
    查报警
    刷显示
    处理串口命令
}
```

这种写法前期快，但后期所有功能挤在一个循环里。

RT-Thread 的结构：

```text
sensor_service  有自己的采样周期
control_service 等待消息队列
alarm_service   等待事件通知
shell_cmd       运行时修改参数
igw_model       统一保存共享状态
board_port      屏蔽板级硬件细节
```

这不是为了把简单事情复杂化，而是为了让系统变大以后还能维护。

本阶段最大的收获是：

```text
RTOS 的优势不是“能创建线程”
而是每个模块都有自己的时间节奏、阻塞边界和通信方式
```

具体到本项目：

```text
传感器慢，不影响 Shell
Shell 等用户输入，不影响控制线程
报警线程平时睡眠，不浪费 CPU
控制线程没有数据时阻塞等待，不空转
```

这才是 RT-Thread API 真正带来的工程价值。

### 下一阶段

下一阶段要把温湿度从模拟值替换成真实 SHT30/SHT31。

明天元件到后按这个顺序验证：

```text
1. 接线：VCC → 3.3V，GND → GND，SCL → PB8，SDA → PB9
2. list device 确认有 i2c1
```

如果 SHT30 成功，第一阶段可以升级为：

```text
真实温湿度 + 真实光照 + PWM输出 + 蜂鸣器报警 + Shell状态查看
```

到那时才可以说：

**智能家居环境监测核心版第一阶段完整闭环。**

当前阶段的准确说法是：

**第一阶段核心控制链路已闭环，真实温湿度输入等待硬件实测。**
