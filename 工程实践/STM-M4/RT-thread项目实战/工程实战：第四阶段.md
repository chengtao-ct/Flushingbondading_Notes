1. 状态轮询
   - 默认每 1000ms 请求 /api/status
   - 连续失败 3 次判定离线
   - 离线时禁用 Apply，但保留 Emergency Safe 的尝试按钮

2. 参数编辑
   - 用户修改模式、目标温度、手动 PWM 后，不立即下发
   - UI 显示 Pending Changes
   - 点击 Apply 后统一 POST /api/control

3. 安全控制
   - Emergency Safe 是单独按钮
   - 点击后需要二次确认
   - 成功后模式切到 safe
   - 失败时显示高优先级错误

4. 控制反馈
   - Apply 后进入 Sending
   - 收到成功响应后显示 Applied
   - 下一次 /api/status 返回新状态后，才认为状态同步完成
   - 超时或失败时保留用户编辑值，并提示未生效

5. 报警事件
   - V1 先显示报警和事件
   - 是否做 Ack/确认可以后续决定



STM32 负责：
  - 计算 alarm_mask
  - 返回当前报警状态
  - 可选返回报警文本/等级

PC Web HMI 负责：
  - 展示 Active Alarms
  - 根据 alarm_mask 映射报警名称
  - 在 Events 区记录“报警出现/恢复”的前端事件
  - 不向 STM32 写报警确认状态

方法：
  POST

Content-Type：
  application/json

可写字段：
  mode: "auto" | "manual" | "safe"
  target_x10: 0..800
  manual_pwm: 0..1000
  reason: optional string

规则：
  - 字段可以部分提交
  - unknown key 拒绝
  - 数值越界拒绝
  - mode 非法拒绝
  - 成功后返回当前快照或 applied 字段

对，你这个质疑成立：**从物理世界来说，temp 确实也是 environment。**

我前面把它单独叫 `Thermal`，不是因为它不属于环境，而是因为在你的项目里它的“控制地位”不一样。

更准确地说应该分成：

**1. Control Process / Temperature Loop**

这里放：

- current temperature
- target temperature
- fan PWM
- manual PWM 可选
- mode 可选

原因是温度在你的系统里不是普通环境量，它是闭环控制对象。你的 `control_service` 是围绕温度误差来算 PWM 的，所以温度更像“被控过程量 PV”，目标温度是“设定值 SP”，风扇 PWM 是“控制输出 MV”。

这在工业控制里是一个经典三元组：

```
PV: Process Value      当前温度
SP: Set Point          目标温度
MV: Manipulated Value  风扇 PWM
```

所以它不应该和湿度、光照、雨水完全平级。

**2. Auxiliary Environment / Environmental Sensors**

这里放：

- humidity
- light
- rain
- rain_detected

这些也是环境量，但它们现在主要用于监测或报警，不直接进入 PID 控制主回路。

所以我会改名，不叫 `Thermal Trend`，而叫：

- `Control Loop Trend`
- `Environment Trend`

这样就清楚了：

```
Control Loop Trend:
temperature / target / fan PWM

Environment Trend:
humidity / light / rain
```

你说得很对，问题不是 temp 是否属于 environment，而是它在这个系统里的角色：**温度既是环境数据，也是主控制回路的核心过程量。**

所以我的修正建议是：中间区域不要按“物理类别”分，而按“工程角色”分。这样更像工业 HMI。



我基本同意你的方向，但我要小小反驳一个点：**右下角那块不是“没必要”，而是“不应该以现在这种大小和位置出现在 Dashboard”。**

它的信息有价值：连接状态、延迟、失败次数、最后错误，对调试很重要。但它不应该抢主操作台的黄金区域。它更适合：

你提出的新版结构更像真正的工业 HMI：

```
左侧：Command 控制区
中间：igw_model 变化曲线区
右侧上方：实时返回值
右侧下方：Events / 报警 / 操作记录
```

这个方向我认为更合理。

**为什么中间应该放曲线**

你的系统核心不是单点数据，而是一个过程控制系统。`igw_model` 里的状态是持续变化的：

- temperature
- target temperature
- humidity
- light
- rain
- fan_pwm
- manual_pwm
- mode
- alarm_mask
- sample_sequence

这些东西只用数字卡片展示，会少掉“趋势判断”能力。比如：

- 温度是否正在接近目标
- 风扇 PWM 是否长期打满
- 光照下降是否触发低光报警
- 雨水传感器是否抖动
- humidity 是否和温度变化有关
- sample sequence 是否中断

所以中间主区域应该给曲线，这是你这个项目最有“上位机感”的部分。

**但我不同意把所有 igw_model 字段塞进一张曲线**

这里我反驳你一下：如果“一张图显示所有变化”，会很乱。温度是 `x10`，湿度是 `% x10`，光照是 `%`，雨水是 `%`，PWM 是 `0-1000`，mode 是枚举，alarm_mask 是位图。单位不同，量级不同，放在同一个 y 轴上没有工程意义。

我建议中间区域做成“曲线组”，而不是“一张万能曲线”：

- Thermal：温度 + 目标温度 + 风扇 PWM
- Environment：湿度 + 光照 + 雨水
- Control：模式、manual_pwm、fan_pwm、报警 mask 的阶梯/状态图

Dashboard 默认显示前两个：上面 Thermal，下面 Environment。Control 细节可以在 Trends 页面展开。

**右侧实时返回值应该更像状态寄存器表**

右侧上方不要再叫 `Process Overview`，可以改成 `Live Snapshot` 或 `Realtime Values`。

它显示当前 `/api/status` 的最新值：

```
Temp        25.3 C
Humi        61.2 %
Light       78 %
Rain        12 % / DRY
Fan PWM     350
Manual PWM  400
Mode        MANUAL
Target      28.0 C
Seq         42
Alarm Mask  0x00
```

这个区域的价值是“看当前值”，中间曲线的价值是“看变化过程”。两者分工明确。

**右侧下方 Events 更合理**

Events 应该记录“发生了什么”，而不是混在实时值里：

- command sent
- command synchronized
- command failed
- connection offline / recovered
- alarm appeared / recovered
- mode changed
- emergency safe requested

这样右侧就是一个“当前状态 + 最近事件”的操作员信息栏。

**我推荐的 Dashboard 新布局**

```
┌──────────────┬──────────────────────────────┬──────────────────┐
│ Command      │ Thermal Trend                │ Realtime Values  │
│              │ Temp / Target / Fan PWM      │ igw_model latest │
│              ├──────────────────────────────┤                  │
│              │ Environment Trend            ├──────────────────┤
│              │ Humi / Light / Rain          │ Events / Alarms  │
└──────────────┴──────────────────────────────┴──────────────────┘
```

底部不再放大块 Diagnostics。通信状态收进顶部状态栏，详细诊断去 `Diagnostics` 页面。
