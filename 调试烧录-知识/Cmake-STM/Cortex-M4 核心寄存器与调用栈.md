# Cortex-M4 核心寄存器与调用栈

> [!abstract] 核心本质
> 调试器的"上帝视角"源于三个核心寄存器（PC/LR/SP）与栈的协作机制。PC 定位"我在哪"，LR 记录"我从哪来"，SP 管理"现场保护所"，三者共同构成了程序执行流的完整追踪链路。

---

## 一、核心寄存器：CPU 的"导航系统"

### 1.1 三大核心寄存器

| 寄存器 | 全称 | 职责 | 类比 |
| --- | --- | --- | --- |
| **PC** | Program Counter | 指向当前执行的指令地址 | CPU 的"眼睛" |
| **LR** | Link Register | 存放函数返回地址 | 调用的"回程票" |
| **SP** | Stack Pointer | 指向栈顶位置 | 内存的"记账员" |

### 1.2 PC：程序计数器

PC 是 CPU 的"眼睛"，它永远指向**当前正在执行**的指令地址。

```
Flash 代码区：
┌──────────────────┐ 0x0800 1000
│ MOV R0, #10      │ ← PC 指这里，CPU 正在执行
├──────────────────┤ 0x0800 1004
│ BL func_B        │
├──────────────────┤ 0x0800 1008
│ MOV R0, #20      │
└──────────────────┘
```

**关键特性**：
- 每执行一条指令，PC 自动增加（指向下一条）
- 遇到分支指令（B/BL），PC 跳转到目标地址
- 调试器读 PC 就知道"程序停在哪里"

### 1.3 LR：链接寄存器

LR 存放的是**函数返回地址**——即调用指令的下一条指令地址。

```c
void func_A(void) {
    int a = 10;       // 地址 0x0800 1000
    func_B();         // 地址 0x0800 1004，BL 指令
    a = 20;           // 地址 0x0800 1008 ← LR = 这个！
}
```

**执行 `BL func_B` 时**：
1. 硬件把 `0x0800 1008`（下一条指令地址）存入 LR
2. PC 跳转到 `func_B` 的入口

> [!warning] LR 只有一个
> 当发生嵌套调用时，LR 会被覆盖。必须把旧的 LR 压栈保存！

### 1.4 SP：栈指针

SP 是一个**寄存器**，它只存一个值：**栈顶的地址**。

```
SP 寄存器：存着 "0x2000 1FF0" 这个数字
RAM 的 0x2000 1FF0 位置：才是真正存数据的地方

类比：
SP = 你的手指（指向某个柜子）
RAM = 一排储物柜（实际存放物品）
```

> [!tip] SP 的本质
> SP 是"指针"，不是"容器"。它告诉你"栈用到哪了"，数据实际存在 RAM 里。

## 二、栈机制：函数调用的"现场保护所"

### 2.1 栈的本质

**栈是一块连续的 RAM 区域**，用于存储：
- 函数返回地址
- 局部变量
- 寄存器现场（中断/函数调用时）

```
RAM 地址空间
┌──────────────────┐ 0x2000 2000  ← 栈区最高地址
│                  │
│     （空闲）      │
│                  │
├──────────────────┤ 0x2000 1FF0  ← SP 当前位置（栈顶）
│   返回地址        │
├──────────────────┤ 0x2000 1FEC
│   局部变量 y      │
├──────────────────┤ 0x2000 1FE8
│   局部变量 x      │
└──────────────────┘ 0x2000 1FE4
```

### 2.2 压栈与出栈

| 操作 | 英文 | SP 变化 | 类比 |
| --- | --- | --- | --- |
| 存数据 | **PUSH（压栈）** | SP 减小 | 再放一个盘子 |
| 取数据 | **POP（出栈）** | SP 增大 | 拿走最上面的盘子 |

**"压栈"不是"进入新的栈"**，而是在同一个栈上，SP 往下移动，腾出空间存新数据。

就像一摞盘子：

```
你有一张桌子（RAM）
桌上放着一摞盘子（栈）
"压栈" = 再放一个盘子上去
"出栈" = 拿走最上面的盘子

桌子没变，盘子数量在变。
```

### 2.3 函数调用的完整流程

```c
void func_A(void) {      // 入口 0x0800 1000
    int a = 10;          // 地址 0x0800 1002
    func_B();            // 地址 0x0800 1004，返回点 0x0800 1008
    a = 20;              // 地址 0x0800 1008
}

void func_B(void) {      // 入口 0x0800 2000
    int x = 100;
    int y = 200;
}
```

**执行过程**：

```
Step 1: func_A 执行，局部变量 a 入栈
┌──────────────────┐ 0x2000 1FFC ← SP
│   a = 10         │
└──────────────────┘

Step 2: 调用 func_B，返回地址入栈
┌──────────────────┐ 0x2000 1FFC
│   a = 10         │
├──────────────────┤ 0x2000 1FF8 ← SP
│   返回地址        │ = 0x0800 1008
└──────────────────┘

Step 3: func_B 执行，局部变量入栈
┌──────────────────┐ 0x2000 1FFC
│   a = 10         │
├──────────────────┤ 0x2000 1FF8
│   返回地址        │
├──────────────────┤ 0x2000 1FF4
│   x = 100        │
├──────────────────┤ 0x2000 1FF0 ← SP
│   y = 200        │
└──────────────────┘

Step 4: func_B 返回，SP 上移
┌──────────────────┐ 0x2000 1FFC
│   a = 10         │
├──────────────────┤ 0x2000 1FF8 ← SP（回到这里）
│   返回地址        │ （读出 0x0800 1008，跳转执行）
├──────────────────┤ 0x2000 1FF4
│   x = 100        │ ← 数据还在，但变成"野数据"
├──────────────────┤ 0x2000 1FF0
│   y = 200        │
└──────────────────┘
```

### 2.4 局部变量的"失效"真相

> [!danger] 关键误区
> 函数返回后，局部变量**没有被删除**，只是被"遗弃"了！

**真相**：
- 数据还在 RAM 里
- 只是 SP 移走了，不再"指向"它
- 后续函数调用会**覆盖**这块内存

**危险实验**：

```c
int *p = NULL;  // 全局指针

void func_B(void) {
    int x = 100;
    p = &x;          // 把 x 的地址存到全局指针
}

void func_A(void) {
    func_B();
    printf("%d", *p);   // 可能打印 100，也可能打印乱码！
}
```

**打个比方**：

```
你住酒店（RAM），开了一个房间（栈空间）。
你在房间里放了一个行李箱（x = 100）。

退房时（函数返回）：
- 你没有把行李箱拿走
- 酒店也没有清理房间
- 行李箱还在房间里！

但：
- 你已经"退房"了，这个房间不再属于你
- 下一个客人（其他函数）入住时，会把你的行李箱扔出去，放他自己的东西

所以：
- 行李箱没有被"删除"
- 它只是被"遗弃"了，随时可能被"覆盖"
```

## 三、普通函数调用 vs 中断/异常

> [!important] 核心区别
> 普通函数调用由**编译器**生成压栈代码，中断/异常由**硬件自动**完成压栈。

### 3.1 普通函数调用

```
func_A 调用 func_B：

┌─────────────────────────────────────────────────────┐
│  func_A:                                            │
│      ...                                            │
│      BL func_B        ; 硬件：LR = 返回地址         │
│      ...                                            │
│                                                      │
│  func_B:                                            │
│      PUSH {R7, LR}    ; 编译器：把 LR 压栈          │
│      SUB SP, SP, #8   ; 编译器：给局部变量腾空间    │
│      ...函数体...                                   │
│      ADD SP, SP, #8   ; 编译器：释放局部变量空间    │
│      POP {R7, PC}     ; 编译器：返回地址弹到 PC     │
└─────────────────────────────────────────────────────┘
```

**关键点**：
- `BL` 指令：硬件把返回地址写入 LR
- `PUSH/POP`：**编译器生成**的代码，负责保存/恢复现场

### 3.2 中断/异常

```
中断发生：

┌─────────────────────────────────────────────────────┐
│  主程序正在执行...                                   │
│                                                      │
│  中断信号来了！                                      │
│                                                      │
│  ┌─────────────────────────────────────────────┐    │
│  │ 硬件自动压栈（不需要编译器生成代码）：        │    │
│  │   R0, R1, R2, R3, R12, LR, PC, xPSR         │    │
│  └─────────────────────────────────────────────┘    │
│                                                      │
│  LR = 0xFFFFFFF9（EXC_RETURN，特殊标记）            │
│  PC = 中断处理函数地址                               │
│                                                      │
│  中断处理函数执行...                                 │
│                                                      │
│  BX LR               ; CPU 认出 EXC_RETURN          │
│                       ; 硬件自动出栈 8 个寄存器      │
│                       ; PC 恢复，继续执行主程序      │
└─────────────────────────────────────────────────────┘
```

**关键点**：
- 压栈：**硬件自动完成**，压入固定 8 个寄存器
- LR：变成 **EXC_RETURN**（如 0xFFFFFFF9），不是返回地址
- 返回：`BX LR` 触发硬件自动出栈

### 3.3 EXC_RETURN：中断返回的"暗号"

**普通函数返回**：

```asm
BX LR    ; CPU 跳转到 LR 存的地址
```

**中断返回**：

```asm
BX LR    ; LR = 0xFFFFFFF9
```

CPU 看到这个值，**不是跳转到 0xFFFFFFF9**（那是无效地址）。

**而是触发特殊行为**：

```
CPU 看到 0xFFFFFFF9：
"哦，这是 EXC_RETURN！"
"我要从中断返回了！"
"返回地址在哪？在栈里！"
"从栈里弹出 8 个寄存器，恢复现场！"
```

| 场景 | LR 存什么 | 返回时做什么 |
| --- | --- | --- |
| 普通函数 | 返回地址（如 0x0800 1008） | 跳转到该地址 |
| 中断 | EXC_RETURN（如 0xFFFFFFF9） | 从栈中恢复现场 |

### 3.4 中断时栈中保存的 PC

假设主程序正在执行：

```c
while(1) {
    a = 10;        // 地址 0x0800 1000
    b = 20;        // 地址 0x0800 1004 ← 中断在这里发生！
    c = 30;        // 地址 0x0800 1008
}
```

**中断发生时**：

```
1. CPU 正要执行 0x0800 1004（b = 20）
2. 中断信号来了！
3. 硬件自动压栈：
   - PC = 0x0800 1004（被打断的指令地址）← 关键！
4. LR 变成 0xFFFFFFF9（EXC_RETURN）
5. PC 跳转到中断处理函数
```

**中断返回时**：

```
1. CPU 执行 BX LR（LR = 0xFFFFFFF9）
2. CPU 认出这是 EXC_RETURN
3. 从栈中弹出 8 个寄存器
4. PC = 0x0800 1004（从栈中恢复）
5. CPU 继续执行 b = 20
```

> [!tip] 核心结论
> 栈中保存的 PC = **被打断的那条指令的地址**
> 
> 这就是为什么中断返回后，程序能"无缝衔接"——因为 CPU 记住了"刚才执行到哪"。

### 3.5 完整对比表

| 对比项 | 普通函数调用 | 中断/异常 |
| --- | --- | --- |
| 触发方式 | 软件指令（BL） | 硬件信号 |
| 压栈时机 | 编译器在函数入口生成 | 硬件自动完成 |
| 压栈内容 | LR + 编译器决定的其他寄存器 | 固定 8 个寄存器 |
| LR 的值 | 返回地址 | EXC_RETURN（特殊标记） |
| 返回方式 | `POP {PC}` 或 `BX LR` | `BX LR`（触发硬件出栈） |
| 栈中 PC | 调用者的返回地址 | 被打断的指令地址 |

## 四、栈帧与调用链

### 4.1 栈帧结构

每个函数在栈上占用的区域叫**栈帧**，包含：

```
┌─────────────────────────────────────┐
│           func_A 的栈帧              │
│  ┌─────────────────────────────┐    │
│  │ 局部变量 a                   │    │
│  ├─────────────────────────────┤    │
│  │ 返回地址（指向 func_A 调用者）│    │
│  └─────────────────────────────┘    │
├─────────────────────────────────────┤
│           func_B 的栈帧              │
│  ┌─────────────────────────────┐    │
│  │ 局部变量 x, y                │    │
│  ├─────────────────────────────┤    │
│  │ 返回地址（指向 func_A）       │    │
│  └─────────────────────────────┘    │
├─────────────────────────────────────┤
│           func_C 的栈帧              │
│  ┌─────────────────────────────┐    │
│  │ 局部变量 m                   │    │
│  └─────────────────────────────┘    │
└─────────────────────────────────────┘
```

### 4.2 调试器如何还原调用链

当程序停在断点时，调试器显示：

```
#0  func_C () at main.c:25
#1  0x08001234 in func_B () at main.c:18
#2  0x08001100 in func_A () at main.c:10
#3  0x08001000 in main () at main.c:5
```

**原理**：

```
1. 读 PC → 知道当前在 func_C 执行
2. 读 SP → 知道栈顶位置
3. 查 .debug_frame → 知道栈帧大小
4. 计算：SP + 栈帧大小 = 返回地址位置
5. 读返回地址 → 知道调用者是 func_B
6. 重复 2-5 → 还原完整调用链
```

### 4.3 调试信息的来源

栈帧信息存在 **ELF 文件的 `.debug_*` 段**：

```
.elf 文件
├── .text         → 机器指令
├── .data         → 数据
└── .debug_*      → 调试信息
    ├── .debug_info    → 函数名、变量名、类型
    ├── .debug_line    → 代码行号
    └── .debug_frame   → 栈帧大小、寄存器保存位置
```

| 编译选项 | 文件大小 | 调试能力 |
| --- | --- | --- |
| `-O0 -g` | 大 | ✅ 完整调试信息 |
| `-O2` | 小 | ❌ 优化后难以调试 |
| `-O2 -g` | 中等 | ⚠️ 部分信息丢失 |

## 五、断点机制：让 CPU "暂停"的艺术

### 5.1 软件断点

**原理**：把目标指令替换为 `BKPT` 指令

```
设置断点前：
0x0800 1004:  BL func_B    ; 原指令

设置断点后：
0x0800 1004:  BKPT #0      ; 被替换成断点指令
```

**CPU 执行到 `BKPT` 时**：
1. 触发 Debug 异常
2. CPU 进入"暂停状态"
3. 调试器获得控制权

**缺点**：
- 需要修改 [[../../嵌入式/硬件与芯片/内存/Flash]]
- Flash 擦写寿命有限（约 10000 次）
- 无法用于 ROM 代码

### 5.2 硬件断点

**原理**：使用 CPU 内部的 **FPB（Flash Patch and Breakpoint）** 比较器

```
┌─────────────────────────────────────┐
│           CPU 内部                   │
│  ┌─────────────────────────────┐    │
│  │  FPB 比较器                  │    │
│  │  ┌─────────────────────┐    │    │
│  │  │ 地址：0x0800 1004    │    │    │
│  │  │ 使能：1              │    │    │
│  │  └─────────────────────┘    │    │
│  └─────────────────────────────┘    │
└─────────────────────────────────────┘

CPU 每次取指时检查：
"PC 等于 0x0800 1004 吗？是的？触发断点！"
```

**优点**：不修改 Flash
**缺点**：数量有限（Cortex-M4 最多 6 个）

### 5.3 数据断点（观察点）

**原理**：使用 **DWT（Data Watchpoint and Trace）** 单元监控内存访问

```
┌─────────────────────────────────────┐
│           DWT 单元                   │
│  ┌─────────────────────────────┐    │
│  │ 观察点比较器                 │    │
│  │ ┌─────────────────────┐     │    │
│  │ │ 地址：0x2000 0100    │     │    │
│  │ │ 类型：写             │     │    │
│  │ │ 使能：1              │     │    │
│  │ └─────────────────────┘     │    │
│  └─────────────────────────────┘    │
└─────────────────────────────────────┘

CPU 每次访问内存时检查：
"正在写入 0x2000 0100 吗？是的？触发断点！"
```

**典型应用**：定位变量被意外修改的 Bug

```c
int motor_speed = 100;  // 某处被改成 0，导致电机停转

// 设置数据断点监控写入 motor_speed
// 程序会在任何写入 motor_speed 的地方暂停
```

### 5.4 三种断点对比

| 类型 | 硬件单元 | 触发条件 | 数量限制 | 适用场景 |
| --- | --- | --- | --- | --- |
| 软件断点 | 无（修改 Flash） | 执行某地址 | 无限 | 普通代码调试 |
| 硬件执行断点 | FPB | 执行某地址 | 4-6 个 | ROM 代码、Flash 寿命敏感 |
| 数据断点 | DWT | 读/写某地址 | 2-4 个 | 变量被篡改定位 |

## 六、HardFault 定位：从崩溃现场找真凶

### 6.1 HardFault 发生时的状态

当 HardFault 发生时，硬件自动：

1. **压栈 8 个寄存器**：R0, R1, R2, R3, R12, LR, PC, xPSR
2. **LR 被赋予 EXC_RETURN**（如 0xFFFFFFF9）
3. **跳转到 HardFault_Handler**

**此时寄存器状态**：

| 寄存器 | 值 | 有用吗？ |
| --- | --- | --- |
| PC | `HardFault_Handler` 地址 | ❌ 没用 |
| LR | `0xFFFFFFF9`（EXC_RETURN） | ❌ 不是返回地址 |
| SP | 指向栈顶 | ✅ **关键线索！** |

### 6.2 栈中寄存器布局

```
SP →  ┌──────────────┐
      │  R0          │ 偏移 0
      ├──────────────┤
      │  R1          │ 偏移 4
      ├──────────────┤
      │  R2          │ 偏移 8
      ├──────────────┤
      │  R3          │ 偏移 12
      ├──────────────┤
      │  R12         │ 偏移 16
      ├──────────────┤
      │  LR          │ 偏移 20  ← 调用者的返回地址
      ├──────────────┤
      │  PC          │ 偏移 24  ← 崩溃时的指令地址！
      ├──────────────┤
      │  xPSR        │ 偏移 28
      └──────────────┘
```

### 6.3 工程实践：HardFault 定位代码

```c
void HardFault_Handler(void) {
    // 1. 读 SP，找到栈顶位置
    uint32_t *sp;
    __asm volatile ("MRS %0, MSP" : "=r"(sp));
    
    // 2. 从栈中读取"案发时刻"的关键寄存器
    uint32_t r0  = sp[0];
    uint32_t r1  = sp[1];
    uint32_t r2  = sp[2];
    uint32_t r3  = sp[3];
    uint32_t r12 = sp[4];
    uint32_t lr  = sp[5];  // 调用者的返回地址
    uint32_t pc  = sp[6];  // 崩溃时的指令地址！
    uint32_t psr = sp[7];
    
    // 3. 打印或通过调试器查看
    printf("HardFault!\n");
    printf("PC = 0x%08X\n", pc);  // 用这个值在 .map 文件中定位
    printf("LR = 0x%08X\n", lr);
    
    while(1);  // 死循环，等待调试器连接
}
```

> [!tip] 定位方法
> 用 PC 值在 `.map` 文件或 `.lst` 文件中查找，定位到具体的函数和代码行。

## 七、工程实践建议

### 7.1 调试策略

| 场景 | 推荐方案 |
| --- | --- |
| 普通代码调试 | 软件断点（数量无限） |
| ROM 代码调试 | 硬件断点（无法修改 Flash） |
| 变量被篡改 | 数据断点（DWT 观察点） |
| 程序崩溃 | 栈回溯 + HardFault 处理函数 |

### 7.2 避坑指南

> [!warning] 栈溢出
> 局部变量过大、递归过深都会导致栈溢出，覆盖其他内存区域。建议：
> - 大数组使用 `static` 或动态分配
> - RTOS 任务栈预留足够余量

> [!warning] 优化导致调试困难
> `-O2` 及以上优化会重排指令、消除死代码、内联函数，导致变量"消失"。
> 
> 调试时建议使用 `-O0 -g`

### 7.3 调试器资源限制

| 资源 | 数量 | 备注 |
| --- | --- | --- |
| 硬件执行断点（FPB） | 最多 6 个 | 取决于芯片实现 |
| 数据断点（DWT） | 最多 4 个 | 通常 2-4 个 |
| 软件断点 | 无限 | 受 Flash 大小限制 |
