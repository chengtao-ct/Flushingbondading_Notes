# 栈与上下文

> [!abstract]
> 栈是保存 CPU 工作现场的内存区域，上下文 = 寄存器组，上下文切换的本质是换 SP 指针。理解栈帧布局和伪造栈帧，是掌握 RTOS 任务切换原理的根基。

> [!info] 面试开场句
> "栈的本质是一块连续 RAM 加一个 SP 指针。Cortex-M 中断时硬件自动压栈 8 个寄存器，用 EXC_RETURN 机制触发返回。RTOS 任务切换通过伪造栈帧，让 CPU '以为'任务被中断过，从而自然地切换到新任务。"

---

## 一、栈的硬件本质

数据结构里的栈是"后进先出"的抽象。但在 MCU 里，栈非常具体：

**栈 = 一块连续的 RAM + 一个 SP 寄存器（Stack Pointer）**

```
SRAM（STM32F407 有 192KB）
┌──────────────┐ 0x20030000  ← SRAM 顶部
│              │
│  （未使用）   │
│              │
├──────────────┤ ← SP 指向这里（栈顶）
│  压入的数据   │  SP 每次压栈 -= 4（向下生长）
│  压入的数据   │
│  压入的数据   │
└──────────────┘ 0x20000000  ← SRAM 底部
```

- **SP 不是数据本身**，它是一个指针，告诉 CPU "当前栈顶在哪"
- ARM Cortex-M 用 **满递减栈（Full Descending）**：先减 SP，再写入数据
- 压栈等价于 `SP -= 4; *SP = data;`
- 出栈等价于 `data = *SP; SP += 4;`

> [!tip] 中断保存现场的本质，就是"往栈里压一堆寄存器值"。理解了栈的物理结构，后续所有机制都水到渠成。

## 二、两个栈指针：MSP 与 PSP

### 2.1 物理关系

Cortex-M 内部 R13（SP）有**两个物理副本**，同一时刻只用一个：

```
R13 = SP（别名）
  ├── MSP（Main Stack Pointer）     ← 系统栈
  └── PSP（Process Stack Pointer）   ← 进程栈

CONTROL 寄存器 Bit[1] 决定当前用哪个：
  CONTROL[1] = 0 → 用 MSP
  CONTROL[1] = 1 → 用 PSP
```

### 2.2 使用规则

| 场景 | main / 任务 | 中断（Handler 模式） |
|------|------------|-------------------|
| **裸机** | MSP | MSP（同一个） |
| **RTOS** | PSP（每个任务独立） | MSP（所有中断共享） |

```
RTOS 下的内存布局：

MSP（系统栈）              PSP_A（任务A）       PSP_B（任务B）
┌──────────┐              ┌──────────┐        ┌──────────┐
│ 中断共用  │              │ 任务A的   │        │ 任务B的   │
│ 的栈空间  │              │ 局部变量  │        │ 局部变量  │
│          │              │ 调用链    │        │ 调用链    │
└──────────┘              └──────────┘        └──────────┘

切换任务 = 切换 PSP，MSP 永远不动
```

### 2.3 为什么 RTOS 要隔离

- 任务栈溢出不会破坏中断系统（MSP 在独立的内存区域）
- 每个任务的栈大小独立配置，互不影响
- 中断栈使用量可以精确预估（嵌套深度 × 8 个寄存器 + ISR 临时变量）

### 2.4 中断时的 SP 切换过程

```
任务A 正在执行（用 PSP_A）
  │
  ├── 中断来了
  │
  ├── CPU 硬件自动做：
  │   1. 用当前 SP（PSP_A）压入 8 个寄存器 ← 注意：压到任务栈上！
  │   2. SP 切换到 MSP
  │   3. LR = EXC_RETURN（0xFFFFFFFD → 返回 Thread 模式，用 PSP）
  │
  ├── ISR 执行（用 MSP）
  │
  ├── ISR 执行完 BX LR
  │   CPU 硬件自动做：
  │   1. 识别 EXC_RETURN → 触发出栈
  │   2. SP 切换回 PSP_A
  │   3. 从 PSP_A 的栈帧弹出 8 个寄存器
  │   4. 任务A 继续，完全不知道被打断过
```

> [!important] 进入中断时 8 个寄存器压到的是**任务栈（PSP）**上，不是 MSP 上。切换到 MSP 是压栈之后的事。ISR 执行期间用的是 MSP，ISR 的局部变量、函数调用消耗的是 MSP 空间。

## 三、上下文 = 寄存器组

想象你在做数学题，写到一半被打断。你需要记住什么才能回来继续？

| 需要记住的 | 对应寄存器 | 含义 |
|-----------|-----------|------|
| 算到哪一步了 | PC（R15） | 程序计数器，下一条要执行的指令地址 |
| 中间结果 | R0-R3 | 通用寄存器，函数参数和返回值 |
| 用了什么规则 | xPSR | 程序状态（条件标志 + Thumb标志 + 中断号） |
| 做完该去哪 | LR（R14） | 链接寄存器，函数返回地址 |
| 临时工具 | R12 | 通用寄存器（IP） |
| 自己的工具箱 | R4-R11 | 通用寄存器，callee-saved |

### caller-saved vs callee-saved

AAPCS（ARM 架构过程调用标准）把寄存器分成两类：

| 类别 | 寄存器 | 含义 | 谁保存 |
|------|--------|------|--------|
| caller-saved | R0-R3, R12 | 调用者保存 | 被调函数可以自由使用，调用者负责保存 |
| callee-saved | R4-R11 | 被调者保存 | 被调函数如果用了，必须在使用前保存，返回前恢复 |

> [!tip] Cortex-M 硬件自动压栈的 8 个恰好就是 caller-saved 组。R4-R11 不保存，因为 ISR（被调函数）如果要用，编译器会自动生成 push/pop 指令保存。

## 四、硬件自动压栈（Cortex-M 核心机制）

中断发生时，CPU 硬件自动把 8 个寄存器压栈，不需要软件介入：

```
压栈顺序（从高地址到低地址）：

  ┌──────────┐
  │ xPSR     │  ← SP + 0x1C   程序状态寄存器
  │ PC       │  ← SP + 0x18   返回地址（被中断的指令+2 或+4）
  │ LR       │  ← SP + 0x14   原来的链接寄存器值
  │ R12      │  ← SP + 0x10   通用寄存器
  │ R3       │  ← SP + 0x0C
  │ R2       │  ← SP + 0x08
  │ R1       │  ← SP + 0x04
  │ R0       │  ← SP + 0x00   ← SP 指向这里
  └──────────┘

总共 8 × 4 = 32 字节
```

### 为什么只保存 8 个

ARM 的设计哲学是 **"按需付费"**：

```
如果硬件保存全部 16 个（R0-R11 + R12, LR, PC, xPSR）：
  - 每次中断多消耗 32 字节（R4-R11）
  - 嵌套 3 层多消耗 96 字节
  - 大部分 ISR 只用 R0-R3，根本不需要 R4-R11
  - 对栈空间紧张的嵌入式系统是浪费
  - 硬件复杂度增加，芯片面积和功耗增大

所以 ARM 只保存最低必需（caller-saved），R4-R11 由软件按需保存。
```

## 五、EXC_RETURN 与中断返回

### 5.1 LR 去哪了

进入中断时，CPU 做了两件事：
1. 把原来的 LR 作为栈帧的第 3 个元素**压栈保存**
2. 把 LR 寄存器本身**覆盖**为 EXC_RETURN 特殊值

```
时间线：
  ① CPU 把 R0, R1, R2, R3, R12, LR, PC, xPSR 依次压栈 ← LR 保存在栈帧里
  ② CPU 把 LR 寄存器覆盖为 EXC_RETURN（如 0xFFFFFFF9）
  ③ CPU 跳转到 ISR 执行
  ④ ISR 执行 BX LR → CPU 识别到 0xFFFFFFF9 → 触发出栈
  ⑤ 出栈弹出 8 个寄存器 → 其中第 3 个恢复到 LR → 原来的 LR 回来了！
```

> [!tip] LR 在中断期间反正不能做函数调用（ISR 里调用函数会覆盖 LR），不如利用它传递 EXC_RETURN，省去增加额外硬件状态位的成本。

### 5.2 EXC_RETURN 值含义

| EXC_RETURN 值 | 返回到 | 使用的 SP | 典型场景 |
|--------------|--------|----------|---------|
| 0xFFFFFFF1 | Handler 模式 | MSP | 中断嵌套，返回到上一级中断 |
| 0xFFFFFFF9 | Thread 模式 | MSP | 裸机程序，返回到 main |
| 0xFFFFFFFD | Thread 模式 | PSP | RTOS 任务，返回到任务代码 |

### 5.3 返回机制

```
ISR 最后一行 BX LR：
  CPU 检测到 LR >= 0xFFFFFFF0
  → 这不是普通地址！这是 EXC_RETURN！
  → 触发中断返回序列
  → 根据 EXC_RETURN 值决定切换到 MSP 还是 PSP
  → 自动出栈 8 个寄存器
  → 恢复到被打断的地方
```

## 六、伪造栈帧 — RTOS 任务切换的根基

### 6.1 为什么需要伪造

RTOS 创建新任务时，这个任务**从来没运行过**，没有"被打断的现场"。但 Cortex-M 的中断返回机制要求栈上有完整的 8 个寄存器才能恢复执行。

**解决方法：伪造一个栈帧，让 CPU 以为这个任务被中断过。**

### 6.2 xPSR Bit[24] 必须为 1

```
xPSR 寄存器中 Bit[24] = T flag（Thumb 模式标志）
  T = 1 → CPU 处于 Thumb 状态
  T = 0 → CPU 处于 ARM 状态

Cortex-M 砍掉了 ARM 状态，只支持 Thumb/Thumb-2
如果伪造栈帧中 xPSR 的 Bit[24] = 0
  → CPU 尝试进入不存在的 ARM 状态
  → 直接触发 HardFault

所以 0x01000000（Bit[24] = 1）是必须的。
```

### 6.3 Thumb 状态 vs ARM 状态

ARM 状态和 Thumb 状态不是"指令集 vs 编码"，而是 **CPU 的两种硬件取指/解码模式**：

| | ARM 状态 | Thumb 状态 | Thumb-2 |
|--|---------|-----------|---------|
| 指令宽度 | 固定 32 位 | 固定 16 位 | 16 位 + 32 位混合 |
| 功能 | 完整指令集 | 子集 | 接近完整 |
| 代码密度 | 低 | 高 | 高 |
| 性能 | 最高 | 略低 | 接近 ARM 状态 |

```
经典 ARM 处理器（ARM7TDMI, Cortex-A）：
  ARM 状态 ↔ Thumb 状态    可通过 BX 指令切换
  CPSR 的 Bit[5] = T flag 决定当前模式

Cortex-M 处理器：
  只有 Thumb/Thumb-2 状态   永远在 Thumb 模式
  硬件上不存在 ARM 状态解码器
```

### 6.4 完整代码示例

```c
// 伪造栈帧（RTOS 创建新任务时调用）
// entry:     任务入口函数
// parameter: 任务参数
// stack_addr: 栈顶地址（高地址）
void *rt_hw_stack_init(void *entry, void *parameter, void *stack_addr) {
    uint8_t *stk = (uint8_t *)stack_addr;

    // 模拟硬件压栈的 8 个寄存器（顺序必须和硬件完全一致）
    stk -= 4; *(uint32_t *)stk = 0x01000000;       // xPSR: Thumb 模式位必须置 1
    stk -= 4; *(uint32_t *)stk = (uint32_t)entry;   // PC:  任务入口函数地址
    stk -= 4; *(uint32_t *)stk = 0xFFFFFFFF;        // LR:  任务不会返回，填死值
    stk -= 4; *(uint32_t *)stk = 0x12121212;        // R12
    stk -= 4; *(uint32_t *)stk = 0x03030303;        // R3
    stk -= 4; *(uint32_t *)stk = 0x02020202;        // R2
    stk -= 4; *(uint32_t *)stk = (uint32_t)parameter; // R1: 任务参数
    stk -= 4; *(uint32_t *)stk = 0x00000000;        // R0

    // R4-R11 由 PendSV ISR 手动保存，也要预留空间
    stk -= 4; *(uint32_t *)stk = 0x11111111;        // R11
    stk -= 4; *(uint32_t *)stk = 0x10101010;        // R10
    stk -= 4; *(uint32_t *)stk = 0x09090909;        // R9
    stk -= 4; *(uint32_t *)stk = 0x08080808;        // R8
    stk -= 4; *(uint32_t *)stk = 0x07070707;        // R7
    stk -= 4; *(uint32_t *)stk = 0x06060606;        // R6
    stk -= 4; *(uint32_t *)stk = 0x05050505;        // R5
    stk -= 4; *(uint32_t *)stk = 0x04040404;        // R4

    return stk; // 返回伪造后的 SP，存入任务的 TCB
}
```

### 6.5 伪造栈帧如何被消费

```
1. RTOS 创建任务 → 调用 rt_hw_stack_init → 得到伪造的 SP → 存入 TCB
2. 某个时刻触发 PendSV（任务切换）
3. PendSV ISR：
   a. 把当前任务的 R4-R11 压入当前任务的栈（保存旧任务）
   b. 把当前 SP 存入旧任务的 TCB
   c. 从新任务的 TCB 取出 SP（指向伪造的栈帧）
   d. 从新栈帧弹出 R4-R11
   e. BX LR → CPU 出栈 8 个寄存器 → PC 弹出 = 任务入口函数 → 新任务开始执行
```

新任务第一次运行时，它"以为"自己是从中断返回的，PC 指向 entry 函数，R1 携带参数。**这就是 RTOS 任务启动的真相。**

## 七、中断嵌套时的栈布局

嵌套 N 次中断，就压 N × 8 个寄存器：

```
假设 main() 在执行，中断A来了，中断A执行中，更高优先级的中断B又来了：

高地址
┌──────────────┐
│ main 的 8 个  │  ← 第一次硬件压栈（中断A 打断 main 时）
│ 寄存器        │     SP_A 指向这里
├──────────────┤
│ 中断A 的      │  ← 第二次硬件压栈（中断B 打断中断A 时）
│ 8 个寄存器    │     SP_B 指向这里
├──────────────┤
│ ISR-B 的      │  ← ISR-B 执行中的临时数据
│ 局部变量      │     当前 SP 指向这里
└──────────────┘
低地址

返回过程：
  1. 中断B 执行 BX LR → 出栈 8 个 → SP 回到 SP_A → CPU 回到中断A 继续
  2. 中断A 执行 BX LR → 出栈 8 个 → SP 回到 main → CPU 回到 main 继续
```

> [!warning] 中断嵌套深度受栈空间限制。每次嵌套至少消耗 32 字节（8 个寄存器），加上 ISR 自身的栈消耗。嵌套过深会导致栈溢出。

## 八、ISR 中途为什么不能切任务

```
ISR 执行到一半时，MSP 上有两层数据：

MSP ─→ ┌──────────────┐
        │ 硬件压的 8 个  │  ← 这部分完整，不会丢
        ├──────────────┤
        │ ISR 的临时数据 │  ← ISR 执行中产生的（局部变量、函数调用栈）
        └──────────────┘

此刻强行切任务（换 PSP）：
  - PSP 从 A 切到 B ✅ 任务B 可以正常运行
  - 但 MSP 上 ISR 的临时数据还在！
  - ISR 永远不会执行完了（任务B 在跑）
  - 下一次中断进来，继续往 MSP 压栈 → 覆盖上一次 ISR 的残留数据
  - MSP 被污染，系统栈结构崩塌
```

**本质：ISR 执行了一半被"遗弃"，它占着的 MSP 空间永远不会被释放。**

RTOS 的正确做法：
```
ISR 中唤醒高优先级任务
  → 只设一个 flag（需要调度）
  → ISR 正常执行完
  → 中断退出，嵌套计数归零（rt_interrupt_leave 检查）
  → MSP 上的 ISR 数据已经清理干净（ISR 正常返回了）
  → 安全触发 PendSV → 切换 PSP → 任务切换完成
```

## 九、不同架构的现场保存对比

| 架构 | 现场保存方式 | 保存哪些寄存器 | 谁负责 | 中断延迟 |
|------|------------|--------------|--------|---------|
| **ARM Cortex-M** | 硬件自动压栈 | 8 个（R0-R3, R12, LR, PC, xPSR） | 硬件 | **最低**（~12 周期） |
| **ARM Cortex-A** | 软件保存 | 所有通用寄存器（13 个 + 状态） | 向量入口处手写汇编 | 较高 |
| **RISC-V** | 软件保存 | 上下文由软件决定 | ISR 入口手动保存 | 较高 |
| **x86** | 硬件部分保存 | EFLAGS, CS, EIP, ErrorCode | 硬件 + 软件配合 | 中等 |

Cortex-A 的中断入口（软件保存）：
```asm
irq_handler:
    sub sp, sp, #72         @ 手动分配栈空间
    stmia sp, {r0-r12}      @ 保存所有通用寄存器
    mrs r0, spsr            @ 保存被中断的状态寄存器
    str r0, [sp, #60]
    @ ... 处理中断 ...
    ldmia sp, {r0-r12}      @ 恢复所有寄存器
    add sp, sp, #72
    subs pc, lr, #4         @ 返回被中断的地方
```

> [!tip] Cortex-M 中断延迟远低于 Cortex-A，根因就是硬件自动保存了 8 个寄存器，省去了 ~8 次内存写入的时间。

## 十、栈溢出的实际行为

### 函数调用的真实栈消耗

局部变量大小 ≠ 实际栈消耗，函数调用有隐藏开销：

```
每层函数调用的真实栈消耗：
  局部变量         → 编译器分配的空间
+ 4 字节          → 压入 LR（函数调用必须保存返回地址）
+ 4~8 字节        → 编译器保存的 callee-saved 寄存器（R4-R11）
+ 0~8 字节        → 对齐填充（AAPCS 要求 8 字节栈对齐）
────────────────────
≈ 局部变量 + 12~20 字节/层
```

### 溢出时 CPU 的行为

```
任务栈空间：512 字节（0x20001000 ~ 0x20001200）

递归调用，每层真实消耗约 72 字节：
  层1：SP -= 72  →  ✅
  层2：SP -= 72  →  ✅
  ...
  层8：SP -= 72 = 576 字节 > 512 → ❌ 超出栈空间

但 CPU 不知道越界！它只管减 SP 然后写数据：
  → 写入的数据覆盖了栈空间之外的内存
```

| 场景 | 后果 |
|------|------|
| 无 MPU 保护 | 静默覆盖其他任务栈 / 全局变量，极难定位的幽灵 Bug |
| 有 MPU 保护 | 写入非法区域触发 MemManage 异常，进入 HardFault_Handler |
| RTOS 栈溢出检测 | 任务切换时检查栈底 canary 标记（FreeRTOS 的 stack overflow hook） |
| RTOS 水线检测 | 记录栈的最大使用深度，用于调优栈大小 |

## 十一、面试高频问题

> [!example]- Q1：MSP 和 PSP 的区别和使用场景？
> MSP（Main Stack Pointer）和 PSP（Process Stack Pointer）是 Cortex-M R13 寄存器的两个物理副本，由 CONTROL[1] 选择使用哪个。裸机中全程用 MSP；RTOS 中任务用 PSP（每个任务独立），中断用 MSP（所有中断共享）。切换任务只需要切换 PSP，中断栈不受影响。

> [!example]- Q2：为什么 ISR 中途不能切任务？
> ISR 执行到一半时，MSP 上有 ISR 的临时数据（局部变量、函数调用栈）。此时如果切换 PSP 去跑新任务，MSP 上的 ISR 永远不会执行完，它占着的空间永远不会被释放，后续中断继续往 MSP 压栈会覆盖残留数据，导致系统栈结构崩塌。所以 RTOS 用延迟调度：ISR 中只设 flag，等所有中断退出（嵌套计数归零）后再触发 PendSV 切换任务。

> [!example]- Q3：伪造栈帧的原理？xPSR 为什么必须设 0x01000000？
> 新任务从未运行过，没有"被中断的现场"。RTOS 在任务栈中伪造 8 个寄存器的值，模拟硬件压栈的结果。xPSR 的 Bit[24] 是 T flag（Thumb 模式标志），Cortex-M 只支持 Thumb 状态，如果 Bit[24] = 0，CPU 尝试进入不存在的 ARM 状态，直接 HardFault。

> [!example]- Q4：栈溢出时 CPU 会怎样？
> CPU 不知道栈越界，它只管减 SP 然后写数据。没有 MPU 时会静默覆盖其他内存区域，导致极难定位的幽灵 Bug；有 MPU 时触发 MemManage 异常；RTOS 层面可以通过 canary 标记或水线检测发现。

> [!example]- Q5：为什么硬件只保存 8 个而不是 16 个寄存器？
> ARM 的"按需付费"设计哲学。8 个是 caller-saved 寄存器（AAPCS 最低要求），大部分 ISR 只用 R0-R3。如果保存全部 16 个，每次中断多消耗 32 字节，嵌套 3 层就多 96 字节，对栈空间紧张的嵌入式系统是浪费。R4-R11 由编译器按需生成 push/pop，用多少保存多少。

> [!example]- Q6：中断嵌套时栈帧怎么排列？
> 嵌套 N 次就压 N 组 8 个寄存器，在内存中从高到低依次排列。返回时从最内层开始逐层出栈，LIFO 顺序。每次出栈 SP 回到上一层的位置，CPU 恢复到上一层的中断上下文。嵌套深度受栈空间限制。
