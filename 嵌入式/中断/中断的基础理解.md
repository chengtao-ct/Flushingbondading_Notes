# 中断系统基础理解

## 概述

中断是 MCU 响应外部事件的机制。当硬件信号到来时，CPU 自动保存当前现场，跳转到预设的处理函数（ISR）执行，完毕后恢复现场继续原来的程序。中断系统的核心是**优先级管理**（谁能打断谁）和**ISR 设计**（快、纯、安全）。

> [!info] 面试开场句
> "中断本质是硬件驱动的函数调用。CPU 自动压栈保存现场，查向量表跳转 ISR，执行完自动出栈恢复。优先级分抢占和子优先级，ISR 设计原则是快、纯、安全。"

---

## 中断的完整机制

### 执行流程

```
CPU 正在执行 main() 第 47 行
        │
        ├── 硬件信号来了（按键/定时器/UART 等）
        │
        ├── CPU 硬件自动完成：
        │   1. 把当前寄存器压栈（PC, PSR, LR 等 8 个）
        │   2. 查中断向量表，找到 ISR 函数地址
        │   3. 跳转到 ISR 执行
        │
        ├── ISR 执行完毕
        │
        └── CPU 硬件自动出栈，回到 main() 第 47 行继续
            （CPU 不知道自己被打断过）
```

> [!tip] 压栈出栈由硬件自动完成，不需要手动写代码。这是 ARM Cortex-M 的特性，其他架构可能需要软件保存现场。

### 中断向量表

向量表是一张函数指针数组，存储每个中断源对应的 ISR 地址：

```
地址          内容
0x00000000    栈顶地址（MSP）
0x00000004    Reset_Handler（上电入口）
0x00000008    NMI_Handler
0x0000000C    HardFault_Handler
...
0x0000004C    EXTI0_IRQHandler    ← EXTI Line0 中断
0x00000050    EXTI1_IRQHandler    ← EXTI Line1 中断
...
0x00000xxx    USART1_IRQHandler   ← UART1 中断
```

每个中断源对应一个固定地址，硬件自动查表跳转，不需要手动判断是哪个中断。

> [!important] 通用结论
> 所有 MCU 都有中断向量表，ARM Cortex-M 格式统一，ESP32 的 Xtensa 也有类似结构。

## NVIC —— 中断总调度

NVIC（Nested Vectored Interrupt Controller）是 Cortex-M 内核内置的**中断管理器**，所有中断都要经过它才能到达 CPU。

```
各种中断源                   NVIC（调度中心）              CPU
──────────                  ┌──────────────┐
定时器溢出    ───┐           │  优先级排序    │           ┌──────┐
UART 收数据   ───┤           │  嵌套管理     │──────────→│ CPU  │
按键按下      ───┼─────────→│  屏蔽控制     │           └──────┘
ADC 转换完成  ───┤           │  向量表索引   │
DMA 传输完成  ───┘           └──────────────┘

NVIC 决定：
  1. 这个中断能不能进？（是否使能、是否被屏蔽）
  2. 现在能不能打断？（优先级比较）
  3. 跳到哪个 ISR？（查向量表）
```

> [!tip] NVIC 是内核自带的，所有 Cortex-M 芯片（STM32/NXP/GD32）共用同一个 NVIC。

## EXTI —— 外部中断路由器

EXTI（External Interrupt/Event Controller）负责把 **GPIO 引脚的电平变化**转成中断信号送给 NVIC。

### 路由规则

GPIO 有多组端口，但 EXTI Line 数量有限，需要 SYSCFG 寄存器做路由选择：

```
EXTI 路由表：
  EXTI Line0  ← PA0 或 PB0 或 PC0 ...（SYSCFG 选择）
  EXTI Line1  ← PA1 或 PB1 或 PC1 ...
  ...
  EXTI Line15 ← PA15 或 PB15 或 PC15 ...

规则：同一 Line 只能选一个端口，不同 Line 可以选不同端口
  ✅ PA0 用 Line0, PB1 用 Line1
  ❌ PA0 和 PB0 都用 Line0 → 冲突，只能选一个
```

### EXTI vs 外设中断

| 中断源 | 是否经过 EXTI | 说明 |
|--------|-------------|------|
| GPIO 引脚电平变化 | **是** | EXTI 检测边沿 → 通知 NVIC |
| UART 收发完成 | 否 | 外设直接通知 NVIC |
| 定时器溢出 | 否 | 外设直接通知 NVIC |
| ADC 转换完成 | 否 | 外设直接通知 NVIC |
| DMA 传输完成 | 否 | 外设直接通知 NVIC |

> [!important] NVIC 管所有中断，EXTI 只管 GPIO 引脚中断。

## 优先级管理

### 抢占优先级 vs 子优先级

每个中断有一个优先级寄存器（8 位，Cortex-M4 用高 4 位），拆成两部分：

```
[7] [6] [5] [4] | [3] [2] [1] [0]
 抢占优先级      |   子优先级
 (Preemption)    |   (Sub)

分组方式（以 STM32 为例）：
  Group 0: 0位抢占 + 4位子优先级
  Group 1: 1位抢占 + 3位子优先级
  Group 2: 2位抢占 + 2位子优先级 ← 最常用
  Group 3: 3位抢占 + 1位子优先级
  Group 4: 4位抢占 + 0位子优先级

数字越小，优先级越高。
```

**两种优先级的作用完全不同：**

| | 抢占优先级 | 子优先级 |
|--|----------|---------|
| 作用 | 能不能打断别人 | 同时来谁先走 |
| 特性 | 高可打断低 → **嵌套** | 只排队，**不能打断** |

| 场景               | 结果            |
| ---------------- | ------------- |
| 高抢占来了，CPU 在执行低抢占 | 打断，高优先级先执行    |
| 两个同抢占同时来         | 高子优先级先执行，不能打断 |
| 低抢占来了，CPU 在执行高抢占 | 排队等           |

```
举例（Group 2: 2位抢占 + 2位子优先级）：

中断A: 抢占=0, 子=1  （最高抢占）
中断B: 抢占=1, 子=0
中断C: 抢占=1, 子=2
中断D: 抢占=2, 子=0  （最低抢占）

CPU 执行 main()
  → B 来了 → 执行 B
    → A 来了 → A 抢占更高 → 打断 B，执行 A
      → A 完成 → 回到 B
    → B 完成 → 回到 main
  → B 和 C 同时来 → 抢占相同，看子优先级 → B(0) 先，C(2) 排队
```

## 信号链路

从按键到 ISR 的完整路径：

```
物理层          路由层           控制层          内核层          代码层
──────         ──────          ──────         ──────         ──────

按键按下
  │
  ▼
PC13 引脚
电平变化
  │
  ▼
GPIO 输入 ──→ SYSCFG 路由 ──→ EXTI 检测 ──→ NVIC 排队 ──→ CPU 跳转
               "PC13 接到      下降沿触发     优先级判断    ISR 函数
                Line13"        PR 置 1
```

**详细步骤：**

1. PC13 引脚电平变化（硬件自动检测）
2. SYSCFG 寄存器把 PC13 路由到 EXTI Line13
3. EXTI 检测到下降沿，PR13 标志位置 1
4. EXTI 把中断请求发给 NVIC
5. NVIC 检查：使能了没？优先级够不够？能不能打断？
6. NVIC 通知 CPU，CPU 自动压栈
7. 查向量表，跳转到 EXTI15_10_IRQHandler
8. ISR 中清 PR13 标志位
9. 执行完毕，CPU 自动出栈，回到被打断的地方

## ISR 设计原则

### 三条铁律

```
1. 快 —— 越短越好（< 几 us）
2. 纯 —— 不调用阻塞函数
3. 安全 —— 共享数据要保护
```

### 不能做的事

| 不能做 | 原因 | 后果 |
|--------|------|------|
| printf | UART 阻塞发送，一个字符 ~1ms | 其他中断被阻塞 |
| HAL_Delay | 依赖 SysTick，ISR 中可能卡死 | 永远等不到 |
| 大量计算 | 占 CPU 太久 | 低优先级中断无响应 |
| malloc/free | 堆操作不可重入 | 堆损坏 |
| 读写 Flash | Flash 操作阻塞几~几十 ms | 系统卡死 |

### 正确做法：上半部 / 下半部

```
上半部（ISR）：
  1. 清中断标志位
  2. 读数据到缓冲区
  3. 设标志位 / 塞环形缓冲区
  → 极速完成

下半部（main 循环）：
  1. 检查标志位 / 取缓冲区数据
  2. printf、计算、状态机等
  → 不影响中断响应
```

> [!tip] 这是通用设计模式，Linux 的 Top Half / Bottom Half、FreeRTOS 的 FromISR + Task 都是同一思路。

## 中断安全性

### 竞态条件

ISR 和 main 共享变量时，main 的非原子操作可能被 ISR 打断导致数据不一致：

```
main: 读取 flag → [被ISR打断] → ISR 修改 flag → main 用的是旧值
```

### 解决方案

| 方案 | 做法 | 适用场景 | 缺点 |
|------|------|---------|------|
| **关中断** | `__disable_irq()` 保护共享变量 | 简单标志位 | 影响实时性 |
| **volatile** | 告诉编译器每次从内存重读 | ISR 写、main 只读 | 不保证原子性 |
| **环形缓冲区** | ISR 塞数据，main 取数据 | 多事件、高频 | 实现稍复杂 |

```c
// 方案一：临界区（关中断）
__disable_irq();
shared_var = 0;
__enable_irq();

// 方案二：volatile
volatile uint8_t flag = 0;  // 编译器不优化，每次重新读

// 方案三：环形缓冲区（最佳）
// ISR 只写 tail，main 只写 head → 天然无竞态
```

> [!warning] volatile 不是万能的
> volatile 只保证"每次从内存读"，不保证原子性。竞态条件仍存在。volatile + 临界区才是完整方案。

## 关键概念速查

| 概念 | 说明 |
|------|------|
| 中断向量表 | 函数指针数组，每个中断源对应一个 ISR 地址 |
| NVIC | 内核中断控制器，管理优先级、嵌套、屏蔽 |
| EXTI | GPIO 外部中断路由器，把引脚电平变化转发给 NVIC |
| 抢占优先级 | 决定能不能打断正在执行的中断 |
| 子优先级 | 决定同时来的中断谁先执行 |
| ISR | 中断服务函数，必须快、纯、安全 |
| 临界区 | 关中断保护的代码段，防止竞态条件 |
| volatile | 防止编译器优化共享变量 |

## 面试高频问题

> [!example]- Q1：中断的执行流程？
> 硬件信号 → CPU 自动压栈（8个寄存器）→ 查向量表跳转 ISR → 执行完毕 → 自动出栈恢复。整个过程硬件自动完成保存和恢复。

> [!example]- Q2：抢占优先级和子优先级的区别？
> 抢占优先级决定能不能打断（高可打断低，实现嵌套）；子优先级决定同时来谁先执行（只能排队不能打断）。数字越小优先级越高。

> [!example]- Q3：NVIC 和 EXTI 的关系？
> NVIC 管所有中断的优先级和调度；EXTI 只负责把 GPIO 引脚电平变化转成中断信号送给 NVIC。UART、定时器等外设中断直接到 NVIC，不经过 EXTI。

> [!example]- Q4：ISR 里不能做什么？
> 不能做耗时操作：printf、HAL_Delay、大量计算、malloc/free、Flash 读写。ISR 只做清标志位、读数据、设标志，耗时逻辑放 main 循环。

> [!example]- Q5：ISR 和 main 共享变量怎么处理？
> 竞态条件是核心问题。简单场景用 volatile + 关中断临界区；高频场景用环形缓冲区（ISR 写 tail，main 写 head，天然无竞态）。
