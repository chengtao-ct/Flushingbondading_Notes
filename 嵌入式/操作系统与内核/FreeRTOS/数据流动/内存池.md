# 内存池

> [!info] 核心概念
> **内存池** 是一种预先分配一大块连续内存，然后按固定大小切分管理的机制。它的核心目的是**避免内存碎片**，提高分配效率。

## 🎯 通俗理解：停车场的智慧

想象你经营一个停车场：

| 方式 | 类比 | 特点 |
| ---- | ---- | ---- |
| 动态分配 | 随意停车 | 车辆大小不一，停得乱七八糟，空间浪费严重 |
| 内存池 | 划定车位 | 每个车位固定大小，整齐划一，空间利用率高 |

**内存池就像划定好固定车位的停车场**：
- 车位大小固定 → 内存块大小固定
- 车位数量固定 → 内存块数量固定
- 车来了直接停 → 分配速度快（O(1)）
- 车走了车位还在 → 不会产生碎片

## 🔧 FreeRTOS 中的内存管理

FreeRTOS 本身**没有直接提供**名为"内存池"的 API，但它提供了多种堆管理方案，其中 `heap_4.c` 最常用：

### FreeRTOS 堆管理方案对比

| 方案 | 特点 | 适用场景 |
| ---- | ---- | ---- |
| [[heap_1]] | 只分配不释放 | 极简单系统 |
| [[heap_2]] | 可释放但不合并碎片 | 已废弃 |
| [[heap_3]] | 封装标准库 malloc | 不推荐 |
| [[heap_4]] | 可释放且合并碎片 | **最常用** |
| [[heap_5]] | 多段不连续内存 | 外部 SRAM 场景 |

### 如何在 FreeRTOS 中实现内存池效果？

```c
/* 方法一：静态分配（真正的内存池思想） */
#define POOL_BLOCK_SIZE  64    // 每个块 64 字节
#define POOL_BLOCK_COUNT 10    // 共 10 个块

static uint8_t mem_pool[POOL_BLOCK_SIZE * POOL_BLOCK_COUNT];  // 预分配内存池
static uint8_t block_used[POOL_BLOCK_COUNT];  // 使用标记

/* 从内存池申请一块 */
void* pool_alloc(void)
{
    for (int i = 0; i < POOL_BLOCK_COUNT; i++)
    {
        if (block_used[i] == 0)  // 找到空闲块
        {
            block_used[i] = 1;
            return &mem_pool[i * POOL_BLOCK_SIZE];  // 返回块地址
        }
    }
    return NULL;  // 池已满
}

/* 释放回内存池 */
void pool_free(void *ptr)
{
    int index = ((uint8_t*)ptr - mem_pool) / POOL_BLOCK_SIZE;
    if (index >= 0 && index < POOL_BLOCK_COUNT)
    {
        block_used[index] = 0;  // 标记为空闲
    }
}
/* 方法二：使用 FreeRTOS 的 pvPortMalloc */
void task_example(void *pvParameters)
{
    /* 注意：heap_4 方案下，这会从 FreeRTOS 管理的堆中分配 */
    uint8_t *buffer = (uint8_t*)pvPortMalloc(128);
    
    if (buffer != NULL)
    {
        /* 使用内存 */
        memset(buffer, 0, 128);
        
        /* 用完必须释放 */
        vPortFree(buffer);
    }
}
```
## ⚡ 软硬结合：STM32F407 上的实际表现

### 内存布局（以 STM32F407 为例）

```
┌─────────────────────────────────────┐ 0x20000000
│         SRAM1 (112 KB)              │ ← 主内存，存放堆、栈、全局变量
├─────────────────────────────────────┤
│         SRAM2 (16 KB)               │ ← 可用于特定外设
├─────────────────────────────────────┤
│         SRAM3 (64 KB)               │ ← CCM 内存，DMA 无法访问！
└─────────────────────────────────────┘ 0x20030000
```

### 如何验证内存池在工作？

```c
/* 在 Keil/IAR 中查看内存使用情况 */
extern uint32_t _estack;      // 栈顶
extern uint32_t _Min_Heap_Size;  // 堆大小

/* FreeRTOS 任务栈溢出检测 */
void vApplicationStackOverflowHook(TaskHandle_t xTask, char *pcTaskName)
{
    /* 如果跑到这里，说明某个任务栈爆了！ */
    while(1);  // 死循环，方便调试器定位
}
```

**用调试器观察**：

1. 打开 Memory 窗口，输入 `0x20000000`
2. 观察内存池区域的数据变化
3. 申请时看到数据被写入，释放后数据还在（只是标记为可用）

---

避坑经验

1. **CCM 内存陷阱**：STM32F407 的 CCM RAM（64KB）不能被 DMA 访问！如果把内存池放在 CCM，再用 DMA 搬运会直接卡死。
2. **忘记释放**：内存池虽然减少了碎片，但忘记释放仍会导致"池满"，后续申请全部失败。
3. **线程安全问题**：多任务环境下，内存池的分配/释放操作必须加 [互斥锁](app://obsidian.md/%E4%BA%92%E6%96%A5%E9%94%81) 或关中断保护。

## 📊 内存池 vs 动态分配

|特性|内存池|动态分配|
|---|---|---|
|分配速度|极快（O(1)）|较慢（需搜索合并）|
|内存碎片|无|可能产生|
|灵活性|固定块大小|任意大小|
|适用场景|实时性要求高|通用场景|
