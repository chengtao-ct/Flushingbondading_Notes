# 队列

队列是 FreeRTOS 中最基础、最通用的通信机制。在 [[任务通知]] 和 [[../数据流动/流缓冲区]] 出现之前，它是任务间通信（IPC）的唯一标准答案。

虽然现在我们有了更轻量的选择，但队列依然是处理 **多对多通信** 和 **结构化数据传递** 的核心支柱。

> [!info] 核心本质
> 队列是一个 **FIFO（先进先出）** 的环形缓冲区。
> 它的核心特性是 **数据拷贝**：发送数据时，数据被拷贝进队列；接收数据时，数据从队列拷贝出来。这种机制保证了数据的独立性，但也带来了内存和 CPU 的开销。

---

## 1. 原理剖析：为什么它是“万能”的？

### 1.1 数据拷贝机制
不同于 [环形缓冲区](../数据流动/环形缓冲区.md)直接存取指针，队列操作的是数据本身。
- **发送**：`xQueueSend` 将数据从发送者的内存空间 **拷贝** 到队列内部的存储区域。
- **接收**：`xQueueReceive` 将数据从队列内部 **拷贝** 到接收者的变量中。

**优势**：
- **解耦**：发送者发送完数据后可以立即修改或销毁原变量，无需担心接收者读到脏数据。
- **安全**：避免了指针悬挂和内存生命周期管理的复杂性。

**代价**：
- 如果传递的数据结构很大（如 1KB 的结构体），每次发送都会触发 `memcpy`，消耗大量 CPU 周期。

### 1.2 多任务并发安全
队列天生支持 **多写多读**。
- 多个任务可以同时向同一个队列发送数据。
- 多个任务可以同时从同一个队列接收数据。
- 内部通过 [[临界区]] 和关中断机制自动处理并发竞争，用户无需手动加锁。

### 1.3 阻塞机制
- **满队阻塞**：当队列满时，试图发送的任务会进入阻塞态，直到队列有空位或超时。
- **空队阻塞**：当队列空时，试图接收的任务会进入阻塞态，直到队列有数据或超时。

## 2. 实战链路：结构化数据传递

**场景**：一个传感器采集任务将数据打包，传递给显示任务进行 UI 刷新。这是一个典型的“一对一”结构体传递场景。

### 2.1 定义数据结构

```c
// 定义传感器数据结构体
typedef struct {
    uint8_t sensor_id;
    float temperature;
    float humidity;
    uint32_t timestamp;
} SensorData_t;

// 定义队列句柄
QueueHandle_t xSensorQueue;
```
### 2.2 生产者：打包发送

```c
void vTask_SensorReader(void *pvParameters) {
    SensorData_t data;

    while(1) {
        // 1. 采集数据
        data.sensor_id = 1;
        data.temperature = Read_Temp();
        data.humidity = Read_Humid();
        data.timestamp = Get_Tick();

        // 2. 发送到队列（拷贝整个结构体）
        // portMAX_DELAY 表示如果队列满了，无限期等待
        if (xQueueSend(xSensorQueue, &data, portMAX_DELAY) != pdPASS) {
            // 发送失败处理（通常不会发生，除非队列句柄无效）
        }
        
        vTaskDelay(1000);
    }
}
```

### 2.3 消费者：阻塞接收

```c
void vTask_Display(void *pvParameters) {
    SensorData_t receivedData;

    while(1) {
        // 1. 阻塞等待数据
        // 收到数据后，内核会将数据从队列拷贝到 receivedData 变量中
        if (xQueueReceive(xSensorQueue, &receivedData, portMAX_DELAY) == pdPASS) {
            
            // 2. 处理数据
            Update_UI(receivedData.temperature, receivedData.humidity);
        }
    }
}
```

### 2.4 初始化

```c
void App_Init(void) {
    // 创建队列：能容纳 10 个 SensorData_t 大小的数据项
    xSensorQueue = xQueueCreate(10, sizeof(SensorData_t));
}
```

## 3. 避坑指南：架构师的忠告

性能与陷阱

### 3.1 大数据的拷贝灾难

**问题**：如果你通过队列传递一个巨大的结构体（比如 1KB 的图像数据块），频繁的 `memcpy` 会导致 CPU 负载飙升，且队列本身需要巨大的 RAM 空间来缓存这些数据。

**对策**：**传递指针，而非数据**。

- 发送时发送 `&data_ptr`（指针的拷贝，仅 4 字节）。
- 接收方收到指针后直接操作内存。
- **风险**：此时变成了共享内存模式，必须确保发送方在接收方使用完数据前，不能修改或释放该内存块。通常需要配合 [内存池](app://obsidian.md/%E5%86%85%E5%AD%98%E6%B1%A0) 或 [互斥锁 Mutex](app://obsidian.md/%E4%BA%92%E6%96%A5%E9%94%81%20Mutex) 使用。

### 3.2 ISR 中的禁忌

**问题**：在中断中调用 `xQueueSend` 会导致系统崩溃。  
**对策**：必须使用带有 `_FromISR` 后缀的 API，如 `xQueueSendFromISR`。详见 [_FROM_ISR](_FROM_ISR.md)。

### 3.3 死锁风险

**场景**：任务 A 持有资源 X，等待队列 Q；任务 B 持有队列 Q 的数据，等待资源 X。  
**后果**：两个任务互相等待，系统卡死。  
**对策**：设计时理清资源依赖图，或使用 [互斥锁 Mutex](app://obsidian.md/%E4%BA%92%E6%96%A5%E9%94%81%20Mutex) 的优先级继承机制（但队列本身不具备优先级继承功能）。

## 4. 选型决策矩阵

随着 FreeRTOS 的演进，队列不再是唯一解。我们需要根据场景精准选型。

| 场景                      | 推荐方案            | 理由                 |
| :---------------------- | :-------------- | :----------------- |
| **低频、结构化数据传递**          | **队列**          | 安全、解耦、支持多对多。       |
| **高频、字节流数据** (UART/ADC) | [流缓冲区](../数据流动/流缓冲区.md) | 零拷贝、无锁、极速。         |
| **单纯的事件同步** (A做完通知B)    | [任务通知](任务通知.md) | 极致轻量，无额外内存开销。      |
| **多任务广播** (一个事件唤醒多个任务)  | [事件组](事件组.md)   | 队列只能唤醒一个接收者。       |
| **大数据块传递**              | 队列传指针           | 避免拷贝开销，但需自行管理内存安全。 |
|                         |                 |                    |

**大师建议**：  
队列是嵌入式系统的“瑞士军刀”，功能全面但略显笨重。在简单的“一对一”同步场景下，请优先考虑 [任务通知](app://obsidian.md/%E4%BB%BB%E5%8A%A1%E9%80%9A%E7%9F%A5)；在高速数据流场景下，请优先考虑 [流缓冲区](app://obsidian.md/%E6%B5%81%E7%BC%93%E5%86%B2%E5%8C%BA)。只有在需要 **多任务共享** 或 **数据缓冲** 时，才动用队列这把重武器。
