FreeRTOS 的临界区（Critical Section）通过配置 `BASEPRI` 寄存器，屏蔽了**优先级数值大于等于 `configMAX_SYSCALL_INTERRUPT_PRIORITY`** 的所有中断。  
这意味着：**低优先级的硬件中断会被屏蔽，但高优先级的硬件中断（如电机急停、安全气囊）依然可以打断临界区。**

---

### 1. 硬件原理：BASEPRI 寄存器

在 Cortex-M 架构中，中断屏蔽不是简单的“开/关”，而是通过 **优先级阈值** 来管理的。

- **`BASEPRI` 寄存器**：它设定了一个“门槛”。
    - 优先级数值 **大于或等于** 这个门槛的中断，全部被屏蔽（CPU 不响应）。
    - 优先级数值 **小于** 这个门槛的中断，正常响应。

注意：数值越小，优先级越高

在 STM32 中，优先级 0 是最高，优先级 15 是最低。  
假设 `BASEPRI` 设为 5：

- 优先级 5, 6...15（低优先级）：**被屏蔽**（进不来）。
- 优先级 0, 1...4（高优先级）：**不屏蔽**（可以打断）。

---

### 2. 实战配置：FreeRTOS 的“分界线”

在 `FreeRTOSConfig.h` 中，有一个极其关键的宏定义：`configMAX_SYSCALL_INTERRUPT_PRIORITY`（系统调用最大中断优先级）。

这就是临界区的“门槛”。

```c
// FreeRTOSConfig.h 中的典型配置
#define configMAX_SYSCALL_INTERRUPT_PRIORITY  5 // 假设设为 5
```

**这意味着：**

1. **受 OS 管理的中断（优先级 5~15）**：
    - 这些中断可以调用 FreeRTOS 的 API（如 `xQueueSendFromISR`）。
    - **进入临界区时，这些中断会被屏蔽**。因为如果它们触发并调用了 API，可能会破坏临界区的数据完整性。
2. **不受 OS 管理的中断（优先级 0~4）**：
    - 这些中断**不允许**调用任何 FreeRTOS API。
    - **进入临界区时，这些中断不会被屏蔽**。它们拥有“特权”，即使代码在临界区内，也能立即抢占 CPU。

---

### 3. 生活类比：VIP 通道

把 CPU 比作一个**演唱会现场**，临界区就是**歌手正在独唱**（不能被打扰）。

- **普通观众（低优先级硬件中断）**：
    - 想进场必须买票，还要排队。歌手独唱时（临界区），保安把大门关了，**屏蔽**。
- **保安队长/消防员（高优先级硬件中断）**：
    - 他们有最高权限（优先级 0~4），不归演唱会主办方（OS）管。
    - 即使歌手在独唱，如果发生火灾（紧急事件），他们可以直接冲进去打断表演。**不屏蔽**。

---

### 4. 避坑指南：为什么要有这个机制？

如果临界区屏蔽了**所有**中断（包括最高优先级的），那么当你的代码在临界区内执行稍微久一点（比如几十微秒），电机控制的 PWM 保护中断就无法响应，可能导致电机过流烧毁。

**FreeRTOS 这种设计是为了“保命”：**

- 让普通的、耗时的、涉及 OS 数据结构的中断（如 UART 接收、USB 通信）在临界区外排队。
- 让极其紧急、不依赖 OS 的中断（如硬件故障保护、高频电机控制）随时能抢占。

### 5. 代码验证

你可以通过配置 NVIC 来验证这一点：

```c
// 假设 configMAX_SYSCALL_INTERRUPT_PRIORITY 为 5

// 1. 配置 UART 中断优先级为 6 (低优先级)
NVIC_SetPriority(USART1_IRQn, 6); 
// 结果：进入临界区后，UART 中断被挂起，无法执行。

// 2. 配置 电机急停 中断优先级为 2 (高优先级)
NVIC_SetPriority(EXTI0_IRQn, 2); 
// 结果：进入临界区后，EXTI0 中断依然可以触发，打断临界区代码！
// 注意：EXTI0 的 ISR 中绝对不能调用 xQueueSend 等 FreeRTOS API，否则系统会崩溃。
```
