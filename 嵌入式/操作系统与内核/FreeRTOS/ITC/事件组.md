# 事件组

在实际工程中，我们经常遇到复杂的同步场景：系统的主控制任务必须等待“IMU传感器校准完成” **且** “网络连接成功” **且** “电池电压正常”这三个条件同时满足后，才能启动。

如果用 [[二值信号量]] 或 [[FreeRTOS Queue]] 来实现，你需要创建多个对象，并在应用层写一堆复杂的轮询代码。**事件组** 就是为了解决这种**多任务联合协调**而生的。

> [!info] 核心本质
> 事件组就像是立在系统中央的一块“**全局布告栏**”。
> - **生产者**：任何任务都可以在上面贴标签（设置 Bit）。
> - **消费者**：任何任务都可以站在布告栏前等待一个或多个标签组合出现。
> - **特性**：只传递**状态**，不传递数据。天生支持**多对多**同步。

---

## 1. 原理剖析：24 位的位图

### 1.1 数据模型
事件组在底层本质上是一个 `EventBits_t` 类型的变量。
- 在大多数 32 位架构中，它是一个 32 位整数。
- **高 8 位**：被内核保留用于控制状态。
- **低 24 位**：提供给用户使用。这意味着一个事件组可以管理 **24 个独立的事件**。

### 1.2 逻辑等待
消费者任务调用 `xEventGroupWaitBits()` 时，可以指定“等待逻辑”：
- **逻辑与（AND）**：必须等待所有指定的 Bit 都置位才唤醒。
    - 场景：启动系统前，必须所有子系统都就绪。
- **逻辑或（OR）**：只要任意一个指定的 Bit 置位就唤醒。
    - 场景：等待“按键按下”或“触摸屏点击”任意一个输入事件。

## 2. 实战链路：多传感器同步启动

**场景**：环境监测设备，需同步采集温度和湿度，两者都准备好后才开始处理。

### 2.1 定义事件位

```c
// 定义事件位（使用宏提高可读性）
#define EVENT_TEMP_READY    ( 1 << 0 ) // Bit 0
#define EVENT_HUMID_READY   ( 1 << 1 ) // Bit 1

// 全局句柄
EventGroupHandle_t xSensorEvents;
```
### 2.2 生产者：贴标签

```c
// 温度采集任务
void vTask_TempSensor(void *pvParameters) {
    while(1) {
        Read_Temperature();
        // 数据准备好，设置 Bit 0
        xEventGroupSetBits(xSensorEvents, EVENT_TEMP_READY);
        vTaskDelay(1000);
    }
}

// 湿度采集任务
void vTask_HumidSensor(void *pvParameters) {
    while(1) {
        Read_Humidity();
        // 数据准备好，设置 Bit 1
        xEventGroupSetBits(xSensorEvents, EVENT_HUMID_READY);
        vTaskDelay(1000);
    }
}
```

### 2.3 消费者：守株待兔

```c
void vTask_DataProcess(void *pvParameters) {
    while(1) {
        // 阻塞等待，不消耗 CPU
        EventBits_t uxBits = xEventGroupWaitBits(
            xSensorEvents,               // 事件组句柄
            EVENT_TEMP_READY | EVENT_HUMID_READY, // 等待 Bit0 和 Bit1
            pdTRUE,                      // xClearOnExit: 唤醒后自动清除这些位
            pdTRUE,                      // xWaitForAllBits: pdTRUE=逻辑与(必须全满足)
            portMAX_DELAY                // 无限期等待
        );

        // 执行到这里，说明两个传感器数据均已就绪
        Process_All_Data();
    }
}
```

## 3. 避坑指南：架构师的血泪教训

致命陷阱：中断中的“推迟执行”

这是 FreeRTOS 中极其隐蔽、坑杀过无数老手的陷阱。

### 3.1 ISR 中的特殊机制

如果你在硬件中断（ISR）中调用 `xEventGroupSetBitsFromISR()`，**实际的置位动作并不会在中断上下文中立刻发生！**

**原因**：事件组的逻辑复杂（可能需要遍历链表唤醒多个任务），如果在 ISR 中直接处理，会导致中断执行时间过长，破坏实时性。

**机制**：FreeRTOS 将置位请求发送给 **定时器守护任务**，由这个后台任务代为执行。

### 3.2 灾难后果

如果你的 `FreeRTOSConfig.h` 中，定时器任务的优先级 `configTIMER_TASK_PRIORITY` 设置得很低：

1. 中断触发，发送指令给定时器任务。
2. 中断退出。
3. 定时器任务因优先级低，抢不到 CPU。
4. **结果**：事件明明发生了，等待的高优先级任务却迟迟无法被唤醒！

### 3.3 解决方案

- **检查优先级**：确保 `configTIMER_TASK_PRIORITY` 足够高（通常仅次于最高实时控制任务）。
- **替代方案**：如果对实时性要求极高，考虑在 ISR 中使用 [任务通知](app://obsidian.md/%E4%BB%BB%E5%8A%A1%E9%80%9A%E7%9F%A5) 代替事件组（但任务通知不支持多对多广播）。

### 3.4 原子清除陷阱

- **错误做法**：任务唤醒后，手动调用 `xEventGroupClearBits()` 清除标志。
- **风险**：在唤醒和清除之间，可能发生任务切换，导致新发生的事件被错误清除（竞态条件）。
- **正确做法**：使用 `xEventGroupWaitBits()` 的 `xClearOnExit` 参数设为 `pdTRUE`，内核会在临界区内自动原子清除。

## 4. 进阶：惊群效应

事件组支持**广播**。如果三个任务都在等待 `BIT2`，当 `BIT2` 被置位时，这三个任务会**同时**被唤醒。

- **优点**：非常适合做“全局状态广播”（如：向所有任务广播“系统断电警告”）。
- **缺点**：瞬间唤醒大量任务会造成 CPU 算力峰值冲击，需评估系统负载能力。

## 5. 选型决策矩阵

|特性|事件组|任务通知|队列|
|:--|:--|:--|:--|
|**通信模式**|多对多 (广播)|一对一 (单播)|多对多|
|**传输内容**|状态|32位值 / 状态|数据 / 结构体|
|**内存开销**|中|极低 (TCB自带)|高|
|**逻辑判断**|支持 AND / OR|需软件实现|FIFO|
|**典型场景**|多条件同步、全局广播|高频 ISR 唤醒|数据流传输|

**大师建议**：

- 只有在需要**多条件同步**（AND/OR）或**多任务广播**时，才动用事件组。
- 简单的一对一同步，首选 [任务通知](app://obsidian.md/%E4%BB%BB%E5%8A%A1%E9%80%9A%E7%9F%A5)，速度更快且无 ISR 陷阱。
