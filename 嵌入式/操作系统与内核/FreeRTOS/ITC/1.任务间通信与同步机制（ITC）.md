# 任务间通信与同步机制（ITC）

在 FreeRTOS 的多任务世界里，任务不是孤岛。当多个执行流并发运行时，我们需要一套严密的“交通规则”来协调它们的执行顺序和资源访问——这就是 ITC（Inter-Task Communication）。

> [!info] 核心区分：同步 vs 互斥
> 很多新人容易混淆这两个概念，这会导致架构设计上的根本性错误：
> - **同步**：控制执行顺序。“你做完了通知我”。（像接力棒）
> - **互斥**：保护资源独占。“我在用，你别进来”。（像厕所钥匙）


---

## 1. 核心机制详解

### 1.1 [[信号量]]—— 专职“通知”

信号量本质上是一个维护计数的机制，主要用于任务间、任务与中断间的**同步**。

- **二值信号量**：
    - **原理**：计数仅为 0 或 1。
    - **特性**：**无所有权**。任务 A 释放，任务 B 获取。
    - **场景**：**中断唤醒任务**。例如 ADC 转换完成中断触发，ISR 释放信号量，唤醒阻塞的数据处理任务。

- **计数信号量**：
    - **原理**：计数范围 0 ~ N。
    - **场景**：**资源池管理**。例如系统有 3 个 DMA 通道，初始化计数为 3。任务申请时 Take（减 1），释放时 Give（加 1）。计数为 0 时，新任务排队等待。

### 1.2 互斥锁—— 专职“保护”

虽然 Mutex 行为类似二值信号量，但它拥有两个灵魂特性，专门用于**互斥**：

1.  **所有权**：只有拿到锁的任务才能释放锁。这防止了“张三占了厕所，李四把门打开了”的荒谬情况。
2.  **优先级继承**：这是 Mutex 的杀手锏。当高优先级任务等待低优先级任务持有的锁时，系统会临时提升低优先级任务的优先级，以解决 [优先级翻转](../Problems/优先级翻转.md)问题。

> [!warning] 严禁在 ISR 中使用 Mutex
> 中断没有任务上下文（TCB），无法持有“所有权”，且中断不能阻塞。ISR 中请使用 `xSemaphoreGiveFromISR` 操作二值信号量。

## 2. 底层防线：[[临界区]]与原子操作

除了 OS 级别的对象，我们还有更底层的硬件级保护手段，适用于极短时间的保护。

### 2.1 临界区
- **机制**：操作 `BASEPRI` 寄存器，直接屏蔽中断。
- **代价**：破坏实时性。**临界区内代码必须极短**，严禁调用 `vTaskDelay` 等阻塞 API。
- **适用**：结构体多成员赋值、极短的寄存器配置。

### 2.2 原子操作
- **机制**：利用 CPU 的硬件指令（如 CAS、LDREX/STREX），在汇编层面保证“读-改-写”不可分割。
- **适用**：简单的全局计数器、标志位翻转。效率最高，不涉及调度。

## 3. 进阶利器：[任务通知](任务通知.md)

在 FreeRTOS V8.2.0 之后，引入了更轻量级的机制——**[任务通知](任务通知.md)**。

> [!tip] 为什么推荐任务通知？
> 相比信号量，任务通知直接向目标任务发送一个事件或数值，**不需要创建队列对象**。
> - **速度更快**：通常快 45%。
> - **更省内存**：每个任务控制块（TCB）内部自带 8 字节通知值，无需额外分配 RAM。

### 3.1 代码对比
**传统信号量方式**：
```c
// 需要先创建对象
SemaphoreHandle_t xSem = xSemaphoreCreateBinary();

// ISR 中释放
xSemaphoreGiveFromISR(xSem, &xHigherPriorityTaskWoken);
```

**任务通知方式**：

```c
// 无需创建对象，直接通知目标任务句柄
xTaskNotifyFromISR(xTaskToNotify, 0x01, eSetBits, &xHigherPriorityTaskWoken);
```

**局限性**：任务通知只能用于 **1 对 1** 同步，无法实现 1 对多或多对 1。

## 4. 实战推演：生产者-消费者模型

**场景**：传感器采集任务与串口发送任务。  
**需求**：采集完数据后，通知发送任务处理（同步）。

### 4.1 架构设计

- **生产者**：读取传感器 -> 将数据放入[环形缓冲区](../数据流动/环形缓冲区.md) -> 释放信号量。
- **消费者**：等待信号量（阻塞） -> 被唤醒 -> 从缓冲区取数据发送。

### 4.2 代码实现

```c
#include "freertos/FreeRTOS.h"
#include "freertos/task.h"
#include "freertos/semphr.h"

SemaphoreHandle_t xDataReadySem; // 信号量句柄
uint16_t sensor_buffer[10];      // 模拟共享数据

// 任务 A：生产者
void vTask_Producer(void *pvParameters) {
    while (1) {
        // 1. 读取传感器数据到缓冲区
        Read_Sensor(sensor_buffer);
        
        // 2. 数据就绪，释放信号量（计数 +1）
        // 此时若有任务在等待，会被唤醒进入就绪态
        xSemaphoreGive(xDataReadySem);
        
        vTaskDelay(pdMS_TO_TICKS(1000));
    }
}

// 任务 B：消费者
void vTask_Consumer(void *pvParameters) {
    while (1) {
        // 1. 等待信号量（计数 -1，若为 0 则阻塞）
        // portMAX_DELAY 表示死等，直到收到通知
        if (xSemaphoreTake(xDataReadySem, portMAX_DELAY) == pdTRUE) {
            // 2. 被唤醒，处理数据
            Send_UART(sensor_buffer);
        }
    }
}

void app_main(void) {
    // 必须在使用前创建
    xDataReadySem = xSemaphoreCreateBinary(); 
    
    xTaskCreate(vTask_Producer, "Producer", 2048, NULL, 1, NULL);
    xTaskCreate(vTask_Consumer, "Consumer", 2048, NULL, 2, NULL);
}
```

## 5. 避坑指南：架构师的血泪教训

常见陷阱

1. **死锁**：任务 A 拿了锁 1，想去拿锁 2；任务 B 拿了锁 2，想去拿锁 1。两人互相等待，系统卡死。
    - **对策**：规定锁的获取顺序（如必须先拿锁 1 再拿锁 2），或设置超时时间。
2. **优先级翻转误用**：用二值信号量去保护 I2C 总线（互斥场景），导致高优先级任务被中优先级任务阻塞。
    - **对策**：保护共享资源必须用 **Mutex**。
3. **临界区滥用**：在临界区内调用 `vTaskDelay` 或打印大量日志。
    - **后果**：中断被屏蔽时间过长，看门狗复位，系统实时性崩溃。

## 6. 选型决策矩阵

|保护机制|核心用途|ISR 支持|内存开销|执行效率|典型场景|
|:--|:--|:--|:--|:--|:--|
|**原子操作**|单变量保护|✅|极低|极高|全局计数器、标志位|
|**临界区**|极短代码块|✅ (专用API)|低|高|结构体赋值、寄存器配置|
|**任务通知**|1对1 同步|✅|极低 (TCB自带)|极高|中断唤醒特定任务|
|**二值信号量**|任务同步|✅|中|中|生产者-消费者、中断同步|
|**互斥锁**|资源互斥|❌|中|中|I2C/SPI 总线、串口打印|
|**队列**|数据传递|✅|高|低|大量数据的跨任务传输|

**架构建议**：优秀的嵌入式设计应遵循“**数据流驱动**”。首选 [FreeRTOS Queue](app://obsidian.md/FreeRTOS%20Queue) 或 [流缓冲区](../数据流动/流缓冲区.md) 进行数据传递，从架构上减少共享资源，从而规避锁的使用。
