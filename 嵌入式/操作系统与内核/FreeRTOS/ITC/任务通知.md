# 任务通知

在 FreeRTOS 的世界里，任务通知是一场“降维打击”。在它出现之前，我们做任务间同步或中断唤醒，通常依赖信号量或事件组。但这类传统的 IPC（进程间通信）机制非常“重”。

> [!info] 核心本质
> 任务通知是一种**“直达任务”**的轻量级信令机制。
> 传统的信号量/队列是“公共信箱”模式（需要第三方对象），而任务通知是“拨打私人手机”模式（直接修改目标任务 TCB 内部变量）。

---

## 1. 原理剖析：为什么它是最快的？

### 1.1 抛弃“中间商”
当你创建一个队列或信号量时，内核需要单独分配 RAM，维护复杂的链表。每次操作都涉及链表遍历和状态检查。

**任务通知的做法**：
直接查看任务的“身份证”—— [[TCB|任务控制块]]。在 TCB 结构体中，FreeRTOS 直接硬编码植入了专属变量：
- `ulNotifiedValue`：一个 32 位的通知值。
- `ucNotifyState`：通知状态（等待、待处理等）。

**优势**：
1.  **零额外内存**：不需要创建额外的 OS 对象，直接“借用”任务的 TCB 空间。
2.  **极致速度**：解除阻塞的速度比传统信号量快 **45%** 以上（官方数据）。因为省去了链表操作，直接进行内存读写。

### 1.2 灵活的多面手
虽然只有一个 32 位变量，但它可以模拟多种机制：
- **二值信号量**：仅使用 1 个 bit 或简单的非零判断。
- **计数信号量**：让 `ulNotifiedValue` 进行累加。
- **事件组**：将 32 位值的每一位当作独立的事件标志。
- **邮箱**：直接传递一个 32 位的数据（如指针或状态码）。

## 2. 实战链路：极速唤醒机制

**场景**：ADC 转换完成中断唤醒电机控制任务。这是一对一的紧密耦合场景，任务通知是完美选择。

### 2.1 代码实现

```c
// 1. 定义全局任务句柄（ISR 需要拿着它去“拨号”）
TaskHandle_t xMotorControlTask = NULL;

// 2. 消费者：等待通知（替代 xSemaphoreTake）
void vTask_MotorControl(void *pvParameters) {
    while(1) {
        // 参数 pdTRUE：退出阻塞时自动清零通知值（模拟二值信号量）
        // 参数 portMAX_DELAY：无限期等待
        ulTaskNotifyTake(pdTRUE, portMAX_DELAY);

        // --- 醒来，执行关键算法 ---
        Update_Motor_FOC();
    }
}

// 3. 生产者：发送通知（替代 xSemaphoreGiveFromISR）
void ADC_IRQHandler(void) {
    BaseType_t xHigherPriorityTaskWoken = pdFALSE;

    // 清除硬件中断标志
    ADC_ClearITPendingBit(ADC1, ADC_IT_EOC); 

    // [核心] 直接向指定任务的 TCB 发送通知
    // 这会使得 xMotorControlTask 的通知值 +1，并解除其阻塞
    vTaskNotifyGiveFromISR(xMotorControlTask, &xHigherPriorityTaskWoken);

    // 触发上下文切换
    portYIELD_FROM_ISR(xHigherPriorityTaskWoken);
}
```

## 3. 避坑指南：架构师的忠告

任务通知虽快，但绝不是万能的。它的设计决定了其天然的局限性。

致命限制与陷阱

### 3.1 单播限制

- **原理**：任务通知必须指定一个明确的 `TaskHandle`。
- **限制**：**只能用于单一接收者**。
- **场景**：如果 Wi-Fi 连接成功，需要同时唤醒“日志任务”、“UI 任务”和“上传任务”，绝对不能用任务通知。请使用 [事件组](app://obsidian.md/%E4%BA%8B%E4%BB%B6%E7%BB%84) 或 [信号量](app://obsidian.md/%E4%BF%A1%E5%8F%B7%E9%87%8F)。

### 3.2 无缓冲能力

- **原理**：TCB 里只有一个 `ulNotifiedValue` 变量。
- **限制**：它不是 FIFO 队列。如果 ISR 触发太快，任务还没来得及处理，新的通知可能会覆盖旧的（取决于配置），或者仅仅是计数增加。
- **场景**：如果你需要传递一串字节流，必须使用 [FreeRTOS Queue](app://obsidian.md/FreeRTOS%20Queue) 或 [流缓冲区](app://obsidian.md/%E6%B5%81%E7%BC%93%E5%86%B2%E5%8C%BA)。

### 3.3 发送方不能阻塞

- **原理**：发送通知是直接写内存，速度极快。
- **限制**：如果接收方来不及处理，发送方（如 ISR）无法像写队列那样“阻塞等待”。发送操作是“即发即忘”的。

### 3.4 驱动解耦陷阱

- **错误做法**：为了追求性能，在通用的 UART 驱动底层直接写死 `vTaskNotifyGiveFromISR(xAppTask)`。
- **后果**：这个驱动库彻底绑死了上层应用，无法移植到其他项目。
- **建议**：通用驱动库应保持对 OS 的无知，或仅依赖标准的 OSAL 对象。任务通知只应用在**应用层特定的 ISR** 或紧密耦合的任务间。

## 4. 进阶：索引通知 (FreeRTOS V10.4.0+)

在较新的 FreeRTOS 版本中，TCB 里的通知值从“单一变量”变成了一个**数组**（默认大小为 1）。

这意味着你可以对同一个任务使用不同的“通道”：

- `ulTaskNotifyTakeIndexed(0, ...)`：通道 0 用于 ADC 唤醒。
- `ulTaskNotifyTakeIndexed(1, ...)`：通道 1 用于按键唤醒。

这极大地增强了单个任务处理多路事件的能力，且依然保持了极低的内存开销。

## 5. 选型决策矩阵

|特性|任务通知|二值信号量|消息队列|
|:--|:--|:--|:--|
|**内存开销**|极低 (0 额外)|低|高|
|**执行速度**|极快|中|慢|
|**接收者数量**|**仅 1 个**|多个|多个|
|**数据传递**|32 位值 (或指针)|无|任意大小|
|**缓冲能力**|无 (仅状态/计数)|无|有 (FIFO)|
|**典型场景**|高频 ISR 唤醒、任务同步|多任务广播|数据流传输|

**大师建议**：

- 90% 的“中断唤醒任务”场景，任务通知都是最优解。
- 涉及“数据流”或“多任务广播”时，请退回到队列或事件组。
