> [!abstract] 核心本质
> 进程是**资源分配**的基本单位，线程是**执行调度**的基本单位，协程是**用户态轻量级线程**。在嵌入式 RTOS 中，"任务"本质上就是线程——共享一个地址空间，每个任务独立栈，内核调度器切换。理解三者的区别和切换代价，是搞懂 [[进程调度算法]]、[[互斥与同步]]、[[IPC-进程间通信]] 的前提。

---

## 一、资源隔离模型

### 1.1 进程、线程、协程的内存视图

```
进程 A（独立地址空间，有 MMU）                进程 B（独立地址空间）
┌──────────────────────────┐                ┌──────────────────────────┐
│  CODE / TEXT             │                │  CODE / TEXT             │
│  DATA / BSS              │                │  DATA / BSS              │
│  ┌────────────────────┐  │                │  ┌────────────────────┐  │
│  │  HEAP（共享）       │  │                │  │  HEAP              │  │
│  └──┬─────┬─────┬─────┘  │                │  └──┬─────────────────┘  │
│     │     │     │        │                │     │                    │
│  ┌──▼──┐┌─▼──┐┌─▼───┐   │                │  ┌──▼──┐                 │
│  │线程1││线程2││线程3│   │                │  │线程1│                 │
│  │Stack││Stack││Stack│   │                │  │Stack│                 │
│  │ TCB ││ TCB ││ TCB │   │                │  │ TCB │                 │
│  └─────┘└────┘└─────┘   │                │  └─────┘                 │
└──────────────────────────┘                └──────────────────────────┘

关键区别：
  - 进程间：地址空间完全隔离（靠 MMU 翻译）
  - 线程间：共享进程的堆、代码、数据，但各自有独立栈和 TCB
  - 协程间：共享线程的栈，由用户程序自己管理上下文
```

### 1.2 三者核心对比

| 维度 | 进程 | 线程 | 协程 |
|------|------|------|------|
| **定义** | 资源分配的基本单位 | 执行调度的基本单位 | 用户态轻量级线程 |
| **地址空间** | 独立（MMU 隔离） | 共享所属进程 | 共享所属线程 |
| **切换开销** | 大（页表切换 + TLB 刷新 + Cache 失效） | 中（寄存器 + 栈指针） | 小（仅几个寄存器，纯用户态） |
| **切换者** | 内核 | 内核 | 用户程序 |
| **通信方式** | [[IPC-进程间通信]]（管道、共享内存、消息队列） | 全局变量 / 锁 | 共享变量（天然同线程） |
| **适用场景** | 隔离性要求高 | 并发执行 | 高并发 IO |
| **嵌入式常见度** | 少（多数 MCU 无 MMU） | 核心（RTOS 任务模型） | 特定场景（Protothreads） |

> [!tip] 嵌入式视角
> 绝大多数 Cortex-M MCU **没有 MMU**，跑不了真正的进程。RTOS 里的"任务"本质上就是线程——共享一个扁平地址空间。只有跑 Linux 的嵌入式处理器（如 Cortex-A、ESP32 的某些配置）才有进程的概念。

## 二、切换机制详解

### 2.1 进程切换（最重）

```
用户态                          内核态                         硬件
─────                          ─────                          ────

用户程序执行中
    │
    ├── 系统调用 / 中断 ──────→  进入内核态
    │                           │
    │                     ┌─────▼──────┐
    │                     │ 保存当前进程 │
    │                     │ 上下文：     │
    │                     │  PC, SP,     │
    │                     │  通用寄存器  │
    │                     │  页表基址    │
    │                     └─────┬──────┘
    │                           │
    │                     ┌─────▼──────┐     ┌─────────────┐
    │                     │ 刷新 TLB   │────→│ Cache 失效   │
    │                     └─────┬──────┘     │ → 性能抖动   │
    │                           │            └─────────────┘
    │                     ┌─────▼──────┐
    │                     │ 切换内核栈  │
    │                     │ 加载新进程  │
    │                     │ 上下文      │
    │                     │ 新页表      │
    │                     └─────┬──────┘
    │                           │
    │◄──── 返回用户态 ──────────┘
    │
用户程序恢复执行（新进程）
```

**代价**：切换页表 → TLB 全部失效 → Cache 大面积冷启动 → 一次切换可能消耗**数千到上万时钟周期**。

### 2.2 线程切换（中等）

```
用户态                          内核态
─────                          ─────

用户程序执行中
    │
    ├── 时间片中断 / 系统调用 ─→  进入内核态
    │                             │
    │                       ┌─────▼──────┐
    │                       │ 保存当前线程 │
    │                       │ 上下文：     │
    │                       │  PC, SP,     │
    │                       │  通用寄存器  │
    │                       └─────┬──────┘
    │                             │
    │                       ┌─────▼──────┐
    │                       │ 切换栈指针  │
    │                       │ 加载新线程  │
    │                       │ 上下文      │
    │                       └─────┬──────┘
    │                             │
    │◄──── 返回用户态 ────────────┘
    │
 同一进程内，地址空间不变
 TLB 不用刷新，Cache 影响小
```

**代价**：只保存/恢复寄存器组 + 切换栈指针，通常**几百时钟周期**。同进程内的线程切换远比进程切换轻量。

### 2.3 协程切换（最轻）

```
应用程序（纯用户态，内核无感知）
───────

协程 A 执行中
    │
    ├── yield（主动让出 CPU）
    │
    │   ┌──────────────────────────┐
    │   │ 保存协程 A 上下文：       │
    │   │ 仅 SP + 几个寄存器       │
    │   │ 无系统调用，无内核参与    │
    │   └────────┬─────────────────┘
    │            │
    │   ┌────────▼─────────────────┐
    │   │ 恢复协程 B 上下文        │
    │   └────────┬─────────────────┘
    │            │
    ├── 协程 B 开始执行
    │
    │   关键：整个切换在用户态完成
    │   内核根本不知道发生了切换
    │   代价：仅几十个时钟周期
```

**代价**：纯用户态，无系统调用，无内核参与。切换仅**几十时钟周期**。

> [!important] 切换代价量级对比
> - 进程切换：~10,000 周期（毫秒级）
> - 线程切换：~100-500 周期（微秒级）
> - 协程切换：~20-50 周期（纳秒级）
> 
> 在 RTOS 中，任务切换属于线程级别，FreeRTOS 在 Cortex-M4 上一次切换大约 **200-400 周期**。

## 三、RTOS 任务模型（嵌入式核心）

### 3.1 RTOS 任务 ≈ 线程

RTOS 的任务模型本质上是**线程模型**，因为它在无 MMU 的扁平内存模型上运行：

```
RTOS 扁平内存模型（无 MMU）：
┌──────────────────────────────────────────────┐
│  全局共享地址空间                              │
│  ┌────────┐  ┌────────┐  ┌────────┐          │
│  │ 任务1  │  │ 任务2  │  │ 任务3  │  ...      │
│  │ Stack  │  │ Stack  │  │ Stack  │          │
│  │ + TCB  │  │ + TCB  │  │ + TCB  │          │
│  └────────┘  └────────┘  └────────┘          │
│  共享：代码区、全局变量、堆、外设寄存器        │
│  隔离：各自有独立栈                           │
└──────────────────────────────────────────────┘
```

```c
// FreeRTOS 任务创建
void vTask1(void *pvParameters) {
    while (1) {
        // 任务逻辑
        vTaskDelay(pdMS_TO_TICKS(100));
    }
}

void vTask2(void *pvParameters) {
    while (1) {
        // 任务逻辑
    }
}

int main(void) {
    xTaskCreate(vTask1, "Task1", 128, NULL, 1, NULL);
    xTaskCreate(vTask2, "Task2", 128, NULL, 1, NULL);
    vTaskStartScheduler();
}
```

### 3.2 RTOS 任务 vs Linux 进程

| 特性 | RTOS 任务 | Linux 进程 |
|------|----------|-----------|
| 地址空间 | 共享（无 MMU，扁平模型） | 独立（有 MMU，[[MMU(内存管理单元)]] 隔离） |
| 内存保护 | 无或弱（MPU 可选） | 强（MMU 硬件隔离） |
| 切换开销 | 微秒级 | 毫秒级 |
| 典型数量 | 几十到上百个 | 上百到上千个 |
| 通信方式 | [[../04_FreeRTOS/同步与通信/队列]]、[[../04_FreeRTOS/同步与通信/信号量]]、[[../04_FreeRTOS/同步与通信/任务通知]] | [[IPC-进程间通信]] 全套（管道、socket、共享内存） |

### 3.3 嵌入式中的协程应用

在资源极度受限的 MCU 上（RAM 只有几 KB），连 RTOS 任务栈都开不起时，协程是很好的替代方案：

```c
// Protothreads 库 —— 无栈协程，适合极度受限 MCU
#include "pt.h"

static struct pt pt_led;

PT_THREAD(led_thread(struct pt *pt)) {
    PT_BEGIN(pt);
    
    while (1) {
        LED_ON();
        PT_SLEEP(pt, 500);
        LED_OFF();
        PT_SLEEP(pt, 500);
    }
    
    PT_END(pt);
}
```

**核心优势**：
- 无栈开销：所有协程共享调用者的栈
- 实现极简：基于 `switch-case` 的 Duff's Device 技巧
- 代价：不能有局部变量跨 `yield` 保持（因为栈是共享的）

> [!warning] 协程的限制
> 协程中**绝对不能**调用阻塞操作（如 `vTaskDelay`、`HAL_Delay`），否则整个线程被卡住。协程只适合 IO 等待场景，用 `yield` 让出控制权。

## 四、并发与并行

### 4.1 概念辨析

```
并发 (Concurrency)                     并行 (Parallelism)
─────────────────                      ─────────────────
单核 CPU                               多核 CPU
┌────────────────────┐                ┌──────────┬──────────┐
│  时间片轮转          │                │ Core 0   │ Core 1   │
│  ┌───┐ ┌───┐ ┌───┐ │                │          │          │
│  │ T1│→│T2 │→│T3 │ │                │  T1 执行  │  T2 执行  │
│  └───┘ └───┘ └───┘ │                │  真正的   │  真正的   │
│  宏观同时，微观交替  │                │  同时执行  │  同时执行  │
└────────────────────┘                └──────────┴──────────┘

一句话：并发是"看起来同时"，并行是"真的同时"
```

### 4.2 多线程 ≠ 一定更快

```
开销来源：
  1. 线程创建和销毁（TCB 分配、栈分配）
  2. 上下文切换（寄存器保存恢复）
  3. Cache 失效（详见 [[../02_多核与底层机制/多核并发与Cache一致性]] 中伪共享问题）
  4. 锁竞争（[[互斥与同步]] 的核心问题）

适用场景分析：
  ✅ IO 密集型（等待时切换出去，CPU 不闲着）
  ✅ 多核并行计算（真正的并行加速）
  ❌ 计算密集型 + 单核（切换开销 > 收益，反而更慢）
```

**感悟**：在实际嵌入式项目中，不要为了"多任务"而多任务。很多场景一个主循环 + [[../03_裸机架构模式/FSM-有限状态机]] + 中断就足够了。RTOS 的引入应该是**有明确需求**（实时性要求、多个独立功能模块需要并发），而不是一开始就上 RTOS。

## 五、工程避坑指南

> [!danger] 致命陷阱：多任务环境下的常见坑

| 陷阱 | 后果 | 解决方案 | 相关笔记 |
|------|------|---------|---------|
| 多任务共享数据不加锁 | 竞态条件、数据损坏 | 互斥锁、原子操作、临界区 | [[互斥与同步]] |
| 死锁（循环等待资源） | 系统卡死 | 锁获取顺序一致、超时锁 | [[../04_FreeRTOS/锁与并发保护/锁与并发保护机制]] |
| 栈溢出 | HardFault / 跑飞 | RTOS 任务栈预留充足空间，用 `uxTaskGetStackHighWaterMark()` 监控 | [[../04_FreeRTOS/任务管理/任务栈与溢出防护]] |
| 优先级翻转 | 高优先级任务被低优先级阻塞 | 优先级继承协议（Mutex 自带） | [[../04_FreeRTOS/任务管理/优先级翻转]] |
| 协程中调用阻塞 API | 整个线程被卡死 | 协程中只用非阻塞 API | — |
| ISR 中调用阻塞 API | 系统卡死或崩溃 | 只用 `FromISR` 后缀 API | [[../04_FreeRTOS/锁与并发保护/FromISR机制]] |
| 任务栈里的局部变量过大 | 栈溢出 | 大缓冲区用全局数组或内存池 | [[../03_裸机架构模式/内存池]] |

### 5.1 栈溢出的隐蔽性

```c
// 这个任务看起来没问题，但 512 字节栈可能不够
void vSensorTask(void *pvParameters) {
    while (1) {
        float buffer[128];  // 128 * 4 = 512 字节！栈直接爆了
        read_sensor(buffer);
        vTaskDelay(pdMS_TO_TICKS(100));
    }
}

// 正确做法：大缓冲区用全局变量或内存池
static float g_sensor_buffer[128];  // 放全局区

void vSensorTask(void *pvParameters) {
    while (1) {
        read_sensor(g_sensor_buffer);
        vTaskDelay(pdMS_TO_TICKS(100));
    }
}
```

### 5.2 优先级翻转实例

```
时间线 →→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→

低优先级任务 L    ████████ 锁持有中 ████████
                          ↓ 被抢占
中优先级任务 M              ████████████████ 执行
                                    ↓
高优先级任务 H          ██ 请求锁 → 被阻塞！等待 L 释放
                        但 L 被 M 抢占了 → H 被 M 间接阻塞
                        = 优先级翻转！

解决方案（Mutex 的优先级继承）：
L 持有锁 + H 等待 → L 的优先级临时提升到 H 的级别
→ L 不会被 M 抢占 → L 尽快释放锁 → H 尽快获得执行
```

详见 [[../04_FreeRTOS/任务管理/优先级翻转]]

## 六、面试高频问题

> [!example]- Q1：进程、线程、协程的区别？
> 进程是资源分配的基本单位（独立地址空间），线程是执行调度的基本单位（共享地址空间），协程是用户态轻量级线程（无内核参与切换）。切换代价：进程 > 线程 > 协程。

> [!example]- Q2：为什么 RTOS 的任务更像线程而不是进程？
> 因为多数 MCU 无 MMU，所有任务共享一个扁平地址空间，没有独立的虚拟地址空间。每个任务只是有独立的栈和 TCB，本质就是线程。

> [!example]- Q3：协程为什么比线程轻量？
> 协程切换在用户态完成，不需要系统调用、不需要内核参与、不需要切换栈。只需要保存恢复几个寄存器（SP 为主）。代价是协程不能利用多核，且不能有阻塞操作。

> [!example]- Q4：进程切换为什么这么慢？
> 进程切换需要：保存旧进程上下文 → 切换页表（修改 TTBR 寄存器）→ 刷新 TLB → Cache 大面积失效 → 加载新进程上下文。其中 TLB + Cache 重建是最大的性能杀手。

> [!example]- Q5：嵌入式项目中什么时候该用 RTOS 多任务，什么时候用裸机？
> 简单场景（< 3 个并发功能、实时性要求低）：裸机 + 主循环 + 中断 + [[../03_裸机架构模式/FSM-有限状态机]]。复杂场景（多并发模块、有实时性约束、需要同步/通信机制）：RTOS 多任务。核心判断标准是**系统是否有多个独立的、需要并发执行的功能模块**。
