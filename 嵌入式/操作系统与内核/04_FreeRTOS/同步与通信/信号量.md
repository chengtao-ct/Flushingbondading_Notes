信号量是 RTOS 中最古老的同步机制之一。如果说 [[互斥锁 Mutex]] 是用来“抢厕所”的，那么信号量更多是用来“发信号”或“数数”的。

> [!info] 核心本质
> 信号量本质上是一个 **整数计数器** + **等待队列**。
> 它不传输数据，只管理“许可”。
> - **P 操作**：申请许可。如果计数器 > 0，减 1 并继续；否则阻塞等待。
> - **V 操作**：释放许可。计数器加 1，如果有任务在等待，唤醒其中一个。

---

## 1. 分类与场景：二值 vs 计数

在 FreeRTOS 中，信号量主要分为两类，应用场景截然不同。

### 1.1 二值信号量
- **数值范围**：仅 0 或 1。
- **核心用途**：**任务同步**，特别是“中断唤醒任务”。
- **类比**：**接力棒**。ISR 跑完把棒子扔下，任务捡起棒子接着跑。
- **特性**：没有“所有权”概念。任何任务都可以 Give，任何任务都可以 Take。

### 1.2 计数信号量
- **数值范围**：0 ~ N。
- **核心用途**：**资源计数** 或 **事件计数**。
- **类比**：**停车场显示屏**。
    - 车位总数 = N。
    - 车进库：Take（计数减 1）。
    - 车出库：Give（计数加 1）。
    - 计数为 0：车位满，新车必须排队（阻塞）。
- **典型场景**：管理 DMA 通道、内存池块、I2C 总线占用数等有限资源。

## 2. 实战链路：中断同步（二值信号量）

这是二值信号量最经典的用法：让高优先级的任务在 ISR 触发后立即执行，实现 [[../内存管理/中断推迟处理]]。

### 2.1 代码实现

```c
// 1. 创建二值信号量
SemaphoreHandle_t xBinarySem;

void App_Init(void) {
    xBinarySem = xSemaphoreCreateBinary();
}

// 2. ISR：生产者（释放信号）
void EXTI0_IRQHandler(void) {
    BaseType_t xHigherPriorityTaskWoken = pdFALSE;

    if(EXTI_GetITStatus(EXTI_Line0)) {
        EXTI_ClearITPendingBit(EXTI_Line0);
        
        // [核心] 释放信号量，计数从 0 变为 1
        // 如果有任务在等，会将其唤醒
        xSemaphoreGiveFromISR(xBinarySem, &xHigherPriorityTaskWoken);
        
        // 请求上下文切换
        portYIELD_FROM_ISR(xHigherPriorityTaskWoken);
    }
}

// 3. Task：消费者（获取信号）
void vTask_Handler(void *pvParameters) {
    while(1) {
        // [核心] 等待信号量。计数为 0 时阻塞，为 1 时获取并减为 0
        if(xSemaphoreTake(xBinarySem, portMAX_DELAY) == pdTRUE) {
            // 被唤醒，处理紧急事件
            Process_Event();
        }
    }
}
```
## 3. 经典模型：生产者-消费者

你提供的笔记中包含了生产者-消费者代码，这是操作系统教学的经典案例。在 FreeRTOS 工程中，这通常用于数据缓冲场景。

**模型要素**：

- `mutex`：互斥锁，保护缓冲区操作。
- `empty`：计数信号量，记录空闲缓冲区数量。
- `full`：计数信号量，记录已填充缓冲区数量。

### 3.1 FreeRTOS 实现版

```c
#define BUFFER_SIZE 10

SemaphoreHandle_t xMutex;      // 互斥锁
SemaphoreHandle_t xEmptySlots; // 计数信号量：初始为 BUFFER_SIZE
SemaphoreHandle_t xFullSlots;  // 计数信号量：初始为 0

void Producer_Task(void *pvParameters) {
    while(1) {
        int item = Produce_Item();

        // 1. 申请一个空位（empty - 1）。如果满了，阻塞
        xSemaphoreTake(xEmptySlots, portMAX_DELAY);
        
        // 2. 获取缓冲区锁
        xSemaphoreTake(xMutex, portMAX_DELAY);
        
        Insert_Item(item); // 写入数据
        
        // 3. 释放锁
        xSemaphoreGive(xMutex);
        
        // 4. 增加已填充计数（full + 1），唤醒消费者
        xSemaphoreGive(xFullSlots);
    }
}

void Consumer_Task(void *pvParameters) {
    while(1) {
        // 1. 申请一个数据（full - 1）。如果空了，阻塞
        xSemaphoreTake(xFullSlots, portMAX_DELAY);
        
        // 2. 获取锁
        xSemaphoreTake(xMutex, portMAX_DELAY);
        
        int item = Remove_Item(); // 读取数据
        
        // 3. 释放锁
        xSemaphoreGive(xMutex);
        
        // 4. 增加空位计数（empty + 1），唤醒生产者
        xSemaphoreGive(xEmptySlots);
        
        Consume_Item(item);
    }
}
```

## 4. 避坑指南：信号量 vs 互斥锁

这是新人最容易混淆的地方，也是面试必问。

致命误区

**不要用二值信号量去做互斥保护！**

### 4.1 为什么？

假设有两个任务 A 和 B，优先级 A > B。

1. 任务 B 拿到了二值信号量（开始使用资源）。
2. 任务 A 就绪，抢占了 B。
3. 任务 A 也想用资源，去拿信号量，结果阻塞了。
4. **问题**：任务 B 本来应该继续运行把资源释放，但因为优先级低于 A，被 A 抢占了。现在 A 阻塞等 B，B 被抢占等 A（等 A 释放 CPU）。
5. **结果**：虽然不会死锁，但造成了不必要的上下文切换和优先级反转风险。

### 4.2 核心区别

|特性|二值信号量|互斥锁|
|:--|:--|:--|
|**核心用途**|**同步**（通知事件发生）|**互斥**（保护资源访问）|
|**所有权**|**无**。谁都可以 Give|**有**。谁 Take 谁 Give|
|**优先级继承**|**无**|**有**（解决优先级翻转）|
|**ISR 中使用**|✅ 可以 (`FromISR`)|❌ 绝对禁止|
|**典型场景**|中断唤醒任务、任务间通知|保护 I2C/SPI 总线、串口打印|

## 5. 总结

信号量是 RTOS 的交通信号灯。

- **二值信号量**：像红绿灯，控制“能不能走”（同步）。
- **计数信号量**：像停车场计数牌，控制“还有多少坑位”（资源管理）。
- **互斥锁**：像厕所钥匙，控制“谁在里面”（互斥）。

**大师建议**：  
在 FreeRTOS 开发中，如果是单纯的“ISR 唤醒 Task”，首选 [任务通知](app://obsidian.md/%E4%BB%BB%E5%8A%A1%E9%80%9A%E7%9F%A5)（更快、更省内存）。只有在需要“多对多同步”或“资源计数”时，才动用信号量。
