在裸机（Super-loop）时代，我们只有一个执行流，最多加个中断。但在 [[RTOS|实时操作系统]] 的多任务抢占式环境中，“锁”是我们维持系统秩序、防止数据被撕裂的最后一道防线。

> [!info] 核心本质
> 锁存在的唯一目的，是为了解决 **[[竞态条件]]**。即：多个执行流（任务/中断）同时访问同一块内存资源，导致数据逻辑混乱。

---

## 1. 原理剖析：为什么纯软件锁很难？

在硬件不支持原子操作之前，人们尝试用纯软件算法（如 Peterson 算法）实现锁，但逻辑极其复杂且效率低下。

现代嵌入式系统（如 STM32F407）完全依赖硬件级的 **原子指令** 来实现高效的锁机制。

### 1.1 硬件原子指令

原子指令是指一条指令完成“读-改-写”操作，中间不可被中断。

#### Test-and-Set (TSL)
读一个值并写一个新值，一步完成。这是 [[自旋锁]] 的基础。

```c
void lock() {
    // TSL 指令：尝试把 flag 设为 1，并返回旧值
    // 如果返回旧值是 1，说明本来就是锁的，继续循环（自旋）
    // 如果返回旧值是 0，说明刚才没锁，现在被我锁住了(变成了1)，循环结束
    while (TestAndSet(&flag) == 1); 
}
````

#### Compare-and-Swap (CAS)

比较内存里的值是不是我期望的，如果是就修改，不是就失败。

```c
void lock() {
    // 尝试把 0 改成 1
    // 如果 flag 原本是 0，改成 1，成功 -> 拿到锁
    // 如果 flag 原本是 1，失败 -> 继续试
    while (CAS(&flag, 0, 1) == false);
}
```

## 2. 实战推演：数据撕裂的灾难

假设你的智能小车有一个全局变量 `uint32_t total_mileage;`（总里程），Task A（底盘测速）和 Task B（里程统计）都会去更新它。

在 C 语言中 `total_mileage++` 是一行代码，但在 ARM Cortex-M4 汇编下是三条指令：

1. `LDR R0, [地址]` （读到寄存器）
2. `ADD R0, R0, #1` （寄存器加1）
3. `STR R0, [地址]` （写回内存）

**灾难场景**：  
如果在 Task A 执行完 `ADD` 但还没执行 `STR` 时，SysTick 中断触发，调度器切换到 Task B。Task B 完整执行了这三条指令。当 CPU 切回 Task A 时，Task A 继续执行旧的 `STR`，覆盖了 Task B 的结果。  
**后果**：两次加 1 操作，最终结果只加了 1。这就是经典的 **Read-Modify-Write 撕裂**。

## 3. 解决方案：RTOS 的两把“武器”

为了保护临界区资源，FreeRTOS 提供了两种截然不同的机制。

### 武器 1：[[临界区]]—— 简单粗暴的“核武器”

- **机制**：直接操作 Cortex-M4 的 `BASEPRI` 寄存器，屏蔽所有受 OS 管理的中断。中断没了，调度器瘫痪，绝对不会发生上下文切换。
- **代码**：
    
    ```c
    taskENTER_CRITICAL();
    total_mileage++; // 此时是绝对安全的原子操作
    taskEXIT_CRITICAL();
    ```
    
- **代价**：破坏实时性！如果临界区代码执行太久，会延迟电机控制等关键中断响应。

### 武器 2：互斥锁—— OS 级的“文明排队”

- **机制**：基于软件标志。Task A 拿到锁后，Task B 再请求会被内核踢进 **阻塞态**。等 A 释放锁，B 才被唤醒。
- **代码**：
    
    ```c
    if(xSemaphoreTake(xMutex, portMAX_DELAY) == pdTRUE) {
        // 安全地访问共享资源，比如操作 I2C 读取 MPU6050
        Update_MPU6050_Data(); 
        xSemaphoreGive(xMutex);
    }
    ```
    
- **代价**：开销大。涉及队列操作和上下文切换，可能消耗数百条指令周期。

### 选型指南

|资源类型|操作耗时|推荐方案|理由|
|:--|:--|:--|:--|
|全局变量、标志位|极短（几个时钟周期）|**临界区**|速度快，无调度开销|
|I2C 总线、长串口指令|较长（毫秒级）|**互斥锁**|临界区会卡死中断，必须用锁让出 CPU|

## 4. 避坑指南：中断中的禁忌

致命陷阱：ISR 中绝对不能使用 Mutex

**不要把操作系统的“软件对象（任务）”和处理器的“硬件机制（中断）”混为一谈。**

### 4.1 为什么 ISR 不能用 Mutex？

1. **所有权冲突**：Mutex 拥有“所有权”概念，绑定任务的 [任务控制块](app://obsidian.md/TCB)。中断没有 TCB，无法记录所有权，也无法执行 [优先级翻转](app://obsidian.md/%E4%BC%98%E5%85%88%E7%BA%A7%E7%BF%BB%E8%BD%AC) 中的优先级继承机制。
2. **阻塞机制冲突**：Mutex 获取失败会进入 **阻塞态**。硬件中断（ISR）运行在 Handler 模式，必须最快速度执行完毕并退出，绝对不允许“休眠”或“排队”。

### 4.2 正确的中断同步方案

**方案 A：二值信号量**  
用于“事件通知”，没有所有权概念。

```c
// 任务中等待
void vTask_MotorControl(void *pvParameters) {
    while(1) {
        if (xSemaphoreTake(xEncoderSemaphore, portMAX_DELAY) == pdTRUE) {
            // 处理逻辑...
        }
    }
}

// ISR 中释放
void EXTI15_10_IRQHandler(void) {
    BaseType_t xHigherPriorityTaskWoken = pdFALSE;
    EXTI_ClearITPendingBit(EXTI_Line11); 
    // 必须使用 FromISR 后缀
    xSemaphoreGiveFromISR(xEncoderSemaphore, &xHigherPriorityTaskWoken);
    portYIELD_FROM_ISR(xHigherPriorityTaskWoken); 
}
```

**方案 B：队列**  
用于“数据传递”。

```c
void USART1_IRQHandler(void) {
    BaseType_t xHigherPriorityTaskWoken = pdFALSE;
    uint8_t rx_data;
    if(USART_GetITStatus(USART1, USART_IT_RXNE) != RESET) {
        rx_data = USART_ReceiveData(USART1);
        xQueueSendFromISR(xUartQueue, &rx_data, &xHigherPriorityTaskWoken);
    }
    portYIELD_FROM_ISR(xHigherPriorityTaskWoken);
}
```

## 5. 进阶：中断中的临界区

如果在 ISR 中需要操作共享数据（如环形缓冲区），不能用 Mutex，只能用临界区。但 FreeRTOS 专门提供了 [[FromISR机制]] 版本。

### 5.1 为什么不能用普通版本？

普通 `taskEXIT_CRITICAL()` 会直接将 `BASEPRI` **清零**（完全开放中断）。  
如果当前处于中断嵌套中（优先级 5 的中断被打断），清零会导致优先级更低的中断（6, 7...）错误地抢占进来，破坏硬件中断嵌套逻辑。

### 5.2 正确代码实现

`_FROM_ISR` 版本采用“现场保护”机制：保存当前 `BASEPRI` 值，退出时恢复原值。

```c
void USART1_IRQHandler(void) {
    uint8_t rx_byte;
    // 1. 声明变量保存当前中断状态
    UBaseType_t uxSavedInterruptStatus;

    if(USART_GetITStatus(USART1, USART_IT_RXNE) != RESET) {
        rx_byte = USART_ReceiveData(USART1);
        
        // 2. 进入临界区（保存现场）
        uxSavedInterruptStatus = taskENTER_CRITICAL_FROM_ISR();
        
        RingBuffer_Push(&my_buffer, rx_byte);
        
        // 3. 退出临界区（恢复现场，而不是清零）
        taskEXIT_CRITICAL_FROM_ISR(uxSavedInterruptStatus);
    }
}
```
