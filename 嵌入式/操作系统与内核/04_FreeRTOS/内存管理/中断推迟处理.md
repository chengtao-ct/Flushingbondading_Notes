# 中断推迟处理

在调试 STM32F407ZGT6 的底盘电机控制时，你必然会遇到需要处理高频编码器脉冲（通过 EXTI 或定时器输入捕获）的场景。如果你按照裸机的思维，在中断里直接进行电机转速的浮点数运算和 PID 闭环更新，整个系统很快就会崩溃。

> [!info] 核心矛盾
> **硬件要求立刻响应** vs **软件需要大量计算**。
> ISR 运行在 Cortex-M4 的 Handler 模式下，它会无情地抢占所有软件任务。如果 ISR 执行时间过长，会破坏操作系统的实时调度，导致其他低优先级硬件中断（如串口接收、I2C 传感器读取）被饿死或丢失数据。

---

## 1. 架构范式：上半部与下半部

为了解决上述矛盾，Linux 内核引入了“上半部”和“下半部”的概念，FreeRTOS 完美继承了这一架构：

- **上半部（Top Half / ISR）**：
    - **职责**：只做绝对必要、极度耗时短的事情。
    - **动作**：清除硬件中断标志位、读取外设寄存器数据、发送同步信号。
    - **原则**：快进快出，严禁阻塞，严禁进行复杂运算。

- **下半部（Bottom Half / Task）**：
    - **职责**：处理繁重的逻辑。
    - **动作**：浮点运算、PID 算法、格式化打印、访问共享资源（使用 [[互斥锁 Mutex]]）。
    - **原则**：在普通任务上下文中运行，允许阻塞，允许被更高优先级抢占。

## 2. 实战策略：三种工程解法

根据性能需求和资源限制，FreeRTOS 提供了三种主流的实现路径。

### 策略 A：极致性能 —— [任务通知](../ITC/任务通知.md)

**适用场景**：高频事件（如电机编码器、高速 ADC），要求极低延迟。
**优势**：FreeRTOS V8.2+ 推荐方案。直接修改目标任务 TCB 内部变量，速度比信号量快 45%，且无需额外分配内存。

```c
// 定义任务句柄，用于 ISR 定位目标任务
TaskHandle_t xMotorPIDTaskHandle = NULL;

// 【下半部】：处理复杂逻辑
void vTask_MotorPID(void *pvParameters) {
    while(1) {
        // 阻塞等待通知（类似二值信号量，但更快）
        // pdTRUE 表示退出时自动清零计数
        ulTaskNotifyTake(pdTRUE, portMAX_DELAY);

        // --- 安全区域：任务上下文 ---
        // 执行耗时的浮点 PID 运算
        Calculate_PID_And_Update_PWM();
    }
}

// 【上半部】：硬件中断
void EXTI15_10_IRQHandler(void) {
    BaseType_t xHigherPriorityTaskWoken = pdFALSE;

    if(EXTI_GetITStatus(EXTI_Line11) != RESET) {
        EXTI_ClearITPendingBit(EXTI_Line11); // 1. 清除标志

        // 2. 直接向目标任务发送通知
        vTaskNotifyGiveFromISR(xMotorPIDTaskHandle, &xHigherPriorityTaskWoken);

        // 3. 申请上下文切换
        portYIELD_FROM_ISR(xHigherPriorityTaskWoken);
    }
}
```

### 策略 B：数据缓冲 —— 消息队列

**适用场景**：需要传递具体数据（而非仅信号），且涉及共享资源保护。  
**优势**：解耦生产者（ISR）与消费者，天然缓冲数据，配合互斥锁解决资源竞争。

案例：ADC 采样与 OLED 显示

假设 ADC 采样后需通过 I2C 总线显示在 OLED 上。I2C 操作耗时且需要 [互斥锁 Mutex](app://obsidian.md/%E4%BA%92%E6%96%A5%E9%94%81%20Mutex)，绝不能在 ISR 中执行。

**标准执行链路**：

1. **ISR 阶段**：读取 ADC 寄存器 -> `xQueueSendFromISR` 压入队列 -> 退出（耗时 < 5us）。
2. **Task 阶段**：任务阻塞解除 -> 从队列取数据 -> 获取 Mutex -> 操作 I2C -> 释放 Mutex
```c
// 定义回调函数（将在 Timer 任务上下文中执行）
void vKeyDebounceCallback(void *pvParameter1, uint32_t ulParameter2) {
    // 这里可以安全地进行延时、打印等操作
    printf("Key Pressed, Debounce Handled.\n");
}

// 【上半部】：ISR
void EXTI0_IRQHandler(void) {
    BaseType_t xWoken = pdFALSE;
    if(EXTI_GetITStatus(EXTI_Line0)) {
        EXTI_ClearITPendingBit(EXTI_Line0);
        
        // 将回调函数“委托”给 Timer 任务执行
        xTimerPendFunctionCallFromISR(vKeyDebounceCallback, NULL, 0, &xWoken);
        portYIELD_FROM_ISR(xWoken);
    }
}
```
## 3. 避坑指南：大师的工程建议

性能与稳定性红线

### 3.1 如何量化 ISR 的“短”？

不要凭感觉。教你在示波器下看真相：

1. 在 ISR 入口第一行代码拉高某个 GPIO。
2. 在 `portYIELD_FROM_ISR` 前拉低该 GPIO。
3. 用示波器测量脉宽。

- **标准**：STM32F407 @ 168MHz 下，ISR 脉宽应严格控制在 **1~5 微秒**以内。如果达到几十微秒，必须重构！

### 3.2 警惕“中断风暴”

如果使用[任务通知](../ITC/任务通知.md)，且 ISR 触发频率（如电机飞车）超过了下半部任务的处理速度：

- **后果**：下半部任务一直处于就绪态，抢占 CPU，低优先级任务（如通信、日志）被饿死，形成“活锁”。
- **对策**：硬件层面加滤波，或在极高频场景下改用定时器轮询模式，而非每个脉冲都触发中断。

### 3.3 资源竞争的隔离

永远不要在 ISR 中尝试获取 [互斥锁 Mutex](app://obsidian.md/%E4%BA%92%E6%96%A5%E9%94%81%20Mutex)。ISR 没有任务上下文，一旦阻塞将导致系统死锁。所有涉及互斥锁的操作，必须通过“队列”或“任务通知”转移到下半部任务中执行。
