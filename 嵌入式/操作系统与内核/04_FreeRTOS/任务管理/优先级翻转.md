# 优先级翻转

> [!info] 核心定义
> **优先级翻转**是指在[[RTOS|实时操作系统]]中，高优先级任务被低优先级任务阻塞的现象。其本质是：**低优先级任务用“硬资源（互斥锁）”绑架了高优先级任务，而中优先级任务用“软资源（CPU 时间片）”剥夺了低优先级任务的执行权**。

---

## 1. 原理剖析：为什么会发生？

### 1.1 场景模型
假设系统中有三个任务：
- **任务 H (High)**：优先级最高，紧急数据上报。
- **任务 M (Medium)**：优先级中等，后台计算（不涉及共享资源）。
- **任务 L (Low)**：优先级最低，读取 I2C 传感器（涉及共享资源）。

### 1.2 现象复现（医院类比）
为了方便理解，我们可以将 CPU 比作**诊室**，共享资源（I2C 总线）比作**医生**：

1.  **低优先级任务 L 进场**：慢性病患者 L 进入诊室，占用医生（获得锁）。
2.  **高优先级任务 H 到达**：急诊患者 H 到了，必须马上看医生。但医生被 L 占着，H 只能在门口排队（阻塞）。
3.  **中优先级任务 M 插队**：此时，普通感冒患者 M 来了。虽然 M 没有要看医生，但他想用诊室（CPU）做广播体操。
4.  **翻转发生**：
    - 因为 L 的优先级低于 M，调度器强行把 L 赶出诊室，让 M 进去广播体操。
    - 结果：急诊患者 H（高优先级）不仅被 L（低优先级）阻塞，现在还要被迫等待 M（中优先级）做完体操！
    - **后果**：H 的实时性被彻底破坏，仿佛它的优先级变成了最低。

### 1.3 根本原因
[[任务调度|调度器]]是“无情”的，它只看当前谁的优先级高、谁在就绪态。调度器**并不知道**低优先级任务 L 手里握着高优先级任务 H 急需的资源。

---

## 2. 实战方案 A：互斥锁与优先级继承

这是最常用、成本最低的解决方案。

### 2.1 解决逻辑
摒弃[[二值信号量]]，改用 `xSemaphoreCreateMutex()` 创建[[互斥锁 Mutex]]。
当 H 因等锁被阻塞时，FreeRTOS 内核触发**优先级继承**：
- 内核将 L 的优先级临时“拔高”到和 H 一样的级别。
- 此时 M 无法抢占 L，L 会以最高优先级执行完临界区，释放锁。
- H 瞬间拿到锁，系统恢复正常调度。

### 2.2 代码实现 (ESP32 / FreeRTOS)

```c
#include "freertos/FreeRTOS.h"
#include "freertos/task.h"
#include "freertos/semphr.h"
#include <stdio.h>

SemaphoreHandle_t sharedMutex;

// 优先级 1（最低）：读取 I2C 传感器
void LowPriorityTask(void *pvParameters) {
    while (1) {
        // 尝试获取互斥锁
        if (xSemaphoreTake(sharedMutex, portMAX_DELAY)) {
            printf("Low Task: 拿到锁，当前优先级: %u\n", uxTaskPriorityGet(NULL));
            
            // 模拟耗时的 I2C 读取操作
            // 在优先级继承机制下，此时若 H 任务请求锁，L 的优先级会被临时抬高
            vTaskDelay(pdMS_TO_TICKS(2000)); 
            
            printf("Low Task: 释放锁，优先级恢复: %u\n", uxTaskPriorityGet(NULL));
            xSemaphoreGive(sharedMutex);
        }
        vTaskDelay(pdMS_TO_TICKS(500));
    }
}

// 优先级 2（中等）：处理其他后台计算（不请求锁）
void MediumPriorityTask(void *pvParameters) {
    while (1) {
        printf("Medium Task: 正在执行后台计算...\n");
        vTaskDelay(pdMS_TO_TICKS(500));
    }
}

// 优先级 3（最高）：紧急数据上报
void HighPriorityTask(void *pvParameters) {
    while (1) {
        vTaskDelay(pdMS_TO_TICKS(1000)); // 先休眠，让 Low 任务有机会先拿到锁
        
        printf("High Task: 需要共享资源\n");
        // 尝试获取锁，如果锁被 Low 占用，触发优先级继承
        if (xSemaphoreTake(sharedMutex, portMAX_DELAY)) {
            printf("High Task: 终于拿到锁了！\n");
            vTaskDelay(pdMS_TO_TICKS(1000));
            printf("High Task: 释放资源\n");
            xSemaphoreGive(sharedMutex);
        }
    }
}

void app_main(void) {
    // [关键]：必须创建 Mutex，而不是 Binary Semaphore！
    // 只有 Mutex 才内置了优先级继承机制
    sharedMutex = xSemaphoreCreateMutex(); 
    
    // 创建任务，优先级 High(3) > Medium(2) > Low(1)
    xTaskCreate(LowPriorityTask, "LowPrio", 2048, NULL, 1, NULL);
    xTaskCreate(MediumPriorityTask, "MedPrio", 2048, NULL, 2, NULL);
    xTaskCreate(HighPriorityTask, "HighPrio", 2048, NULL, 3, NULL);
}
```
### 2.3 时序微观推演

|时刻|事件|调度器行为|L 任务优先级|
|:--|:--|:--|:--|
|**T0**|L 拿到锁，开始 I2C 操作|L 运行|1|
|**T1**|M 延时结束，进入就绪态|M 抢占 L，L 挂起|1|
|**T2**|H 延时结束，请求锁失败|H 阻塞，**触发优先级继承**|**1 → 3 (跃升)**|
|**T3**|L 优先级变为 3|L 反向抢占 M，继续 I2C|3|
|**T4**|L 释放锁|L 优先级**瞬间跌回** 1，H 被唤醒|3 → 1|
|**T5**|H 拿到锁|H 运行|1|

## 3. 实战方案 B：守护任务

对于复杂的系统（如智能小车多传感器融合），这是一种架构级的根治方案。

### 3.1 解决逻辑

彻底废除外设级的 Mutex。单独起一个优先级极高的“I2C 守护任务”。

- **架构**：任何应用层任务都不允许直接操作硬件寄存器。
- **通信**：需要读取传感器时，把请求打包扔进 [FreeRTOS Queue](app://obsidian.md/FreeRTOS%20Queue)。
- **执行**：守护任务串行地从队列取指令、操作 I2C、再把数据通过回调或通知扔回给请求方。
- **优势**：从物理架构上彻底消灭资源竞争，免疫死锁和优先级翻转。

---

## 4. 避坑指南

导师的经验之谈

1. **Mutex 不能在中断中使用**：`xSemaphoreCreateMutex()` 创建的锁不能在 ISR（中断服务程序）中操作。如果必须在 ISR 中同步，请使用[二值信号量](app://obsidian.md/%E4%BA%8C%E5%80%BC%E4%BF%A1%E5%8F%B7%E9%87%8F)或[任务通知](app://obsidian.md/%E4%BB%BB%E5%8A%A1%E9%80%9A%E7%9F%A5)。
2. **优先级继承不是万能药**：它只能缓解翻转，不能完全消除。如果 L 任务在临界区内调用了 `vTaskDelay`，即便优先级抬高了，L 依然会阻塞，H 还是得等。**临界区内严禁调用任何能引起阻塞的 API！**
3. **死锁风险**：如果任务连续两次 Take 同一个 Mutex 而不 Give，或者 A 等 B，B 等 A，Mutex 也救不了你。设计时务必理清资源依赖图。

---

## 5. 总结与选型建议

|方案|适用场景|优点|缺点|
|:--|:--|:--|:--|
|**方案 A (Mutex)**|系统简单，资源竞争偶发|开发快，改动小|仅缓解翻转，仍需小心死锁|
|**方案 B (Gatekeeper)**|系统复杂，多传感器并发|彻底免疫翻转与死锁，架构清晰|初期构建成本高，通信开销略大|

**导师建议**：如果你正在开发的智能小车未来要加入更多的传感器融合，且任务数量激增，极力推荐使用 **方案 B**。无锁化设计是大型复杂 RTOS 系统保持时序稳定的终极防御手段。
