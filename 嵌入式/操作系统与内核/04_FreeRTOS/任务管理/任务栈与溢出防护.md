> [!abstract] 核心本质
> 任务栈溢出是嵌入式系统最隐蔽的崩溃源之一。FreeRTOS提供两种检测方法，但**本质都是事后检测**——真正防护需要精确估算 + 运行期监控 + 架构设计三管齐下。

## 一、核心定义

**任务栈**：每个任务独立的私有内存区域，用于存储：
- 局部变量
- 函数调用链（返回地址、参数）
- 任务切换时的上下文（寄存器快照）
- 中断嵌套时的现场保护

**栈溢出**：当 `[[SP栈指针]]` 越过栈边界，破坏相邻内存区域（TCB、其他任务栈、全局变量）。

## 二、底层原理

### 2.1 栈的内存布局

```
低地址
┌─────────────────────┐ ← pxStack (栈底)
│  栈尾标记区 (可选)   │ ← 方法2检测用
├─────────────────────┤
│                     │
│    未使用区域       │ ← High Water Mark 测量此处
│                     │
├─────────────────────┤ ← SP 当前位置
│    已使用栈空间     │
│  ┌───────────────┐  │
│  │ 局部变量      │  │
│  │ 返回地址      │  │
│  │ 寄存器现场    │  │
│  └───────────────┘  │
└─────────────────────┘ ← pxTopOfStack (栈顶，高地址)
```

### 2.2 两种溢出检测方法对比

| 特性 | 方法1 (`=1`) | 方法2 (`=2`) |
|------|--------------|--------------|
| 检测时机 | 上下文切换时 | 上下文切换时 |
| 检测原理 | 检查 SP 是否越界 | 检查栈尾16字节是否被改写 |
| 检测能力 | 只能检测"越界后" | 能检测"越界附近" |
| 性能开销 | 低 | 略高（每次切换检查16字节） |
| 误报率 | 可能漏报 | 更可靠 |
| 推荐场景 | 资源紧张设备 | 开发调试阶段 |

### 2.3 源码级检测逻辑

```c
// tasks.c - 方法1：检查栈指针
#if (configCHECK_FOR_STACK_OVERFLOW == 1)
    #define taskCHECK_FOR_STACK_OVERFLOW() \
    { \
        if (pxCurrentTCB->pxTopOfStack <= pxCurrentTCB->pxStack) { \
            vApplicationStackOverflowHook((TaskHandle_t)pxCurrentTCB, \
                                          pxCurrentTCB->pcTaskName); \
        } \
    }
#endif

// tasks.c - 方法2：检查栈尾标记
#if (configCHECK_FOR_STACK_OVERFLOW == 2)
    #define taskCHECK_FOR_STACK_OVERFLOW() \
    { \
        const uint32_t *pulStack = (uint32_t *)pxCurrentTCB->pxStack; \
        /* 检查栈尾16字节是否仍为已知模式 0xA5A5A5A5 */ \
        for (uint8_t i = 0; i < 4; i++) { \
            if (pulStack[i] != 0xA5A5A5A5) { \
                vApplicationStackOverflowHook(...); \
                break; \
            } \
        } \
    }
#endif
```

> [!warning] 关键认知
> 两种方法都**只在任务切换时检测**。如果栈溢出后立即触发HardFault，Hook函数根本没机会执行！

### 2.4 High Water Mark 原理

```c
// 返回"历史最小剩余栈空间"（单位：word，即4字节）
UBaseType_t uxTaskGetStackHighWaterMark(TaskHandle_t xTask)
{
    TCB_t *pxTCB = prvGetTCBFromHandle(xTask);
    uint32_t *pulStack = (uint32_t *)pxTCB->pxStack;
    
    // 从栈底向上扫描，找到第一个非 0xA5 的位置
    while (*pulStack == 0xA5A5A5A5) {
        pulStack++;
    }
    
    // 计算剩余空间（word数）
    return (UBaseType_t)(pulStack - (uint32_t *)pxTCB->pxStack);
}
```

## 三、实战应用落地

### 3.1 栈空间估算公式

```
任务栈需求 = 局部变量最大值 
           + 函数调用链深度 × 单帧开销(约16~32字节)
           + 最大中断嵌套开销
           + FPU上下文(若启用, 约136字节)
           + 安全冗余(建议20%~30%)
```

**典型开销参考**（Cortex-M4）：

| 项目 | 开销 |
|------|------|
| 任务切换上下文 | 约64字节 |
| 单层函数调用 | 16~32字节 |
| 单层中断嵌套 | 约32字节 + 局部变量 |
| FPU寄存器组 | 136字节（若启用） |
| HardFault现场 | 额外约32字节 |

### 3.2 配置与Hook实现

```c
// FreeRTOSConfig.h
#define configCHECK_FOR_STACK_OVERFLOW    2
#define configRECORD_STACK_HIGH_ADDRESS   1  // 记录栈地址信息

// main.c 或 stack_monitor.c
#include "FreeRTOS.h"
#include "task.h"

// 必须实现的Hook函数
void vApplicationStackOverflowHook(TaskHandle_t xTask, char *pcTaskName)
{
    // ⚠️ 此时系统已不稳定，只能做最简单的操作！
    
    // 1. 记录故障信息到全局变量（方便调试器读取）
    extern struct {
        char taskName[16];
        uint32_t tick;
        void *stackBase;
        void *stackTop;
    } gStackOverflowInfo;
    
    strncpy(gStackOverflowInfo.taskName, pcTaskName, 15);
    gStackOverflowInfo.tick = xTaskGetTickCount();
    gStackOverflowInfo.stackBase = pxTaskGetStackStart(xTask);
    
    // 2. 翻转错误指示LED
    GPIO_WriteBit(ERR_LED_GPIO, ERR_LED_PIN, Bit_SET);
    
    // 3. 进入死循环等待调试器
    taskDISABLE_INTERRUPTS();
    while (1) {
        __NOP();
    }
    
    // 或者直接复位
    // NVIC_SystemReset();
}
```

### 3.3 栈监控任务

```c
void vStackMonitorTask(void *pvParameters)
{
    const TickType_t xInterval = pdMS_TO_TICKS(5000);  // 5秒检查一次
    TaskStatus_t *pxTaskStatusArray;
    UBaseType_t uxTaskCount;
    uint32_t ulTotalRunTime;
    
    for (;;) {
        uxTaskCount = uxTaskGetNumberOfTasks();
        pxTaskStatusArray = pvPortMalloc(uxTaskCount * sizeof(TaskStatus_t));
        
        if (pxTaskStatusArray != NULL) {
            uxTaskGetSystemState(pxTaskStatusArray, uxTaskCount, &ulTotalRunTime);
            
            printf("\n=== Stack Monitor ===\n");
            printf("%-16s %8s %8s %8s\n", "Task", "Stack", "HWM", "Status");
            printf("----------------------------------------\n");
            
            for (UBaseType_t i = 0; i < uxTaskCount; i++) {
                // HWM单位是word，转换为字节
                uint32_t stackHWMBytes = pxTaskStatusArray[i].usStackHighWaterMark * 4;
                uint32_t stackSizeBytes = pxTaskStatusArray[i].xStackSize * 4;
                
                printf("%-16s %8u %8u %8s\n",
                       pxTaskStatusArray[i].pcTaskName,
                       stackSizeBytes,
                       stackHWMBytes,
                       stackHWMBytes < 64 ? "⚠️ LOW" : "OK");
            }
            
            vPortFree(pxTaskStatusArray);
        }
        
        vTaskDelay(xInterval);
    }
}
```

### 3.4 常见溢出场景

```c
// ❌ 场景1：栈上放大数组
void vBadTask(void *pvParameters)
{
    uint8_t buffer[2048];  // 直接撑爆小任务栈
    process_data(buffer);
}

// ✅ 修正：使用静态区或堆
void vGoodTask(void *pvParameters)
{
    static uint8_t buffer[2048];  // 静态区
    // 或 uint8_t *buffer = pvPortMalloc(2048);  // 堆
    process_data(buffer);
}

// ❌ 场景2：深递归
uint32_t factorial(uint32_t n)
{
    if (n <= 1) return 1;
    return n * factorial(n - 1);  // 每层递归消耗栈帧
}

// ❌ 场景3：printf家族
void vLogTask(void *pvParameters)
{
    char buf[256];
    snprintf(buf, sizeof(buf), "Value: %d, String: %s", ...);  // 可能耗尽栈
}
```

## 四、避坑与边界条件

> [!danger] 致命陷阱
> 1. **Hook函数中禁止做复杂操作**：不能调用 `printf()`、`vTaskDelay()`、申请内存等
> 2. **检测方法2依赖栈初始化模式**：如果代码手动改写了栈底内容，会误报
> 3. **中断中的栈使用的是当前任务栈**：中断嵌套深度直接影响所有任务栈需求

> [!warning] 避坑指南
> - `uxTaskGetStackHighWaterMark()` 返回值单位是 **word（4字节）**，不是byte
> - 开启优化后，编译器可能复用栈空间，HWM测量值会波动
> - 不要在Release版本关闭溢出检测，建议保持 `=1`
