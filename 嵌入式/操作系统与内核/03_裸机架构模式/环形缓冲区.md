# 全局缓冲区与环形缓冲区

在 [[../04_FreeRTOS/内存管理/中断推迟处理]] 的架构中，ISR（生产者）产生数据的速度往往和 Task（消费者）处理数据的速度不匹配。全局缓冲区就是解决这个“速率匹配”问题的核心组件。

> [!info] 核心定义
> **全局缓冲区**是一块预先分配好的、固定大小的连续内存区域。它通常被定义为全局数组，生命周期贯穿整个程序运行期。
> 在高性能场景下，我们通常将其封装为 **环形缓冲区**，以实现数据的循环写入和读取。

---

## 1. 原理剖析：为什么需要“环形”？

### 1.1 朴素数组的困境
假设我们定义一个简单的数组 `uint8_t buffer[100]`：
- **写入**：ISR 不断往里填数据，索引 `index` 不断增加。
- **问题**：当 `index` 达到 99 时，数组满了。此时 ISR 必须停止写入，或者从头开始覆盖旧数据。
- **管理成本**：如果从头开始，你需要复杂的逻辑来搬运剩余数据，或者重置索引，这在实时系统中是极大的开销。

### 1.2 环形缓冲区的智慧
环形缓冲区通过两个指针（或索引）将线性的数组逻辑上“首尾相连”：
- **头指针**：指向下一个要写入的位置。
- **尾指针**：指向下一个要读取的位置。

**生活类比：传送带**
想象一个首尾相接的传送带。
- **生产者（ISR）**：站在一个位置往带上放包裹，放完后往前走一步。
- **消费者**：站在另一个位置取包裹，取完后往前走一步。
- 只要消费者跟得上生产者的速度，传送带永远转不完，不需要停下来“倒带”。

## 2. 代码实战：构建一个高效的 Ring Buffer

这是一个工业级的环形缓冲区结构体实现，支持任意数据类型。

### 2.1 结构定义

```c
#include <stdint.h>
#include <stdbool.h>
#include <string.h>

// 定义缓冲区大小（必须是 2 的 n 次幂，方便取模优化）
#define BUFFER_SIZE 256

typedef struct {
    uint8_t  buffer[BUFFER_SIZE]; // 数据存储区
    uint16_t head;                // 头指针：写入位置
    uint16_t tail;                // 尾指针：读取位置
    uint16_t count;               // 当前数据量（可选，用于判断满/空）
} RingBuffer_t;

// 全局实例
RingBuffer_t g_UartRxBuffer;

```

### 2.2 核心操作函数

```c
// 初始化
void RingBuffer_Init(RingBuffer_t *rb) {
    rb->head = 0;
    rb->tail = 0;
    rb->count = 0;
    memset(rb->buffer, 0, BUFFER_SIZE);
}

// 写入数据（通常在 ISR 中调用）
bool RingBuffer_Push(RingBuffer_t *rb, uint8_t data) {
    // 1. 检查是否满（如果满了，策略可以是丢弃新数据或覆盖旧数据）
    if (rb->count >= BUFFER_SIZE) {
        return false; // 缓冲区满，丢弃数据
    }

    // 2. 写入数据
    rb->buffer[rb->head] = data;

    // 3. 移动头指针（取模运算实现环形）
    // 如果 BUFFER_SIZE 是 2^n，可以用 (rb->head + 1) & (BUFFER_SIZE - 1) 代替取模，效率更高
    rb->head = (rb->head + 1) % BUFFER_SIZE;

    // 4. 更新计数
    rb->count++;

    return true;
}

// 读取数据（通常在 Task 中调用）
bool RingBuffer_Pop(RingBuffer_t *rb, uint8_t *pData) {
    // 1. 检查是否空
    if (rb->count == 0) {
        return false; // 缓冲区空
    }

    // 2. 读取数据
    *pData = rb->buffer[rb->tail];

    // 3. 移动尾指针
    rb->tail = (rb->tail + 1) % BUFFER_SIZE;

    // 4. 更新计数
    rb->count--;

    return true;
}
```

## 3. 实战链路：ISR 与 Task 的协作

结合中[中断推迟处理](../Problems/中断推迟处理.md)，我们来看看如何用全局缓冲区替代队列。

### 3.1 场景：高频 UART 接收

```c
// ISR：生产者（极速写入，不阻塞）
void USART1_IRQHandler(void) {
    uint8_t rx_byte;
    BaseType_t xWoken = pdFALSE;

    if(USART_GetITStatus(USART1, USART_IT_RXNE)) {
        rx_byte = USART_ReceiveData(USART1);
        
        // [核心] 直接压入全局缓冲区，无需调用复杂的 Queue API
        // 这里的操作仅涉及数组赋值和索引加法，耗时极短
        RingBuffer_Push(&g_UartRxBuffer, rx_byte);
        
        // 可选：只有当收到换行符时，才唤醒任务处理（减少切换开销）
        if (rx_byte == '\n') {
             vTaskNotifyGiveFromISR(xUartTaskHandle, &xWoken);
             portYIELD_FROM_ISR(xWoken);
        }
    }
}

// Task：消费者（安全读取，处理业务）
void vUartTask(void *pvParameters) {
    uint8_t byte;
    while(1) {
        // 等待通知（阻塞）
        ulTaskNotifyTake(pdTRUE, portMAX_DELAY);

        // 批量处理缓冲区数据
        // 此时 ISR 可能还在继续写入，需注意并发保护
        while (g_UartRxBuffer.count > 0) {
            // [关键] 进入临界区保护，防止读取过程中 ISR 修改 count
            taskENTER_CRITICAL();
            RingBuffer_Pop(&g_UartRxBuffer, &byte);
            taskEXIT_CRITICAL();

            // 处理业务（如解析协议）
            Process_Byte(byte);
        }
    }
}
```

## 4. 避坑指南：并发与性能

必须要懂的陷阱

### 4.1 竞态条件

**问题**：`count` 变量是共享资源。

- Task 正在读取 `count` 判断是否为空。
- 此时 ISR 突然触发，修改了 `count`。
- Task 的判断逻辑失效，可能导致读取到垃圾数据或死循环。

**对策**：

- **简单场景**：在 Task 读取时使用 `taskENTER_CRITICAL()` 关中断保护。因为读取操作极快，关中断时间极短，不影响实时性。
- **高性能场景**：如果不想关中断，可以使用 **原子操作** 更新 `count`，或者设计“无锁环形缓冲区”（Lock-Free Ring Buffer），但这需要深厚的底层知识。

### 4.2 缓冲区溢出

**问题**：ISR 接收数据的速度 > Task 处理速度，导致缓冲区写满。  
**后果**：新数据被丢弃，或者覆盖了旧数据（取决于策略）。  
**对策**：

- 增大 `BUFFER_SIZE`。
- 在 Task 中提高处理优先级。
- 实现流控机制（如硬件流控 RTS/CTS）。

### 4.3 Volatile 关键字

虽然结构体被多个上下文访问，但在 FreeRTOS 的临界区保护下，通常不需要给整个结构体加 `volatile`。但如果你的 `count` 是通过原子操作无锁访问的，则必须声明为 `volatile` 或使用 `volatile uint16_t`。

## 5. 选型对比：全局缓冲区 vs FreeRTOS Queue

|特性|全局缓冲区|FreeRTOS Queue|
|:--|:--|:--|
|**内存开销**|极低（仅数组+索引）|较高（需链表节点、事件列表等管理开销）|
|**数据拷贝**|**零拷贝**（直接操作内存）|有拷贝（Send 时拷入，Receive 时拷出）|
|**执行速度**|极快（几条汇编指令）|较慢（涉及内核调度、锁机制）|
|**并发安全**|需手动加锁或原子操作|内核自动管理，天然安全|
|**适用场景**|高速数据流（UART/SPI）、DMA传输|低频事件、结构体传递、跨任务通信|

**大师建议**：

- 如果是 **字节流传输**（如串口接收、传感器原始数据流），首选 **全局环形缓冲区**。
- 如果是 **结构化事件传递**（如“按键按下”、“电机过热”），首选 **FreeRTOS Queue**。
